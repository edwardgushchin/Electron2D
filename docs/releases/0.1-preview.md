Первая версия реализует **первый законченный 2D-движок с полным Godot 4.7-compatible public API contract для публичной поверхности Electron2D `0.1-preview`**. Решение о включении Godot-типов в публичную поверхность определяется ручным профилем `data/api/electron2d-public-api-profile.json`, который утверждает project owner; generated artifacts, manifest, Wiki и задачи публичных классов только проверяют, публикуют и доказывают этот профиль.

## Цель

Разработчик на C# должен суметь:

> создать проект, собрать сцену в редакторе, написать игровую логику, импортировать ресурсы, запустить игру и экспортировать один проект под Windows, Linux, macOS, Android, iOS и WebAssembly browser без переписывания игрового кода.

Electron2D также должен соответствовать позиционированию **Agent-native cross-platform 2D game engine**:

> человек и AI-агент должны суметь создать небольшую законченную 2D-игру на C#, работая в открытом `Electron2D.Editor` с одной живой моделью проекта; headless-режим остаётся обязательным для CI, тестов и автономных запусков, но не заменяет основной совместный workflow.

Agent-native cross-platform 2D game engine в `0.1-preview` означает не встроенный чат и не привязку к конкретной LLM, а архитектуру, где важные операции проходят через живый `ProjectWorkspace`, общий слой инструментальных операций `Tooling`, локальный MCP-сервер, CLI для headless-сценариев, текстовые project files, структурированную диагностику, API manifest, Editor Capability Manifest, карту совместимости и видимое управление запущенной игрой.

Версия считается успешной, когда Electron2D подходит для создания и выпуска небольшой законченной 2D-игры, а не только для демонстрации отдельных подсистем.

Целевая аудитория первой версии:

* C#-разработчики;
* одиночные разработчики и небольшие команды;
* программисты, которым нужен node/resource API в полностью управляемой .NET-экосистеме;
* проекты малого и среднего масштаба;
* AI-агенты, CI и IDE-интеграции, работающие через публичный tooling API без GUI automation и подключающиеся к активной Editor-сессии, когда она открыта.

Редактор предназначен для desktop. Android, iOS и WebAssembly browser являются платформами запуска и экспорта, но не редактирования проектов.

## Матрицы платформ

`0.1-preview` различает три набора:

- `runtimeTargets` - платформы запуска и экспорта игр: Windows x64, Linux x64, macOS arm64, Android arm64, iOS arm64 и WebAssembly browser.
- `editorTargets` - платформы редактора: Windows, Linux и macOS. Редактор для мобильных устройств и браузера не входит в первую версию без отдельного продуктового решения.
- `releaseVerificationTargets` - платформы, на которых именно этот релиз требует проверяемый результат быстрой и длительной проверки перед выпуском.

Продуктовое решение для `0.1-preview`: `releaseVerificationTargets` не сокращает `runtimeTargets`. Все шесть платформ запуска и экспорта остаются в релизной проверке, пока этот документ не будет изменён явно. iOS и WebAssembly browser нельзя убирать из матрицы запуска/экспорта или пресетов экспорта ради упрощения проверок. Если окружение недоступно, проверка должна оставлять машинно-читаемый отчёт о заблокированном окружении (`blocked-environment artifact`): недостающем SDK, устройстве, симуляторе, браузерной автоматизации или ОС сборочной машины. Такой отчёт объясняет причину блокировки, но не засчитывает релизную проверку как пройденную.

# Что можно сделать на первой версии

Electron2D 0.1 должен позволять создавать:

* простые платформеры;
* top-down игры;
* аркады;
* головоломки;
* карточные и настольные игры;
* визуальные новеллы;
* UI-heavy приложения;
* небольшие RPG без сложной навигации и освещения;
* локальные игры с клавиатурой, мышью, геймпадом или touch-вводом.

Первая версия не обязана поддерживать крупные открытые миры, сложное динамическое освещение, rollback multiplayer или тысячи физических объектов.

# Основной пользовательский сценарий

Полный рабочий цикл должен выглядеть так:

1. Разработчик создаёт проект Electron2D.
2. Открывает его в `Electron2D.Editor`.
3. Создаёт сцену.
4. Добавляет узлы.
5. Импортирует текстуры, шрифты и звук.
6. Настраивает свойства через Inspector.
7. Создаёт C#-скрипт и прикрепляет его к узлу.
8. Настраивает Input Map.
9. Запускает текущую сцену или весь проект.
10. Исправляет ошибки по сообщениям редактора.
11. Создаёт export preset.
12. Получает запускаемую сборку для выбранной платформы.

Этот сценарий должен работать на чистой машине после установки документированных зависимостей.

Совместный сценарий человека и AI должен использовать тот же `ProjectWorkspace`, что и ручные операции Editor:

1. Разработчик открывает проект в `Electron2D.Editor`.
2. Разработчик создаёт задачу в Project Tasks или выбирает существующую задачу как active task.
3. AI-агент подключается к активной Editor-сессии через MCP/IPC.
4. Созданные агентом файлы сразу появляются в FileSystem dock без ручного refresh.
5. Изменения сцен и ресурсов сразу отражаются в Scene Tree, Inspector и viewport.
6. Ручные изменения разработчика и изменения AI объединяются структурно, а конфликтующие правки показываются в conflict panel.
7. Агент запускает текущую сцену в видимом игровом процессе, отправляет input, ставит игру на паузу, делает frame step и получает screenshot/runtime tree.
8. Агент связывает transactions, jobs, diagnostics и artifacts с active task.
9. Diagnostics, тесты и visual diff, то есть визуальное сравнение результата, видны в Editor и доступны агенту через MCP.
10. Агент отправляет задачу в `Awaiting Acceptance`.
11. Разработчик просматривает changeset, tests, diagnostics и screenshots, затем принимает результат или запрашивает изменения.
12. Последняя AI-транзакция может быть отменена обычным Undo.

## Project Tasks

Project Tasks являются обязательной подсистемой Editor:

1. Пользователь видит доску задач с колонками `Backlog`, `Ready`, `In Progress`, `Blocked`, `Review`, `Awaiting Acceptance`, `Done`, `Cancelled`.
2. Задачи, board и activity хранятся как first-class документы `ProjectWorkspace`, а не как отдельный Markdown-backlog.
3. Task mutations поддерживают `expectedRevision`, dirty state, grouped Undo/Redo, external synchronization и conflict handling.
4. `Review` означает техническую проверку, а `Awaiting Acceptance` означает ожидание решения разработчика.
5. AI может отправить задачу на приёмку, но не может сам поставить `Done`; приёмка требует доверенный interactive Editor context, а не самодекларируемый `ActorKind`.
6. Любое изменение task document, включая `ExternalImport`, migration, CLI import и crash recovery, проходит через `TaskTransitionValidator` и `TaskAcceptanceService`; прямая правка файла не может установить `Done`, `AcceptedAt`, `AcceptedBy`, `AcceptanceState`, audit-поля activity или привилегированный reopen без structured diagnostic и pending-import/conflict state.
7. `.taskboard/**` является `EditorMetadata`: эти документы доступны Editor, Tooling, CLI и MCP, но не импортируются как игровые ресурсы, не попадают в production asset packs, APK, AAB, app bundle или desktop distribution и не материализуются в runtime snapshot как игровые файлы.
8. Если task data экспортируется во внешний отчёт, это выполняется только отдельной явной report-командой; такой экспорт не является обязательной возможностью `0.1-preview` и не становится canonical storage. Jobs могут хранить `TaskId`, но не копируют task activity в build directory по умолчанию.
9. Изменение task status, activity или board rank не делает build/test/run artifacts `stale`, если игровые входные документы и build/run configuration не изменились.
10. `Reopen` является человеческим действием, а не отдельным статусом; история приёмки сохраняется в activity.
11. Ручной `Blocked` и блокировка из-за незавершённых dependencies различаются: закрытие dependency не снимает ручной blocker.
12. Отдельного действия `Reject` в `0.1-preview` нет: `Accept` переводит задачу в `Done`, `Request changes` возвращает её в `In Progress`, а `Cancel` переводит в `Cancelled`, если задача больше не нужна.

Headless-сценарий должен использовать тот же слой модели проекта, когда Editor закрыт или команда явно запущена для CI:

1. `e2d project create` создаёт проект, локальный git-репозиторий, project-local `AGENTS.md`, starter skills и начальную доску встроенного `ProjectTaskManager`.
2. Template `.gitignore` не игнорирует `.electron2d/` целиком; он явно игнорирует generated/local-only каталоги `.electron2d/import-cache/`, `.electron2d/workspaces/`, `.electron2d/context/`, `.electron2d/session/` и `.electron2d/user/`, включая локальные breakpoints.
3. `e2d scene` создаёт и изменяет сцену в стабильном текстовом формате.
4. `e2d build`, `e2d validate` и `e2d test` возвращают структурированные результаты.
5. `e2d run` умеет выполнить фиксированное число кадров, применить input trace и сохранить diagnostics, screenshot, итоговое scene tree и performance данные.
6. `e2d export` собирает проект в headless-режиме.
7. `e2d api compare-godot` и локальная документация помогают не использовать неподдержанный Godot API.

# Публичная модель движка

API следует согласованной node/resource модели Godot `4.7-stable` и корневому контракту `T-0241`.

Необходимо сразу зафиксировать:

> Публичный runtime API Electron2D `0.1-preview` воспроизводит Godot `4.7-stable` .NET/C# API для утверждённой публичной поверхности Electron2D под namespace `Electron2D`. Для Godot members должны совпадать наследование, свойства, методы, overloads, события, сигналы, enum, constants, типы параметров и возвращаемых значений, значения по умолчанию и наблюдаемое поведение. Намеренно отсутствующий или несовместимый Godot member обязан иметь явное `Deferred`/`Unsupported` решение. Отдельный Electron2D-specific public extension допустим только при точном owner-approved member/value решении `electronExtension`, рабочем поведении и `parity = not_applicable`; такой extension нельзя выдавать за совпадающий Godot member или использовать для молчаливого замещения отсутствующего Godot API. GDScript не поддерживается; публичный путь скриптов - C#, публичный shader source language - HLSL.

Формула релиза:

```text
Не ручной Markdown-профиль и не фактический список из сборки.
Ручной JSON-профиль project owner-а + Godot 4.7 C# compatibility для утверждённых Godot members + явные Deferred/Unsupported решения для исключённых Godot members + отдельные owner-approved Electron2D extensions с `parity = not_applicable`.
```

Решение baseline от 2026-06-20: реализация `0.1-preview` пишется заново после удаления старого каталога `src/Electron2D/`. Старый component-код, compatibility layer, `IComponent`, `SpriteRenderer`, legacy physics components и любые публичные API, которых нет в generated Godot `4.7-stable` packets или которые не утверждены задачей совместимости, не переносятся в новую реализацию.

Wiki-материалы релиза должны готовиться как GitHub Wiki source, а не как локальный сайт внутри репозитория движка.

Нельзя добавлять пустые классы только ради похожего списка API. Каждый публичный класс из утверждённой публичной поверхности должен:

* быть функционально реализован;
* иметь документацию;
* иметь хотя бы один тест или пример;
* иметь подтверждённый паритет с Godot `4.7-stable` .NET/C# API.

Для Wiki нужна таблица совместимости:

| Статус       | Значение                           |
| ------------ | ---------------------------------- |
| Supported    | Утверждён в manual profile; full Godot `4.7-stable` parity доказывается owning class tasks and final gates |
| Deferred     | Явно исключён из текущего релиза и отложен для будущего решения |
| Unsupported  | Явно исключён из текущего релиза |
| Unapproved   | Экспортируется runtime assembly без manual profile decision; это failing gate state |

Для любого публичного API, включённого в обязательную поверхность `0.1-preview`, допустим только статус `Supported` с `profile_approved` в manual profile. Полная strict parity не считается доказанной текущим manifest и требует отдельного evidence-контракта. `Deferred` и `Unsupported` являются явными решениями manual profile, но такие типы не могут оставаться exported public types текущего релиза. `Unapproved` означает отсутствие решения и всегда ломает gate.

# Обязательные подсистемы

## 1. Объектная модель и SceneTree

Первая версия должна поддерживать:

* иерархию узлов;
* добавление, удаление и перемещение узлов;
* владение узлами сценой;
* пути к узлам;
* группы;
* сигналы;
* отложенные вызовы;
* lifecycle узлов;
* обычный и физический update;
* pause mode;
* загрузку и инстанцирование сцен;
* смену активной сцены;
* освобождение узлов;
* обработку ошибок внутри пользовательского кода.

Точный набор публичных node/resource типов, callbacks и методов node API для этого среза не фиксируется вручную в release note. Источник истины для решения о включении типа: `data/api/electron2d-public-api-profile.json`. `data/api/electron2d-api-manifest.json`, Wiki `API-Compatibility.md`, generated reports `T-0242`/`T-0243`/`T-0244` и задачи публичных классов под gate `T-0245` являются производными проверками и evidence.

Этот раздел фиксирует не список имён, а обязательное поведение: базовая иерархия объектов и ресурсов, lifecycle узлов, дерево сцены, владение, пути, группы, сигналы, deferred-вызовы, обычный/физический update, ввод, освобождение узлов и загрузка/инстанцирование сцен должны следовать Godot `4.7-stable` для утверждённой публичной поверхности.

## 2. Математические и Variant-типы

Нужен законченный 2D-набор математических, идентификаторных, callable/signal, RID и `Variant`-связанных возможностей. Конкретные публичные типы утверждаются только manual API profile, а members и signatures доказываются generated API artifacts; release note фиксирует только поведенческую обязанность: 2D-геометрия, цвет, mathematical helpers, deterministic random, пути/имена, сигналы, callable bridge, RID-граница и `Variant` должны быть совместимы с Godot `4.7-stable` в утверждённом срезе.

`Variant` первой версии не обязан поддерживать все трёхмерные типы Godot. Должен существовать документированный закрытый список поддерживаемых значений:

* null;
* bool;
* целые и вещественные числа;
* string;
* перечисления;
* основные 2D-типы;
* массивы;
* словари;
* Object и Resource;
* NodePath и StringName.

Сериализация этих типов должна быть стабильной и проверяемой round-trip тестами.

* корректное восстановление ресурсов после потери или пересоздания устройства.

Минимальные rendering/UI узлы не перечисляются здесь как public API source of truth. Manual API profile утверждает типы, generated artifacts доказывают точные имена и signatures; release contract требует покрыть спрайты, покадровую 2D-анимацию, камеру, tilemap layer, polygon/line drawing, texture/color primitives, nine-patch и текстовый вывод.

`CanvasItem` должен предоставлять базовый immediate drawing API:

* линии, полилинии, прямоугольники, круги и полигоны;
* текстуры и прямоугольники текстур;
* строки с импортированным шрифтом;
* явный redraw request.

### Шейдеры

Первая версия может поддерживать только canvas-шейдеры:

* vertex stage;
* fragment stage;
* uniforms;
* texture samplers;
* engine-provided built-ins;
* импорт и компиляцию через внутренний shader translation backend;
* диагностику с указанием файла и строки.

Не требуется:

* совместимость с языком шейдеров Godot;
* compute shaders;
* geometry shaders;
* визуальный редактор шейдеров;
* полноценный render graph.

Компиляция должна выполняться во время импорта или экспорта, а не полагаться на runtime-компиляцию на iOS.

# Два профиля рендеринга

## Compatibility

Гарантированный минимальный набор для всех поддерживаемых платформ:

* спрайты;
* анимации;
* TileMap;
* UI;
* текст;
* примитивы;
* камера;
* clipping;
* стандартные blend modes;
* render targets, если backend это позволяет.

Не гарантируются:

* пользовательские шейдеры;
* постобработка;
* сложные материалы;
* GPU particles;
* идентичность нестандартного blending.

## Standard

Полный Standard backend:

* custom CanvasItem shaders;
* ShaderMaterial;
* render-to-texture;
* несколько проходов;
* расширенное смешивание;
* постобработка в пределах поддерживаемой первой версии.

Проект должен явно объявлять минимальный профиль:

```text
rendering/required_profile = compatibility
```

или:

```text
rendering/required_profile = standard
```

При запросе `standard` движок не должен молча отключать эффекты. На неподдерживаемом Android-устройстве он должен:

* показать понятную ошибку;
* записать причину в лог;
* завершить запуск либо применить явно разрешённый пользователем fallback.

## 4. Android fallback

Последовательность запуска:

1. Попытка создать графическое устройство с мобильным совместимым профилем.
2. Выполнить внутренний GPU smoke test.
3. Проверить создание texture, pipeline, command buffer и первый submit.
4. При ошибке активировать compatibility backend.
5. Передать игре фактический набор возможностей.

Фактический профиль рендеринга и набор возможностей должны быть доступны игре через approved API surface из manual profile. Логирование должно включать:

* выбранный backend;
* GPU и драйвер;
* причины fallback;
* отключённые возможности.

## 7. C# scripting

Electron2D поддерживает только C# как публичный scripting path. Это постоянная политика проекта, а не временное ограничение первой версии; GDScript, visual scripting и другие пользовательские script languages не входят в публичный authoring path.

Игровой скрипт:

* наследуется от `Node` или его потомка;
* компилируется обычным .NET toolchain;
* может иметь экспортируемые свойства;
* может объявлять и принимать сигналы;
* может обращаться к сцене и сервисам движка;
* получает lifecycle callbacks.

Metadata markers для exported properties, signals и editor-time tool behavior должны оставаться marker API. Release note не фиксирует attribute syntax или class names вручную; связь с serialization/Inspector должна идти через AOT-safe metadata bridge, без обязательного runtime assembly scan.

Tool-behavior может быть ограниченным и экспериментальным. Baseline от 2026-06-21 вводит внутренний execution host для явно зарегистрированной tool metadata: editor-time callbacks запускаются только явным tooling-вызовом, exceptions изолируются в result object, dynamic assembly load не используется, обычный runtime traversal остаётся отдельным путём.

Внешняя IDE не является обязательной частью workflow и не может закрывать обязательные требования `0.1-preview`. `Electron2D.Editor` должен включать центральное workspace `Script` со встроенной базовой C# IDE: разработчик может создать, написать, исследовать, собрать и пошагово отладить игровой код без выхода из редактора.

Обязательный минимум редактирования:

* создание, открытие, редактирование, переименование и удаление `.cs` files;
* вкладки открытых документов;
* line numbers;
* C# syntax highlighting;
* автоматические отступы;
* настройка tabs/spaces;
* matching скобок и кавычек;
* code folding;
* выделение текущей строки;
* поиск и замена в файле;
* поиск по проекту;
* переход к строке;
* clipboard;
* Undo/Redo;
* сохранение одного файла и `Save All`;
* отображение dirty state;
* восстановление открытых вкладок после перезапуска Editor.

Ручной ввод в `.cs` buffer проходит через `TextBufferEditSession`, lightweight `CodeDocumentChanged` events и `TextBufferUndo`, а не через полноценную `WorkspaceTransactionEngine`-транзакцию на каждый символ. AI/refactoring/multi-file операции идут через `WorkspaceTransactionEngine` и `WorkspaceUndo`. Save выполняется как persistence transaction с atomic write; агентский `script_save` не сохраняет ручные unsaved changes, сделанные после базовой revision агента, без structured conflict или интерактивного подтверждения разработчика.

Обязательный минимум language services:

* project-aware autocomplete для типов, методов, свойств, events и локальных переменных;
* completion для API Electron2D;
* signature help с активным параметром;
* hover/Quick Info;
* XML documentation;
* live compiler diagnostics;
* подчёркивание ошибок и предупреждений;
* go to definition;
* find references;
* rename symbol;
* document formatting;
* basic code actions;
* добавление недостающего `using`;
* навигация по классам, методам и свойствам текущего файла;
* diagnostics panel с переходом к строке ошибки.

Semantic model должна учитывать текущий `.csproj`, references проекта, Electron2D assemblies, NuGet dependencies, conditional compilation symbols, unsaved C# buffers, source-generated API и фактическую build configuration.

Language services находятся в отдельном `Electron2D.CSharpLanguageServices` assembly, работающем внутри Editor process, не в игровом runtime и не в Editor UI. Отдельный language-service process не входит в обязательный scope `0.1-preview`. Baseline реализации — Roslyn-backed live workspace, синхронизированный с `CodeDocument`; completion, hover, diagnostics и navigation используют `DocumentRevision`, `SemanticVersion` и `ConfigurationHash`, чтобы устаревшие ответы не применялись к новому buffer state. `WorkspaceSnapshot` используется для build/test/run/debug и пакетного анализа, а не для каждого IDE-запроса.

Обязательный минимум managed C# debugger для local desktop run на Windows, Linux и macOS:

* установка, включение, отключение и удаление breakpoint в gutter;
* сохранение breakpoints между Editor sessions;
* breakpoint identity через `BreakpointId`, `DocumentId` и `SourceAnchor`;
* rebase breakpoint anchor после text edits или перевод breakpoint в unverified state при неоднозначном переносе;
* локальное хранение breakpoints в `.electron2d/user/breakpoints.e2debug` без попадания в Git, `WorkspaceSnapshot`, runtime export или production package;
* запуск текущей сцены или проекта под debugger;
* attach к игровому процессу, запущенному Editor;
* pause, continue, stop, restart;
* step into, step over, step out;
* подсветка текущей исполняемой строки;
* call stack;
* выбор stack frame;
* threads;
* locals;
* arguments;
* watches;
* остановка на необработанном exception;
* переход из stack trace к исходному коду;
* Debug Output.

Runtime debug bridge и managed C# debugger — разные подсистемы. Runtime debug bridge инспектирует scene tree, node properties, input, screenshots, frame step и performance metrics. Managed C# debugger отвечает за source breakpoints, call stack, locals, watches, exceptions и step into/over/out.

Debug session запускается на конкретном immutable `WorkspaceSnapshot`: build с Portable PDB, game process, `SnapshotId`, source revisions, build configuration, symbols path и mapping snapshot source к открытому `CodeDocument`. Если пользователь или AI меняет код после старта, session помечается `stale`, Editor предлагает rebuild/restart и не сохраняет несохранённые изменения автоматически.

Managed debugger подключается через обязательную DAP-границу к packaged .NET debug adapter. Отклонение от DAP допускается только новым архитектурным решением, если P0 technical spike докажет отсутствие подходящего распространяемого adapter. Выбор adapter является отдельным P0 technical spike до полноценной реализации debugger и проверяет Tier 1 desktop platforms, Portable PDB, launch/attach, breakpoints, stepping, locals/watches, exceptions, restart, DAP capability matrix, лицензию и способ обновления adapter binaries. Agent `debug_attach` ограничен game process активной Editor play/debug session; произвольный attach к другому процессу отсутствует либо требует явного интерактивного подтверждения разработчика.

Tooling/MCP parity для `Script` workspace и debugger должен покрывать не только text edits и breakpoints, но и rename/delete/search script files, document symbols, code actions, debugger restart, breakpoint update, stacks всех threads, watch definitions, явное `debug_evaluate_watches(frameId)` и frame-scoped locals/arguments. Если Editor поддерживает эти действия интерактивно, AI получает эквивалентную семантическую команду без keyboard/mouse automation.

Hot Reload и Edit and Continue не являются обязательным критерием первой версии. Допустим быстрый rebuild и restart play/debug session после изменения кода.

## 9. Физика 2D

Первая версия должна поддерживать базовую физическую игру, а не только AABB-проверки.

Минимальные physics nodes и shape resources не перечисляются здесь как public API source of truth. Manual API profile утверждает конкретные типы, generated API artifacts доказывают имена и signatures; release contract требует статические/динамические/управляемые тела, области, collision shapes, ray queries и базовые 2D-формы для production gameplay.

`ConcavePolygonShape2D` разрешается использовать только для статической геометрии.

Обязательные возможности:

* collision layers и masks;
* friction и restitution;
* gravity;
* sleeping;
* sensors/areas;
* body entered/exited;
* area entered/exited;
* raycast;
* point query;
* shape query;
* fixed physics timestep;
* one-way platforms;
* базовое continuous collision detection для быстрых тел;
* корректное удаление тел во время physics step через deferred queue.

Управляемое 2D-тело должно иметь практически применимое поведение:

* collision response для одиночного движения;
* slide movement;
* floor/wall/ceiling state;
* floor normal, floor snap и up-direction behavior.

Не требуется:

* битовая детерминированность между платформами;
* rollback physics;
* полный набор joints;
* soft bodies;
* fluid simulation.

Начальный baseline от 2026-06-21: physics domain уже предоставляет public RID-границу для space/body/area/joint/shape creation, lifecycle, transform sync, collision filter helpers, material/body-state snapshot, validation, AOT-safe serialization metadata, overlap signals/helper behavior, AABB ray/point/shape query baseline, fixed physics tick `1/60`, начальное движение динамических тел, basic AABB sweep для быстрых тел, one-way platform checks, управляемое движение с floor/wall/ceiling state, floor snap, platform velocity baseline, debug collision shape hooks и безопасное удаление во время physics traversal. Точные class/member names задаются manual profile и generated API artifacts; полноценная симуляция столкновений, запись geometry/material в production backend, gravity integration, точный narrow-phase и shape-level overlap signals остаются следующими задачами физического домена.

Box2D.NET candidate validation от 2026-06-21: отдельный smoke-проект `tests/Electron2D.Tests.PhysicsBox2DSmoke` и verifier `tools/Verify-Box2DPhysicsCandidate.ps1` проверяют `Box2D.NET 3.1.654` в Release/JIT и NativeAOT, измеряют allocations per tick и не добавляют Box2D.NET в public runtime dependency graph. Локальный `win-x64` результат: Release/JIT `AllocatedBytesPerTick=464.633`, NativeAOT `AllocatedBytesPerTick=354.333`. Android arm64 Release/AOT и iOS arm64 Release/AOT остаются явными mobile gaps до задач export/toolchain.

# Electron2D.Editor

Godot 4 является каноническим UX- и layout-референсом для `Electron2D.Editor 0.1-preview`. Electron2D.Editor воспроизводит знакомую структуру редактора Godot: верхнее меню и переключатель рабочих пространств, вкладки сцен, центральное рабочее пространство, `Scene` и `FileSystem` docks слева, `Inspector`/`Node` справа и инструментальную нижнюю панель. Трёхмерные рабочие пространства, инструменты, типы и настройки полностью исключены.

Любая задача `Electron2D.Editor`, которая создаёт или меняет видимый UI, имеет обязательный визуальный acceptance gate. Исполнитель должен открыть реальное окно редактора или documented automated harness, создать screenshot, сохранить его как артефакт проверки и проанализировать результат до закрытия задачи. Анализ должен подтверждать как минимум: элемент находится в ожидаемой области layout, доступен pointer/keyboard interaction там, где это применимо, текст не выходит за границы контейнера, активные/hover/focus states не ломают layout, UI соответствует `docs/editor/godot4-editor-reference.md`, а запрещённые `3D`, `AssetLib` и GDScript элементы не появились в видимом интерфейсе. Задача, изменяющая Editor UI, не считается готовой к приёмке, если есть только build, model test или smoke-команда без созданного и рассмотренного screenshot.

# Формальные критерии готовности

## Функциональность

Версия готова, когда сторонний разработчик может:

* установить Electron2D;
* создать новый проект;
* собрать сцену без ручного редактирования служебных файлов;
* написать C#-логику;
* запустить игру из редактора;
* сохранить и повторно открыть проект;
* экспортировать его на все заявленные платформы.

## Редактор

* Electron2D.Editor полностью работает на Electron2D.
* Нет обязательной зависимости от внешнего UI framework.
* Desktop executable `Electron2D.Editor` использует брендовую `.ico`-иконку из `data/assets/branding/`.
* Default layout соответствует `docs/editor/godot4-editor-reference.md`: `2D`, `Script`, `Game`, `Tasks`; `Scene`/`FileSystem` слева; `Inspector`/`Node` и `Agent Workspace` справа; bottom panel снизу.
* В Editor отсутствуют workspace `3D`, `AssetLib`, GDScript UI и disabled 3D controls.
* `Script` workspace позволяет редактировать C# code, получать IntelliSense, ставить breakpoints, запускать local desktop debug session и смотреть stacks всех threads, frame-scoped locals/arguments и watch values без внешней IDE.
* Нет известных ошибок, приводящих к повреждению сцен или ресурсов.
* Undo/redo работает для основных операций.
* Editor выдерживает многократный цикл run/stop.

## Платформы

Для каждой Tier 1 платформы:

* CI собирает runtime;
* reference game запускается;
* production package contents не содержит `.taskboard/**` и другие `EditorMetadata`;
* выполнен тест на реальном устройстве либо машине;
* проверены input, audio, rendering и filesystem;
* выполнен минимум 30-минутный soak test;
* нет монотонного роста native или managed memory;
* Android/iOS выдерживают многократный background/foreground cycle.

## Внутренние проверки репозитория

* `eng/Electron2D.Build` является внутренним C#-инструментом репозитория для будущего единого запуска проверок, генерации, подготовки пакетов и проверки релизного кандидата.
* `0.1-preview` переводит внутренние проверки репозитория на C#-команды `eng/Electron2D.Build`: `T-0207` добавил переносимый каркас команд `test`, `verify`, `update wiki --check`, `package --rid <rid>` и `release verify`, `T-0213` перенесла `verify readme`, `verify docs`, `update docs --check` и `update docs`, а `T-0214` переносит манифест API, вывод GitHub Wiki, правила лицензий и проверки манифестов без PowerShell как целевого интерфейса. Оставшиеся старые `.ps1`-скрипты являются миграционным долгом для `T-0210`, а не совместимой целевой поверхностью.
* `audit package` создаёт и проверяет внешний аудиторский ZIP из явной конфигурации, детерминированного patch, контрольных сумм, манифеста восстановления и сырых доказательств проверок. Ручная сборка таких ZIP после этой задачи считается нарушением workflow.
* Внешний аудит `T-0213` r05 принял перенос README/docs verifier-ов без blocker-ов.
* Внешний аудит `T-0228` r04 принял детерминированный инструмент сборки audit package без blocker-ов.
* Команды подготовки пакета и релизной проверки до реализации фактической упаковки завершаются закрытым отказом: не создают архивы, теги или GitHub Release и возвращают структурированную диагностику.
* Внешний аудит `T-0207` r04 принял внутренний C#-инструмент репозитория без blocker-ов; фактический перенос оставшихся проверок, CI и подготовки релиза остаётся за последующими задачами `T-0208`-`T-0210`.

## Release metadata

* README, NuGet package metadata и Editor executable используют брендовые ассеты из `data/assets/branding/`.

## API

* нет публичных заглушек;
* все публичные типы имеют XML-документацию;
* все публичные типы отражены в manual API profile, generated manifest и generated Wiki;
* каждый exported public type имеет manual `approved` decision и статус `Supported`/`profile_approved`; `Deferred`/`Unsupported` не могут оставаться exported public types текущего релиза;
* `Unapproved`, `Deferred` и `Unsupported` являются `outOfProfile` состояниями и не закрывают финальный gate `T-0980`;
* `e2d api compare-godot` читает полный manual profile: `approved` возвращает `profile_approved` и отдельную runtime availability, `deferred`/`unsupported` сохраняют точное решение, неизвестное имя возвращает `type_not_found`; `parityEvidence.status = not_verified`, а полные счётчики строгого сравнения с Godot 4.7 остаются будущим evidence-контрактом class tasks/final gates;
* breaking changes допустимы в `0.x`, но фиксируются в changelog.

## Производительность

Минимальный критерий:

* reference games работают в 60 FPS на опубликованном наборе эталонных устройств;
* интерактивный `RuntimeHost` использует измеряемый `DeltaTime`, накопитель fixed physics, ограничение догоняющих physics-шагов и ожидание только до срока кадра либо через синхронизацию показа, а не фиксированное ожидание после показа кадра;
* runtime-диагностика разделяет время `input`, `physics`, `process`, `render-plan`, `submit`, `present`, запрошенное и фактически прошедшее ожидание, а также ожидание во время паузы окна, чтобы живая проверка производительности могла измерять настоящую причину просадок;
* пустая сцена и типовая sprite-сцена не создают постоянных managed allocations каждый кадр после прогрева;
* batching действительно уменьшает количество draw calls;
* многократная загрузка и выгрузка сцен не приводит к утечкам GPU, audio и physics ресурсов;
* производительность редактора остаётся приемлемой на сцене примерно из 1 000 узлов.

## Стабильность данных

* 100 последовательных циклов save/load не повреждают тестовую сцену;
* переименование и перемещение ресурсов не ломают UID-ссылки;
* импорт можно полностью пересоздать из исходных файлов;
* формат проекта и импортированный cache разделены;
* ошибки сериализации не приводят к молчаливой потере свойств.

# Краткое определение релиза

**Electron2D 0.1-preview — это C#-движок и desktop-редактор, позволяющие создать небольшую законченную 2D-игру с node/resource архитектурой и экспортировать один проект под Windows, Linux, macOS, Android, iOS и WebAssembly browser.**

Главный критерий первой версии — не количество реализованных классов, а наличие полностью рабочего пути:

```text
Project → Scene → C# code → Run → Debug → Export → Play
```

Версию `1.0` следует выпускать позднее: после стабилизации публичного API, нескольких реальных проектов, сформированной migration policy и подтверждённой эксплуатации Android/iOS в production.

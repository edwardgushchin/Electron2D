## Назначение

`Tasks` — центральное рабочее пространство `Electron2D.Editor` для задач пользовательского проекта. Оно показывает встроенный `ProjectTaskManager` как обычную доску задач, а не как Markdown-файл, dock-панель или часть Agent Workspace.

Текущая реализация model-first: `ProjectTasksBoardSnapshot` описывает видимую доску, выбранную карточку, правый `Task Details`, фильтры, действия и visual harness. Постоянный desktop event loop, live drag-and-drop pointer events и полная привязка к открытому окну добавляются следующими задачами поверх этого snapshot-контракта.

## Snapshot model

Snapshot доски содержит:

- workspace switcher `2D`, `Script`, `Game`, `Tasks` и выбранный workspace `Tasks`;
- колонки `Ready`, `In Progress`, `Blocked`, `Review`, `Done`, `Cancelled`;
- карточки с task id, title, priority, labels, readiness, blocking reasons, rank и archived state;
- выбранную карточку `T-0155` и правый `Inspector` с заголовком `Task Details`;
- description, acceptance criteria, subtasks, activity kinds, linked transactions, jobs, diagnostics, artifacts и linked scenes/resources/nodes;
- drag-and-drop intent с target status, target rank, allowed result и diagnostic code для запрещённого transition;
- actions `Accept`, `Request Changes`, `Cancel`, `Create`, `Edit`, `Archive`, `Hard Delete`;
- filters `Status`, `Priority`, `Labels`, `Text`, `Linked Object`;
- признаки stable rank round-trip, archive view, hidden archived task, destructive confirmation, trusted human acceptance и работы без AI/MCP-сессии.

`Accept` и `Request Changes` помечены как trusted interactive Editor actions: они требуют доверенный контекст действия человека внутри редактора. Агентский payload не может включить `AgentAcceptActionAvailable` или принять задачу за пользователя.

## VS Code extension v1

Расширение живёт в `src/Electron2D.VsCode`, собирается TypeScript/Node 24 toolchain через `esbuild` и упаковывается в platform-neutral VSIX без встроенного `e2d`. Команда `Electron2D: Open Task Board` открывает один editor-area `WebviewPanel`; taskboard не регистрируется как custom editor и не записывает JSON напрямую. Editor title панели локализован как `Task Board` для английского интерфейса и `Доска задач` для русского и не включает folder/workspace name: имя проекта уже добавляется VS Code отдельным workspace segment, поэтому window title не должен повторять `Electron2D`. Project label внутри webview и multi-root folder selection при этом сохраняют выбранное имя workspace.

Язык расширения не имеет отдельной настройки и всегда следует `vscode.env.language`. Значения `ru` и `ru-*` выбирают русский словарь, `en` и `en-*` — английский; любая другая или пустая locale использует английский fallback. Первая версия локализации обязана полностью поддерживать `en` и `ru`: Extension Host, TreeView/sidebar, standard alerts/dialogs, status labels/descriptions, loading/error states, Task Details, board toolbar, filters, dependency/activity/file surfaces и безопасное отображение Markdown используют один типизированный словарь с одинаковым набором ключей. Интерполяция допускает только позиционные значения и не выполняет HTML. `webviewHtml` получает уже выбранную locale, задаёт корректный `<html lang>` и безопасно вставляет локализованные attributes/text; runtime `webview.ts` выбирает тот же словарь из `document.documentElement.lang`, поэтому до и после первого board snapshot язык не меняется.

Manifest contributions локализуются стандартным механизмом VS Code NLS: `package.json` содержит только `%key%` для `displayName`, description, capability descriptions, command title/category, Activity Bar title, view name, configuration title и setting description; `package.nls.json` является английским default, `package.nls.ru.json` содержит русский набор с теми же ключами. Оба файла входят в VSIX. Машиночитаемые `TaskStatus`, readiness/acceptance values, task IDs, CLI arguments, diagnostic codes, paths и task data никогда не переводятся. Смена display language требует обычной перезагрузки окна VS Code, после которой новая panel instance и sidebar используют обновлённую `vscode.env.language`.

Расширение регистрирует отдельный container `Task Board` в VS Code Activity Bar с монохромным SVG icon и view `Tasks`. Показ container через клик по значку автоматически выполняет `electron2d.openTaskBoard`: первый клик открывает доску в editor area, последующие клики раскрывают уже существующий `WebviewPanel` без создания дубликата. Команда из Command Palette остаётся равноправным способом открытия. SVG icon является частью VSIX; sidebar не хранит и не редактирует task data.

Manifest также активирует расширение по `workspaceContains:.taskboard/board.e2tasks`. Во время активации доверенной рабочей области extension host тихо проверяет папки в их исходном порядке и открывает первую папку с канонической доской в активной editor column. Эта панель получает фокус и становится текущей страницей вместо Welcome; отдельная команда закрытия Welcome не вызывается, восстановленные пользовательские вкладки не закрываются. В недоверенной рабочей области или при отсутствии канонической доски автоматического открытия нет. Startup path не показывает warning или Quick Pick, а ручная команда и Activity Bar сохраняют выбор папки для multi-root и повторно используют уже открытую панель.

Task list view получает compact board snapshot только через `e2d tasks board` и показывает каждую неархивированную задачу отдельной строкой `taskId · title`. Справа отображаются `status · priority`, tooltip содержит полное название и те же metadata. Порядок строк следует каноническому порядку статусов, затем placement rank и task id. Клик по строке открывает или раскрывает editor-area board и показывает modal details выбранной задачи. Watcher-ы board, active task и completed task files обновляют список с debounce; loading, empty и CLI error состояния должны быть видимы в sidebar, а не превращаться в пустую неразличимую область.

Все CLI-вызовы TreeView и WebviewPanel для одной пары workspace/executable проходят через общую FIFO-очередь. Следующая команда запускается только после success или failure предыдущей; ошибка не отравляет очередь, а разные workspace/executable не блокируют друг друга. Одинаковые compact board reads дополнительно coalesce-ятся общим in-flight broker и получают один Promise вместо дублирующего процесса. Сериализация охватывает board/task reads, create/edit/move/comment/attachment/lifecycle mutations и trusted human acceptance bridge: прямого обходного запуска `e2d tasks` в extension host нет.

Каждая команда внутри своей позиции очереди ограниченно повторяется после точной transient-диагностики `E2D-CLI-0002: Taskboard is locked by another writer` с возрастающей короткой задержкой. Позиция команды при повторе сохраняется, поэтому следующая mutation не может обогнать временно заблокированную. Конфликт task revision, запрещённый transition, malformed board и остальные semantic diagnostics не повторяются и сразу передаются стандартному VS Code alert. После исчерпания лимита последняя lock error также показывается пользователю. Это требуется потому, что каждый `TaskBoardDiskStore` constructor выполняет recovery под exclusive `.taskboard/.lock`: сериализация устраняет внутреннюю конкуренцию расширения, а bounded retry покрывает кратковременного внешнего writer-а.

Точный `Taskboard revision conflict` для rank-only `tasks move` является отдельным безопасным recovery-контрактом. Не покидая текущую позицию общей workspace-очереди, extension host перечитывает `tasks board --compact true --include-archived true`, публикует свежий `BoardView`, проверяет, что перемещаемая задача и указанная целевая группа всё ещё существуют, и повторяет исходный placement intent с актуальной `board.revision`. Допускается не более трёх CLI-попыток вместе с первой; перед каждым повтором обязателен новый canonical read. Если intent больше неприменим, конфликт повторяется после лимита или diagnostic не является точным конфликтом board revision, автоматический повтор прекращается и причина показывается стандартным VS Code alert. Task revision conflicts автоматически не перебазируются: edit/comment/lifecycle intent может зависеть от изменившегося содержимого и остаётся fail-closed. После точного `Task revision conflict` исходное действие остаётся невыполненным, но extension host обязательно перечитывает compact board и, если конфликтующая задача по-прежнему открыта, её полные details через `tasks get`; один локализованный VS Code alert сообщает, что snapshot обновлён и действие нужно повторить. Другие diagnostics не запускают этот recovery path.

Новая задача и `reopen` по умолчанию получают canonical `status=Ready`. Compact snapshot отдельно публикует effective `boardStatus`: для `Ready` он становится `Blocked`, если readiness равен `BlockedByDependencies`/`DependencyCancelled` или присутствует manual/environment/decision/external blocker. После завершения dependencies или снятия ручных причин следующий read автоматически возвращает `boardStatus=Ready`, не переписывая dependent task. Переход в `InProgress` и submit из `Review` по-прежнему требуют вычисляемый readiness `Ready`.

VS Code Task Board локализует названия колонок и status metadata в sidebar, но не меняет machine-readable значения в `.e2task`, `board.e2tasks`, CLI и webview messages. Заголовок каждой колонки содержит русскую подпись и tooltip с условиями размещения:

| Machine-readable status | Русская подпись | Когда задача размещается в колонке |
| --- | --- | --- |
| `Ready` | `Готово к работе` | Новая или reopened задача получает canonical status `Ready`. Карточка показывается здесь, когда readiness равен `Ready` и нет ручных blocking reasons. |
| `InProgress` | `В работе` | Работа явно начата из `Ready` или возобновлена из `Blocked`; обычный переход разрешён только при вычисляемом readiness `Ready`, то есть когда все dependencies имеют status `Done`. Сюда также возвращают задачу через trusted-human `request-changes` или `reopen --status InProgress`. |
| `Blocked` | `Заблокировано` | Сюда попадает явно остановленная задача, а также canonical `Ready` с `BlockedByDependencies`, `DependencyCancelled` или ручным blocker. После снятия причин вычисляемая карточка автоматически возвращается в `Ready`. |
| `Review` | `На проверке` | Рабочая фаза завершена. `acceptanceState=Open` означает внутреннюю проверку, `Submitted` — ожидание решения пользователя; `submit` не перемещает карточку в другую колонку. При замечаниях задача возвращается в `InProgress`, при принятии — в `Done`. |
| `Done` | `Завершено` | Доверенный человек принял `Review` с `acceptanceState=Submitted`. Обычный агентский или noninteractive CLI transition не может назначить этот status. |
| `Cancelled` | `Отменено` | Задачу явно отменили из `Ready`, `InProgress` или `Blocked`. `reopen` по умолчанию возвращает её в canonical `Ready`; при blockers карточка вычисляемо окажется в `Blocked`. |

Sidebar click имеет отдельный low-latency path. Если board panel уже существует, extension host немедленно раскрывает её и отправляет compact task из текущего `BoardView` в modal без повторного `tasks board`; полный `tasks get <task-id>` выполняется после immediate render. Compact task используется только для заголовка доступного loading state и не изображает отсутствующие description, criteria, activity, attachments или actions как готовые пустые данные. Ответ старого details request применяется только если пользователь всё ещё выбрал ту же задачу. Initial panel creation по-прежнему выполняет единственный обязательный board load до открытия выбранной task. Визуальная реакция на клик не должна ждать CLI startup или полного details snapshot.

Task details modal имеет одинаковую конечную ширину и высоту до и после hydration. Первый кадр содержит task id/title, Windows-подобную кнопку закрытия и отдельную content area с `aria-busy=true`, status-текстом `Загрузка подробностей задачи…` и skeleton placeholders. Loading state не показывает edit/comment/decision actions и не выводит заголовки пустых секций. После успешного `tasks get` skeleton атомарно заменяется полным содержимым в той же scroll-area без изменения внешней геометрии modal, скрытия backdrop или повторного открытия окна.

Успешно hydrated task сохраняется в памяти только на время жизни конкретного `TaskBoardPanel`. Повторный выбор той же задачи немедленно показывает этот полный snapshot и параллельно выполняет новый canonical `tasks get`; свежий ответ применяется только при неизменившемся selection token и обновляет cache. При публикации любого нового compact board snapshot extension host сравнивает revision выбранной карточки с revision hydrated cache. Если они различаются, cache этой задачи удаляется и открытый modal без закрытия и изменения navigation history получает canonical `tasks get`; до завершения hydration stale revision больше не считается актуальной для следующего действия. Это proactive обновление покрывает watcher, manual refresh и refresh после собственной mutation. Stale cache не является writer-ом и не обходит optimistic concurrency: выигравшая гонку mutation по-прежнему fail-closed отклоняет устаревшее действие и запускает описанный canonical refresh без автоматического повтора intent. Cache не сериализуется, не записывается в `.taskboard` и исчезает при закрытии panel.

Если initial hydration без cache завершается ошибкой, skeleton заменяется inline error state в том же modal с `role=alert`, понятным текстом и кнопкой повторной загрузки; ошибка не скрывается за backdrop и окно сохраняет геометрию. Если cache уже был показан, неуспешный background refresh не удаляет полные task details: cached snapshot остаётся доступным, а diagnostic показывается стандартным VS Code error notification. Error response, как и successful response, применяется только пока выбран тот же task id.

CLI разрешается в таком порядке: user-level setting `electron2d.taskboard.cliPath`, затем команда `e2d` из `PATH`. Каждый запуск использует массив аргументов и `shell: false`. Полный refresh вызывает `e2d tasks board --compact true --include-archived true`: карточки получают поля для колонок, фильтров, зависимостей и revisions, а также каталог тегов, дедлайн, сводку критериев и число вложений. Полные `description`, activity, тексты criteria, execution contract, attachment metadata и legacy fragments в compact response не входят. При открытии карточки extension отдельно вызывает `e2d tasks get <task-id>` и передаёт полный task snapshot с каноническим `description` в details. V3 storage хранит placements, parent и relations только по `taskUid`; CLI consumer projection вычисляет привычные `taskId`-поля для UI и не записывает их обратно. Mutations передают task/board revisions из последнего snapshot и проходят schema, whole-board semantic и transition validation. Webview, watcher и extension host не меняют `.taskboard/board.e2tasks`, `.e2task` или attachment metadata самостоятельно.

Extension выполняет taskboard operations только в trusted workspace. Автоматический startup выбирает первую папку с `.taskboard/board.e2tasks` без диалога; при ручном открытии в multi-root workspace пользователь по-прежнему выбирает конкретную папку, а выбор сохраняется только в workspace state. Watcher наблюдает `.taskboard/board.e2tasks`, `.taskboard/tasks/*.e2task` и `.taskboard/completed/*.e2task`, debounce-ит события и автоматически инициирует полный CLI refresh editor board и Activity Bar task list. Пользователь не должен нажимать `Refresh` после успешной внешней команды `e2d tasks`.

Автоматический refresh сериализован отдельно для каждого consumer. Одновременно выполняется не более одного compact board read; несколько событий во время одного чтения объединяются в один обязательный trailing refresh после его завершения. Поэтому watcher event, пришедший во время shared in-flight read, не может потеряться вместе с устаревшим snapshot, а более старый результат не может перезаписать более новый. Ошибка отдельного чтения остаётся видимой через стандартный VS Code alert, но coordinator освобождается и обслуживает следующий запрос. Собственные webview mutations после успешного CLI write используют тот же coordinator. Успешная публикация snapshot также синхронизирует выбранные details при изменившейся task revision. Конфликт board revision не перезаписывает чужое состояние: safe placement recovery сначала публикует canonical refresh и только затем повторяет всё ещё применимый intent; невосстановимый конфликт показывается через `vscode.window.showErrorMessage`.

Общие command/refresh diagnostics принадлежат VS Code workbench, а не webview layout. Ошибки create/edit/move/comment/attachment/archive/submit/accept/request-changes, manual refresh, watcher refresh и cached details refresh сохраняются в `lastError` и показываются ровно одним стандартным `vscode.window.showErrorMessage`. Extension host не отправляет общий `type=error` message в webview; HTML/CSS/JavaScript не содержат inline `#notice` banner. Initial uncached task-details error является исключением: он остаётся внутри modal как content-specific `role=alert` с кнопкой retry. Sidebar load failure остаётся TreeView error row, чтобы скрытый view не создавал повторяющиеся notifications.

Доска отображает фиксированные шесть колонок `Готово к работе`, `В работе`, `Заблокировано`, `На проверке`, `Завершено`, `Отменено`; размещение использует machine-readable `boardStatus`, а details и mutations сохраняют canonical `status`. В колонке `На проверке` карточка визуально различает внутреннюю проверку и ожидание пользовательского решения через `acceptanceState`, не создавая отдельную workflow-колонку. В VS Code-представлении задачи внутри каждой статусной колонки образуют единый плоский список без заголовков и контейнеров Epoch/Milestone. Канонические `groups` и `groupId` остаются в compact snapshot и `.taskboard`: скрытие групп не меняет данные. Drag-and-drop является только операцией placement: карточку можно переставить внутри её текущей effective-status колонки, extension host вызывает только `e2d tasks move` с новым `rank` и сохранённым `groupId`. Move message не содержит `status`, не вызывает `set-status` и не меняет task revision/activity. При устаревшей board revision host выполняет описанный canonical refresh/rebase/retry, не меняя target rank или group; drop в другую колонку по-прежнему отклоняется локально без CLI mutation. Прежний `Alt+ArrowLeft/Right` status move отсутствует. Статус меняется только отдельным lifecycle action/CLI command. Фильтры, archive view, dependency tree и details с description, criteria, comments/activity и attachment list сохраняются. Create, edit, placement move, comment, attachment add/remove, reopen, archive/unarchive, submit, accept и request changes отправляются в extension host как allowlisted typed messages и затем выполняются через CLI.

Footer подробностей зависит только от канонического состояния загруженной задачи. Для `Review` сохраняется действие приёмки. Активная неархивированная задача со статусом `Cancelled` показывает вторичную destructive-кнопку `В архив` и основную кнопку `Вернуть в работу`; другие активные статусы этих действий не получают. `Вернуть в работу` вызывает `e2d tasks reopen` с task revision и локализованной причиной, поэтому canonical результатом остаётся `Ready/NotSubmitted`, а не прямой переход в `InProgress`. `В архив` сначала требует локализованного modal confirmation, затем передаёт task revision и board revision в `e2d tasks archive`; отмена подтверждения не запускает CLI. Уже архивированная задача независимо от terminal-status показывает только отдельное действие `Вернуть из архива`, которое передаёт обе revision в `e2d tasks unarchive` и не выполняет `reopen`. После успешной lifecycle-операции extension host заново читает board и открытые details; ошибка или revision conflict показываются стандартным VS Code alert без оптимистического изменения DOM.

### Reference-like visual system VS Code

Основная доска следует утверждённому reference screenshot, не копируя VS Code chrome и не изменяя lifecycle semantics:

- верхняя область является единой компактной полосой. Слева она содержит фирменную SVG-эмблему Electron2D и breadcrumb `folder.name / Доска задач`: имя проекта берётся только из выбранной workspace folder, а название доски локализуется. Устаревшие `Electron2D Task Board` и `taskboard (main)` не возвращаются;
- по центру toolbar содержит широкое поле поиска с SVG-лупой и видимой подсказкой `Ctrl+F`; справа располагаются текстовая кнопка фильтров с badge количества активных фильтров, компактный archive toggle, SVG-кнопки управления тегами и обновления, затем одна обычная primary-кнопка `Создать задачу` без split-секции. На широком viewport все элементы остаются в одной строке, а на узком переходят в явно заданную адаптивную сетку без переполнения;
- webview не резервирует строку для общих diagnostics и не рисует собственный error banner; стандартный VS Code alert остаётся поверх workbench и не меняет геометрию board;
- status column имеет тонкий цветовой accent, компактный header с локализованным title, count badge, overflow affordance и keyboard-accessible collapse button; collapsed state не удаляет данные и не меняет machine-readable status;
- Epoch/Milestone headers, nested lanes и отдельные group drop zones в VS Code-представлении не создаются; порядок карточек определяется статусом и rank, а сохранённый `groupId` остаётся доступен CLI и другим потребителям taskboard;
- collapse controls колонок и фильтр тегов используют настоящие SVG-chevron с квадратным `viewBox`, stroke path и `aria-hidden=true`; Unicode-стрелки в интерактивных controls запрещены. Поворот задаётся состоянием `aria-expanded`, а `transform-box: fill-box` и `transform-origin: center` удерживают геометрический центр. Кнопка `Создать задачу` не имеет SVG, `aria-expanded`, `aria-controls`, соседней toggle-секции и выпадающего меню;
- длинный card list сначала показывает компактный preview из шести карточек и кнопку `+ N ещё`; раскрытие выполняется локально в webview, не меняет board placement и не скрывает задачи от filter/count semantics. Раскрытые списки запоминаются в памяти текущей webview отдельно по `TaskStatus`, поэтому watcher refresh, мутация, ручное обновление, изменение фильтров и любой повторный `state` render показывают актуальный полный набор карточек без возврата к preview. Новая webview начинает с компактного состояния; этот presentation-флаг не записывается в TaskBoard, task activity, workspace state или настройки проекта;
- над названием карточки показываются цветные теги, назначенные задаче. Они разрешаются через общий каталог доски и полностью отсутствуют, если задача не имеет тегов;
- если у задачи есть безопасное растровое вложение, верх карточки занимает полноширинная обложка. Изображение целиком масштабируется до доступной ширины с сохранением пропорций: горизонтальное кадрирование, фиксированная высота и `object-fit: cover` не допускаются. По умолчанию используется первое изображение, а явно выбранное имеет приоритет. Карточка без изображений не создаёт пустую область. Изображение не выходит за границы карточки и имеет осмысленный `alt`;
- нижняя строка карточки показывает только имеющиеся данные: локализованную дату дедлайна, прогресс критериев в виде `Passed/Total` и число прикреплённых файлов. Связанные пути к файлам не считаются вложениями. Пустая карточка не резервирует места под эти показатели;
- card anatomy показывает теги и переносимый title в основной части, а task id — только в нижней мета-строке перед priority badge, доступными показателями и status indicators. Номер не резервирует отдельную верхнюю строку и не дублируется. Каждый tag chip и priority badge использует собственный `inline-flex`-контейнер с явным центрированием по обеим осям, компактной одинаковой минимальной высотой и `line-height = 1`; подпись не зависит от font baseline, а priority badge остаётся на одной вертикальной оси со всей мета-строкой. Dependency, blocking, terminal и acceptance indicators остаются компактными; полные тексты criteria и attachment metadata для карточки не загружаются;
- status accents используют разные VS Code theme variables (`charts-*`, testing/error/warning и disabled foreground) для всех шести колонок; literal hex palette в board/modal presentation запрещена. Текст, фон, границы, focus и hover также опираются на VS Code theme variables и остаются читаемыми в light/dark themes;
- board остаётся внутренней двумерной scroll-area. Шесть развёрнутых status columns используют единый адаптивный track: `calc(13.25rem + 10px)` является минимальной шириной, а `20rem` — максимальной. Между этими пределами колонки расширяются равномерно и вместе с неизменными gap/padding заполняют доступную ширину board viewport. На узком viewport колонки не сжимаются ниже минимума и сохраняют нижний horizontal scrollbar; на сверхшироком viewport они прекращают рост после `20rem`. Card/long token containment, порядок статусов и высота колонок по содержимому не меняются;
- русские status labels остаются presentation source of truth; reference English captions не возвращаются в UI, а внутренние identifiers `Ready`…`Cancelled` не локализуются.

Кнопка управления тегами открывает стандартный VS Code Quick Pick. Из него можно создать тег, выбрать существующий тег для изменения имени или цвета и запросить удаление. Используемый тег CLI удалить не позволяет. Цвет вводится как одно из legacy-имён либо точный `#RRGGBB`; поле показывает текущий canonical value, отклоняет остальные CSS-строки до mutation и передаёт значение только в `e2d tasks tag create/update`. В подробностях задачи действие `Добавить тег` предлагает доступные определения из общего каталога и пункт создания. При выборе нового пункта VS Code запрашивает название и цвет, после чего extension вызывает атомарный `tasks tag create --assign-to`: тег появляется в каталоге доски и сразу назначается задаче. Обычное редактирование задачи также позволяет задать или очистить календарный дедлайн.

Визуальное обновление не изменяет modal details, skeleton/cache/error states, CSP, trusted-workspace guard и CLI-only mutation boundary. Drag-and-drop payload является rank-only placement intent и не служит lifecycle transition.

Высота каждой status column определяется только её header и фактическим content: пустая колонка заканчивается после header/body padding, короткая — после последней card. Viewport-filling `min-height` для column запрещён. Board сохраняет `align-items: start` и общий `overflow: auto`, поэтому длинная колонка по-прежнему расширяет vertical scroll range всей доски.

Все шесть колонок сохраняют минимальную читаемую ширину и не сжимаются до ширины editor viewport. Центральная область доски занимает оставшуюся высоту webview и является самостоятельной двумерной scroll-area: горизонтальная полоса доступна у нижней границы видимой области, а длинная status column сохраняет intrinsic content height и увеличивает общий `scrollHeight` доски выше `clientHeight`. Общая вертикальная полоса board позволяет дойти до последней карточки; column не растягивается только до высоты viewport и не обрезает нижнюю часть card list. Для вертикальной полосы используется односторонний `scrollbar-gutter: stable`: место резервируется только у правой границы scroll-area, а зеркальный левый gutter не создаётся, поэтому первая колонка выровнена с левой границей header content. Каждая task card рассчитывает padding и border внутри ширины родительской колонки; grid min-content не может расширить column/list, а длинные URL, пути и технические идентификаторы переносятся внутри карточки.

Открытая задача не занимает отдельный track и не меняет геометрию доски. `Task Details` показывается как modal window по центру webview поверх затемнённого backdrop. Окно ограничено размерами viewport, имеет фиксированную title bar и отдельную прокручиваемую content area; заголовок и Windows-подобная кнопка `×` остаются видимыми при прокрутке длинного description/activity. Кнопка имеет доступное имя `Close task details`, закрытие доступно также по `Escape`, после чего focus возвращается на карточку, открывшую modal, если она ещё находится в DOM.

Task Details хранит локальную browser-like историю только на время открытого modal. Прямой выбор карточки на доске или строки VS Code sidebar открывает задачу как новый navigation root и очищает прежний stack. Переход по dependency или parent внутри modal добавляет текущий `taskId` в stack, если целевая задача отличается от текущей; цепочка `T-A → T-B → T-C` последовательно возвращается через `T-B` к `T-A`. Повторное открытие текущей задачи не создаёт запись. Close через `×`, backdrop или `Escape` синхронно очищает host history и текущий selection token, поэтому запоздавшая lazy hydration не может снова показать закрытую или предыдущую задачу. Back немедленно выбирает предыдущий compact/cached snapshot, не добавляя обратный переход в stack; late hydration применяется только если task id и selection token всё ещё текущие.

Кнопка настроек доски открывает отдельное доступное окно управления тегами. В нём можно создать тег, изменить его название и цвет или удалить неиспользуемый тег. Все действия отправляются в Extension Host как allowlisted messages и выполняются через `e2d tasks tag`; webview не меняет каталог самостоятельно. Редактирование задачи показывает существующие теги из каталога. Пользователь может там же ввести новое название: расширение вызывает атомарное создание глобального тега с назначением текущей задаче, поэтому новый тег сразу появляется в настройках и становится доступен другим задачам. Одноразового тега только для одной карточки не существует.

В левой части верхней строки title bar находится Back control. Он имеет `aria-label="Назад к предыдущей задаче"`, совпадающий tooltip, центрированный DOM-created SVG arrow и disabled state, когда stack пуст. Unicode arrow запрещён. Размер, foreground/background, hover, disabled и `focus-visible` используют только `--vscode-*` theme variables. Overflow и Windows-like close controls остаются справа; Back доступен также во время lazy hydration целевой задачи, если предыдущий history entry уже существует.

Reference-like modal использует full task snapshot и разделяет данные на устойчивую оболочку и четыре keyboard-accessible вкладки. В фиксированной верхней области находятся breadcrumb `TASKS / T-ID`, title, close/overflow controls, chips canonical status/priority/labels/parent и tablist `Обзор`, `Детали`, `Зависимости`, `Активность`. Каждый label является ссылкой на глобальный каталог: renderer разрешает `task.labels` через `BoardView.tags`, показывает `tag.name` и theme-compatible цвет `tag.color`; сырой `tagId` не считается пользовательской подписью, а отсутствующая запись каталога не создаёт ложный chip. Активная вкладка имеет `aria-selected=true`, связанный `tabpanel` и theme-derived focus/accent. После hydration по умолчанию открывается `Обзор`. Для задачи в `Review` fixed footer остаётся вне прокручиваемого tabpanel и содержит только одну зелёную кнопку `Принять`, выровненную справа. Кнопка доступна, когда все имеющиеся критерии имеют состояние `Passed`; в других workflow-статусах footer не отображается. Прежние нижние действия редактирования, тегов, вложений, submit, request changes, архивации и закрытия отсутствуют.

Четыре основные вкладки следуют отдельным reference-композициям:

- `Обзор` — responsive two-column workspace с пропорцией около `1.5:1`. Левая колонка является одной непрерывной bordered panel и содержит три borderless-секции: `Цель`, нумерованный блок `Что нужно сделать` из `executionContract.requiredOutputs` и `Контекст` из task type/readiness/stop conditions. Правая колонка содержит отдельные `Сводка`, `Связанные файлы`, `Прикреплённые файлы` и `Каталоги/области проекта` именно в этом порядке. `Сводка` показывает ровно шесть видов сведений: номер задачи, статус, приоритет, теги, связанные задачи и дату создания. Рядом с `taskId` расположена кнопка с DOM-created SVG copy icon, которая копирует точный номер и имеет локализованное доступное имя. Теги используют тот же каталог, цветовой accent и компактный `.card-tag`, что карточки доски. Прямые parent/dependency/child/dependent связи отображаются списком внутренних ссылок и продолжают историю текущего окна; рекурсивная dependency closure в краткую сводку не входит. Строка `Тип задачи` в `Сводке` отсутствует. `Критерии приёмки` и `Команды` в `Обзоре` не дублируются: полный canonical список критериев и `executionContract.requiredCommands` доступны во вкладке `Детали` как `Критерии приёмки` и `Обязательные проверки`. Surface panels совпадают с editor/modal background и отделяются только theme border, без тяжёлой literal заливки.
- Ниже двух колонок `Обзора` находится полноширинный блок `Чат с агентом`. Он показывает canonical `conversation.messages` текущей задачи в порядке `sequence`, визуально различает пользователя и `Agent / Codex`, сохраняет переносы безопасного Markdown-текста и не смешивает переписку с системной timeline `Активности`. Нижняя composer-строка содержит многострочное поле, кнопку отправки и во время generation — действие отмены. Промежуточные reasoning summary/commentary, tool, permission и status события показываются как transient presentation текущего run; в TaskFile сохраняются только исходная реплика пользователя и итоговая реплика агента. Перезагрузка панели восстанавливает canonical историю и понятное состояние отсутствующей, отменённой либо разорванной backend-сессии, но не выдаёт незавершённый transient текст за финальный ответ.
- Webview отправляет Extension Host только allowlisted intents `sendAgentMessage`, `cancelAgentRun` и `respondAgentPermission`; прямой `fetch`, произвольная команда, provider credential, capability token и OpenCode endpoint в webview не попадают. До обращения к модели Extension Host записывает human message через краткоживущий private stdio capability bridge с явным UTF-8 без BOM независимо от Windows console/OEM code page и получает новую task revision. Только точный `Task revision conflict` для ещё не сохранённой human-реплики допускает ограниченное восстановление: host перечитывает ту же задачу, проверяет `taskUid` и повторяет ровно этот append с новой revision; прочие конфликты, повторный conflict и неоднозначный результат fail closed, поэтому уже выигравшая реплика не дублируется. После успешного ответа final message дописывается через trusted agent route с `agentRunId`. Перед каждым run host строит canonical task context, фиксирует checkpoint с revision/sequence/digest и передаёт агенту task ID, постановку, критерии, execution contract, conversation, activity summary и manifest вложений. Если задача изменилась после checkpoint, продолжение требует явного rebase на свежий digest. Terminal presentation предыдущего run не блокирует новый `connecting`: новый `runId` сбрасывает transient state, а поздние события прежнего run игнорируются.
- `OpenCodeBackend` живёт только в Extension Host: для каждой workspace folder существует не более одного принадлежащего расширению локального `opencode serve`, а каждой задаче соответствует отдельная OpenCode session. Session key включает canonical workspace URI и `taskUid`; переключение задачи не переносит сообщения, context или разрешения между sessions. Provider registry и authentication берутся из обычной пользовательской конфигурации OpenCode, а каждый context/prompt request чата T-1222 явно выбирает доступную в этом registry модель `openai/gpt-5.6-sol` (`GPT-5.6 Sol`). Выбор на уровне request обязателен: восстановленная session хранит последнюю использованную модель и иначе может проигнорировать новый server default, снова отправив запрос в LM Studio. TaskBoard не задаёт `enabled_providers`, provider allowlist, provider endpoint или credential и не выполняет fallback на отдельный локальный inference server; отсутствие требуемой модели/authentication даёт структурированную диагностику. Ошибка, которую OpenCode вернул внутри assistant response, не подменяется сообщением о пустом финале: Extension Host сохраняет безопасную причину, включая нехватку контекстного окна модели, и удаляет из неё credentials. Потеря транспорта инвалидирует runtime и привязанный session cache. Однозначный connection-refused до подтверждённой доставки prompt допускает один запуск нового owned runtime и одну повторную попытку; неоднозначная ошибка не приводит к слепому replay. Raw `fetch failed` преобразуется в локализованную безопасную диагностику с понятным действием. Подписка на события привязана к времени жизни backend через `AbortSignal`, поэтому закрытие панели или workspace не оставляет SSE-подключение и дочерний server process. TaskBoard хранит canonical user/final messages и минимальные идентификаторы run/session, необходимые для восстановления связи. Stock OpenCode сохраняет собственные session/message records локально в user data; строгий no-disk режим не заявляется и показывается как ограничение, пока backend не предоставляет доказуемую ephemeral retention.
- Backend принудительно отключает OpenCode sharing и применяет fail-closed permission policy поверх обычной OpenCode-конфигурации, не заменяя её provider registry или authentication; единственный request-level model override фиксирует требуемый `openai/gpt-5.6-sol`. Чтение project context может быть разрешено, а изменение файлов, shell/task execution, network и доступ вне workspace требуют явного пользовательского подтверждения или запрещаются. Карточка permission показывает конкретное действие и scope; ответы `один раз`, `для сессии` и `отклонить` проходят только через Extension Host. Secrets, environment values, authorization headers и raw tool payload не передаются в webview, task conversation, activity или output logs. Отмена вызывает `session.abort`, закрытие панели отменяет подписку на SSE, а dispose workspace завершает принадлежащий расширению server process.
- Presentation adapter не подменяет canonical storage, но исправляет уже мигрированные snapshots. Если canonical `description` точно распознаётся как полный legacy task envelope с heading текущего `taskId` и секцией `### Самодостаточное описание`, в `Цели` отображается только содержимое этой секции до следующего heading; обычный description показывается целиком без Markdown-интерпретации. Overview классифицирует typed `linkedArtifacts`: `Связанные файлы` содержит только file paths, которые агент или другой task writer явно посчитал относящимися к задаче, а `Каталоги/области проекта` — только directory paths. `task.attachments` никогда не примешивается к этому списку и не копируется в `linkedArtifacts`. Отдельный нижний блок `Прикреплённые файлы` строится только из canonical `task.attachments` и показывает каждое вложение независимо от его происхождения: добавленное агентом или автоматически прикреплённое платформой, когда пользователь отправил файл в чат. `executionContract.requiredCommands` не читается и не отображается в Overview. Command-like и неизвестная произвольная строка из `linkedArtifacts` не отображается как файл; нормализация данных выполняется отдельно через CLI. Каждый file/directory/attachment path получает детерминированный DOM-created SVG icon по Explorer-like semantics: folder, Markdown/README, JSON, C#, C# project, image, archive или generic file; attachment дополнительно использует media type. Иконка имеет собственный theme-compatible цветовой класс, не является Unicode-глифом, не использует `innerHTML` и не изменяет безопасный текст пути. Оба списка имеют раздельные локализованные empty states. Имя каждого связанного и прикреплённого файла является keyboard-accessible ссылкой. Текст ссылки всегда занимает одну строку: не помещающийся путь скрывается внутри доступной ширины и завершается многоточием без горизонтального скролла секции, а точное полное значение сохраняется в `aria-label` и tooltip. Для связанного файла и нерастрового вложения click или `Enter` передаёт Extension Host только allowlisted project-relative path; host повторно отклоняет absolute path, URI и traversal, проверяет containment внутри выбранной workspace folder и открывает существующий обычный файл через стандартный VS Code editor. Растровое вложение с доверенным `previewUri` не покидает Task Board: ссылка открывает поверх Task Details внутреннее theme-compatible окно просмотра с полным изображением, именем файла и доступной кнопкой закрытия; viewer закрывается по `Escape` или клику по backdrop и возвращает фокус к ссылке. Ошибка host-открытия показывается стандартным VS Code alert. Допустимые cover/remove actions располагаются только в строках `Прикреплённых файлов` и не меняют семантику ссылки на сам файл. Отдельное действие `Показать в проводнике` отсутствует; workspace reveal не входит в webview message contract. Для проверки presentation используются реальные `e2d tasks get`, а не visual fixture.
- `Детали` — плотная reference-форма: первая строка параметров status/priority/readiness/task type, вторая строка createdAt/updatedAt; ниже идут full-width criteria rows, четырёхколоночный execution contract `Можно менять`/`Нельзя менять`/`Ожидаемый результат`/`Стоп-условия`, checklist `Готово к началу` и required command chips. Поля status/readiness являются отображением canonical snapshot, а не обходным editor-ом task state.

Actions имеют явную визуальную иерархию: недоступная мутация `Добавить связь` остаётся компактной secondary-кнопкой в toolbar, использует DOM-created SVG plus вместо Unicode и theme-derived disabled state с пояснением CLI-only boundary; `Открыть задачу` является full-width primary-кнопкой inspector с DOM-created SVG перехода. Обе кнопки используют единые размеры icon/text, выраженную theme-derived поверхность, высоту не меньше стандартного VS Code action, border radius и theme hover/active/focus states; browser-default button surface, тонкая чёрная рамка и текстовый Unicode-глиф запрещены. Удаление composer не оставляет отдельной нижней панели или зарезервированного зазора: timeline следует сразу после toolbar и использует общую прокручиваемую область tabpanel. VS Code Task Board не использует модель назначения задач. Overview и Details не показывают `Назначен`, `Исполнитель`, `Не назначен` или assignee-derived metadata; edit dialog не запрашивает assignee, webview message/security allowlist не принимает его, extension host не передаёт `--assignee`, а text search не учитывает это значение. Nullable schema v2 field может читаться для backward compatibility старых task documents, но не является частью UI или mutation contract расширения и не вызывает массовой миграции taskboard.

Пользовательский текст задачи поддерживает одну безопасную inline-конвенцию: фрагмент между парными обратными кавычками является указанием на жирное выделение. Например, исходное `` `.taskboard` `` отображается как **.taskboard** без видимых обратных кавычек. Конвенция применяется к заголовкам задач, критериям приёмки, значениям параметров, execution contract, обязательным командам, dependency labels, description и activity payload. Парсер не интерпретирует HTML или остальной Markdown, не связывает delimiters через перевод строки и программно создаёт только текстовые узлы и `strong`; непарная обратная кавычка сохраняется буквально.

Закрытый details layer обязан быть полностью исключён из layout и hit-testing тройным состоянием: HTML `hidden`, DOM `inert`/`aria-hidden=true` и author CSS `.details[hidden] { display: none !important; pointer-events: none !important; }`. Открытие снимает все три ограничения до создания dialog content; close button, `Escape` и backdrop click используют один visibility helper. Reveal существующей editor-area панели явно выполняется с `preserveFocus: false`, поэтому клик по Activity Bar возвращает фокус в Task Board, а невидимый modal никогда не затемняет и не блокирует доску.

`Принять` требует явного modal confirmation VS Code. Capability не приходит из webview payload: extension host создаёт краткоживущий private stdio bridge context после подтверждения, а actor/time определяются host-ом. Если `Review` ещё имеет `acceptanceState=InternalReview`, extension host внутри одной позиции общей FIFO-очереди сначала выполняет обычный `submit`, берёт новую task revision и затем вызывает trusted human `accept`; для `Submitted` повторный `submit` не нужен. Правило TaskBoard «принимать только переданную задачу» не ослабляется. Noninteractive CLI, MCP и агент не могут получить human acceptance через пользовательский JSON payload.

Webview использует строгий CSP с nonce, `localResourceRoots`, allowlisted message discriminators и безопасное построение DOM без `innerHTML`: обычный пользовательский текст создаётся через text nodes, а разрешённое inline-выделение — через `strong`. Inline preview и обложка карточки разрешены только для raster image MIME types; SVG, HTML, архивы и прочие вложения показываются как metadata/download action и никогда не исполняются. Compact snapshot содержит путь, но не готовый URI: Extension Host повторно проверяет raster allowlist и префикс `.taskboard/attachments/<taskId>/`, после чего выдаёт webview URI. Скрипт webview не получает Node API и не выполняет команды оболочки.

Agent avatar создаётся через безопасные SVG DOM API без `innerHTML` и внешней загрузки, наследует цвета активной темы и имеет доступное имя Codex. Буквенный `C` для таких записей не создаётся. Человек, service и прочие участники не проходят agent predicate: их подпись продолжает строиться из canonical identity, а аватар — из первой безопасной буквы `actorId`. Будущий выбор провайдера должен заменять presentation provider в одной точке, но UI настройки и другие провайдеры в текущую задачу не входят.

## Читаемое представление изменений задачи в Activity

Событие `TaskPatched` хранит неизменяемый JSON Patch как техническое доказательство перехода между ревизиями, но вкладка `Активность` не должна показывать этот документ пользователю целиком. Presentation layer разбирает только массив `patch`, извлекает из каждого JSON Pointer первое поле пути и сохраняет порядок первого появления. Повторные операции над вложенными элементами одного раздела объединяются в одну подпись.

Известные поля отображаются локализованными названиями разделов задачи. Например, изменения `/title`, `/description`, `/acceptanceCriteria`, `/links` и `/executionContract` должны выглядеть как короткое резюме «Задача обновлена: заголовок задачи, описание, критерии приёмки, связанные файлы, контракт выполнения». Значения операций, JSON-разметка и escape-последовательности Unicode в UI не выводятся. Резюме участвует в поиске по активности вместо служебного payload.

Parser не меняет canonical activity и не интерпретирует HTML. Если payload не является корректным объектом с массивом `patch`, событие остаётся видимым и его исходный текст проходит через прежний безопасный text-node renderer. Пустой корректный patch получает нейтральную подпись «Задача обновлена».

Событие `StatusChange` также хранит технический JSON с полями `previous`, `next` и `reason`, но вкладка `Активность` показывает его как локализованное предложение. Например, переход `Review` → `InProgress` выглядит как «Статус изменён: На проверке → В работе. Причина: …». Parser `StatusChange` не изменяет canonical activity и не доверяет произвольному объекту. Если payload не является объектом с допустимыми `previous`/`next` и непустой строкой `reason`, timeline сохраняет прежний безопасный буквальный fallback без интерпретации HTML и без исключения при отрисовке.

## Модальная форма создания задачи

Обычная кнопка `Создать задачу` открывает внутри webview одно модальное окно вместо последовательности `window.prompt`. Удалённые ранее split-секция, стрелка и дополнительное меню не возвращаются. Форма содержит обязательное однострочное название, многострочное Markdown-описание, выбор `P0`–`P3` с начальным значением `P2` и необязательный календарный срок. Пользовательские значения назначаются только через свойства DOM `value` и передаются как данные; HTML из полей не интерпретируется.

Название нормализуется через `trim`; до непустого значения кнопка `Создать` недоступна. Срок передаётся только как строгая существующая дата `YYYY-MM-DD`; локальная ошибка показывается возле поля и не закрывает форму. `Ctrl+Enter` отправляет валидную форму, а `Отмена`, кнопка закрытия, backdrop и `Escape` закрывают только непоставленную в очередь форму, очищают её и возвращают focus исходной кнопке. После подтверждения webview отправляет ровно одно allowlisted сообщение `{ type: "create", title, description, priority, deadline? }`, переводит controls в pending-state и запрещает повторную отправку и закрытие до результата. Extension Host выполняет публичный `e2d tasks create` с board revision recovery. После canonical refresh он отправляет отдельное успешное acknowledgement: форма закрывается, черновик очищается, focus возвращается кнопке. Dialog использует `role=dialog`, `aria-modal=true`, связанный заголовок, видимые labels и theme tokens VS Code. Контейнер ограничен viewport, длинное описание прокручивается внутри формы, а на узкой ширине поля и кнопки переходят в одну колонку без горизонтального overflow. ## Центрирование счётчика статусной колонки

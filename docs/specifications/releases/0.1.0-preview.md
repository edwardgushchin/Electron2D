# Electron2D 0.1.0 Preview

Статус: целевая спецификация релиза.
Источник: перенесено из корневого `GOAL-0.1.0.md`.
Последнее обновление: 2026-06-22.
Связанные документы: [Архитектура и платформенный стек Electron2D](../architecture/engine-platform-stack.md); [Agent-native cross-platform 2D game engine workflow Electron2D 0.1](../architecture/agent-native-workflow.md).

Первая версия реализует **первый законченный 2D-профиль движка**, а не весь API Godot.

## Цель

Разработчик на C# должен суметь:

> создать проект, собрать сцену в редакторе, написать игровую логику, импортировать ресурсы, запустить игру и экспортировать один проект под Windows, Linux, macOS, Android и iOS без переписывания игрового кода.

Electron2D также должен соответствовать позиционированию **Agent-native cross-platform 2D game engine**:

> человек и AI-агент должны суметь создать небольшую законченную 2D-игру на C#, работая в открытом `Electron2D.Editor` с одной живой моделью проекта; headless-режим остаётся обязательным для CI, тестов и автономных запусков, но не заменяет основной совместный workflow.

Agent-native cross-platform 2D game engine в `0.1.0` означает не встроенный чат и не привязку к конкретной LLM, а архитектуру, где важные операции проходят через живый `ProjectWorkspace`, общий слой инструментальных операций `Tooling`, локальный MCP-сервер, CLI для headless-сценариев, текстовые project files, структурированную диагностику, API manifest, Editor Capability Manifest, карту совместимости и видимое управление запущенной игрой.

Версия считается успешной, когда Electron2D подходит для создания и выпуска небольшой законченной 2D-игры, а не только для демонстрации отдельных подсистем.

Целевая аудитория первой версии:

* C#-разработчики;
* одиночные разработчики и небольшие команды;
* программисты, которым нужен node/resource API в полностью управляемой .NET-экосистеме;
* проекты малого и среднего масштаба;
* AI-агенты, CI и IDE-интеграции, работающие через публичный tooling API без GUI automation и подключающиеся к активной Editor-сессии, когда она открыта.

Редактор предназначен для desktop. Android и iOS являются платформами запуска и экспорта, но не редактирования проектов.

# Что можно сделать на первой версии

Electron2D 0.1 должен позволять создавать:

* простые платформеры;
* top-down игры;
* аркады;
* головоломки;
* карточные и настольные игры;
* визуальные новеллы;
* UI-heavy приложения;
* небольшие RPG без сложной навигации и освещения;
* локальные игры с клавиатурой, мышью, геймпадом или touch-вводом.

Первая версия не обязана поддерживать крупные открытые миры, сложное динамическое освещение, rollback multiplayer или тысячи физических объектов.

# Основной пользовательский сценарий

Полный рабочий цикл должен выглядеть так:

1. Разработчик создаёт проект Electron2D.
2. Открывает его в `Electron2D.Editor`.
3. Создаёт сцену.
4. Добавляет узлы.
5. Импортирует текстуры, шрифты и звук.
6. Настраивает свойства через Inspector.
7. Создаёт C#-скрипт и прикрепляет его к узлу.
8. Настраивает Input Map.
9. Запускает текущую сцену или весь проект.
10. Исправляет ошибки по сообщениям редактора.
11. Создаёт export preset.
12. Получает запускаемую сборку для выбранной платформы.

Этот сценарий должен работать на чистой машине после установки документированных зависимостей.

Совместный сценарий человека и AI должен использовать тот же `ProjectWorkspace`, что и ручные операции Editor:

1. Разработчик открывает проект в `Electron2D.Editor`.
2. Разработчик создаёт задачу в Project Tasks или выбирает существующую задачу как active task.
3. AI-агент подключается к активной Editor-сессии через MCP/IPC.
4. Созданные агентом файлы сразу появляются в FileSystem dock без ручного refresh.
5. Изменения сцен и ресурсов сразу отражаются в Scene Tree, Inspector и viewport.
6. Ручные изменения разработчика и изменения AI объединяются структурно, а конфликтующие правки показываются в conflict panel.
7. Агент запускает текущую сцену в видимом игровом процессе, отправляет input, ставит игру на паузу, делает frame step и получает screenshot/runtime tree.
8. Агент связывает transactions, jobs, diagnostics и artifacts с active task.
9. Diagnostics, тесты и visual diff, то есть визуальное сравнение результата, видны в Editor и доступны агенту через MCP.
10. Агент отправляет задачу в `Awaiting Acceptance`.
11. Разработчик просматривает changeset, tests, diagnostics и screenshots, затем принимает результат или запрашивает изменения.
12. Последняя AI-транзакция может быть отменена обычным Undo.

## Project Tasks

Project Tasks являются обязательной подсистемой Editor:

1. Пользователь видит доску задач с колонками `Backlog`, `Ready`, `In Progress`, `Blocked`, `Review`, `Awaiting Acceptance`, `Done`, `Cancelled`.
2. Задачи, board и activity хранятся как first-class документы `ProjectWorkspace`, а не как отдельный Markdown-backlog.
3. Task mutations поддерживают `expectedRevision`, dirty state, grouped Undo/Redo, external synchronization и conflict handling.
4. `Review` означает техническую проверку, а `Awaiting Acceptance` означает ожидание решения разработчика.
5. AI может отправить задачу на приёмку, но не может сам поставить `Done`; приёмка требует доверенный interactive Editor context, а не самодекларируемый `ActorKind`.
6. Любое изменение task document, включая `ExternalImport`, migration, CLI import и crash recovery, проходит через `TaskTransitionValidator` и `TaskAcceptanceService`; прямая правка файла не может установить `Done`, `AcceptedAt`, `AcceptedBy`, `AcceptanceState`, audit-поля activity или привилегированный reopen без structured diagnostic и pending-import/conflict state.
7. `.electron2d/tasks/**` является `EditorMetadata`: эти документы доступны Editor, Tooling, CLI и MCP, но не импортируются как игровые ресурсы, не попадают в production asset packs, APK, AAB, app bundle или desktop distribution и не материализуются в runtime snapshot как игровые файлы.
8. Если task data экспортируется во внешний отчёт, это выполняется только отдельной явной report-командой; такой экспорт не является обязательной возможностью `0.1.0` и не становится canonical storage. Jobs могут хранить `TaskId`, но не копируют task activity в build directory по умолчанию.
9. Изменение task status, activity или board rank не делает build/test/run artifacts `stale`, если игровые входные документы и build/run configuration не изменились.
10. `Reopen` является человеческим действием, а не отдельным статусом; история приёмки сохраняется в activity.
11. Ручной `Blocked` и блокировка из-за незавершённых dependencies различаются: закрытие dependency не снимает ручной blocker.
12. Отдельного действия `Reject` в `0.1.0` нет: `Accept` переводит задачу в `Done`, `Request changes` возвращает её в `In Progress`, а `Cancel` переводит в `Cancelled`, если задача больше не нужна.

Headless-сценарий должен использовать тот же слой модели проекта, когда Editor закрыт или команда явно запущена для CI:

1. `e2d project create` создаёт проект, локальный git-репозиторий, project-local `AGENTS.md`, starter skills и начальную доску встроенного `ProjectTaskManager`.
2. Template `.gitignore` не игнорирует `.electron2d/` целиком; он явно игнорирует generated/local-only каталоги `.electron2d/import-cache/`, `.electron2d/workspaces/`, `.electron2d/context/`, `.electron2d/session/` и `.electron2d/user/`, включая локальные breakpoints.
3. `e2d scene` создаёт и изменяет сцену в стабильном текстовом формате.
4. `e2d build`, `e2d validate` и `e2d test` возвращают структурированные результаты.
5. `e2d run` умеет выполнить фиксированное число кадров, применить input trace и сохранить diagnostics, screenshot, итоговое scene tree и performance данные.
6. `e2d export` собирает проект в headless-режиме.
7. `e2d api compare-godot` и локальная документация помогают не использовать неподдержанный Godot API.

# Публичная модель движка

API следует согласованной node/resource модели, но первая версия реализует только утверждённый 2D-поднабор.

Необходимо сразу зафиксировать:

> Публичный runtime API Electron2D `0.1.0` воспроизводит утверждённый 2D-профиль Godot `4.7-stable` .NET/C# API один к одному под namespace `Electron2D`. Для каждого типа, включённого в профиль, должны совпадать публичное наследование, свойства, методы, overloads, события, сигналы, enum, constants, типы параметров и возвращаемых значений, значения по умолчанию и наблюдаемое поведение. Типы и подсистемы вне 2D-профиля не входят в контракт Electron2D `0.1.0`. GDScript не поддерживается.

Формула релиза:

```text
Не 100% всего Godot API.
100% Godot C# API внутри утверждённого 2D-профиля.
```

Решение baseline от 2026-06-20: реализация `0.1.0 Preview` пишется заново после удаления старого каталога `src/Electron2D/`. Старый component-код, compatibility layer, `IComponent`, `SpriteRenderer`, legacy physics components и любые публичные API, которых нет в согласованном 2D-поднаборе, не переносятся в новую реализацию.

Wiki-материалы релиза должны готовиться как GitHub Wiki source, а не как локальный сайт внутри репозитория движка.

Нельзя добавлять пустые классы только ради похожего списка API. Каждый публичный класс из утверждённого 2D-профиля должен:

* быть функционально реализован;
* иметь документацию;
* иметь хотя бы один тест или пример;
* иметь подтверждённый паритет с Godot `4.7-stable` .NET/C# API.

Для Wiki нужна таблица совместимости:

| Статус       | Значение                           |
| ------------ | ---------------------------------- |
| Supported    | Реализовано и поддерживается       |
| Partial      | Реализована указанная часть API вне обязательного 2D-профиля |
| Experimental | API или поведение вне обязательного 2D-профиля может измениться |
| Planned      | Ещё не входит в пакет              |
| Not planned  | Осознанно не поддерживается        |

Для любого публичного API, включённого в обязательный 2D-профиль `0.1.0`, допустим только статус `Supported` с подтверждённым `Parity verified`. Статусы `Partial`, `Experimental` и `Planned` применимы только к API вне обязательного профиля или к будущим расширениям.

# Два профиля рендеринга

## Compatibility

Гарантированный минимальный набор для всех поддерживаемых платформ:

* спрайты;
* анимации;
* TileMap;
* UI;
* текст;
* примитивы;
* камера;
* clipping;
* стандартные blend modes;
* render targets, если backend это позволяет.

Не гарантируются:

* пользовательские шейдеры;
* постобработка;
* сложные материалы;
* GPU particles;
* идентичность нестандартного blending.

## Standard

Полный Standard backend:

* custom CanvasItem shaders;
* ShaderMaterial;
* render-to-texture;
* несколько проходов;
* расширенное смешивание;
* постобработка в пределах поддерживаемой первой версии.

Проект должен явно объявлять минимальный профиль:

```text
rendering/required_profile = compatibility
```

или:

```text
rendering/required_profile = standard
```

При запросе `standard` движок не должен молча отключать эффекты. На неподдерживаемом Android-устройстве он должен:

* показать понятную ошибку;
* записать причину в лог;
* завершить запуск либо применить явно разрешённый пользователем fallback.

## 4. Android fallback

Последовательность запуска:

1. Попытка создать графическое устройство с мобильным совместимым профилем.
2. Выполнить внутренний GPU smoke test.
3. Проверить создание texture, pipeline, command buffer и первый submit.
4. При ошибке активировать compatibility backend.
5. Передать игре фактический набор возможностей.

Должен существовать API вида:

```csharp
RenderingServer.CurrentProfile
RenderingServer.HasFeature(RenderingFeature feature)
```

или:

```csharp
RenderingDeviceFeatures.CustomShaders
RenderingDeviceFeatures.RenderTargets
RenderingDeviceFeatures.AnisotropicFiltering
```

Логирование должно включать:

* выбранный backend;
* GPU и драйвер;
* причины fallback;
* отключённые возможности.

## 6. Сериализация

Первая версия должна уметь сохранять:

* сцены;
* встроенные и внешние ресурсы;
* экспортированные свойства пользовательских классов;
* ссылки между объектами;
* массивы и словари;
* перечисления;
* nullable-значения;

Для инспектора и сериализации следует использовать source-generated metadata либо иной AOT-safe механизм.

Нельзя делать обязательными:

* `Reflection.Emit`;
* runtime-компиляцию;
* генерацию IL;
* динамическую загрузку пользовательского кода на iOS.

Обязательны тесты:

```text
load → modify → save → load
```

Результат должен сохранять значения и ссылки без повреждения данных.

## 7. C# scripting

Первая версия поддерживает только C#.

Игровой скрипт:

* наследуется от `Node` или его потомка;
* компилируется обычным .NET toolchain;
* может иметь экспортируемые свойства;
* может объявлять и принимать сигналы;
* может обращаться к сцене и сервисам движка;
* получает lifecycle callbacks.

Минимальные атрибуты:

```csharp
[Export]
[Signal]
[Tool]
```

Атрибуты должны оставаться marker API. Связь с serialization/Inspector должна идти через AOT-safe metadata bridge, без обязательного runtime assembly scan.

`[Tool]` может быть ограниченным и экспериментальным. Baseline от 2026-06-21 вводит внутренний execution host для явно зарегистрированной tool metadata: editor-time callbacks запускаются только явным tooling-вызовом, exceptions изолируются в result object, dynamic assembly load не используется, обычный runtime traversal остаётся отдельным путём.

Внешняя IDE не является обязательной частью workflow и не может закрывать обязательные требования `0.1.0`. `Electron2D.Editor` должен включать центральное workspace `Script` со встроенной базовой C# IDE: разработчик может создать, написать, исследовать, собрать и пошагово отладить игровой код без выхода из редактора.

Обязательный минимум редактирования:

* создание, открытие, редактирование, переименование и удаление `.cs` files;
* вкладки открытых документов;
* line numbers;
* C# syntax highlighting;
* автоматические отступы;
* настройка tabs/spaces;
* matching скобок и кавычек;
* code folding;
* выделение текущей строки;
* поиск и замена в файле;
* поиск по проекту;
* переход к строке;
* clipboard;
* Undo/Redo;
* сохранение одного файла и `Save All`;
* отображение dirty state;
* восстановление открытых вкладок после перезапуска Editor.

Ручной ввод в `.cs` buffer проходит через `TextBufferEditSession`, lightweight `CodeDocumentChanged` events и `TextBufferUndo`, а не через полноценную `WorkspaceTransactionEngine`-транзакцию на каждый символ. AI/refactoring/multi-file операции идут через `WorkspaceTransactionEngine` и `WorkspaceUndo`. Save выполняется как persistence transaction с atomic write; агентский `script_save` не сохраняет ручные unsaved changes, сделанные после базовой revision агента, без structured conflict или интерактивного подтверждения разработчика.

Обязательный минимум language services:

* project-aware autocomplete для типов, методов, свойств, events и локальных переменных;
* completion для API Electron2D;
* signature help с активным параметром;
* hover/Quick Info;
* XML documentation;
* live compiler diagnostics;
* подчёркивание ошибок и предупреждений;
* go to definition;
* find references;
* rename symbol;
* document formatting;
* basic code actions;
* добавление недостающего `using`;
* навигация по классам, методам и свойствам текущего файла;
* diagnostics panel с переходом к строке ошибки.

Semantic model должна учитывать текущий `.csproj`, references проекта, Electron2D assemblies, NuGet dependencies, conditional compilation symbols, unsaved C# buffers, source-generated API и фактическую build configuration.

Language services находятся в отдельном `Electron2D.CSharpLanguageServices` assembly, работающем внутри Editor process, не в игровом runtime и не в Editor UI. Отдельный language-service process не входит в обязательный scope `0.1.0`. Baseline реализации — Roslyn-backed live workspace, синхронизированный с `CodeDocument`; completion, hover, diagnostics и navigation используют `DocumentRevision`, `SemanticVersion` и `ConfigurationHash`, чтобы устаревшие ответы не применялись к новому buffer state. `WorkspaceSnapshot` используется для build/test/run/debug и пакетного анализа, а не для каждого IDE-запроса.

Обязательный минимум managed C# debugger для local desktop run на Windows, Linux и macOS:

* установка, включение, отключение и удаление breakpoint в gutter;
* сохранение breakpoints между Editor sessions;
* breakpoint identity через `BreakpointId`, `DocumentId` и `SourceAnchor`;
* rebase breakpoint anchor после text edits или перевод breakpoint в unverified state при неоднозначном переносе;
* локальное хранение breakpoints в `.electron2d/user/breakpoints.e2debug` без попадания в Git, `WorkspaceSnapshot`, runtime export или production package;
* запуск текущей сцены или проекта под debugger;
* attach к игровому процессу, запущенному Editor;
* pause, continue, stop, restart;
* step into, step over, step out;
* подсветка текущей исполняемой строки;
* call stack;
* выбор stack frame;
* threads;
* locals;
* arguments;
* watches;
* остановка на необработанном exception;
* переход из stack trace к исходному коду;
* Debug Output.

Runtime debug bridge и managed C# debugger — разные подсистемы. Runtime debug bridge инспектирует scene tree, node properties, input, screenshots, frame step и performance metrics. Managed C# debugger отвечает за source breakpoints, call stack, locals, watches, exceptions и step into/over/out.

Debug session запускается на конкретном immutable `WorkspaceSnapshot`: build с Portable PDB, game process, `SnapshotId`, source revisions, build configuration, symbols path и mapping snapshot source к открытому `CodeDocument`. Если пользователь или AI меняет код после старта, session помечается `stale`, Editor предлагает rebuild/restart и не сохраняет несохранённые изменения автоматически.

Managed debugger подключается через обязательную DAP-границу к packaged .NET debug adapter. Отклонение от DAP допускается только новым архитектурным решением, если P0 technical spike докажет отсутствие подходящего распространяемого adapter. Выбор adapter является отдельным P0 technical spike до полноценной реализации debugger и проверяет Tier 1 desktop platforms, Portable PDB, launch/attach, breakpoints, stepping, locals/watches, exceptions, restart, DAP capability matrix, лицензию и способ обновления adapter binaries. Agent `debug_attach` ограничен game process активной Editor play/debug session; произвольный attach к другому процессу отсутствует либо требует явного интерактивного подтверждения разработчика.

Tooling/MCP parity для `Script` workspace и debugger должен покрывать не только text edits и breakpoints, но и rename/delete/search script files, document symbols, code actions, debugger restart, breakpoint update, stacks всех threads, watch definitions, явное `debug_evaluate_watches(frameId)` и frame-scoped locals/arguments. Если Editor поддерживает эти действия интерактивно, AI получает эквивалентную семантическую команду без keyboard/mouse automation.

Hot Reload и Edit and Continue не являются обязательным критерием первой версии. Допустим быстрый rebuild и restart play/debug session после изменения кода.

# Electron2D.Editor

Godot 4 является каноническим UX- и layout-референсом для `Electron2D.Editor 0.1.0`. Electron2D.Editor воспроизводит знакомую структуру редактора Godot: верхнее меню и переключатель рабочих пространств, вкладки сцен, центральное рабочее пространство, `Scene` и `FileSystem` docks слева, `Inspector`/`Node` справа и инструментальную нижнюю панель. Трёхмерные рабочие пространства, инструменты, типы и настройки полностью исключены.

Любая задача `Electron2D.Editor`, которая создаёт или меняет видимый UI, имеет обязательный визуальный acceptance gate. Исполнитель должен открыть реальное окно редактора или documented automated harness, создать screenshot, сохранить его как артефакт проверки и проанализировать результат до закрытия задачи. Анализ должен подтверждать как минимум: элемент находится в ожидаемой области layout, доступен pointer/keyboard interaction там, где это применимо, текст не выходит за границы контейнера, активные/hover/focus states не ломают layout, UI соответствует `docs/specifications/editor/godot4-editor-reference.md`, а запрещённые `3D`, `AssetLib` и GDScript элементы не появились в видимом интерфейсе. Задача, изменяющая Editor UI, не считается готовой к приёмке, если есть только build, model test или smoke-команда без созданного и рассмотренного screenshot.

# Образцовые игры для приёмки

Первая версия должна поставляться минимум с двумя законченными примерами.

Каждая reference game является полноценным валидным проектом `Electron2D.Editor`, а не отдельным demo folder, test fixture или набором runtime files. Приёмка каждой игры должна подтверждать, что проект создан или мигрирован через тот же project format, который открывает Editor: есть валидные `project.e2d.json`, `.csproj`, main scene, project settings, Input Map, export presets, импортируемые resources, scripts, `.electron2d/tasks/**` project-task metadata и пользовательские editor metadata там, где они нужны workflow. Проект должен открываться через Project Manager, отображать сцены и ресурсы в Editor, сохраняться и повторно открываться без ручного исправления служебных файлов, проходить `e2d project validate`/Editor validation, запускаться из Editor и headless CLI, экспортироваться через обычные presets и не зависеть от ignored локальных workflow files вроде `TASKS.md`, `dev-diary/` или `completed-tasks/`.

## 1. Небольшой платформер

Проверяет:

* TileMap;
* CharacterBody2D;
* one-way platforms;
* Camera2D;
* AnimatedSprite2D;
* AnimationPlayer;
* звук;
* gamepad;
* touch controls;
* pause menu;
* сохранение прогресса.

## 2. UI-heavy puzzle или карточная игра

Проверяет:

* Control;
* containers;
* кнопки;
* списки;
* локализацию;
* разные разрешения;
* touch;
* текст;
* сохранение;
* scene transitions;
* Android compatibility renderer.

Оба проекта должны собираться из одной кодовой базы под все пять платформ. Допускаются только platform-specific настройки, иконки, signing и storefront metadata.

# Формальные критерии готовности

## Функциональность

Версия готова, когда сторонний разработчик может:

* установить Electron2D;
* создать новый проект;
* собрать сцену без ручного редактирования служебных файлов;
* написать C#-логику;
* запустить игру из редактора;
* сохранить и повторно открыть проект;
* экспортировать его на все заявленные платформы.

## Редактор

* Electron2D.Editor полностью работает на Electron2D.
* Нет обязательной зависимости от внешнего UI framework.
* Desktop executable `Electron2D.Editor` использует брендовую `.ico`-иконку из `data/assets/branding/`.
* Default layout соответствует `docs/specifications/editor/godot4-editor-reference.md`: `2D`, `Script`, `Game`, `Tasks`; `Scene`/`FileSystem` слева; `Inspector`/`Node` и `Agent Workspace` справа; bottom panel снизу.
* В Editor отсутствуют workspace `3D`, `AssetLib`, GDScript UI и disabled 3D controls.
* `Script` workspace позволяет редактировать C# code, получать IntelliSense, ставить breakpoints, запускать local desktop debug session и смотреть stacks всех threads, frame-scoped locals/arguments и watch values без внешней IDE.
* Нет известных ошибок, приводящих к повреждению сцен или ресурсов.
* Undo/redo работает для основных операций.
* Editor выдерживает многократный цикл run/stop.

## Платформы

Для каждой Tier 1 платформы:

* CI собирает runtime;
* reference game запускается;
* production package contents не содержит `.electron2d/tasks/**` и другие `EditorMetadata`;
* выполнен тест на реальном устройстве либо машине;
* проверены input, audio, rendering и filesystem;
* выполнен минимум 30-минутный soak test;
* нет монотонного роста native или managed memory;
* Android/iOS выдерживают многократный background/foreground cycle.

## Release metadata

* README, NuGet package metadata и Editor executable используют брендовые ассеты из `data/assets/branding/`.

## API

* нет публичных заглушек;
* все публичные типы имеют XML-документацию;
* все публичные типы отражены в Wiki;
* каждый тип из утверждённого 2D-профиля имеет статус `Supported` и `Parity verified`;
* `Partial`, `Experimental` и `Planned` допустимы только для API вне обязательного 2D-профиля `0.1.0`;
* `e2d api compare-godot` для типов профиля возвращает `missing types = 0`, `missing members = 0`, `signature mismatches = 0`, `inheritance mismatches = 0`, `default mismatches = 0`, `unexpected changes = 0`;
* breaking changes допустимы в `0.x`, но фиксируются в changelog.

## Производительность

Минимальный критерий:

* reference games работают в 60 FPS на опубликованном наборе эталонных устройств;
* пустая сцена и типовая sprite-сцена не создают постоянных managed allocations каждый кадр после прогрева;
* batching действительно уменьшает количество draw calls;
* многократная загрузка и выгрузка сцен не приводит к утечкам GPU, audio и physics ресурсов;
* производительность редактора остаётся приемлемой на сцене примерно из 1 000 узлов.

## Стабильность данных

* 100 последовательных циклов save/load не повреждают тестовую сцену;
* переименование и перемещение ресурсов не ломают UID-ссылки;
* импорт можно полностью пересоздать из исходных файлов;
* формат проекта и импортированный cache разделены;
* ошибки сериализации не приводят к молчаливой потере свойств.

# Краткое определение релиза

**Electron2D 0.1.0 Preview — это C#-движок и desktop-редактор, позволяющие создать небольшую законченную 2D-игру с node/resource архитектурой и экспортировать один проект под Windows, Linux, macOS, Android и iOS.**

Главный критерий первой версии — не количество реализованных классов, а наличие полностью рабочего пути:

```text
Project → Scene → C# code → Run → Debug → Export → Play
```

Версию `1.0` следует выпускать позднее: после стабилизации публичного API, нескольких реальных проектов, сформированной migration policy и подтверждённой эксплуатации Android/iOS в production.

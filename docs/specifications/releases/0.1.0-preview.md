# Electron2D 0.1.0 Preview

Статус: целевая спецификация релиза.
Источник: перенесено из корневого `GOAL-0.1.0.md`.
Последнее обновление: 2026-06-21.
Связанные документы: [Архитектура и платформенный стек Electron2D](../architecture/engine-platform-stack.md); [AI-friendly workflow Electron2D 0.1](../architecture/ai-friendly-workflow.md).

Первая версия — не «аналог Godot со всеми классами», а **первый законченный вертикальный срез движка**.

## Цель

Разработчик на C# должен суметь:

> создать проект, собрать сцену в редакторе, написать игровую логику, импортировать ресурсы, запустить игру и экспортировать один проект под Windows, Linux, macOS, Android и iOS без переписывания игрового кода.

Electron2D также должен быть AI-friendly движком:

> человек или AI-агент должен суметь создать небольшую законченную 2D-игру на C#, используя редактор либо полностью автоматизированный рабочий процесс, а затем запустить, протестировать и экспортировать проект без управления редактором мышью.

AI-friendly в `0.1.0` означает не встроенный чат и не привязку к конкретной LLM, а архитектуру, где важные операции доступны через общий tooling layer, CLI, локальный MCP-сервер, текстовые project files, structured diagnostics, API manifest, compatibility map и автоматизированный runtime capture.

Версия считается успешной, когда Electron2D подходит для создания и выпуска небольшой законченной 2D-игры, а не только для демонстрации отдельных подсистем.

Целевая аудитория первой версии:

* C#-разработчики;
* одиночные разработчики и небольшие команды;
* программисты, которым нужен node/resource API в полностью управляемой .NET-экосистеме;
* проекты малого и среднего масштаба;
* AI-агенты, CI и IDE-интеграции, работающие через публичный tooling API без GUI automation.

Редактор предназначен для desktop. Android и iOS являются платформами запуска и экспорта, но не редактирования проектов.

# Что можно сделать на первой версии

Electron2D 0.1 должен позволять создавать:

* простые платформеры;
* top-down игры;
* аркады;
* головоломки;
* карточные и настольные игры;
* визуальные новеллы;
* UI-heavy приложения;
* небольшие RPG без сложной навигации и освещения;
* локальные игры с клавиатурой, мышью, геймпадом или touch-вводом.

Первая версия не обязана поддерживать крупные открытые миры, сложное динамическое освещение, rollback multiplayer или тысячи физических объектов.

# Основной пользовательский сценарий

Полный рабочий цикл должен выглядеть так:

1. Разработчик создаёт проект Electron2D.
2. Открывает его в `Electron2D.Editor`.
3. Создаёт сцену.
4. Добавляет узлы.
5. Импортирует текстуры, шрифты и звук.
6. Настраивает свойства через Inspector.
7. Создаёт C#-скрипт и прикрепляет его к узлу.
8. Настраивает Input Map.
9. Запускает текущую сцену или весь проект.
10. Исправляет ошибки по сообщениям редактора.
11. Создаёт export preset.
12. Получает запускаемую сборку для выбранной платформы.

Этот сценарий должен работать на чистой машине после установки документированных зависимостей.

Автоматизированный сценарий должен использовать тот же проектный model layer:

1. `e2d project create` создаёт проект, локальный git-репозиторий, project-local `AGENTS.md` и starter skills.
2. `e2d scene` создаёт и изменяет сцену в стабильном текстовом формате.
3. `e2d build`, `e2d validate` и `e2d test` возвращают структурированные результаты.
4. `e2d run` умеет выполнить фиксированное число кадров, применить input trace и сохранить diagnostics, screenshot, итоговое scene tree и performance данные.
5. `e2d export` собирает проект в headless-режиме.
6. `e2d api compare-godot` и локальная документация помогают не использовать неподдержанный Godot API.

# Публичная модель движка

API следует согласованной node/resource модели, но первая версия реализует только 2D-поднабор.

Необходимо сразу зафиксировать:

> Electron2D фиксирует собственный C# API-контракт для 2D-поднабора и не гарантирует исходную совместимость с проектами, написанными для других движков или языков сценариев.

Решение baseline от 2026-06-20: реализация `0.1.0 Preview` пишется заново после удаления старого каталога `src/Electron2D/`. Старый component-код, compatibility layer, `IComponent`, `SpriteRenderer`, legacy physics components и любые публичные API, которых нет в согласованном 2D-поднаборе, не переносятся в новую реализацию.

Wiki-материалы релиза должны готовиться как GitHub Wiki source, а не как локальный сайт внутри репозитория движка.

Нельзя добавлять пустые классы только ради похожего списка API. Каждый публичный класс должен:

* быть функционально реализован;
* иметь документацию;
* иметь хотя бы один тест или пример;
* иметь явно описанные отличия от Godot.

Для Wiki нужна таблица совместимости:

| Статус       | Значение                           |
| ------------ | ---------------------------------- |
| Supported    | Реализовано и поддерживается       |
| Partial      | Реализована указанная часть API    |
| Experimental | API или поведение может измениться |
| Planned      | Ещё не входит в пакет              |
| Not planned  | Осознанно не поддерживается        |

# Два профиля рендеринга

## Compatibility

Гарантированный минимальный набор для всех поддерживаемых платформ:

* спрайты;
* анимации;
* TileMap;
* UI;
* текст;
* примитивы;
* камера;
* clipping;
* стандартные blend modes;
* render targets, если backend это позволяет.

Не гарантируются:

* пользовательские шейдеры;
* постобработка;
* сложные материалы;
* GPU particles;
* идентичность нестандартного blending.

## Standard

Полный Standard backend:

* custom CanvasItem shaders;
* ShaderMaterial;
* render-to-texture;
* несколько проходов;
* расширенное смешивание;
* постобработка в пределах поддерживаемой первой версии.

Проект должен явно объявлять минимальный профиль:

```text
rendering/required_profile = compatibility
```

или:

```text
rendering/required_profile = standard
```

При запросе `standard` движок не должен молча отключать эффекты. На неподдерживаемом Android-устройстве он должен:

* показать понятную ошибку;
* записать причину в лог;
* завершить запуск либо применить явно разрешённый пользователем fallback.

## 4. Android fallback

Последовательность запуска:

1. Попытка создать графическое устройство с мобильным совместимым профилем.
2. Выполнить внутренний GPU smoke test.
3. Проверить создание texture, pipeline, command buffer и первый submit.
4. При ошибке активировать compatibility backend.
5. Передать игре фактический набор возможностей.

Должен существовать API вида:

```csharp
RenderingServer.CurrentProfile
RenderingServer.HasFeature(RenderingFeature feature)
```

или:

```csharp
RenderingDeviceFeatures.CustomShaders
RenderingDeviceFeatures.RenderTargets
RenderingDeviceFeatures.AnisotropicFiltering
```

Логирование должно включать:

* выбранный backend;
* GPU и драйвер;
* причины fallback;
* отключённые возможности.

## 7. C# scripting

Первая версия поддерживает только C#.

Игровой скрипт:

* наследуется от `Node` или его потомка;
* компилируется обычным .NET toolchain;
* может иметь экспортируемые свойства;
* может объявлять и принимать сигналы;
* может обращаться к сцене и сервисам движка;
* получает lifecycle callbacks.

Минимальные атрибуты:

```csharp
[Export]
[Signal]
[Tool]
```

Атрибуты должны оставаться marker API. Связь с serialization/Inspector должна идти через AOT-safe metadata bridge, без обязательного runtime assembly scan.

`[Tool]` может быть ограниченным и экспериментальным. Baseline от 2026-06-21 вводит внутренний execution host для явно зарегистрированной tool metadata: editor-time callbacks запускаются только явным tooling-вызовом, exceptions изолируются в result object, dynamic assembly load не используется, обычный runtime traversal остаётся отдельным путём.

Разрешается использовать внешний IDE. Встроенный полноценный редактор C# в первую версию не входит.

Минимальная интеграция:

* создать скрипт;
* прикрепить скрипт к узлу;
* открыть его во внешнем IDE;
* собрать проект;
* показать compiler errors;
* перезапустить игровую сцену после сборки.

Hot Reload не является обязательным критерием первой версии.

## 9. Физика 2D

Первая версия должна поддерживать базовую физическую игру, а не только AABB-проверки.

Минимальные узлы:

```text
StaticBody2D
RigidBody2D
CharacterBody2D
Area2D
CollisionShape2D
RayCast2D
```

Формы:

```text
RectangleShape2D
CircleShape2D
CapsuleShape2D
SegmentShape2D
ConvexPolygonShape2D
ConcavePolygonShape2D
```

`ConcavePolygonShape2D` разрешается использовать только для статической геометрии.

Обязательные возможности:

* collision layers и masks;
* friction и restitution;
* gravity;
* sleeping;
* sensors/areas;
* body entered/exited;
* area entered/exited;
* raycast;
* point query;
* shape query;
* fixed physics timestep;
* one-way platforms;
* базовое continuous collision detection для быстрых тел;
* корректное удаление тел во время physics step через deferred queue.

`CharacterBody2D` должен иметь практически применимое поведение:

```text
MoveAndCollide
MoveAndSlide
IsOnFloor
IsOnWall
IsOnCeiling
FloorNormal
FloorSnapLength
UpDirection
```

Не требуется:

* битовая детерминированность между платформами;
* rollback physics;
* полный набор joints;
* soft bodies;
* fluid simulation.

Физический backend не должен просачиваться в публичные классы. Пользовательские типы не должны содержать Box2D handles.

Начальный baseline от 2026-06-21: `PhysicsServer2D` уже предоставляет public `Rid`-границу для space/body/area/joint/shape creation, `SpaceParameter`, `ShapeType`, `ProcessInfo`, `FreeRid()` и internal swappable backend. `StaticBody2D`, `RigidBody2D`, `CharacterBody2D`, `Area2D`, `CollisionShape2D`, `RayCast2D`, `Shape2D`, `RectangleShape2D`, `CircleShape2D`, `CapsuleShape2D`, `SegmentShape2D`, `ConvexPolygonShape2D`, `ConcavePolygonShape2D`, `PhysicsMaterial`, `World2D`, `PhysicsDirectSpaceState2D` и query parameter resources уже предоставляют node/resource API surface, RID lifecycle, transform sync, collision filter helpers, material/body-state snapshot, validation, AOT-safe serialization metadata, `Area2D` overlap signals/helper methods, AABB ray/point/shape query baseline, fixed physics tick `1/60`, начальное движение `RigidBody2D` по `LinearVelocity`, basic AABB sweep для быстрых тел, one-way platform checks, `CharacterBody2D.MoveAndCollide()`, `CharacterBody2D.MoveAndSlide()`, floor/wall/ceiling state, floor snap, platform velocity baseline, debug collision shape hooks и безопасное удаление через `QueueFree()` во время physics traversal. Полноценная симуляция столкновений, запись geometry/material в production backend, gravity integration, точный narrow-phase и shape-level overlap signals остаются следующими задачами физического домена.

Box2D.NET candidate validation от 2026-06-21: отдельный smoke-проект `tests/Electron2D.Tests.PhysicsBox2DSmoke` и verifier `tools/Verify-Box2DPhysicsCandidate.ps1` проверяют `Box2D.NET 3.1.654` в Release/JIT и NativeAOT, измеряют allocations per tick и не добавляют Box2D.NET в public runtime dependency graph. Локальный `win-x64` результат: Release/JIT `AllocatedBytesPerTick=464.633`, NativeAOT `AllocatedBytesPerTick=354.333`. Android arm64 Release/AOT и iOS arm64 Release/AOT остаются явными mobile gaps до задач export/toolchain.

# Формальные критерии готовности

## Функциональность

Версия готова, когда сторонний разработчик может:

* установить Electron2D;
* создать новый проект;
* собрать сцену без ручного редактирования служебных файлов;
* написать C#-логику;
* запустить игру из редактора;
* сохранить и повторно открыть проект;
* экспортировать его на все заявленные платформы.

## Редактор

* Electron2D.Editor полностью работает на Electron2D.
* Нет обязательной зависимости от внешнего UI framework.
* Нет известных ошибок, приводящих к повреждению сцен или ресурсов.
* Undo/redo работает для основных операций.
* Editor выдерживает многократный цикл run/stop.

## Платформы

Для каждой Tier 1 платформы:

* CI собирает runtime;
* reference game запускается;
* выполнен тест на реальном устройстве либо машине;
* проверены input, audio, rendering и filesystem;
* выполнен минимум 30-минутный soak test;
* нет монотонного роста native или managed memory;
* Android/iOS выдерживают многократный background/foreground cycle.

## API

* нет публичных заглушек;
* все публичные типы имеют XML-документацию;
* все публичные типы отражены в Wiki;
* для частично совместимых Godot API описаны отличия;
* breaking changes допустимы в `0.x`, но фиксируются в changelog.

## Производительность

Минимальный критерий:

* reference games работают в 60 FPS на опубликованном наборе эталонных устройств;
* пустая сцена и типовая sprite-сцена не создают постоянных managed allocations каждый кадр после прогрева;
* batching действительно уменьшает количество draw calls;
* многократная загрузка и выгрузка сцен не приводит к утечкам GPU, audio и physics ресурсов;
* производительность редактора остаётся приемлемой на сцене примерно из 1 000 узлов.

## Стабильность данных

* 100 последовательных циклов save/load не повреждают тестовую сцену;
* переименование и перемещение ресурсов не ломают UID-ссылки;
* импорт можно полностью пересоздать из исходных файлов;
* формат проекта и импортированный cache разделены;
* ошибки сериализации не приводят к молчаливой потере свойств.

## Документация

Обязательны:

* установка;
* первый проект;
* первая сцена;
* scripting;
* ресурсы;
* физика;
* UI;
* анимация;
* Input Map;
* экспорт для каждой платформы;
* renderer profiles;
* Android fallback;
* различия Electron2D и внешних API;
* API reference;
* troubleshooting.

# Краткое определение релиза

**Electron2D 0.1.0 Preview — это C#-движок и desktop-редактор, позволяющие создать небольшую законченную 2D-игру с node/resource архитектурой и экспортировать один проект под Windows, Linux, macOS, Android и iOS.**

Главный критерий первой версии — не количество реализованных классов, а наличие полностью рабочего пути:

```text
Project → Scene → C# code → Run → Debug → Export → Play
```

Версию `1.0` следует выпускать позднее: после стабилизации публичного API, нескольких реальных проектов, сформированной migration policy и подтверждённой эксплуатации Android/iOS в production.

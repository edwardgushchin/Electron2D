# Политика окончаний строк

Обновлено: 2026-06-28.

Этот файл является доменным документом для репозиторной политики окончаний строк. Он фиксирует ожидаемый контракт, текущее состояние, исключения и проверки.

## Назначение

Репозиторий хранит обычные текстовые файлы с окончанием строки LF. Это делает patch, аудиторские ZIP-архивы и восстановление на отдельной машине независимыми от локальной настройки Git `core.autocrlf`.

## Контракт

`.gitattributes` задаёт общий базовый режим:

```text
* text=auto eol=lf
```

Текстовые файлы, включая Markdown, C#, JSON, XML, `.tmx`, `.tsx`, `.csproj`, `.sln`, скрипты и данные документации, должны попадать в рабочее дерево с LF. Файлы `.bat` и `.cmd` остаются явным исключением с CRLF, потому что они предназначены для запуска штатным обработчиком команд Windows. Исторические verdict-документы в `docs/verdicts/` временно разрешены в рабочем дереве с CRLF: текущая задача не имеет права менять этот архив, поэтому проверка считает такие строки известным ограничением, а не неожиданным нарушением. Остальные исключения должны быть настоящими двоичными форматами: изображения, шрифты, архивы, SQLite-базы, звуковые файлы и другие файлы, где преобразование строк может повредить байты.

Каталог `data/assets/reference-games/` не является двоичным целиком. Его XML-файлы `.tmx` и `.tsx` считаются текстом и должны нормализоваться в LF, а изображения и другие реальные двоичные ресурсы остаются двоичными по расширениям.

## Нормализация

Одноразовая нормализация отслеживаемых текстовых файлов допускается только как байтовое приведение CRLF или одиночного CR к LF без смыслового изменения содержимого. После нормализации не должно оставаться подготовленных к коммиту изменений в индексе Git, если пользователь явно не попросил staging.

## Проверка

Команда:

```bash
dotnet run --project eng/Electron2D.Build -- verify line-endings
```

анализирует `git ls-files --eol` и отклоняет неожиданные `w/crlf` у текстовых файлов. Минимальный список разрешённых CRLF-исключений хранится в реализации проверки: `.bat`, `.cmd` и временный исторический архив `docs/verdicts/`. Если список расширяется, это должно быть обосновано здесь и покрыто focused tests.

Ручная диагностика перед внешним аудитом:

```bash
git ls-files --eol | rg "w/crlf"
```

должна показывать только разрешённые исключения или не показывать строк вообще.

## Связь с аудиторским архивом

`audit package verify` готовит отдельную чистую копию самостоятельно: выставляет локальные `core.autocrlf=false` и `core.eol=lf`, проверяет доступность исходной ревизии `--baseline`, возвращает рабочую копию к этой ревизии, удаляет только отслеживаемые Git файлы (`tracked files`) из списка `git ls-files -z`, заново создаёт их через `git reset --hard <baseline>`, очищает игнорируемые результаты сборки и затем проверяет `git status --porcelain`. После применения patch команда ещё раз удаляет Git-игнорируемые результаты сборки, которые не входят в `repo-file-hashes.json`, перед сравнением восстановленного набора файлов; Git-игнорируемые файлы, явно включённые в модель восстановления через `repoFileAllowlist`, остаются на месте. Поэтому результат проверки восстановления не должен зависеть от настроек Git оператора, от ранее созданных CRLF-байтов в чистой копии или от случайных `obj/`, `bin/` и других игнорируемых файлов.

# Запрос внешнего аудита Electron2D

Вы проводите внешний аудит реализации Electron2D по одному приложенному audit ZIP. Ваша задача — проверить не только упаковку, а само изменение: код, тесты, документацию, соответствие задаче, закрытие прошлых замечаний и оставшиеся риски. Не останавливайтесь на первой найденной проблеме. Нужен полный список доказуемых проблем в пределах текущей области задачи.

Каждый запуск аудита — это полная инженерная проверка текущей области задачи. Это относится к первой итерации, повторным итерациям после `NEEDS_FIXES` и независимому контрольному аудиту после первого `ACCEPT`. Повторная итерация дополнительно проверяет прошлые отчёты и список закрытия прошлых замечаний, но эта проверка не заменяет чтение кода, тестов и документации. Контрольный аудит проверяет чистый контрольный ZIP для той же принятой области независимо, без доверия к предыдущему `ACCEPT` и без прошлых verdict-отчётов в пакете.

Технические имена этого правила: `full current-scope engineering review`, `primary audit`, `control audit`, `metadata.previousVerdictChain`, `metadata.blockerClosureList`.

## Какие материалы использовать

Используйте только файлы основного архива. Не додумывайте отсутствующие материалы и не используйте скрытый контекст.

- `AUDIT-MANIFEST.md` — инвентарь задачи, итерации, исходной ревизии, области, изменённых файлов и доказательств.
- `metadata/audit-package.input.json` — входная конфигурация архива, включая `metadata.scopeTaskIds`, `metadata.scopeSummary`, `metadata.previousVerdictChain` и `metadata.blockerClosureList`.
- `repo-file-hashes.json` — список файлов репозитория, которые относятся к проверяемому изменению.
- `metadata/repo-file-snapshots.json` — индекс полных снимков файлов репозитория.
- `repo-after/` — полные итоговые версии изменённых файлов; это главный материал для чтения кода, тестов и документации.
- `repo-before/` — полные версии файлов из исходной ревизии, если файл существовал до изменения.
- `<task-id>.patch` — карта изменённых мест. Patch помогает найти изменения, но не заменяет чтение полных файлов.
- `evidence/` — сырые доказательства настроенных проверок, если они есть.
- Изменённые файлы репозитория, тесты, документация и прошлые verdict-файлы, если они доступны в архиве.

Подготовка архива выполняется принимающей стороной до отправки. Внешний аудит проверяет содержимое изменения, а не транспорт, не второй служебный архив, не историю отправки и не применимость patch. Если таких материалов нет во входе, это не является проблемой текущего аудита.

Если основной ZIP читается, содержит `metadata/repo-file-snapshots.json` и даёт доступ к `repo-after/`, основная часть аудита — чтение и оценка содержимого по полным файлам. Оценивайте только то, подтверждает ли изменение заявленную задачу и не создаёт ли новых рисков.

## Обязательная проверка содержимого

1. **Проверка реализации.** Прочитайте изменённый код в `repo-after/` по полным итоговым версиям файлов. Оцените, реализует ли он заявленный контракт задачи без скрытых ручных действий. Patch используйте только как карту изменённых мест.
2. **Проверка тестов.** Проверьте по `repo-after/`, что тесты покрывают важные ветки поведения, ограничения и уже найденные проблемы.
3. **Проверка документации.** Проверьте по `repo-after/`, что документация и агентские правила, если они есть в архиве, соответствуют фактическому поведению инструмента.
4. **Проверка соответствия задаче.** Сопоставьте изменение с текущим запросом пользователя и областью задачи.
5. **Проверка секретов и локальных данных.** Проверьте код, patch и доказательства на реальные секреты, приватные ключи, токены, пароли, локальные абсолютные пути и конфиденциальные данные.
6. **Проверка лишних правок.** Проверьте, что изменение не включает файлы и поведение вне заявленной области.
7. **Проверка полноты снимков.** Если `metadata/repo-file-snapshots.json` показывает отсутствующий или неполный снимок важного файла реализации, теста или документации, считайте это недостатком доказательств и блокирующей проблемой, когда без этого файла нельзя проверить задачу. Не называйте чтение patch-а полноценным чтением файла.

Технические имена этих проверок: `implementation content review`, `test coverage review`, `documentation review`, `task compliance review`, `secret scanning`, `scope scanning`, `evidence gap`, `patch-only inspection`, `full file review`.

## Обязательные инженерные критерии

1. **Производительность.** Для игрового движка код в горячем пути выполнения должен быть быстрым. Если текущее изменение ухудшает игровой цикл, отрисовку, ввод, жизненный цикл узлов, загрузку ресурсов, физику, запуск или экспорт, это может блокировать задачу. Заявление о росте производительности без измерения или другого проверяемого доказательства считайте недостатком доказательств. Старый долг производительности вне текущей области оформляйте как отдельное замечание, если он не мешает принять эту задачу.
2. **Совместимость публичного API с выбранным подмножеством Godot 4.7.** Для задач, которые меняют Public API, проверяйте только утверждённый 2D-профиль текущей задачи. Требуйте проверяемую опору: зафиксированную ссылку, снимок, таблицу соответствия или другой материал из пакета. Не расширяйте проверку до всего Godot, если это не входит в область задачи.
3. **Полноценность Public API.** Публичный API не должен быть витриной. Метод, класс или свойство должны иметь рабочий путь выполнения, наблюдаемое поведение, документацию и тест или пример. Если API похож на Godot только формой, но не работает через реальный механизм движка, это блокирующая проблема для Public API задачи.
4. **Реалистичность тестов.** Тесты должны проверять реальное поведение через производственный код или через стабильный внутренний контракт, который действительно используется средой выполнения, редактором или инструментами. Проверка «метод не падает», искусственная фикстура или ветка, добавленная только ради теста, не доказывают задачу.
5. **Качество C#-кода.** Проверяйте правила `AGENTS.md`: понятные имена, полноценную XML-документацию публичного API, лицензионные заголовки, корректное владение ресурсами, nullable/exception-контракты, отсутствие лишней публичной поверхности, placeholder-комментариев, ненужной рефлексии, строковых протоколов вместо типизированной модели и обходов только для тестов.
6. **Согласованность с архитектурой Electron2D.** Изменение не должно создавать параллельный механизм там, где должен использоваться общий путь среды выполнения, редактора, обработки ввода, отрисовки, звука, импорта, сборки или экспорта. Решение должно встраиваться в идею Electron2D как рабочего движка для небольшой законченной 2D-игры.

Технические имена инженерных критериев: `performance`, `Public API`, `Godot 4.7`, `backend path`, `observable behavior`, `test-only branch`, `fixture hook`, `fake backend`, `C# style/best-practices`, `architecture coherence`.

## Проверка области пакета

1. Прочитайте `metadata.scopeTaskIds` и `metadata.scopeSummary` из `metadata/audit-package.input.json`.
2. Если `metadata.scopeTaskIds` пуст или отсутствует, область пакета равна одиночной задаче из metadata.
3. Если `metadata.scopeTaskIds` содержит несколько задач, проверяйте пакет как явно объединённую область. Manifest, metadata, patch, `repo-file-hashes.json`, `metadata/repo-file-snapshots.json`, `repo-after/`, `repo-before/` и evidence должны согласованно объяснять, какие задачи входят в пакет и почему их можно принимать одним verdict-ом.
4. Проверьте, что `AUDIT-MANIFEST.md` перечисляет ту же область, а `metadata.scopeSummary` не противоречит фактическим изменениям.
5. Если изменение содержит правки вне `metadata.scopeTaskIds` или summary, это блокирующая проблема области задачи, даже если отдельные проверки прошли.

Техническое имя явно объединённой области: `combined scope`.

## Проверка прошлых verdict-отчётов

1. Прочитайте `metadata.previousVerdictChain` из `metadata/audit-package.input.json`.
2. Для каждого существующего пути из `metadata.previousVerdictChain` найдите соответствующие прошлые verdict-файлы, если они доступны в архиве или изменении.
3. Проверьте, что прошлые verdict-файлы не были переписаны, сокращены или переоформлены текущим изменением.
4. Прочитайте каждый доступный прошлый verdict-файл и выпишите прошлые блокирующие проблемы.
5. Прочитайте `metadata.blockerClosureList` и проверьте, что каждая прошлая блокирующая проблема имеет проверяемое закрытие в текущих изменениях, тестах, документации или доказательствах. Общая фраза «всё исправлено» не считается закрытием.
6. Если прошлый verdict-файл указан в `metadata.previousVerdictChain`, но отсутствует во входе, проверьте, не скрывает ли это текущую проблему. Само отсутствие старого файла не является проблемой без доказуемого влияния на текущую проверку.

Технические имена этой проверки: `previous verdict files`, `verbatim preservation`, `previous blockers closure`.

## Классы замечаний

Используйте эти классы, но в человекочитаемой части объясняйте их по-русски.

- `current-task blocker` — блокирующая проблема текущей задачи. Она запрещает `VERDICT: ACCEPT`. В отчёте оформляйте её в `BLOCKERS` как `B1`, `B2`, `B3` и далее.
- `follow-up finding` — инженерная проблема или долг, найденный во время проверки, но не блокирующий текущую задачу. В отчёте оформляйте её в `RISKS_AND_NOTES` как `FOLLOW_UP_FINDING F1`, `FOLLOW_UP_FINDING F2` и далее.
- `out-of-scope/info note` — заметка вне текущей области или нейтральная информация. В отчёте оформляйте как `OUT_OF_SCOPE_NOTE N1` или `INFO_NOTE I1`.
- `accepted risk` — явно принятый риск текущего изменения. В отчёте оформляйте как `ACCEPTED_RISK R1` с причиной принятия и моментом следующего пересмотра.
- `global safety blocker` — проблема безопасности, секреты, разрушительные действия, подмена verdict-а или обход аудиторского процесса. Это не заметка и не принятый риск, а блокирующая проблема текущего аудита.
- `unsupported concern` — подозрение, которое не имеет достаточного доказательства в текущем пакете. Оно не запрещает `VERDICT: ACCEPT`. Оформляйте его как `INFO_NOTE` с объяснением, каких доказательств не хватает.

## Правило доказанного blocker-а

Правильный итог аудита может быть `VERDICT: ACCEPT`. Не создавайте блокирующую проблему только потому, что можно представить улучшение, альтернативный дизайн или гипотетический риск.

Запись в `BLOCKERS` допустима только если одновременно выполнены все условия:

1. проблема нарушает явный критерий текущей области аудита;
2. доказательство находится в текущем пакете, а не в догадке о возможном поведении;
3. можно указать точный файл, секцию, символ, команду, артефакт или проверенный фрагмент evidence;
4. влияние делает приёмку текущей задачи небезопасной или некорректной;
5. проблема не закрыта существующим тестом, документированным ограничением, принятым риском или `out-of-scope` решением.

Перед финальным `VERDICT: NEEDS_FIXES` попытайтесь опровергнуть каждый найденный blocker по материалам пакета: проверьте тесты, документацию, metadata, evidence и ограничения текущей области. Если blocker выжил после такой проверки, добавьте в его описание короткую строку `Проверка опровержения`: что именно было проверено и почему это не сняло blocker. Если хотя бы одно из пяти условий выше не доказано, это `unsupported concern`, `FOLLOW_UP_FINDING`, `OUT_OF_SCOPE_NOTE`, `INFO_NOTE` или `ACCEPTED_RISK`, но не blocker.

## Читаемость отчёта

Отчёт должен быть понятен человеку, который принимает задачу. Не пишите объяснения как внутренний машинный протокол.

1. В секциях `TASK_ASSESSMENT`, `BLOCKERS`, `EVIDENCE_REVIEW`, `RISKS_AND_NOTES` и `CLOSURE_DECISION` сначала пишите короткое русское объяснение. Служебные поля, точные имена файлов, команд, metadata-полей и маркеров указывайте после объяснения в отдельном блоке `Техническая привязка`.
2. Не используйте англоязычные обороты в объяснении, если есть нормальная русская фраза. Пишите «способ закрытия замечаний, который можно проверить автоматически», а не `machine-verifiable closure surface`; «штатная команда без ручных обходов», а не `clean command path`; «проверка, что указанная связанная задача действительно существует и подходит по смыслу», а не `semantic existence check target task`; «долг, который не мешает принять эту задачу», а не `non-blocking debt`; «критерий приёмки», а не `acceptance criterion`.
3. Для каждой блокирующей проблемы, последующего замечания, принятого риска и решения о закрытии объясните простыми русскими предложениями: что не так, почему это важно для текущей задачи, что нужно сделать дальше.
4. Технические поля разрешены только после человеческого объяснения. Например: `File/symbol`, `Criterion`, `Evidence`, `Impact`, `Fix`, `Verification`, `metadata.*`, `B1`, `F1`.
5. Видимая подпись причины должна быть русской: `Почему не блокирует текущую задачу`. Если инструментам нужен английский alias `Why not blocker for current task`, укажите его только в технической привязке, а не вместо объяснения.

Пример нормального объяснения:

```text
Почему это не блокирует текущую задачу:
Это не блокирует T-0238. В текущей задаче нужно доказать, что замечания из прошлых аудитов закрываются проверяемым способом, а контрольный аудит запускается без старого контекста. Проверка того, что связанная задача существует и совпадает по смыслу, относится к отдельной доработке. В прошлых отчётах её уже отметили как долг, который не мешает принять T-0238.

Техническая привязка:
- Идентификатор замечания: `F1`
- Связанный отчёт: `r04`
- Служебный класс: `follow-up finding`
- Причина: не входит в критерии приёмки текущей задачи
```

## Структура `RISKS_AND_NOTES`

`RISKS_AND_NOTES` не является свободным текстом. Используйте только явные записи:

- `FOLLOW_UP_FINDING F1` — действие требуется по умолчанию; принимающая сторона должна закрыть запись отдельной задачей, ссылкой на существующую задачу, принятым риском, отметкой о дубле, решением «не требует действия» или повышением до блокирующей проблемы.
- `OUT_OF_SCOPE_NOTE N1` — действие не требуется, если нет строки `Actionable: true`.
- `INFO_NOTE I1` — действие не требуется, если нет строки `Actionable: true`.
- `ACCEPTED_RISK R1` — допустим только с полями `Rationale` и `Next decision point`; без них это недостаток доказательств.

Для каждого `FOLLOW_UP_FINDING` укажите человекочитаемые поля:

- `Идентификатор`: тот же идентификатор, например `F1`.
- `Где найдено`: точный файл, строка, символ, команда или артефакт. Техническое имя поля: `File/symbol`.
- `Проблема`: что не так.
- `Почему не блокирует текущую задачу`: короткое русское объяснение, понятное без знания внутреннего протокола. Техническое имя поля, если оно нужно инструментам: `Why not blocker for current task`.
- `Куда перенести`: существующая задача или новая задача для последующего действия. Технические имена полей: `Suggested existing task`, `Suggested new task`.
- Если указана `Suggested new task`, она должна быть самодостаточной для переноса без скрытого контекста: включите рабочий заголовок задачи, рекомендуемый приоритет, затронутый домен, короткий критерий приёмки и идею проверки. Не ограничивайтесь только номером, короткой меткой или ссылкой.
- `Рекомендуемый приоритет`: рекомендуемый приоритет. Техническое имя поля: `Suggested priority`.
- `Как проверить`: как будущая задача сможет проверить исправление. Техническое имя поля: `Verification idea`.

Текст вида `B1`, `B2` и далее внутри `RISKS_AND_NOTES` сам по себе не является блокирующей проблемой, если рядом прямо сказано, что это историческая ссылка или последующее замечание, а не причина отказа. Нумерованные блокирующие проблемы в секции `BLOCKERS` при `VERDICT: ACCEPT` запрещены.

## Контракт финального ответа

- Отправьте ровно один финальный отчёт. Не отправляйте промежуточные сообщения, черновики, предварительные выводы, отдельные счётчики или частичные списки проблем.
- Не используйте строку `VERDICT:` в промежуточных сообщениях и нигде, кроме первой непустой строки полного финального отчёта.
- Финальный отчёт должен содержать всю информацию для сохранения: блокирующие проблемы, проверенные файлы, доказательства, риски и решение о закрытии.
- В `TASK_ASSESSMENT` явно укажите проверенные `metadata.taskId` и `metadata.iteration` из текущего архива, чтобы сохранённый Markdown можно было отличить от старых отчётов той же задачи.
- Первая непустая строка финального отчёта должна быть строго `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`.
- Если ответ нарушает этот контракт, сторона приёмки обязана считать аудит неполным, даже если в тексте встречается `VERDICT: ACCEPT`.

Машинные маркеры этого контракта: `metadata.taskId`, `metadata.iteration`, `metadata.scopeTaskIds`, `metadata.scopeSummary`, `combined scope`, `metadata.previousVerdictChain`, `metadata.blockerClosureList`, `previous verdict files`, `verbatim preservation`, `previous blockers closure`, `metadata/repo-file-snapshots.json`, `repo-after/`, `repo-before/`, `implementation content review`, `test coverage review`, `documentation review`, `task compliance review`, `secret scanning`, `scope scanning`, `evidence gap`, `patch-only inspection`, `single final report`, `no intermediate VERDICT`, `full current-scope engineering review`, `evidence-backed blocker`, `blocker disproof`, `unsupported concern`, `FOLLOW_UP_FINDING`, `OUT_OF_SCOPE_NOTE`, `ACCEPTED_RISK`, `INFO_NOTE`, `global safety blocker`, `architecture coherence`.

Ответ должен быть строгим и пригодным для сохранения в `docs/verdicts/`. Первая строка ответа должна быть строго одной из двух:

- `VERDICT: ACCEPT`
- `VERDICT: NEEDS_FIXES`

## Формат ответа при принятии

```text
VERDICT: ACCEPT

TASK_ASSESSMENT:
- Кратко по-русски: что проверено и почему изменение можно принять.
- Техническая привязка: `metadata.taskId`, `metadata.iteration`, проверенные файлы и доказательства.

BLOCKERS:
- No blockers found.

EVIDENCE_REVIEW:
- Кратко по-русски: какие файлы, тесты, документация и доказательства были проверены.
- Техническая привязка: точные пути файлов и артефактов.

RISKS_AND_NOTES:
- Человекочитаемое объяснение каждого риска или последующего замечания.
- Техническая привязка: `FOLLOW_UP_FINDING F1`, `OUT_OF_SCOPE_NOTE N1`, `ACCEPTED_RISK R1`, `INFO_NOTE I1` или `None`.

CLOSURE_DECISION:
- Кратко по-русски: почему задачу, текущее изменение или проверяемый пакет можно закрыть.
```

## Формат ответа при отказе

```text
VERDICT: NEEDS_FIXES

TASK_ASSESSMENT:
- Кратко по-русски: что проверено и почему изменение нельзя принять.

  - Почему это важно: почему проблема мешает принять текущую задачу.
  - Что исправить: какое действие требуется.

  - Техническая привязка:

    - `Evidence`: конкретное доказательство из кода, тестов, документации, patch, metadata или evidence.
    - `Impact`: почему это блокирует закрытие задачи.
    - `Fix`: какое исправление требуется.
    - `Verification`: какая команда, проверка или evidence должны подтвердить исправление.

- Кратко по-русски: какие файлы, тесты, документация и доказательства были проверены.
- Техническая привязка: точные пути файлов и артефактов.

CLOSURE_DECISION:
- Кратко по-русски: почему задача остаётся открытой до исправлений.
```

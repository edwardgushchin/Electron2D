# Детерминированный `audit package`

Обновлено: 2026-06-29.

Этот файл является доменным документом для внутренней команды сборки аудиторского архива. Документ фиксирует ожидаемый контракт, текущую модель реализации, ограничения и проверки в одном месте.

## Ведение документа

- Перед изменением команды `audit package` сначала обновите этот документ.
- Затем добавьте точечные тесты в `RepositoryBuildToolTests`, реализуйте изменение в `eng/Electron2D.Build` и снова синхронизируйте этот документ, если фактическое поведение отличается от контракта.
- Доказательства аудита не собираются вручную: архив создаёт только C#-команда репозитория, а внешний аудит допускается только после проверки восстановления.

## Назначение

`audit package` заменяет ручную сборку ZIP-архива для внешнего аудита. Команда создаёт воспроизводимый основной архив из явной конфигурации, запускает заданные проверки, сохраняет сырые доказательства (`evidence`, то есть команды, вывод и коды завершения), строит patch штатным `git diff --binary` через временный индекс Git и проверяет, что чистая копия репозитория восстанавливается из исходной ревизии (`baseline`) до ожидаемого состояния. Текущий контракт не вкладывает в основной ZIP снимок исходной ревизии: чистая копия исходной ревизии является обязательным внешним входом проверки восстановления (`external baseline REQUIRED input`), а отсутствие такого вложенного набора файлов не является требованием контракта (`no baseline payload requirement`). Чтобы отсутствие вложенного снимка не превращалось в неподтверждённое допущение, команда после финальной записи основного ZIP создаёт сопровождающий ZIP `<task-id>-audit-rNN.operator-workflow.zip` с `operator workflow evidence` - переносимыми сырыми доказательствами запуска `audit package verify` и `audit package message` для неизменяемого основного ZIP.

Инструмент является внутренней автоматизацией репозитория. Он не является публичной командой `e2d`, не входит в пакет среды выполнения и не должен публиковаться как часть релиза.

## Границы правил и запросов

`AGENTS.md` задаёт правила работы агента внутри репозитория: безопасность, ведение `TASKS.md`, дневник, проверки, сборку аудиторского архива и обязательный путь внешней приёмки. Он может ссылаться на этот доменный документ, но не должен описывать ручные клики, чтение страницы ChatGPT, работу с cookies или альтернативный браузерный способ вместо `audit submit`.

`docs/release-management/AUDIT-REQUEST.md` является только текстом для внешнего аудитора. Он описывает, какие материалы из одного приложенного audit ZIP нужно проверить и какой финальный отчёт вернуть. Этот файл не описывает Chrome, вкладки, cookies, локальную проверку ZIP, контрольные суммы, ручное скачивание отчёта или операторский порядок отправки.

`.codex/prompts/goal-task-loop.md` является локальным prompt-ом оркестратора. Он описывает порядок ведения задачи на верхнем уровне: предварительное чтение контекста, работу исполнителя, проверки, `audit package`, `audit package verify`, `audit submit`, обработку verdict-а, повторный цикл исправлений, принятие и commit без push. Все детали браузера, режима `Глубокое исследование`, прикрепления ZIP, обновления страницы, экспорта Markdown и закрытия собственной вкладки принадлежат только C#-команде `audit submit` и этому документу.

Любая задача, которую агент доводит до принятия после введения этого правила, остаётся открытой до внешнего отчёта `VERDICT: ACCEPT`, полученного штатной командой `audit submit` и сохранённого в `docs/verdicts/<domain>/<task-id>-audit-rNN.md`. Prompt-ы и агентские правила не должны принимать `VERDICT: ACCEPT` из текста запроса, примера формата, истории чата, обычного предпросмотра или ручного чтения страницы.

## Контракт CLI

Сборка архива:

```bash
dotnet run --project eng/Electron2D.Build -- audit package --task <task-id> --iteration rNN --baseline <sha> --config <path> --out .temp/audit
```

Проверка уже собранного архива на чистой копии:

```bash
dotnet run --project eng/Electron2D.Build -- audit package verify --zip <path> --baseline <sha> --repo <clean-repo-path>
```

Подготовка текста сообщения для внешнего аудита после успешной проверки:

```bash
dotnet run --project eng/Electron2D.Build -- audit package message --zip <path>
```

Отправка проверенного архива во внешний аудит через Electron2D audit-проект ChatGPT:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --out <report-path> --browser-backend codex-chrome
```

`audit package` создаёт файл `<task-id>-audit-rNN.zip` и сопровождающий файл `<task-id>-audit-rNN.operator-workflow.zip` в каталоге `--out`. Если любой из этих файлов уже существует, команда завершается ошибкой. Перезапись разрешена только через явный флаг `--force`; в таком случае диагностическое сообщение фиксирует, что перезапись была разрешена.

Если `--config` не передан, команда читает конфигурацию из `.temp/audit/<task-id>/audit-package.input.json`. Явный `--config` остаётся рекомендуемой формой для внешнего аудита, потому что путь входного файла виден в команде воспроизведения.

`audit package verify` распаковывает основной архив во временный каталог, проверяет контрольные суммы, структуру архива, применимость patch, восстановленные SHA-256 файлов репозитория и защитные правила. Та же команда автоматически требует соседний сопровождающий ZIP `<main-name>.operator-workflow.zip`: если файл отсутствует, не проходит детерминированные правила ZIP, не покрыт `SHA256SUMS.txt`, содержит неполный набор `payload/` или операторских доказательств, ссылается на другой SHA-256 основного ZIP, хранит неверные хэши `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или списка записей основного архива, либо содержит несогласованные метаданные команд, проверка завершается отказом. Команда изменяет переданную чистую копию, поэтому `--repo` должен указывать на отдельную рабочую копию, предназначенную только для проверки восстановления. В этой рабочей копии команда сама выставляет локальные настройки Git `core.autocrlf=false` и `core.eol=lf`, проверяет доступность исходной ревизии `--baseline`, возвращает дерево к этой ревизии, удаляет только отслеживаемые Git файлы (`tracked files`) из списка `git ls-files -z`, снова материализует их через `git reset --hard <baseline>`, удаляет игнорируемые результаты сборки и другие игнорируемые рабочие файлы, а затем проверяет чистоту дерева. Такой порядок принудительно заменяет ранее созданные CRLF-байты на LF даже тогда, когда Git считал рабочее дерево чистым. После успешного применения patch команда повторно очищает Git-игнорируемые результаты сборки, которые не перечислены в `repo-file-hashes.json`, и только затем сравнивает восстановленный набор файлов с моделью архива; намеренно восстановленные Git-игнорируемые файлы из `repoFileAllowlist` сохраняются. Если после подготовки дерево всё ещё содержит изменения, диагностика `E2D-BUILD-AUDIT-REPO-DIRTY` обязана включать фактический вывод `git status --porcelain` без абсолютных путей машины. Команда не создаёт исходную ревизию из содержимого ZIP и не требует, чтобы ZIP содержал Git bundle, архив tar или другой снимок исходного дерева Git.

`audit package message` читает уже собранный ZIP, проверяет корневой `AUDIT-REQUEST.md`, `metadata/audit-package.input.json`, идентификатор задачи, номер итерации и имя файла `<task-id>-audit-rNN.zip`. При успехе команда пишет в `stdout` только готовый текст для вставки в ChatGPT: содержимое `AUDIT-REQUEST.md` из архива без первого Markdown-заголовка первого уровня. Успешный вывод не содержит JSON-диагностик, сгенерированного обрамления задачи и итерации, строк вроде `<task-id> audit rNN`, `Приложен verified audit package` или `Ниже дословный текст`. При ошибке команда завершает работу с ненулевым кодом и пишет только структурированную JSON-диагностику: отсутствующий ZIP, отсутствующий `metadata/audit-package.input.json`, отсутствующий корневой `AUDIT-REQUEST.md`, недействительные обязательные маркеры, недействительные `taskId` или `iteration`, а также несоответствие имени ZIP метаданным считаются блокирующими ошибками.

`audit submit` является операторской C#-автоматизацией поверх проверенного audit package. Команда обязана проверить существование основного ZIP, получить текст сообщения из `--message <path>` или через тот же контракт, что и `audit package message`, открыть одну свежую вкладку Electron2D audit-проекта ChatGPT, дождаться доступности поля сообщения, прокрутить обсуждение вниз, снять baseline уже существующих Deep Research targets, открыть меню кнопкой `+` слева от поля, выбрать реальный пункт `Глубокое исследование`, проверить выбранный connector `connector_openai_deep_research`, прикрепить ровно один основной ZIP-файл, вставить полный текст сообщения без текстового префикса `@Глубокое исследование`, запомнить текущее число сообщений, отправить запрос, дождаться увеличения числа сообщений для текущей отправки, затем обновлять страницу с интервалом не меньше одной минуты до появления готовой карточки отчёта `Углубленный исследовательский отчет` / `Deep research report`. Перед экспортом команда ждёт отрисовку отчётной карточки, прокручивает обсуждение в самый низ и не ищет координату правого верхнего угла карточки. Перед кликом экспорта команда пытается настроить папку загрузки текущего запуска через оба поддерживаемых вызова протокола браузера: `Page.setDownloadBehavior` и `Browser.setDownloadBehavior`; если расширение Chrome отклоняет оба вызова, это не блокирует отправку, потому что результат всё равно можно отличить по новому файлу после клика экспорта. Экспорт выполняется только через DOM-элементы: команда сначала вызывает `button.click()` у видимого пункта `Экспортировать в Markdown` / `Export as Markdown`, если меню уже открыто; кнопка `aria-label="Экспорт"` используется только если меню закрыто или пункт Markdown ещё не виден. Если пункта нет, команда вызывает `button.click()` у видимой кнопки `aria-label="Экспорт"` / `aria-label="Export"` с иконкой скачивания и затем вызывает `button.click()` у видимого пункта `Экспортировать в Markdown` / `Export as Markdown`. Если таких report/export candidates найдено больше одного, команда не выбирает первый элемент, а завершает текущую попытку как неоднозначную. Если уже выбран Deep Research frame или target, пункт Markdown ищется только внутри выбранного контекста; page-level DOM fallback разрешён только когда frame/target surface не выбран вовсе. Если такой элемент недоступен, команда не кликает по вычисленным координатам отчётной карточки и продолжает безопасный цикл ожидания либо завершает `--download-report-only` понятной ошибкой. В обычном submit готовым отчётом считается ровно один полный Markdown-файл, скачанный пунктом `Экспортировать в Markdown` как новый `.md`: если настройка папки загрузки сработала, файл должен быть в управляемой папке текущего запуска; если настройка была отклонена расширением, допустима реальная пользовательская папка загрузок Windows, но только для нового файла, которого не было в baseline-снимке перед кликом экспорта. `--download-report-only` является диагностическим режимом и дополнительно может искать файл в реальной пользовательской папке загрузок Windows. Если после одного export action появляется больше одного нового Markdown-файла в разрешённых для режима папках, команда завершает попытку как неоднозначную и не выбирает более поздний файл. Если новый Markdown появился только вне разрешённых для текущего режима папок, команда завершает попытку ошибкой `E2D-BUILD-AUDIT-SUBMIT-REPORT-DOWNLOAD-FOREIGN`, не обновляет страницу и не нажимает экспорт повторно. Скачанный Markdown в обычном submit и в `--download-report-only` проходит один и тот же строгий валидатор: первая непустая строка этого текста должна быть ровно `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`, и в нём должны присутствовать обязательные секции `TASK_ASSESSMENT:`, `BLOCKERS:`, `EVIDENCE_REVIEW:`, `RISKS_AND_NOTES:` и `CLOSURE_DECISION:`. После первого единственного валидного Markdown-файла команда сразу возвращает отчёт и не выполняет повторный экспорт ради стабилизации. Для `VERDICT: ACCEPT` дополнительно требуется, чтобы секция `BLOCKERS:` не содержала маркеров `B1`, `B2` и далее, а секция `CLOSURE_DECISION:` явно разрешала закрытие задачи. Команда не использует буфер обмена как источник verdict-а и не запускает PowerShell для чтения или записи результата. Поиск verdict-а по всей странице, истории чата, пользовательскому запросу, боковой панели, обычному превью карточки или неполному тексту запрещён. При успехе команда сохраняет отчёт в `--out` и пишет тот же отчёт в `stdout`, затем закрывает только вкладку, созданную текущим запуском. При ошибке команда закрывает только вкладку, созданную текущим запуском, не записывает частичный отчёт как готовый и возвращает структурированную JSON-диагностику. Команда не закрывает и не изменяет уже открытые пользовательские вкладки, даже если они открыты на том же ChatGPT-проекте: закрытие чужих или оставшихся от прежней сессии браузерного протокола (`browser session`) вкладок не является безопасно детерминированным действием через этот прямой путь расширения и поэтому считается невозможным в контракте инструмента.

Если после отправки ChatGPT показывает кнопку остановки ответа, команда считает генерацию активной: она продолжает минутный цикл, делает скриншоты и всё равно обновляет страницу раз в минуту до готового отчёта. Пока генерация активна, команда не принимает отчёт как готовый даже тогда, когда текущий streaming-текст уже начинается с `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`, либо в истории уже видна старая завершённая отчётная карточка. Когда активной генерации нет, команда сначала прокручивает обсуждение вниз, затем пробует нажать уже открытый пункт `Экспортировать в Markdown`; если пункта нет, она открывает кнопку `aria-label="Экспорт"` и нажимает пункт `Экспортировать в Markdown`. После клика команда ждёт новый `.md` в управляемой папке загрузки или в реальной пользовательской папке загрузок Windows. Если генерация стала активной до или после чтения, скачанный текст отбрасывается. Если ChatGPT во время ожидания показывает временный диалог ограничения частоты запросов `Слишком много запросов` / `Too many requests`, команда делает диагностический PNG, нажимает видимую кнопку подтверждения `Понятно` / `OK` и продолжает тот же минутный цикл; такой диалог не считается отчётом и не требует ручного шага. После каждого минутного интервала команда обновляет страницу, ждёт возвращения сообщений обсуждения и только затем продолжает извлечение отчёта. Если расширение Chrome во время длинного ожидания сообщает recoverable CDP detachment, клиент выполняет до пяти повторных `attach` с best-effort `detach` перед каждой попыткой и ждёт `Page.enable`, `Runtime.enable`, `DOM.enable` до 45 секунд на попытку.

Если `Codex Chrome Extension` временно теряет подключение к странице во время чтения состояния страницы, скриншота или обновления страницы, backend `codex-chrome` выполняет ограниченное повторное подключение к собственной вкладке: сначала освобождает отладочное подключение этой вкладки через `detach`, затем снова выполняет `attach` и включает нужные CDP-домены. После успешного восстановления команда продолжает тот же цикл ожидания. Действия, которые могут изменить состояние ChatGPT, например выбор пункта меню, прикрепление файла, вставка текста и отправка, автоматически не повторяются после такого сбоя, чтобы не создать двойное действие.

Pipe-клиент `codex-chrome` отвечает строкой `pong` на входящие служебные JSON-RPC запросы `ping` от `Codex Chrome Extension` во время ожидания своего ответа. Такие запросы не являются действиями браузера, но без совместимого ответа на них расширение может не завершить долгую операцию вкладки.

По умолчанию `audit submit` использует проект `https://chatgpt.com/g/g-p-6950376d4d8c8191a0fe600e98389912-electro2d/project`, backend `--browser-backend codex-chrome`, уже установленный `Codex Chrome Extension` в пользовательском Chrome, интервал обновления `60` секунд, общий предел ожидания отчёта `180` минут и предел ожидания входа в ChatGPT `10` минут. Это единственный штатный браузерный путь. Команда не копирует cookies, `localStorage`, пароли, файлы профиля Chrome или другие файлы сессии: C#-команда подключается к pipe установленного расширения `Codex Chrome Extension`, выбирает backend с типом `extension`, создаёт собственную управляемую вкладку в текущей пользовательской Chrome-сессии, выполняет действия через протокол расширения и закрывает только эту вкладку после результата или ошибки. Она не использует `getUserTabs`, `claimUserTab`, `Page.close` для вкладок пользователя и не завершает чужие сессии браузерного протокола (`browser session`) через `finalizeTabs`. Для воспроизводимости разрешены параметры `--codex-chrome-pipe <path>`, `--codex-session-id <id>` и `--codex-turn-id <id>`; пара `--codex-session-id` и `--codex-turn-id` должна передаваться только вместе. Если пара не передана и в окружении нет одновременно `CODEX_SESSION_ID` и `CODEX_TURN_ID`, команда создаёт независимую сессию браузерного протокола (`browser session`) сама и не использует `CODEX_THREAD_ID` как суррогат такой сессии. Если pipe не передан, команда обнаруживает доступный `codex-browser-use` pipe сама и выбирает только backend `extension`. Отдельная команда проверки входа, другой браузерный backend, закрытие уже открытых пользовательских вкладок и обход через скопированные файлы браузерной сессии не входят в контракт.

Для локальной проверки команды разрешены параметры `--project-url <url>`, `--screenshots-dir <path>`, `--poll-seconds <seconds>`, `--timeout-minutes <minutes>`, `--login-timeout-minutes <minutes>`, `--codex-chrome-pipe <path>`, `--codex-session-id <id>`, `--codex-turn-id <id>` и `--keep-tab-open-on-error`. Значение `--poll-seconds` меньше `60` допускается только вместе с `--allow-fast-poll`, чтобы тесты могли проверять цикл ожидания без изменения рабочего правила внешнего аудита. Флаг `--keep-tab-open-on-error` предназначен только для диагностики реального сайта: при ошибке он оставляет открытой собственную вкладку текущего запуска, чтобы оператор видел фактическое состояние Chrome; при успешном получении отчёта вкладка всё равно закрывается.

Если передан `--screenshots-dir`, команда сохраняет PNG-снимки протокола браузера после каждого ключевого шага: открытия проекта, готовности поля сообщения, открытого меню кнопки `+`, фиксации режима `Глубокое исследование` в поле сообщения, прикрепления файла, вставки текста, отправки запроса, каждого цикла ожидания или обновления страницы, временного диалога ограничения частоты запросов, отсутствующей DOM-кнопки экспорта отчётной карточки, открытого меню экспорта отчётной карточки и обнаружения готового отчёта. Перед каждым таким снимком команда ждёт короткую задержку, чтобы отчётная карточка и меню успели отрисоваться. Эти PNG являются технической диагностикой браузерного протокола, а не самостоятельным доказательством того, что видит пользователь в окне Chrome; если для приёмки нужен снимок реального окна, выполнение проверки блокируется до отдельного пользовательского скриншота. Любые снимки могут содержать текст запроса или отчёта, поэтому их нельзя публиковать без проверки на секреты.

Встроенный операторский слой доказательств не входит в основной ZIP и не меняет его `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или набор записей после выполнения `audit package verify` и `audit package message`. Он создаётся в сопровождающем ZIP `<task-id>-audit-rNN.operator-workflow.zip` после финальной записи и внутренней проверки основного ZIP. Сопровождающий ZIP содержит `OPERATOR-WORKFLOW.md`, `payload/metadata.json`, `payload/sha256.txt`, `payload/AUDIT-MANIFEST.sha256`, `payload/SHA256SUMS.sha256`, `payload/archive-entries.sha256`, `payload/archive-entries.txt`, а также каталоги `evidence/<task-id>-<iteration>/checks/audit-package-verify/` и `evidence/<task-id>-<iteration>/checks/audit-package-message/` с файлами `command.txt`, `stdout.txt`, `stderr.txt`, `exit-code.txt`, `duration-ms.txt`, `metadata.json`, `cwd.txt`, `env.json` и `timeout-seconds.txt`. В `metadata.json` для этих двух каталогов `executionMode` должен быть `subprocess`, ожидаемый и фактический коды завершения должны быть `0`, длительность должна быть больше нуля, а `timeoutSeconds` должен совпадать с `timeout-seconds.txt` и фактическим лимитом времени операторской команды: `180` секунд.

`command.txt` для проверки восстановления обязан показывать переносимую команду с `--zip <relative-zip-path>`, `--baseline <sha>` и плейсхолдером `--repo <clean-repo-path>`, а не абсолютный локальный путь временной рабочей копии. Фактическое выполнение происходит только через отдельный процесс `dotnet run --project eng/Electron2D.Build -- audit package verify --zip <path> --baseline <sha> --repo <clean-repo-path>`; прямой вызов внутреннего метода проверки не считается допустимым evidence. Аналогично `audit package message evidence` создаётся только через отдельный процесс `dotnet run --project eng/Electron2D.Build -- audit package message --zip <path>`, а не через внутренний метод подготовки текста. Это `baseline availability evidence`, то есть проверяемое доказательство того, что пакет был проверен на отдельной чистой копии исходной ревизии. Если архив или изменённые документы утверждают, что исходная ревизия доступна только локально, непубликована или требует ручной догадки, отсутствие вложенного baseline-снимка всё ещё не является отдельным blocker-ом, но успешное `audit package verify evidence` в сопровождающем ZIP становится обязательным доказательством восстановления.

## Окончания строк

Аудиторский архив опирается на общую репозиторную политику из `docs/repository-policy/line-endings.md`: `.gitattributes` задаёт `* text=auto eol=lf`, обычные текстовые файлы должны храниться и восстанавливаться с LF, а CRLF разрешён только для явных исключений `.bat`, `.cmd` и временно для исторических verdict-документов в `docs/verdicts/`, которые текущая задача не изменяет. Настоящие двоичные форматы описываются по расширениям как `binary`; XML-карты `.tmx` и `.tsx` в reference-game данных считаются текстом и нормализуются в LF.

Одноразовая нормализация tracked текстовых файлов допускается только как замена CRLF или одиночного CR на LF без смыслового изменения содержимого. Проверка:

```bash
dotnet run --project eng/Electron2D.Build -- verify line-endings
```

читает `git ls-files --eol` и отклоняет неожиданные `w/crlf` у tracked текстовых файлов. Перед повторным внешним аудитом ручная команда `git ls-files --eol | rg "w/crlf"` должна показывать только описанные исключения или не показывать строк вообще.

## Конфигурация

Источник истины для входных данных - JSON-файл. Путь по умолчанию для рабочих сценариев:

```text
.temp/audit/<task-id>/audit-package.input.json
```

YAML считается допустимым будущим форматом того же контракта, но текущая реализация читает JSON. Конфигурация включается в архив как `metadata/audit-package.input.json`; исходный путь из `.temp` в архив не попадает.

Минимальные поля:

| Поле | Назначение |
| --- | --- |
| `taskId` | Идентификатор задачи. Должен совпадать с `--task` и всеми сгенерированными артефактами. |
| `iteration` | Итерация аудита в форме `rNN`. Должна совпадать с `--iteration`. |
| `baseline` | Исходная ревизия Git. Должна совпадать с `--baseline`. |
| `branch` | Имя рабочей ветки или логическое имя изменения, которое попадёт в запрос аудитору. |
| `domain` | Домен изменения, например `release-management`. |
| `scopeTaskIds` | Машиночитаемая область задач, которые принимает текущий пакет. Если список пуст, область равна одиночному `taskId`; если список содержит несколько значений, пакет объявляет явно комбинированную область (`combined scope`) и должен включать основной `taskId`. |
| `scopeSummary` | Краткое пояснение области аудита. Поле предназначено для объяснения, почему несколько задач входят в один пакет, и не заменяет `repoFileAllowlist`, `previousVerdictChain` или `blockerClosureList`. |
| `repoFileGlobs` | Шаблоны файлов репозитория, которые восстанавливаются через patch. |
| `repoFileAllowlist` | Явный список файлов репозитория, которые разрешено включить даже без совпадения с шаблоном. Список может включать Git-игнорируемые рабочие файлы вроде `TASKS.md` или `dev-diary/...`, если они принадлежат задаче и нужны аудитору. |
| `archiveOnlyEvidenceGlobs` | Шаблоны файлов, которые попадают только в `evidence/` архива и не входят в patch. |
| `checks` | Машиночитаемый список проверок, которые команда запускает до упаковки. |
| `forbiddenPatterns` | Дополнительные запрещённые шаблоны путей сверх встроенной защиты. |
| `secretScanPolicy` | Политика поиска секретов. Поддерживаемое значение текущей реализации: `basic`. |
| `maxFileSize` | Максимальный размер одного входного файла в байтах. |
| `outputDirectory` | Ожидаемый каталог результата. Должен совпадать с `--out`, если указан. |

Дополнительные поля:

| Поле | Назначение |
| --- | --- |
| `previousVerdictChain` | Список предыдущих verdict-документов или пустой список. Каждый путь должен указывать на Markdown-файл внутри `docs/verdicts/`, иначе конфигурация считается ошибочной. ## Содержимое архива

Корневые файлы архива:

- `AUDIT-MANIFEST.md` - инвентарь, а не повествовательный отчёт.
- `<task-id>.patch` - patch, созданный штатным Git через `git diff --binary`.
- `repo-file-hashes.json` - ожидаемые SHA-256 файлов репозитория после применения patch.
- `SHA256SUMS.txt` - контрольные суммы всех файлов архива, кроме самого `SHA256SUMS.txt`, в формате, совместимом с `sha256sum -c`.
- `AUDIT-REQUEST.md` - статический запрос аудитору, дословно скопированный из отслеживаемого Git-файла `docs/release-management/AUDIT-REQUEST.md`.
- `metadata/audit-package.input.json` - нормализованная копия конфигурации.
- `evidence/` - только доказательства настроенных проверок и импортированные файлы доказательств; операторские доказательства `audit package verify` и `audit package message` хранятся в сопровождающем ZIP, чтобы не менять основной архив после проверки.

Архив не содержит и по текущему контракту не обязан содержать снимок исходной ревизии, Git bundle, архив tar или иной вложенный набор файлов исходного дерева. Восстановление проверяется через patch, `repo-file-hashes.json`, `AUDIT-MANIFEST.md` и отдельную чистую копию исходной ревизии, переданную в `audit package verify --repo <clean-repo-path>`.

Сопровождающий ZIP `<task-id>-audit-rNN.operator-workflow.zip` не является частью основного архива и не входит в его `SHA256SUMS.txt`. Он остаётся локальным доказательством предварительной проверки и не прикрепляется командой `audit submit` к внешнему ChatGPT-запросу. Его назначение - доказать локальной предварительной проверке, что уже финальный основной ZIP был проверен и что текст внешнего сообщения был получен штатной командой. Файл `payload/metadata.json` в сопровождающем ZIP содержит относительный путь основного ZIP, SHA-256 основного ZIP, SHA-256 `AUDIT-MANIFEST.md`, SHA-256 `SHA256SUMS.txt`, SHA-256 списка записей архива и сам список записей. Если после создания операторских доказательств основной ZIP будет переписан, эти значения перестанут совпадать с фактическим файлом.

`AUDIT-MANIFEST.md` обязан содержать только проверяемый инвентарь:

- `taskId`, `iteration`, `baseline`, `branch`, `domain`, `scopeTaskIds` и непустой `scopeSummary`, если он задан;
- исходный путь статического запроса аудитору: `docs/release-management/AUDIT-REQUEST.md`;
- `diff name-status`, то есть список изменённых файлов и типов изменения;
- список всех путей архива;
- список repo-owned файлов из `repo-file-hashes.json`, включая существующие предыдущие verdict-документы из `previousVerdictChain`;
- объявленную область задач, цепочку предыдущих verdict-ов и список закрытых blocker-ов;
- модель восстановления;
- список проверок и ссылки на соответствующие файлы `evidence/`, включая найденные TRX-файлы с результатами `dotnet test`;
- политику запрещённых путей и секретов.

## Пути и ZIP

Все пути внутри архива используют UTF-8 POSIX-форму `dir/file.ext`. Backslash, управляющие символы, абсолютные пути, `..`, пустые сегменты и дубли путей запрещены. Манифест использует те же пути, что и архив.

ZIP создаётся в стабильном порядке:

- записи отсортированы по ordinal-сравнению путей;
- у всех записей фиксированное время `2000-01-01T00:00:00Z`;
- внешние атрибуты ZIP-записей, зависящие от операционной системы, должны быть нулевыми;
- ZIP-запись с не-ASCII путём должна иметь `UTF-8 flag`, то есть признак кодировки UTF-8 в центральном каталоге ZIP; текущая реализация пишет имена через UTF-8 и `audit package verify` проверяет этот признак минимум для таких путей;
- текстовые файлы генерируются в стабильном порядке;
- пути машины, абсолютные временные каталоги и текущее время не записываются, кроме явно разрешённых относительных метаданных проверки; Windows-пути с диском по шаблонам `[A-Z]:\...` и `[A-Z]:/...` запрещены в любом текстовом файле архива.
- `audit package verify` отклоняет архив с каталогами-записями, несортированными записями, нестабильной временной меткой ZIP, ненулевыми внешними атрибутами ZIP-записей или отсутствующим UTF-8 flag у не-ASCII пути.

Сопровождающий ZIP операторских доказательств тоже записывается с отсортированными путями, фиксированными временными метками и нулевыми внешними атрибутами ZIP-записей. Его байты не обязаны быть стабильными между разными запусками, потому что `stdout.txt`, `stderr.txt`, `duration-ms.txt` и `metadata.json` отражают реальные отдельные процессы.

## Patch и восстановление

Patch создаётся через Git, без ручной правки заголовков hunk-ов. Для выбранных файлов используется временный индекс Git:

1. В индекс читается дерево `baseline`.
2. В индекс добавляется текущее состояние выбранных файлов репозитория.
3. Команда строит `git diff --cached --binary <baseline>`.

Такой подход включает новые файлы, изменения, удаления и бинарные файлы, которые поддерживает `git diff --binary`. Если выбранный бинарный файл нельзя восстановить через эту модель, команда должна завершиться ошибкой до создания финального архива.

Для текстовых файлов команда добавляет во временный индекс Git байты с LF-окончаниями строк. Это защищает аудиторский patch от массовой перезаписи существующих файлов, когда локальная рабочая копия на Windows хранит те же строки с CRLF-окончаниями. `repo-file-hashes.json` считает SHA-256 по тем же восстановимым байтам, поэтому проверка архива сравнивает состояние, которое реально создаёт patch, а не машинный формат строк в текущем каталоге.

Файлы из `repoFileAllowlist`, которые игнорируются правилами Git, всё равно считаются заявленным входом задачи. Команда обязана принудительно добавить их во временный индекс Git, включить в patch и доказать восстановление через `audit package verify`; правила игнорирования не должны молча исключать такой файл из архива.

Файлы из `previousVerdictChain` считаются частью цепочки внешнего аудита. Когда такой файл есть в рабочей копии, он восстанавливается как обычный файл репозитория даже без совпадения с `repoFileGlobs` и без отдельного `repoFileAllowlist`. Это делает цепочку verdict-ов проверяемой в восстановленной модели файлов репозитория (`restored repo-owned model`): ссылка в `AUDIT-REQUEST.md` совпадает с patch, `repo-file-hashes.json` и инвентарём `AUDIT-MANIFEST.md`. Отсутствующий файл не создаётся автоматически и остаётся только исторической ссылкой, чтобы старые внешние ссылки не блокировали упаковку.

Содержимое существующих verdict-файлов из `previousVerdictChain` сохраняется дословно как историческое внешнее свидетельство внутри восстановленной модели файлов репозитория. Текущий контракт не объявляет `previousVerdictChain` независимым нотариальным доказательством происхождения прошлых внешних отчётов и не требует предыдущие аудиторские ZIP-архивы или отдельные исторические контрольные суммы (`no previous audit packages or detached historical checksums required`), если такой слой явно не введён отдельным контрактом пакета. Поэтому проверка локальных машинных путей и значений, похожих на секреты, не применяет запрет к diff-блокам этих файлов внутри patch: старый ответ аудитора может цитировать путь, имя машины или строку прежнего доказательства, которые не относятся к текущей упаковке. То же ограниченное исключение действует при предварительной проверке самого verdict-файла перед построением patch. Это не выключает `secretScanPolicy` глобально: обычные файлы репозитория, файлы текущей задачи, архивные доказательства, вывод проверок, TRX-файлы, `AUDIT-MANIFEST.md`, `AUDIT-REQUEST.md`, `repo-file-hashes.json` и `metadata/audit-package.input.json` по-прежнему блокируют пакет при новых локальных путях машины или конкретных секретоподобных значениях.

По умолчанию patch остаётся текстовым: чистые изменения обычных текстовых файлов пишутся как обычные diff-блоки. Если обычный diff-блок вне `previousVerdictChain` раскрывает значение, похожее на секрет, только через удалённую или контекстную строку исходной ревизии (`baseline`), команда не передаёт этот текст аудитору. После отдельной проверки текущего содержимого выбранного файла она повторно строит patch и помечает только этот путь как двоичный diff (`GIT binary patch`). Такой блок остаётся применимым через `git apply`, восстанавливает те же байты и не содержит запрещённый текст прежнего состояния. Остальные чистые текстовые изменения остаются обычными diff-блоками. Если секретоподобное значение есть в текущем файле задачи, упаковка всё равно завершается ошибкой до построения patch.

Patch проверяется сразу после создания и до сборки ZIP, а затем повторно при `audit package verify`. Для этой проверки команда временно исключает только diff-блоки существующих исторических verdict-файлов из `previousVerdictChain`; остальные блоки patch сканируются полностью. Это включает добавленные строки, контекстные строки и удалённые строки из `baseline`. Если после точечного двоичного восстановления для проблемных файлов обычный patch всё ещё содержит запрещённый текст, пакет завершается ошибкой вместо передачи такого текста аудитору.

`repo-file-hashes.json` описывает ожидаемое восстановленное состояние:

- `taskId`, `iteration`, `baseline`;
- список существующих repo-owned файлов и их SHA-256;
- список удалённых repo-owned файлов.

Проверка восстановления выполняет:

1. C#-проверку `SHA256SUMS.txt`.
2. Дополнительный запуск `sha256sum -c`, если утилита доступна.
3. `git apply --check`.
4. `git apply`.
5. Сравнение точного набора изменённых, удалённых и новых путей файлов репозитория после применения patch с `repo-file-hashes.json`. Новые Git-игнорируемые файлы из `repoFileAllowlist` тоже входят в это сравнение.
6. Сравнение восстановленных SHA-256 файлов с `repo-file-hashes.json`.
7. Сравнение корневого `AUDIT-REQUEST.md` из архива с восстановленным отслеживаемым Git-файлом `docs/release-management/AUDIT-REQUEST.md`.
8. Проверку, что новые текстовые файлы не являются однострочными заглушками, если они не указаны в `oneLineStubAllowlist`.

## Evidence

Каждая проверка из `checks` создаёт отдельный каталог:

```text
evidence/<task-id>-<iteration>/checks/<check-name>/
```

Минимальные файлы проверки:

- `command.txt` - исполняемый файл и аргументы без shell-строки.
- `cwd.txt` - относительный рабочий каталог.
- `env.json` - только переменные из `envAllowlist`.
- `timeout-seconds.txt` - тайм-аут проверки.
- `exit-code.txt` - ожидаемый и фактический код завершения.
- `duration-ms.txt` - фактическая длительность запуска в миллисекундах, нормализованная как число больше нуля с инвариантным десятичным разделителем.
- `stdout.txt` и `stderr.txt` - полный стандартный вывод и поток ошибок.
- `metadata.json` - машиночитаемая сводка, включая команду, тайм-аут, ожидаемый и фактический код завершения, ту же длительность, SHA-256 файлов вывода и TRX-файлов, если они были найдены.

`metadata.json` должен совпадать с соседними сырыми файлами: `expectedExitCode` и `actualExitCode` берутся из `exit-code.txt`, `timeoutSeconds` - из `timeout-seconds.txt`, а `durationMs` - из `duration-ms.txt`. Эти поля не должны зависеть от порядка аргументов конструктора или от человекочитаемого текста.

Перед записью `stdout.txt` и `stderr.txt` команда заменяет локальный корень репозитория на `<repo-root>`. Это сохраняет полный диагностический вывод проверок, но не раскрывает путь рабочей машины и не требует от оператора вручную подавлять вывод команд.

Для проверок `dotnet test` конфигурация обязана задавать TRX-вывод через штатный logger и включать соответствующий `trxGlobs`, например `TestResults/**/*.trx`. Рекурсивный сегмент `**/` совпадает и с прямыми дочерними файлами, и с файлами во вложенных каталогах, поэтому `TestResults/**/*.trx` включает как `TestResults/result.trx`, так и `TestResults/nested/result.trx`. Вывод команд нельзя подавлять через shell-редиректы вроде `*> $null`, потому что внешний аудитор должен видеть `stdout.txt`, `stderr.txt`, `exit-code.txt`, `timeout-seconds.txt`, `metadata.json` и найденные TRX-файлы без локальных догадок.

TRX-файлы проверяются как XML-доказательства. Сканирование секретов должно читать содержимое результатов тестов: текст стандартного вывода, ошибок, сообщений, трассировок и обычных атрибутов. Служебные отображаемые имена тестов, где тестовый фреймворк может показать параметры проверок вроде синтетических строк `token=...`, не должны блокировать включение штатного TRX. Если настоящее секретное присваивание или блок приватного ключа попадает в текст результата теста, TRX всё равно отклоняется. Перед записью TRX в архив команда нормализует служебные XML-атрибуты с путями сборки, например `codeBase` и `storage`: путь к локальному корню репозитория заменяется на `<repo-root>`, чтобы архив не раскрывал путь машины.

TRX-доказательства получают стабильные имена внутри архива: `trx/test-result-001.trx`, `trx/test-result-002.trx` и далее по отсортированному исходному списку. Исходные имена файлов, которые тестовый раннер может составлять из имени пользователя, имени машины и времени запуска, не записываются в пути архива, `metadata.json`, `AUDIT-MANIFEST.md`, `SHA256SUMS.txt`, `stdout.txt` или `stderr.txt`. Если стандартный вывод или поток ошибок содержит исходный путь TRX, команда заменяет его на стабильный путь архива.

При записи TRX XML команда дополнительно нормализует машинные атрибуты раннера: `TestRun/@name` становится `<test-run>`, а атрибуты `runUser`, `computerName` и `Deployment/@runDeploymentRoot` удаляются из архивной копии TRX. Поэтому ZIP не содержит ни настоящие локальные значения, ни сами проблемные XML-атрибуты, которые внешняя проверка трактует как утечку машинного контекста. Текст результатов тестов не редактируется перед проверкой секретов: настоящее секретное значение в `StdOut`, `StdErr`, сообщении или трассировке по-прежнему блокирует пакет.

Для самого T-0228 внешний пакет должен содержать отдельные checks для:

- минимального запуска `audit package` на тестовом наборе файлов;
- отдельного `audit package verify` на чистой копии baseline.

Эти записи `checks` должны создавать самостоятельные каталоги доказательств в `AUDIT-MANIFEST.md`, а не только строки в общей команде или внешней заметке.

Файлы из `archiveOnlyEvidenceGlobs` копируются под `evidence/<task-id>-<iteration>/archive-only/` и покрываются контрольными суммами. Они не входят в patch и не считаются repo-owned файлами.

После выполнения проверок инструмент фиксирует SHA-256 выбранных файлов репозитория и доказательств. Перед записью ZIP тот же набор пересчитывается снова. Если файлы изменились между проверками и упаковкой, команда завершается ошибкой.

После финальной записи основного ZIP команда больше не изменяет `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или набор его записей. Операторские доказательства создаются только после этого и попадают в сопровождающий ZIP. Поэтому `audit package verify evidence` и `audit package message evidence` относятся к неизменяемому основному ZIP, а не к промежуточной версии архива.

## Защитные правила

Команда отклоняет архив или входные файлы, если путь или содержимое попадает под встроенный запрет:

- `.git`;
- `bin`;
- `obj`;
- `.temp`, `tmp` и временные каталоги;
- вложенные `.zip`, `.tar`, `.tar.gz`, `.7z`;
- `.env` и производные имена;
- опасные имена пути с `secret`, `token` или `private-key`;
- блоки приватных ключей;
- присваивания или JSON/YAML-подобные значения для `secret`, `token`, `password`, `api_key` и `api-key`.

Документальные упоминания `secretScanPolicy`, `secret`, `token`, `password` или `api_key` без секретного значения не считаются утечкой. Синтетические замещающие примеры вроде `token=...` или `password=<redacted>` тоже допустимы, когда они описывают защитное правило. Если после такого замещающего маркера в той же обычной строке текста продолжается предложение, сканирование отделяет маркер по границе пунктуации, кавычек, обратных кавычек или пробела. Любое конкретное непустое значение после `secret`, `token`, `password`, `api_key` или `api-key` всё ещё считается потенциальным секретом, в том числе когда после значения продолжается обычный текст.

`forbiddenPatterns` расширяет этот список для конкретной задачи.

Сканирование содержимого выполняется и при сборке, и при `audit package verify`. Текстовые файлы архива не должны содержать путь к корню локального репозитория, системному временному каталогу или Windows-путь с диском вида `[A-Z]:\...` и `[A-Z]:/...`. Для patch это правило применяется к оставшимся текстовым diff-блокам целиком, включая удалённые строки из `baseline`; исключаются только исторические diff-блоки verdict-файлов из `previousVerdictChain`.

Для TRX-доказательств это правило применяется после безопасной нормализации служебных XML-атрибутов тестового раннера, стабильного переименования TRX-файлов и замены исходных TRX-путей в выводе проверок. Секретные значения в тексте результатов тестов остаются запрещёнными; исключение касается только машинно созданных отображаемых имён тестов и технических путей сборки, которые не являются содержимым проверяемого вывода.

`audit package verify` также завершается ошибкой, если сгенерированные имена путей архива, имя patch, корень `evidence/<task-id>-<iteration>/`, структурные поля `metadata/audit-package.input.json` или блок метаданных `AUDIT-MANIFEST.md` не совпадают с текущими `taskId` и `iteration`. Проверка не сканирует весь Markdown/JSON как свободный текст: связанные задачи вроде `T-0232` допускаются в `scopeTaskIds`, а исторические ссылки на задачи вроде `T-0207` допускаются в `previousVerdictChain` и `blockerClosureList`, потому что эти поля описывают область текущего пакета, цепочку внешних verdict-ов и закрытые blocker-ы, а не имя текущего архива. Корневой `AUDIT-REQUEST.md` больше не содержит сгенерированный заголовок текущей задачи: его соответствие проверяется сравнением с восстановленным отслеживаемым источником.

## Запрос аудитору

`docs/release-management/AUDIT-REQUEST.md` является отслеживаемым исходным контрактом: это постоянный текст запроса внешнему аудитору, который хранится в репозитории, проходит проверку как обычная документация и копируется в корень ZIP как `AUDIT-REQUEST.md` без подстановок, генерации блоков или замены заполнителей. Этот файл также фиксирует контракт проверки области пакета и предыдущих внешних verdict-ов: аудитор обязан читать `metadata.scopeTaskIds`, `metadata.scopeSummary`, `metadata.previousVerdictChain`, `metadata.blockerClosureList`, предыдущие verdict-файлы, дословное сохранение исторических verdict-ов и закрытие каждого предыдущего blocker-а.

Статический запрос обязан требовать проверку содержимого изменения, тестов, документации, секретов и области задачи. Внешний аудитор не проверяет упаковочный слой: целостность ZIP, контрольные суммы, сопровождающий ZIP, операторские доказательства, локальную команду `audit package verify` и применимость patch. Эти проверки остаются локальным предварительным шагом команды репозитория до отправки.

Приёмка verdict-а использует безопасное операторское правило из раздела ниже. Статический запрос внешнему аудитору требует один финальный отчёт и запрещает промежуточные сообщения с `VERDICT:`, но не содержит DOM-селекторы или инструкции по чтению интерфейса.

Динамические сведения о конкретной задаче остаются в проверяемых машинных файлах:

- `AUDIT-MANIFEST.md` - инвентарь архива, исходная ревизия, задача, итерация, список файлов и доказательств;
- `metadata/audit-package.input.json` - нормализованная входная конфигурация;
- `repo-file-hashes.json` - модель восстановления файлов репозитория.

Перед созданием ZIP команда обязана прочитать `docs/release-management/AUDIT-REQUEST.md` из рабочей копии и отклонить пакет, если файл отсутствует, пустой, выглядит как однострочная заглушка, содержит конфиденциальное значение, машинный абсолютный путь или не содержит обязательные маркеры ответа:

- `VERDICT: ACCEPT`;
- `VERDICT: NEEDS_FIXES`;
- `TASK_ASSESSMENT`;
- `BLOCKERS`;
- `EVIDENCE_REVIEW`;
- `RISKS_AND_NOTES`;
- `CLOSURE_DECISION`;
- `metadata.scopeTaskIds`;
- `metadata.scopeSummary`;
- `combined scope`;
- `metadata.previousVerdictChain`;
- `metadata.blockerClosureList`;
- `previous verdict files`;
- `verbatim preservation`;
- `previous blockers closure`;
- `implementation content review`;
- `test coverage review`;
- `documentation review`;
- `task compliance review`;
- `secret scanning`;
- `scope scanning`;
- `single final report`;
- `no intermediate VERDICT`.

`AUDIT-MANIFEST.md` обязан явно указать исходный путь этого статического файла: `docs/release-management/AUDIT-REQUEST.md`. При `audit package verify` после применения patch команда сравнивает `AUDIT-REQUEST.md` из архива с восстановленным `docs/release-management/AUDIT-REQUEST.md`; любое расхождение означает, что архив не соответствует отслеживаемому контракту и должен быть отклонён.

## Отправка внешнему аудитору

Внешний аудит разрешён только после успешного `audit package verify` на отдельной чистой копии исходной ревизии. Штатная отправка выполняется командой `audit submit`: она открывает одну свежую вкладку проекта в пользовательском Chrome через установленный `Codex Chrome Extension`, ждёт доступности поля сообщения, прокручивает обсуждение вниз, снимает baseline уже существующих Deep Research targets, открывает меню кнопкой `+` слева от поля, выбирает реальный пункт `Глубокое исследование`, проверяет выбранный connector `connector_openai_deep_research`, прикрепляет проверенный основной аудиторский ZIP и не прикрепляет сопровождающий ZIP `<task-id>-audit-rNN.operator-workflow.zip`, вставляет полный вывод команды `audit package message --zip <path>` или текст из `--message <path>` без текстового префикса `@Глубокое исследование`, запоминает текущее число сообщений, отправляет сообщение, ждёт появления нового сообщения текущей отправки, обновляет страницу раз в минуту до готового отчёта и сохраняет результат. Финальный текст результата команда получает как Markdown-файл, скачанный пунктом `Экспортировать в Markdown`. Если готовый отчёт отрисован отдельным приложением внутри `iframe[title="internal://deep-research"]` с адресом `connector_openai_deep_research.web-sandbox.oaiusercontent.com`, команда принимает только ровно один видимый встроенный фрейм и ровно один Deep Research frame id из `Page.getFrameTree`; несколько фреймов считаются неоднозначностью. Для выбранного фрейма команда пытается создать отдельный контекст выполнения CDP (`executionContextId`), сначала нажимает уже видимый пункт `Экспортировать в Markdown`, а если меню закрыто, нажимает реальную кнопку экспорта DOM-кликом по `button[aria-label="Экспорт"]` / `button[aria-label="Export"]` и DOM-кликом выбирает `Экспортировать в Markdown` в этом же контексте. Если выбран frame или target, page-level Markdown menu item не используется как fallback. Если Chrome не отдаёт внутренний фрейм через `Page.getFrameTree`, команда не нажимает неизвестные координаты внутри видимого встроенного фрейма, потому что такой клик может попасть в соседнюю кнопку полноэкранного режима. В этом случае `--download-report-only` завершает одну детерминированную попытку понятной ошибкой, а обычный submit остаётся в безопасном цикле ожидания/обновления вместо случайных кликов. Перед экспортом команда best-effort настраивает путь загрузки текущего запуска через `Page.setDownloadBehavior` и `Browser.setDownloadBehavior`; если расширение отклоняет оба вызова, обычный submit расширяет разрешённый список загрузок до реальной Windows-папки загрузок текущего пользователя. После успешного выбора `Экспортировать в Markdown` обычный submit ищет только новые `.md`, которых не было в baseline-снимке перед кликом экспорта; готовым результатом считается ровно один новый Markdown-файл из управляемой папки либо, только при недоступной настройке пути загрузки, из реальной пользовательской папки загрузок. Режим `--download-report-only` дополнительно проверяет реальную Windows-папку загрузок текущего пользователя, включая перенаправление на другой диск через системный Known Folder `Downloads`, потому что это диагностический режим для уже готового отчёта. Если новых файлов нет, команда продолжает ожидание в рамках текущей попытки; если новых Markdown-файлов больше одного в разрешённых папках текущего режима, команда завершает попытку ошибкой неоднозначности и не выбирает более поздний файл. Если новый Markdown появился только вне разрешённых папок текущего режима, команда завершает попытку ошибкой `E2D-BUILD-AUDIT-SUBMIT-REPORT-DOWNLOAD-FOREIGN` и не начинает новый цикл перезагрузки или повторного экспорта того же отчёта. Режим `--download-report-only` не запускает общий цикл обновления страницы и не повторяет скролл/клики после начатого экспорта: он делает одну детерминированную попытку выгрузки Markdown и либо возвращает файл, либо завершает команду понятной ошибкой. Если фрейм не найден, остаётся тот же DOM-путь через кнопку отчётной карточки `aria-label="Экспорт"` в основном DOM страницы. Команда не копирует cookies, `localStorage` или файлы сессии Chrome, не использует буфер обмена как источник verdict-а, не закрывает и не изменяет уже открытые пользовательские вкладки и закрывает только вкладку, созданную текущим запуском.

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Если текст запроса уже подготовлен отдельным файлом, его можно передать явно:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --message <message-path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Если отчёт уже готов в существующем ChatGPT discussion URL, допускается диагностический режим только для скачивания результата без повторной отправки архива:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --download-report-only --project-url <chat-url> --out <report-path> --browser-backend codex-chrome --screenshots-dir <screenshots-path>
```

Короткий ручной запрос (`prompt`, то есть текст, написанный оператором вместо проверенного текста из архива), отправка без прикреплённого основного аудиторского ZIP и отправка без режима `Глубокое исследование` запрещены. Нельзя заменять вывод `audit package message --zip <path>` фразами вроде `Приложен verified audit package`, пересказом задачи или служебным обрамлением `<task-id> audit rNN`: аудитор должен получить один проверенный основной ZIP и дословный текст запроса из архива без первого Markdown H1.

## Извлечение verdict-а - жёсткое правило

Это операторское правило для чтения результата внешнего аудита из текущего audit chat. Оно защищает приёмку от ошибки, когда `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES` найден в пользовательском запросе, примере формата, истории, боковой панели, обычном превью карточки или частично загруженном тексте страницы.

Финальный verdict = первая непустая строка единственного полного Markdown-файла, который ChatGPT скачивает из открытой отчётной карточки `Углубленный исследовательский отчет` / `Deep research report` через пункт меню `Экспортировать в Markdown`. Всё остальное не verdict.

Порядок чтения:

1. Источник verdict-а - только Markdown-файл, скачанный пунктом `Экспортировать в Markdown`, для единственной отчётной карточки `Углубленный исследовательский отчет` / `Deep research report` в текущем audit chat.
2. Запрещено определять verdict по `main.textContent`, всему `article`, `conversation-turn`, sidebar/history, user/tool messages, обычному превью карточки, counters, `verdictCount`, regex over full page или `matches.at(-1)`.
3. Сначала убедитесь, что ChatGPT не показывает видимую активную кнопку остановки ответа текущей отправки; если она видна, генерация ещё активна, даже когда в истории уже есть старый завершённый отчёт. Для встроенного фрейма используйте CDP-контекст этого фрейма и ищите реальную кнопку экспорта внутри него, потому что основной DOM страницы не видит внутренние кнопки cross-origin фрейма. Root-wrapper фрейм и ровно один внутренний готовый фрейм считаются одной отчётной поверхностью; несколько внутренних готовых фреймов или несколько page-level Deep Research frame ids считаются неоднозначностью. Если фрейм или target выбран, оба клика экспорта выполняются только в выбранном контексте; основной DOM страницы не используется как fallback для пункта Markdown. Если фрейм не найден, используйте основной DOM страницы. В обоих случаях экспорт выполняется одинаково: сначала нажмите DOM-элемент `Экспортировать в Markdown` / `Export as Markdown` через `button.click()`, если меню уже открыто; если пункта нет, нажмите DOM-элемент `button[aria-label="Экспорт"]` / `button[aria-label="Export"]` с иконкой скачивания через `button.click()`, затем нажмите DOM-элемент `Экспортировать в Markdown` / `Export as Markdown` через `button.click()`. Не вычисляйте координату правого верхнего угла карточки и не кликайте по неизвестной точке внутри фрейма. Если найдено несколько готовых фреймов, несколько отчётных карточек, несколько export-кнопок или несколько пунктов Markdown, не выбирайте первый кандидат: это неоднозначность источника. Если кнопка экспорта или пункт Markdown недоступны, не нажимайте экспорт и продолжайте минутный цикл. После клика обычный submit должен дождаться ровно одного нового `.md` в разрешённой папке текущего режима, прочитать этот файл, пропустить его через строгую проверку финального отчёта и сохранить полный текст отчёта дословно и без сокращения. Управляемая папка загрузки является основным разрешённым местом; реальная пользовательская папка загрузок становится разрешённой только если расширение Chrome отклонило настройку управляемой папки до отправки. Если новый Markdown появился только вне разрешённых папок текущего режима, остановитесь с ошибкой `E2D-BUILD-AUDIT-SUBMIT-REPORT-DOWNLOAD-FOREIGN`: повторная перезагрузка страницы или повторный клик экспорта в этом состоянии запрещены. Если новых Markdown-файлов больше одного в разрешённых папках текущего режима, остановитесь с ошибкой неоднозначности.
4. Первая непустая строка сохранённого текста должна быть ровно `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`.
5. Текст должен содержать обязательные секции `TASK_ASSESSMENT:`, `BLOCKERS:`, `EVIDENCE_REVIEW:`, `RISKS_AND_NOTES:` и `CLOSURE_DECISION:` в таком порядке.
6. Первый единственный Markdown-файл, который прошёл строгую проверку и получен при неактивной генерации, является готовым отчётом; команда не выполняет второй export только для подтверждения того же текста.
7. Если первая непустая строка не verdict, найдено несколько отчётных карточек или несколько candidate report texts, текст начинается с `Вы сказали`, извлечение включает пользовательский запрос, историю, боковую панель, обычное превью карточки или сообщения инструментов, либо была сжатая передача контекста (`context compaction`) или прерывание (`interruption`), остановитесь: не закрывайте задачу и перечитайте источник заново.
8. `ACCEPT` валиден только если первая строка полного текста отчётной карточки ровно `VERDICT: ACCEPT`, секция `BLOCKERS` не содержит `B1`..`Bn`, а `CLOSURE_DECISION` явно разрешает закрытие. Любой `VERDICT: ACCEPT` из prompt-инструкций, примеров формата, предыдущих сообщений, истории или превью карточки не имеет силы.

Если оператор не может получить Markdown-файл через `Экспортировать в Markdown` отчётной карточки, задача остаётся открытой до повторного чтения источника. Нельзя заменять это правило счётчиком совпадений, предварительным просмотром, regex over full page или любым агрегированным текстом страницы.

## Cleanup

Audit ZIP, сопровождающий ZIP операторских доказательств, временные каталоги и сырые доказательства не коммитятся. После `VERDICT: ACCEPT` рабочие архивы штатно удаляются вручную или будущей командой:

```bash
dotnet run --project eng/Electron2D.Build -- audit cleanup --task <task-id>
```

`audit package` не создаёт финальный ZIP после принятия задачи; новый архив нужен только для следующей итерации внешнего аудита.

Закрытие чужих или оставшихся от старой сессии браузерного протокола (`browser session`) вкладок больше не считается частью реализации: этот путь признан невозможным для детерминированного C#-инструмента через прямой протокол расширения, поэтому команда не использует его и не обещает очистку уже открытых вкладок пользователя.

Ручная сборка ZIP для внешнего аудита запрещена. Внешний аудит допускается только после успешной пары команд `audit package` и `audit package verify` на отдельной чистой копии исходной ревизии; штатная отправка и извлечение отчёта выполняются через `audit submit`, который использует проверенный основной ZIP, текст из `audit package message --zip <path>` и режим `Глубокое исследование`, но не прикрепляет сопровождающий ZIP операторских доказательств.

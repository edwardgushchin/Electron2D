# Детерминированный `audit package`

Обновлено: 2026-07-11.

Этот файл является доменным документом для внутренней команды сборки аудиторского архива. Документ фиксирует ожидаемый контракт, текущую модель реализации, ограничения и проверки в одном месте.

## Ведение документа

- Перед изменением команды `audit package` сначала обновите этот документ.
- Затем добавьте точечные тесты в `RepositoryBuildToolTests`, реализуйте изменение в `eng/Electron2D.Build` и снова синхронизируйте этот документ, если фактическое поведение отличается от контракта.
- Доказательства аудита не собираются вручную: архив создаёт только C#-команда репозитория, а внешний аудит допускается только после проверки восстановления.

## Назначение

`audit package` заменяет ручную сборку ZIP-архива для внешнего аудита. Команда создаёт воспроизводимый основной архив из явной конфигурации, импортирует заранее выполненные preflight-доказательства (`preflight evidence`, то есть файлы вывода уже запущенных проверок), строит patch штатным `git diff --find-renames --binary` через временный индекс Git и проверяет, что чистая копия репозитория восстанавливается из исходной ревизии (`baseline`) до ожидаемого состояния. `audit package` и `audit package verify` не должны повторно запускать тестовые наборы, которые уже прошли отдельным локальным коридором перед упаковкой. Текущий контракт не вкладывает в основной ZIP снимок исходной ревизии: чистая копия исходной ревизии является обязательным внешним входом проверки восстановления (`external baseline REQUIRED input`), а отсутствие такого вложенного набора файлов не является требованием контракта (`no baseline payload requirement`). Чтобы отсутствие вложенного снимка не превращалось в неподтверждённое допущение, команда после финальной записи основного ZIP создаёт сопровождающий ZIP `<task-id>-audit-rNN.operator-workflow.zip` с `operator workflow evidence` - переносимыми сырыми доказательствами запуска `audit package verify` и `audit package message` для неизменяемого основного ZIP.

Инструмент является внутренней автоматизацией репозитория. Он не является публичной командой `e2d`, не входит в пакет среды выполнения и не должен публиковаться как часть релиза.

## Границы правил и запросов

`AGENTS.md` задаёт правила работы агента внутри репозитория: безопасность, управление `.taskboard` только через `e2d tasks`, ведение дневника, проверки, сборку аудиторского архива и обязательный путь внешней приёмки. Он может ссылаться на этот доменный документ, но не должен описывать ручные клики, чтение страницы ChatGPT, работу с cookies или альтернативный браузерный способ вместо `audit submit`.

`docs/release-management/AUDIT-REQUEST.md` является только текстом для внешнего аудитора. Он описывает, какие материалы из одного приложенного audit ZIP нужно проверить и какой финальный отчёт вернуть. Этот файл не описывает Chrome, вкладки, cookies, локальную проверку ZIP, контрольные суммы, ручное скачивание отчёта или операторский порядок отправки.

`.codex/prompts/goal-prompt.md` является коротким active goal prompt-ом оркестратора и должен оставаться не длиннее 3500 символов. Он задаёт цель запуска одной компактной формулировкой и ссылается на workflow-модель вместо копирования её шагов.

`.codex/prompts/goal-task-workflow.md` является локальной workflow-моделью goal-задачи. Он описывает goal-цикл только на уровне состояний и запуска штатных C#-команд: предварительное чтение контекста, работу исполнителя, проверки, `audit package`, `audit package verify`, `audit submit`, обработку сохранённого verdict-а, повторный цикл исправлений, принятие и commit без push. Он не является инструкцией по кликам, полям, вкладкам или другим мелким действиям. Все детали браузера, ordinary ChatGPT-отправки, прикрепления ZIP, чтения ответа и закрытия собственной вкладки принадлежат только C#-команде `audit submit` и этому документу как контракту реализации инструмента.

Любая задача, которую агент доводит до принятия после введения этого правила, остаётся открытой до внешнего отчёта `VERDICT: ACCEPT`, полученного штатной командой `audit submit` и сохранённого в `docs/verdicts/<domain>/<task-id>-audit-rNN.md`. Prompt-ы и агентские правила не должны принимать `VERDICT: ACCEPT` из текста запроса, примера формата, истории чата, обычного предпросмотра или ручного чтения страницы.

## Экономика локального цикла

Внешний аудит является приёмочным барьером, а не инструментом отладки локальной автоматизации. Он должен отвечать на вопрос «можно ли принять проверяемый пакет?», а не искать следующий дефект в прикреплении ZIP, восстановлении архива, ordinary copy action, legacy read-only recovery или распознавании старого отчёта. До создания нового пакета командой `audit package` должны быть закрыты локальные причины отказа, которые можно проверить без внешнего исследования.

Для локальной работы используются три уровня проверок:

- `Fast` - быстрый уровень после мелкой правки. Цель - проверить новый регрессионный тест и ближайший контракт за 1-2 минуты. Этот уровень не должен запускать `audit package`, `audit package verify`, восстановление чистой копии, браузерную отправку, загрузку Markdown-отчёта или широкие фильтры интеграционных тестов без прямой необходимости.
- `Medium` - средний уровень перед созданием пакета. Цель - доказать затронутый внутренний механизм через поведенческий тест, релевантный точечный набор тестов, документационные проверки, `verify audit-followups` при изменении отчётов или заметок о закрытии, проверку лицензий при изменении исходных файлов и `git diff --check`. Ориентир - до 10 минут; если средний уровень регулярно выходит за этот бюджет, это отдельный долг процесса.
- `Heavy` - тяжёлый приёмочный барьер перед реальной отправкой: представительский набор проверок, который реально создаёт audit ZIP, запускает `audit package verify` на отдельной чистой копии, проверяет `audit package message` и сохраняет обязательные `operator workflow evidence`, то есть доказательства запуска операторских команд. Этот уровень запускается только после зелёного `Medium` и не используется после каждой мелкой правки. Полный исчерпывающий набор всех `AuditTier=Heavy` сценариев остаётся отдельным срезом `audit-exhaustive` для изменений самой упаковки, восстановления, ZIP-структуры, сканирования секретов или перед ручным глубоким разбором качества аудиторского инструмента; обычный локальный и CI-коридор не должен платить за этот набор после каждой правки.

Штатная команда быстрого уровня для аудиторских контрактов - `dotnet run --project eng/Electron2D.Build -- verify audit-contracts`. Она читает `docs/release-management/AUDIT-REQUEST.md`, этот документ, `.codex/prompts/goal-prompt.md`, `.codex/prompts/goal-task-workflow.md`, `docs/repository/agent-workflow.md`, `AGENTS.md` и in-process проверяет обязательные маркеры, классификацию отчётов и контракт разбора `RISKS_AND_NOTES`. Команда не создаёт audit ZIP, не распаковывает ZIP, не создаёт чистую копию репозитория, не вызывает `audit package verify`, не запускает браузерную автоматизацию и не вызывает дочерние `dotnet run`. Ожидаемый объём - 10-15 лёгких проверок, ожидаемое внутреннее время выполнения - до 5 секунд, допустимый верхний предел для обычного операторского запуска через `dotnet run` после сборки - 30 секунд. Превышение этого бюджета без явного объяснения означает регрессию быстрого уровня.

Итоговая диагностика `verify audit-contracts` должна содержать человекочитаемую сводку: `AuditTier=Fast`, число проверок, сколько прошло, сколько упало, сколько пропущено, длительность и явное подтверждение, что тяжёлый путь не запускался. Эта сводка нужна для записи в дневник и комментарий задачи через `e2d tasks comment add` в числовом виде, а не общей фразой "точечные проверки прошли".

Интеграционные тесты аудиторской автоматизации в `RepositoryBuildToolTests` должны иметь ровно одну метку `[Trait("AuditTier", "...")]` со значением `Fast`, `Medium` или `Heavy`. `Fast` разрешён только для проверок исходного текста, документов и внутренних контрактов без дочерних процессов; защитный тест `AuditWorkflowFastAuditTierDoesNotUseHeavyHelpers` запрещает в таких тестах вспомогательные методы, которые запускают `dotnet`, Node, `audit package`, `audit package verify`, чистую копию репозитория или браузерную автоматизацию. `Medium` разрешает точечные интеграционные проверки и контролируемые дочерние процессы, но защитный тест `AuditWorkflowMediumAuditTierDoesNotUseHeavyPackagingHelpers` запрещает helper-ы упаковки, восстановления чистой копии, `audit package verify` и `audit package message`. Для части средних и тяжёлых тестов используется дополнительная метка `[Trait("AuditCadence", "Acceptance")]`: она означает не новый уровень качества, а частоту запуска, то есть короткий представительский набор, который должен проходить перед внешней упаковкой и в CI. Штатные срезы `dotnet run --project eng/Electron2D.Build -- test --integration-slice audit-medium` и `dotnet run --project eng/Electron2D.Build -- test --integration-slice audit-heavy` выбирают соответственно `AuditTier=Medium&AuditCadence=Acceptance` и `AuditTier=Heavy&AuditCadence=Acceptance`. Поэтому обычный `audit-medium` не должен быть полным набором из всех `AuditTier=Medium` tests и не должен быть дольше heavy acceptance gate; полный средний набор запускается только явно через `audit-medium-exhaustive`, а срез `audit-exhaustive` выбирает полный `AuditTier=Heavy` без ограничения по частоте запуска. Representative medium tests не должны содержать theory-строки, которые запускают или проверяют `audit-heavy`/`audit-exhaustive`: такие строки относятся к heavy acceptance, иначе `audit-medium` начинает платить за проверку тяжёлого маршрута. Совместимый срез `audit-package` остаётся объединением `AuditTier=Medium|AuditTier=Heavy`, но не используется как основной локальный коридор. При запуске `test --no-build` команда передаёт в дочерний `dotnet test` переменную `ELECTRON2D_BUILD_TOOL_NO_BUILD=1`, чтобы helper-ы внутри integration tests запускали `eng/Electron2D.Build` через `dotnet run --no-build` и не пересобирали build tool десятки раз в широком срезе. Каждый аудиторский срез `test --integration-slice audit-*` печатает `E2D-BUILD-TEST-SLICE-SUMMARY` с именем команды, именем среза, `AuditTier`, числом дочерних процессов, распознанным числом тестов, длительностью, тайм-аутом и причиной сбоя или `failure=none`. Успешный аудиторский срез не имеет права печатать `tests=unknown`, `passed=unknown`, `failed=unknown` или `skipped=unknown`: если `dotnet test` завершился успешно, но числовую сводку не удалось распознать из фактического вывода, команда должна завершиться ошибкой с явным `failure=test-counts-unavailable`. Широкие `FullyQualifiedName~AuditPackage`/`AuditSubmit`/`AuditWorkflow` не являются tier-контрактом. Срез `repository-tooling` исключает все три audit-уровня, чтобы не смешивать обычные проверки build-tool с аудиторским циклом.

`audit package` после создания operator workflow sidecar обязан печатать `E2D-BUILD-AUDIT-OPERATOR-WORKFLOW-SUMMARY`: какие sidecar-команды реально запускались, их коды выхода, длительность каждой команды, общий `totalDurationMs`, тайм-аут и путь сопровождающего ZIP. Эта диагностика не заменяет evidence-файлы внутри sidecar, а даёт оператору числовую стоимость оставшейся тяжёлой части прямо в выводе команды. Сама sidecar-проверка выполняет только `audit package verify` и `audit package message` для готового ZIP; она не запускает `dotnet test`, `eng/Electron2D.Build test` или другие тестовые наборы повторно. `audit package` не должен запускать отдельную вторую clean-verify проверку того же ZIP перед созданием sidecar: единственный обязательный clean-verify pass для готового ZIP выполняется operator workflow subprocess-ом и сохраняется как evidence.

Перед `Heavy` у оператора должна быть краткая матрица доказательств (`proof matrix`) - проверяемая таблица или список вида «замечание или локальный отказ -> изменённый рабочий путь кода или стабильный внутренний драйвер/тестовая обвязка -> имя теста -> команда или фильтр -> ожидаемые доказательства». Если заявленное закрытие blocker-а или локального отказа не связано с поведенческим тестом либо стабильным внутренним контрактом, пакет не считается готовым к созданию.

Для исправительной итерации с непустым `metadata.previousVerdictChain` список `metadata.blockerClosureList` обязателен. Каждая запись закрытия должна называть хотя бы один фактически настроенный `checks[].name` или `preflightChecks[].name` из текущего `audit package` config, который даёт локальное доказательство закрытия. Для существующих прошлых `VERDICT: NEEDS_FIXES` отчётов из `previousVerdictChain` список также должен покрывать каждый найденный blocker: запись закрытия должна назвать путь прошлого отчёта, идентификатор blocker-а вида `B1`, `B2` и имя configured/preflight check текущего пакета. Если существующий прошлый отчёт `VERDICT: NEEDS_FIXES` не содержит в секции `BLOCKERS` распознаваемые `B*` blocker ids, упаковка должна завершиться диагностикой `E2D-BUILD-AUDIT-CONFIG-INVALID`: такой отчёт нельзя считать отсутствием прошлых blocker-ов, потому агент не сможет доказуемо закрыть их пачкой. Это не валидация Markdown-отчёта внешнего аудитора и не требование писать отчёт машинным форматом: проверяется только операторская конфигурация пакета до отправки. Если прошлый blocker закрыт только свободной фразой без связи с configured/preflight check, без пути отчёта или без идентификатора blocker-а, `audit package` должен завершиться диагностикой `E2D-BUILD-AUDIT-CONFIG-INVALID` до запуска внешнего аудита.

Защита от локальных machine paths применяется ко всему текстовому содержимому архива, включая `repo-after/**`, `repo-before/**`, metadata, evidence, request, manifest, checksums и все строки patch. Исторический baseline не является исключением: если исходная ревизия содержит machine-local absolute path в файле, который должен войти в audit scope, сначала нужен отдельный безопасный baseline без этого пути. Подмена или редактирование `repo-before/**` запрещены, потому baseline snapshots должны оставаться полными и дословными.

Обычный исправительный цикл внешнего аудита имеет бюджет внимания: первая отправка без прошлых отчётов, вторая отправка с одним прошлым verdict-отчётом и третья отправка с двумя прошлыми verdict-отчётами. Если для новой упаковки требуется больше двух записей в `metadata.previousVerdictChain`, задача не обрывается, но `audit package` переводит цикл в стабилизационный режим: среди configured или preflight checks должен быть отдельный check с именем `audit-loop-stabilization`. Этот check должен быть локальной пакетной проверкой классов причин, которые уже проявились во внешних отчётах, например driver/harness-тестом для browser DOM path, clean-control context, payload attachment guard, report extraction или restore/package model. В нормальном маршруте он выполняется до упаковки и попадает в `preflightChecks[]` как готовые evidence-файлы, а не запускается повторно внутри `audit package`. Без такого check-а новая упаковка завершается `E2D-BUILD-AUDIT-CONFIG-INVALID`. Смысл правила - не запретить работу после `r03`, а не позволить продолжать `r04`, `r05` и далее как слепую серию внешних отладочных запусков.

Доказательство по тексту исходника (`source-level proof`), то есть тест, который читает C#-файл как текст и ищет строку или порядок строк, не доказывает поведение `audit submit`, DOM-выражений, ordinary copy action, legacy Markdown-export, ожидания отчёта, восстановления архива или сохранения verdict-а. Такой тест допустим только для статического документационного маркера или явного запрета текста, но не для закрытия поведенческого blocker-а.

## Правило остановки (`stop-loss`) для повторяющихся отказов

Класс локального отказа (`failure class`) - это одинаковая причина на одном участке цикла: одинаковый диагностический код, например `E2D-BUILD-AUDIT-SUBMIT-DEEP-RESEARCH-MISSING`, `E2D-BUILD-AUDIT-SUBMIT-REPORT-EXPORT-MISSING`, `E2D-BUILD-AUDIT-SUBMIT-REPORT-STALE`, `E2D-BUILD-AUDIT-RESTORE-MISMATCH`, либо одинаковый этап и смысловая причина, если диагностический код ещё недостаточно точен. Неудачная попытка, которая не создала сохранённый verdict-отчёт, не является полезной внешней итерацией.

Если один класс локального отказа повторился два раза в одной задаче или в одном цикле аудита (`audit-loop`), запрещено собирать или отправлять следующую `rNN` «на удачу»: нужно не увеличивать `rNN`, а остановить цикл упаковки и отправки. Сначала нужно зафиксировать класс отказа комментарием через e2d tasks comment add и в дневнике, добавить локальный регрессионный тест через внутренний драйвер или тестовую обвязку для рабочего пути кода либо стабильного внутреннего контракта, пройти зелёный `Fast`, затем зелёный `Medium`. Только после этого разрешён новый `Heavy`.

Если локальный драйвер или тестовая обвязка для повторяющегося класса отказа невозможны, оператор не должен продолжать увеличивать `rNN`. Нужно явно записать blocker процесса: какая часть не моделируется локально, какие данные нужны, и почему следующий внешний запуск не будет слепым повтором той же попытки.

## Заморозка области цикла аудита

После первого `audit package` или `audit submit` для задачи область пакета заморожена. Нельзя добавлять новые активные задачи, менять несвязанные prompt-ы, соседние документы, дорожную карту, инструменты сборки или критерии приёмки, если это не закрывает сохранённый blocker текущей итерации или явно не включено в `metadata.scopeTaskIds`, `metadata.scopeSummary`, manifest и матрицу доказательств как объединённая область (`combined scope`).

Новые идеи, улучшения процесса и соседние исправления после старта цикла аудита записываются как отдельная задача, запись дневника или `accepted-risk`, но не попадают в текущий пакет без явной объединённой области. Если объединённая область всё же нужна, она должна быть заявлена до упаковки и стать предметом текущей внешней проверки.

## После primary `VERDICT: ACCEPT`

После сохранённого primary `VERDICT: ACCEPT` принятую область нельзя менять до control audit. Запрещены изменения кода, тестов, доменных документов, доказательств, критериев и конфигурации пакета, которые меняют уже принятую область. Разрешены только сохранение отчёта, локальное состояние отправки, сборка отдельно проверенного чистого control ZIP без прошлого контекста, отправка control audit в новый чат проекта и учётное закрытие, которое не меняет принятую область.

Перед любым изменением после primary `ACCEPT` оператор сначала классифицирует `RISKS_AND_NOTES`. Если замечание можно закрыть через CLI-комментарий задачи со state duplicate, accepted-risk, tracked-existing, tracked-new или not-actionable или другое учётное решение без изменения принятой области, код и документы не меняются. Если замечание действительно требует изменения кода, тестов, документов, доказательств или критериев принятой области, право на control audit сбрасывается: нужен новый пакет и новый primary audit.

## Контракт CLI

После создания reservation единая error boundary охватывает browser automation, строгую проверку task/iteration отчёта, создание output-каталога, запись Markdown и финальный переход в `completed`. Любая ошибка на этом участке сначала атомарно переводит попытку в `failed` с точными `failureCode`, `failureMessage` и `completedAtUtc`, затем возвращается вызывающему процессу.

Перед первым `DOM.setFileInputFiles` инструмент получает типизированное состояние composer. Side effect разрешён только для единственного текущего composer с пустым или точным ожидаемым prompt и без вложений. Точный уже готовый payload переиспользуется без повторной установки. Посторонний текст, чужое или неоднозначное вложение, а также ожидаемый ZIP при недопустимом тексте приводят к fail-closed отказу до поиска file input.

Получение ordinary Markdown не требует `navigator.clipboard.readText` или browser clipboard permission. Pre-load capture регистрируется через CDP до навигации и обязан подтвердить готовность уже в загруженном документе; поздний hook не считается достаточным доказательством, потому что обработчик страницы мог сохранить раннюю ссылку на Clipboard API. До единственного copy-click инструмент также обязан доказать структурную принадлежность единственной видимой `button[data-testid="copy-turn-action-button"]` последнему assistant turn текущей отправки. На Windows изменение системного clipboard доказывается сменой native clipboard sequence number; captured `writeText`, `write` или `copy` payload остаётся вторым source-aware доказательством. Если ни один доказанный источник не дал Markdown, команда завершается без второго клика.

Каждая вкладка, созданная `audit submit`, `--download-report-only` или `--dump-dom-only`, безусловно закрывается в `finally` после успешного получения результата либо окончательной ошибки/отмены. Защищённая область начинается до `CreateTabAsync`: если удалённая сторона успела создать вкладку, но ответ `createTab` потерян, session-level `finalizeTabs` всё равно выполняется ровно один раз и не требует известного `tabId`. Recoverable `Debugger unattached` во время ordinary generation polling не считается окончательной ошибкой: инструмент восстанавливает CDP на той же вкладке и продолжает ожидание до общего timeout, не выходя в `finally`. Callback сохранения `tabId` выполняется внутри этой защищённой области. `finalizeTabs` передаёт пустой список `keep`; disconnect или protocol failure при финализации не проглатывается и завершает команду диагностикой `E2D-BUILD-AUDIT-SUBMIT-TAB-FINALIZATION` либо исходной protocol/transport ошибкой. Handoff и диагностическое сохранение вкладки после завершения не входят в детерминированный workflow; пользовательские вкладки инструмент не закрывает.

Быстрая проверка аудиторских контрактов:

```bash
dotnet run --project eng/Electron2D.Build -- verify audit-contracts
```

Сборка архива:

```bash
dotnet run --project eng/Electron2D.Build -- audit package --task <task-id> --iteration rNN --baseline <sha> --config <path> --out .temp/audit
```

Проверка уже собранного архива на чистой копии:

```bash
dotnet run --project eng/Electron2D.Build -- audit package verify --zip <path> --baseline <sha> --repo <clean-repo-path>
```

Подготовка текста сообщения для внешнего аудита после успешной проверки:

```bash
dotnet run --project eng/Electron2D.Build -- audit package message --zip <path>
```

Отправка проверенного архива во внешний аудит через Electron2D audit-проект ChatGPT:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --out <report-path> --browser-backend codex-chrome
```

Та же команда используется для каждой следующей итерации. По последнему сохранённому verdict-отчёту C#-инструмент сам выбирает reuse текущего обсуждения либо новый clean-control чат; агент не выбирает `--reuse-conversation`, `--control-audit`, conversation URL или вкладку.

Повторная отправка новой итерации в новый чат проекта, если сохранённое обсуждение задачи больше нельзя продолжать штатно:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --new-conversation --zip <path> --out <report-path> --browser-backend codex-chrome
```

Новые отправки используют только ordinary ChatGPT. Параметра `--deep-research` нет в публичном CLI-контракте: parser отклоняет его как неизвестную опцию до подключения к браузеру. Deep Research не является резервным audit-route Electron2D.

Контрольный независимый аудит после `VERDICT: ACCEPT` запускается той же командой с clean-control ZIP следующей глобальной итерации. Control route и новый чат выбирает C#-инструмент по сохранённому состоянию.

Read-only восстановление уже готового primary или control отчёта после разрыва browser/CDP-связи:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --download-report-only --project-url <saved-primary-url> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
dotnet run --project eng/Electron2D.Build -- audit submit --download-report-only --project-url <saved-control-url> --out docs/verdicts/<domain>/<task-id>-audit-control-rNN.md --browser-backend codex-chrome
```

Проверка закрытия actionable findings из сохранённых отчётов внешнего аудита:

```bash
dotnet run --project eng/Electron2D.Build -- verify audit-followups
```

Номер `rNN` является общим порядковым номером начатой отправки ZIP для задачи. Primary и control никогда не используют один номер: `r01` начинает новый primary-чат, каждый `NEEDS_FIXES` продолжает тот же чат следующим номером, а каждый `ACCEPT` требует следующую итерацию в новом clean-control чате. Если clean control возвращает `NEEDS_FIXES`, следующая обычная исправительная итерация продолжает именно control-чат с reservation `route=reuse`; её `ACCEPT` не является финальным clean-control и требует ещё один новый control с reservation `route=clean-control`. Задача закрывается только после полного валидного `ACCEPT`, для которого matching reservation имеет `route=clean-control` и `status=completed`; corrective control `ACCEPT` с `route=reuse`, усечённый файл или reservation `report-received`/`failed` не закрывают цикл.

Verdict Markdown публикуется атомарно и неизменяемо. Команда сначала записывает UTF-8 payload во временный файл в том же каталоге, flush-ит его и затем выполняет один `File.Move` без overwrite. Ошибка или отмена удаляет временный файл и не оставляет видимого конечного verdict-а. Если конечный путь уже существует, команда отклоняет запись с `E2D-BUILD-AUDIT-SUBMIT-REPORT-WRITE` и не изменяет существующий файл. При построении следующего route `ReadSavedAuditVerdicts` повторно требует полный строгий report contract и совпадающие `metadata.taskId`/`metadata.iteration`; одной первой строки `VERDICT: ACCEPT` недостаточно.

До первого обращения к браузеру `audit submit` атомарно создаёт локальный машиночитаемый reservation-файл `.temp/audit/<task-id>/submit-attempt-rNN.json`. Reservation фиксирует identity ZIP, выбранный route, `browserSessionId`, `browserTurnId`, время начала, текущий статус, созданный `tabId`, conversation URL и финальную диагностику попытки и навсегда занимает номер итерации, даже если browser/CDP оборвался до Send, conversation sidecar не появился или verdict не был сохранён. Переходы `reserved -> browser-started -> submitted -> report-received -> completed` и `* -> failed` перезаписываются атомарно; `failed` обязан хранить точные `failureCode`, `failureMessage` и время завершения. Тот же ZIP и тот же `rNN` повторно не отправляются. Следующий запуск вычисляет глобальный номер по максимуму сохранённых verdict-ов и reservation-файлов. Если failed reservation не имеет ни `tabId`, ни conversation URL, browser side effect не был подтверждён и следующая глобальная итерация сохраняет route последнего полного verdict-а; то же относится к доказанному `E2D-BUILD-AUDIT-SUBMIT-COMPOSER-STATE`, который происходит до `DOM.setFileInputFiles` и Send. Остальные reservation без verdict-а переводят следующую итерацию в свежий primary-чат. Read-only recovery может сохранить verdict уже зарезервированной итерации и тем самым вернуть обычный переход по verdict-у. `--new-conversation` допускает пропуск ровно одного ожидаемого номера только как аварийную миграцию попытки, случившейся до появления reservation-модели и уже зафиксированной в CLI-комментарии задачи и дневнике; флаг не может обходить финальный clean-control `ACCEPT`.

Контрольный аудит после primary `VERDICT: ACCEPT` отправляет отдельно проверенный чистый контрольный ZIP для той же задачи, следующей rNN и той же принятой области. C#-инструмент всегда начинает этот route с URL проекта ChatGPT и создаёт новый чат; существующий conversation URL или primary-обсуждение для него запрещены. Такой ZIP собирается только штатной командой `audit package`, затем проходит `audit package verify`, но его входная конфигурация должна иметь пустые `metadata.previousVerdictChain` и `metadata.blockerClosureList`, а модель файлов репозитория не должна включать сохранённые Markdown verdict-отчёты под `docs/verdicts/`. Локальный индекс документации `data/documentation/**` не должен индексировать `docs/verdicts/**`; активные контекстные артефакты контрольного ZIP, включая `AUDIT-MANIFEST.md`, `AUDIT-REQUEST.md`, `repo-after/data/documentation/**`, `evidence/**`, `metadata/**`, `repo-file-hashes.json`, `SHA256SUMS.txt` и добавленные строки patch для контекстных файлов, не должны содержать конкретные Markdown-пути сохранённых verdict-отчётов под `docs/verdicts/`, включая нестандартные имена файлов. Mutable process-history files, включая `.taskboard/**`, legacy `TASKS.md` и `data/dev-diary/**`, не входят в clean-control ZIP как repo-owned content: их история содержит предыдущие verdict paths, operator notes и summaries, поэтому контрольный пакет должен переносить принятую область через metadata, domain docs, code/tests и generated evidence без этих ledger-файлов. Обычные repo-owned source/test files под `repo-after/**` могут содержать синтетические fixture paths для проверки самого audit tooling, если эти пути не попали в manifest, request, generated docs, evidence, metadata или repo-file model как контекст старых внешних отчётов. Исторические снимки `repo-before/**`, а также удалённые и контекстные строки patch являются доказательством исходного состояния diff-а, поэтому обычные ссылки из старого содержимого там не считаются текущим контекстом; при этом прямые entries `repo-before/docs/verdicts/**`, metadata-ссылки на `docs/verdicts/**`, snapshot index для сохранённых verdict-отчётов и добавленные строки patch для context-bearing paths остаются запрещены. Чистый контрольный ZIP может отличаться от primary ZIP только удалением контекста прошлых внешних отчётов; код, документы, тесты, доказательства и критерии принятой области менять нельзя. Если между primary `ACCEPT` и control audit изменились код, документы, тесты, доказательства или критерии, нужен новый package и новый primary audit. Между primary `ACCEPT` и control audit разрешены только сохранение verdict report, локальное состояние отправки и сборка чистого контрольного ZIP без прошлых отчётов. Если control audit возвращает `VERDICT: NEEDS_FIXES`, задача остаётся открытой; следующий package должен включить control verdict в `metadata.previousVerdictChain`, закрыть его blocker-ы через `metadata.blockerClosureList`, а очередной `audit submit` автоматически продолжает сохранённый control-чат.

`audit package` создаёт файл `<task-id>-audit-rNN.zip` и сопровождающий файл `<task-id>-audit-rNN.operator-workflow.zip` в каталоге `--out`. Если любой из этих файлов уже существует, команда завершается ошибкой. Перезапись разрешена только через явный флаг `--force`; в таком случае диагностическое сообщение фиксирует, что перезапись была разрешена.

Если `--config` не передан, команда читает конфигурацию из `.temp/audit/<task-id>/audit-package.input.json`. Явный `--config` остаётся рекомендуемой формой для внешнего аудита, потому что путь входного файла виден в команде воспроизведения.

`audit package verify` распаковывает основной архив во временный каталог, проверяет контрольные суммы, структуру архива, применимость patch, восстановленные SHA-256 файлов репозитория, `metadata/repo-file-snapshots.json` и защитные правила. Та же команда автоматически требует соседний сопровождающий ZIP `<main-name>.operator-workflow.zip`: если файл отсутствует, не проходит детерминированные правила ZIP, не покрыт `SHA256SUMS.txt`, содержит неполный набор `payload/` или операторских доказательств, ссылается на другой SHA-256 основного ZIP, хранит неверные хэши `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или списка записей основного архива, либо содержит несогласованные метаданные команд, проверка завершается отказом. Команда изменяет переданную чистую копию, поэтому `--repo` должен указывать на отдельную рабочую копию, предназначенную только для проверки восстановления, а основной ZIP и соседний operator-workflow ZIP обязаны находиться вне этой копии. До изменения Git config, `reset` или `clean` команда проверяет исходный `git status --porcelain --untracked-files=all`; любая tracked или обычная untracked работа приводит к fail-closed `E2D-BUILD-AUDIT-REPO-DIRTY` без изменения файлов, а архив внутри `--repo` отклоняется как `E2D-BUILD-AUDIT-REPO-NOT-ISOLATED`. В этой рабочей копии команда сама выставляет локальные настройки Git `core.autocrlf=false` и `core.eol=lf`, проверяет доступность исходной ревизии `--baseline`, возвращает дерево к этой ревизии, удаляет только отслеживаемые Git файлы (`tracked files`) из списка `git ls-files -z`, снова материализует их через `git reset --hard <baseline>`, удаляет игнорируемые результаты сборки и другие игнорируемые рабочие файлы, а затем проверяет чистоту дерева. Такой порядок принудительно заменяет ранее созданные CRLF-байты на LF даже тогда, когда Git считал рабочее дерево чистым. После успешного применения patch команда повторно очищает Git-игнорируемые результаты сборки, которые не перечислены в `repo-file-hashes.json`, и только затем сравнивает восстановленный набор файлов с моделью архива; намеренно восстановленные Git-игнорируемые файлы из `repoFileAllowlist` сохраняются. Snapshot index должен совпадать с declared scope из `repo-file-hashes.json` и списком удалённых файлов: файл вне этой области, orphan snapshot под `repo-after/` или `repo-before/`, отсутствующий обязательный snapshot, path traversal, duplicate entry или SHA-256 mismatch отклоняют пакет. Для каждой записи `afterSnapshot` обязан быть точным путём `repo-after/<path>`, `beforeSnapshot` - точным путём `repo-before/<path>`, а `status` вычисляется из наличия и SHA-256 `before`/`after`, а не принимается на доверии из JSON. Если после подготовки дерево всё ещё содержит изменения, диагностика `E2D-BUILD-AUDIT-REPO-DIRTY` обязана включать фактический вывод `git status --porcelain` без абсолютных путей машины. Команда не создаёт исходную ревизию из содержимого ZIP и не требует, чтобы ZIP содержал Git bundle, архив tar или другой снимок исходного дерева Git.

`audit package message` читает уже собранный ZIP, проверяет корневой `AUDIT-REQUEST.md`, `metadata/audit-package.input.json`, идентификатор задачи, номер итерации и имя файла `<task-id>-audit-rNN.zip`. При успехе команда пишет в `stdout` только готовый текст для вставки в ChatGPT: содержимое `AUDIT-REQUEST.md` из архива без первого Markdown-заголовка первого уровня. Успешный вывод не содержит JSON-диагностик, сгенерированного обрамления задачи и итерации, строк вроде `<task-id> audit rNN`, `Приложен verified audit package` или `Ниже дословный текст`. При ошибке команда завершает работу с ненулевым кодом и пишет только структурированную JSON-диагностику: отсутствующий ZIP, отсутствующий `metadata/audit-package.input.json`, отсутствующий корневой `AUDIT-REQUEST.md`, недействительные обязательные маркеры, недействительные `taskId` или `iteration`, а также несоответствие имени ZIP метаданным считаются блокирующими ошибками.

`audit submit` является операторской C#-автоматизацией поверх проверенного audit package. Команда обязана проверить существование основного ZIP и по сохранённым verdict-отчётам вместе с reservation-файлами определить единственный допустимый route: первый primary открывает новый чат, `NEEDS_FIXES` продолжает соответствующий сохранённый чат, `ACCEPT` открывает новый clean-control чат следующей глобальной итерации, а незавершённая попытка без verdict-а переводит следующий номер в свежий primary. Агент не выбирает route, вкладку, conversation URL или браузерные действия и не повторяет уже зарезервированный ZIP. После локального разрешения сообщения и сохранённой ссылки reuse, но до обращения к браузеру, команда атомарно резервирует текущий `rNN`; конфликт reservation завершает запуск до browser side effects. Для нового primary/control C#-команда получает текст сообщения из `--message <path>` или через тот же контракт, что и `audit package message`, открывает одну свежую вкладку Electron2D audit-проекта ChatGPT и прикрепляет ровно один основной ZIP-файл единственным non-retriable `DOM.setFileInputFiles`. Этот вызов сам публикует файл в Web uploader; после него запрещены ручные `input`, `change` и любые другие file events. Команда ждёт готовую ZIP-плашку как при пустом composer-е, так и при автоматически восстановленном точном текущем prompt draft-е, затем при необходимости вставляет полный текст сообщения из корневого `AUDIT-REQUEST.md`. Единственный финальный guard непосредственно перед Send требует одновременно точный prompt и точный ZIP: для prompt разрешено только схлопывание последовательностей whitespace и обрезка краёв без изменения регистра, а имя файла сравнивается как отдельное регистрозависимое значение целиком, не как подстрока текста плашки. Только после этого выполняется один Send. Для автоматически выбранного reuse C#-команда прикрепляет только новый ZIP и не вставляет request-текст повторно. Базовый режим `audit submit` использует обычный запрос в ChatGPT: он не вставляет текстовый маркер `@Глубокое исследование`, не выбирает режим `Глубокое исследование` и не ждёт карточку Deep Research report.

Внутри нового primary/control route C#-команда обязана дождаться доступности поля сообщения без предварительного прокручивания страницы проекта; это проверяемый контракт инструмента, а не ручная инструкция агенту.

После отправки обычного запроса команда запоминает число сообщений до отправки, ждёт новое пользовательское сообщение и новый ответ ассистента текущей отправки, затем работает только с последним видимым `data-message-author-role="assistant"` после этого увеличения счётчика. Поиск verdict-а по всей странице, истории чата, пользовательскому запросу, боковой панели, превью карточек или старым ответам запрещён. До единственного copy-click source-scoped capture обязан быть установлен и очищен, а production selector обязан доказать ровно одну видимую `button[data-testid="copy-turn-action-button"]`, структурным владельцем которой является тот же последний assistant turn. Поддерживаются текущие turn wrappers `data-turn`, `conversation-turn-*` и `article`, но глобальная кнопка страницы или кнопка соседнего ответа не является fallback. Повторный DOM `click()`, второй координатный клик и повтор copy action запрещены.

После copy action Windows path принимает системный clipboard только при смене native clipboard sequence number относительно значения непосредственно перед кликом; предварительная запись sentinel не требуется. Вторым доказанным источником служит Markdown, который сама штатная кнопка передала в перехваченный `navigator.clipboard.writeText()`, `navigator.clipboard.write(...)`, `event.clipboardData.setData('text/plain', ...)` или source-scoped `copy` event. `navigator.clipboard.readText()` и выдача Chrome clipboard permission не используются. Fallback на выделенный текст допускается только если anchor/focus находятся внутри текущего assistant-сообщения; полный `active.value`, `active.textContent` и глобальное выделение запрещены. Собственный DOM-to-Markdown renderer не используется. Ошибка после выполненного copy action завершается локальной диагностикой без повторного клика. Если toolbar ещё не гидратирован, разрешён один `Page.reload` только до первого copy action. Полученный текст проходит строгую проверку verdict headings, текущих `metadata.taskId`/`metadata.iteration` и отсутствия numbered blockers в `ACCEPT`.

Параметр `--deep-research` отсутствует в allowlist команды и отклоняется parser-ом как неизвестная опция до подключения к браузеру с `E2D-BUILD-CLI-INVALID-ARGUMENTS`. `--download-report-only` остаётся диагностическим read-only режимом для уже готового отчёта по конкретному ChatGPT conversation URL и не отправляет ZIP повторно: если последний assistant-ответ уже является строгим verdict-отчётом, команда получает его controlled ordinary copy action. Legacy export/download recovery может прочитать старую отчётную карточку, созданную прежним процессом, но не разрешает новую Deep Research отправку. За пределами controlled ordinary copy action и controlled legacy report export команда не использует произвольный буфер обмена как источник verdict-а и не запускает PowerShell для чтения или записи результата. При любом исходе команда в `finally` закрывает только вкладку, созданную текущим запуском; пользовательские вкладки она не закрывает и не изменяет.

Перед нажатием отправки команда проверяет текущее содержимое поля отправки. В течение короткого окна ожидания для нового чата нормализованный только по whitespace полный текст сообщения должен быть равен ожидаемому с сохранением регистра, а рядом с полем должна быть видна ровно одна плашка точного имени основного audit ZIP. Префикс, суффикс, повтор request-текста, изменение регистра и более длинное имя вроде `<name>.zip.backup` отклоняются. Для автоматически выбранного reuse поле отправки должно оставаться пустым или содержать только пробелы: повторный request-текст, ручная просьба или содержимое `AUDIT-REQUEST.md` в composer-е считаются ошибкой pre-send guard-а. Плашка подтверждается точным структурированным значением имени файла и attachment/file/upload metadata, кнопкой удаления, архивным типом либо тем же единственным именем в `input.files` текущей формы; соседний второй ZIP делает payload неоднозначным. Обычная текстовая метка без совпадающего текущего file input не считается прикреплённым ZIP. Если текстовое состояние поля или ZIP-плашка потеряны перед отправкой, команда останавливается с `E2D-BUILD-AUDIT-SUBMIT-PAYLOAD-MISSING`, добавляет в диагностику последнюю причину отказа и не отправляет сомнительный запрос во внешний аудит.

Legacy Deep Research targets учитываются только в read-only recovery старых обсуждений. Новая отправка не создаёт и не выбирает такие targets. Скачанный при legacy recovery Markdown всё равно проходит строгую валидацию `metadata.taskId`, `metadata.iteration`, обязательных секций и признаков текущего отчёта; старый отчёт должен завершиться как stale/invalid report, а не сохраняться в `--out`.

`verify audit-followups` проверяет, что сохранённые отчёты внешнего аудита не оставили actionable `FOLLOW_UP_FINDING` без tracked closure note. Команда требует каноническую доску `.taskboard/board.e2tasks` и принимает TaskFile v2/v3 из `.taskboard/tasks/*.e2task` и `.taskboard/completed/*.e2task`. Для v3 она читает `description`, Markdown/text-части `conversation.messages[].content`, строковые поля typed `activity[].payload` и `legacySourceFragments[].markdown`; v2 остаётся read-only входом совместимости. Команда также читает все Markdown-отчёты под `docs/verdicts/**/*.md`. В Git-репозитории она объединяет отслеживаемые отчёты с фактическими файлами на диске, чтобы только что сохранённые primary/control reports проверялись ещё до commit. `TASKS.md` и `data/completed-tasks/**` больше не являются источниками verifier-а. Saved audit report считается входом проверки, когда первая непустая строка файла строго равна `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`; остальные Markdown-файлы под `docs/verdicts/` игнорируются как невалидные для closure-verifier входы. Для каждого такого отчёта команда извлекает structured entries из секции `RISKS_AND_NOTES`: `FOLLOW_UP_FINDING F1` является actionable по умолчанию; `OUT_OF_SCOPE_NOTE N1` и `INFO_NOTE I1` не требуют closure без явной пометки `Actionable: true`, `Actionable: yes` или `[actionable]`; текст вида `B1` внутри `RISKS_AND_NOTES` не превращается в blocker сам по себе. Уникальный ключ finding-а - пара `(saved report path, finding id)`, поэтому `FOLLOW_UP_FINDING F1` из primary report и `FOLLOW_UP_FINDING F1` из control report закрываются отдельными notes.

Closure note добавляется через `e2d tasks comment add <task-id> --expected-revision <n> --text <markdown>` и в v3 append-only хранится в `conversation.messages`; прямое редактирование `.e2task` запрещено. Рекомендуемый Markdown payload:

```markdown
- audit-followup-closure:
  - source: docs/verdicts/<domain>/<task-id>-audit-rNN.md
  - id: FOLLOW_UP_FINDING F1
  - state: tracked-new
  - target: <task-id>
  - rationale: кратко, почему выбран именно этот способ закрытия.
```

Обязательные поля closure note: `source`, `id`, `state`, `target`, `rationale`. Поле `state` принимает только `tracked-existing`, `tracked-new`, `accepted-risk`, `duplicate`, `not-actionable` или `promoted-blocker`. Для `accepted-risk` тот же note обязан содержать поля risk-register модели, совместимой с T-0105: `affected area`, `impact`, `likelihood`, `mitigation`, `owner/next decision point`, `decision state`. Если accepted risk не содержит rationale, next decision point или decision state, команда возвращает ошибку. Команда пишет только JSON-диагностики `BuildDiagnostic`; незакрытый finding, неверный closure state или неполный accepted risk завершают проверку ненулевым кодом.

Список `checks` в `audit-package.input.json` и notes задачи должен отражать фактически выполненные команды и их состояние. Нельзя записывать в список `checks` внешнюю упаковку, отправку или проверочный шаг, который не запускался в текущей итерации; локальный `verify audit-followups` указывается только после реального запуска.

Если после ordinary-отправки ChatGPT показывает кнопку остановки ответа, команда считает генерацию активной и продолжает минутный цикл до появления нового assistant-сообщения текущей отправки. Пока генерация активна, streaming-текст и старые ответы не принимаются как готовый verdict. После каждого минутного интервала команда обновляет страницу, ждёт возвращения сообщений обсуждения и затем повторяет controlled copy action. Временный диалог ограничения частоты запросов `Слишком много запросов` / `Too many requests` закрывается видимой кнопкой подтверждения `Понятно` / `OK` и не считается отчётом. Legacy Markdown-export запускается только отдельным read-only режимом `--download-report-only` для уже существующего conversation URL.

Когда активной генерации нет, команда сначала прокручивает обсуждение вниз внутри собственной вкладки и только затем ищет новое assistant-сообщение и его copy-кнопку. До заполнения prompt-а новый чат не прокручивается.

Если `Codex Chrome Extension` временно теряет подключение к странице во время чтения состояния страницы или обновления страницы, backend `codex-chrome` выполняет ограниченное повторное подключение к собственной вкладке: сначала освобождает отладочное подключение этой вкладки через `detach`, затем снова выполняет `attach` и включает нужные CDP-домены. После успешного восстановления команда продолжает тот же цикл ожидания. Действия, которые могут изменить состояние ChatGPT, например выбор пункта меню, прикрепление файла, вставка текста и отправка, автоматически не повторяются после такого сбоя, чтобы не создать двойное действие.

После успешного нажатия отправки ZIP команда обязана записать локальную подсказку восстановления с текущим URL созданного audit-чата. Для primary подсказка хранится как `conversation-url-<iteration>.txt` и стабильный `conversation-url.txt`, для control audit - как `control-conversation-url-<iteration>.txt` и `control-conversation-url.txt` под `.temp/audit/<task-id>/`; эти файлы не входят в audit package. Conversation URL является только локальным состоянием отправки: он не включается в `SHA256SUMS.txt`, `AUDIT-MANIFEST.md`, `repo-file-hashes.json`, `metadata/repo-file-snapshots.json`, evidence, saved verdict files или другие repo-owned документы. В комментарии задачи через CLI можно фиксировать место хранения локального состояния, например `.temp/audit/<task-id>/conversation-url.txt`, но не конкретный ChatGPT URL. Если browser/CDP-связь оборвалась после отправки, оператор может запустить `audit submit --download-report-only --project-url <saved-url> ...` без ручного поиска вкладки и без повторной отправки ZIP; primary/control recovery инструмент выводит из строгого имени `--out`. Режим `--download-report-only` принимает только конкретный ChatGPT conversation URL с сегментом `/c/<conversation-id>`; общий URL проекта, корень ChatGPT, пустой conversation id или ссылка без `/c/` отклоняются до подключения к браузеру.

После сохранённого `VERDICT: NEEDS_FIXES` C#-инструмент автоматически продолжает соответствующее обсуждение: после control `NEEDS_FIXES` он читает `.temp/audit/<task-id>/control-conversation-url.txt`, во всех остальных случаях - `.temp/audit/<task-id>/conversation-url.txt`; `<task-id>` выводится из имени ZIP `<task-id>-audit-rNN.zip`. Отсутствие выбранного файла или общий URL проекта блокируют запуск до подключения к браузеру. Повторная отправка в существующий чат прикрепляет только основной ZIP и не вставляет в composer текст `AUDIT-REQUEST.md`, вывод `audit package message`, пользовательский `--message` или любую другую просьбу. Если браузер восстановил в reuse-composer старый или незавершённый черновик, инструмент до прикрепления ZIP обязательно очищает поле; финальный payload содержит пустой composer и ровно один текущий ZIP. Явный `--message` для автоматически выведенного reuse отклоняется до подключения к браузеру, потому контракт повторной итерации уже задан историей существующего чата и корневым `AUDIT-REQUEST.md` внутри ZIP. ZIP-submit не принимает явный `--project-url`: маршрут и sidecar выбирает только state machine. После отправки исправительная итерация пишет `conversation-url-rNN.txt` и обновляет стабильный `conversation-url.txt`, даже если входным discussion был control sidecar.

`--new-conversation` является явным аварийным путём для primary-итерации, когда сохранённое обсуждение задачи больше нельзя продолжить штатной отправкой, например из-за устойчивого отказа именно этого чата до отправки или получения отчёта. Команда открывает корневой URL проекта из собственной конфигурации и создаёт новый audit chat тем же штатным ordinary ChatGPT способом: прикрепляет ровно один проверенный ZIP, вставляет проверенный текст сообщения и сохраняет новый `conversation-url-rNN.txt` плюс стабильный `conversation-url.txt`. Флаг несовместим с `--reuse-conversation`, `--control-audit`, `--download-report-only` и любым явно переданным `--project-url`, потому он всегда означает новый чат проекта, а не переход в уже существующее обсуждение или чужой адрес. Использовать его можно только после локальной фиксации причины в комментарии задачи через `e2d tasks` и дневнике; предыдущие сохранённые verdict reports всё равно должны быть включены в `metadata.previousVerdictChain` самого ZIP, если это исправительная primary-итерация.

После сохранённого primary/corrective `VERDICT: ACCEPT` следующая глобальная итерация автоматически становится независимым clean control. Команда начинает её из корневого URL проекта и не принимает конкретный conversation URL, чтобы контрольный аудит не получил прошлое обсуждение исправлений. Затем команда проверяет сам ZIP: `metadata.previousVerdictChain` и `metadata.blockerClosureList` должны быть массивами длиной ровно `0`, прямые entries `repo-after/docs/verdicts/**` и `repo-before/docs/verdicts/**` запрещены, `.taskboard/**`, legacy `TASKS.md` и `data/dev-diary/**` запрещены как mutable process-history files, а все строки `repo-file-hashes.json` включая `deletedRepoFiles` и все строки `metadata/repo-file-snapshots.json` не должны содержать сохранённые verdict-отчёты `docs/verdicts/`. После структурной проверки команда сканирует контекстные текстовые артефакты ZIP и отклоняет конкретные Markdown-пути под `docs/verdicts/` в `AUDIT-MANIFEST.md`, `AUDIT-REQUEST.md`, `repo-after/data/documentation/**`, `evidence/**`, `metadata/**`, `repo-file-hashes.json`, `SHA256SUMS.txt` или добавленных строках patch для context-bearing repo paths. Обычные source/test snapshots под `repo-after/**`, исторические `repo-before/**` snapshots и удалённые или контекстные строки patch не считаются текущим verdict context. Нарушение этого правила отклоняется до подключения к браузеру диагностикой `E2D-BUILD-AUDIT-SUBMIT-CONTROL-CONTEXT`, потому контрольный аудитор должен получить нулевой контекст прошлых внешних отчётов и самостоятельно искать проблемы в текущей области. URL контрольного обсуждения сохраняется отдельно как `control-conversation-url-<iteration>.txt` и `control-conversation-url.txt`, чтобы не перезаписать основную ссылку задачи.

Все режимы `audit submit`, которые записывают verdict Markdown, принимают только строгий repo-relative путь `--out`: для primary это `docs/verdicts/<domain>/<task-id>-audit-rNN.md`, для control audit это `docs/verdicts/<domain>/<task-id>-audit-control-rNN.md`. Путь не должен иметь произвольный префикс перед `docs/verdicts/`, пропускать доменный сегмент, быть абсолютным или содержать `.`, `..` либо пустые сегменты. Для обычного submit путь дополнительно обязан совпадать с текущими `taskId`/`iteration` и route, автоматически выведенным из сохранённых verdict-ов. `--download-report-only` сам выводит primary/control и ожидаемую task/iteration identity из строгого output path. Если скачанный Markdown явно ссылается на evidence или ZIP другой итерации той же задачи, команда отклоняет его диагностикой `E2D-BUILD-AUDIT-SUBMIT-REPORT-STALE` и не записывает файл как готовый verdict.

DOM-активация кнопки экспорта и Markdown-пункта должна выполнять не только `element.click()`, но и обычную последовательность событий `pointerdown`/`mousedown`/`pointerup`/`mouseup` на найденном элементе. Это сохраняет запрет на координатный клик по отчётной карточке, но покрывает UI, где download action привязан к pointer/mouse event handlers, а не к одному synthetic `click()`. Упоминания `button.click()` в export-контракте ниже являются сокращением для этой полной DOM-активации найденного элемента.

Read-only восстановление отчёта через `--download-report-only` открывает уже существующее обсуждение, поэтому оно не считает существующие legacy Deep Research targets baseline-ом новой отправки. Если в обсуждении осталось несколько готовых старых отчётных targets, read-only путь выбирает последний готовый target в порядке `Target.getTargets` и после экспорта применяет строгую проверку task/iteration.

Legacy-извлечение готового отчёта привязано к текущей вкладке. Команда сначала проверяет видимый Deep Research `iframe` старой отчётной карточки, сопоставляет его с frame id из `Page.getFrameTree` этой же вкладки и пытается экспортировать Markdown в этом контексте. `Target.getTargets` используется только как резервный read-only путь. Если выбран frame или target, кнопка `Экспорт` нажимается только в выбранной отчётной поверхности; пункт `Экспортировать в Markdown` после этого можно искать и в основном DOM страницы, потому что ChatGPT может отрисовать меню поверх документа. Такой fallback не разрешает выбирать другую карточку или создавать новый Deep Research запрос.

Восстановление CDP-связи (`Chrome DevTools Protocol`, то есть протокола управления вкладкой Chrome) применяется к безопасным операциям чтения и подготовки контекста: ordinary generation polling, attach target, enable-команды, `Page.getFrameTree`, создание isolated world и evaluate без побочного эффекта. DOM-клики Send/copy/export и выбор пункта Markdown не повторяются как обычные read-retry, чтобы не отправить сообщение, не скопировать и не экспортировать отчёт дважды. Во время ordinary polling recoverable failures вроде `Debugger unattached` восстанавливаются на той же вкладке до общего timeout; вокруг legacy target/frame короткий локальный recovery повторно выполняет attach/enable перед продолжением чтения.

Строгий валидатор текста отчёта не смягчается, но его диагностика должна объяснять причину `E2D-BUILD-AUDIT-SUBMIT-REPORT-INVALID`. Сообщение обязано различать минимум такие случаи: отчёт ещё генерируется или кандидатов не один; candidate пришёл не из открытой отчётной карточки и не из нового ответа ассистента текущей отправки; Markdown начинается с echo пользовательского prompt-а или похож на prompt-шаблон; первая непустая строка не `VERDICT: ACCEPT` и не `VERDICT: NEEDS_FIXES`; отсутствует или нарушен порядок обязательных heading-ов; `TASK_ASSESSMENT:` не называет текущие `metadata.taskId` и `metadata.iteration`; `VERDICT: ACCEPT` содержит numbered blockers `B1`..`Bn`; submit запущен с `--out`, который не является строгим repo-relative verdict-путём или не совпадает с текущей ZIP identity/mode. Такая диагностика не считается verdict-ом и не разрешает закрытие задачи без полного сохранённого отчёта.

Pipe-клиент `codex-chrome` отвечает строкой `pong` на входящие служебные JSON-RPC запросы `ping` от `Codex Chrome Extension` во время ожидания своего ответа. Такие запросы не являются действиями браузера, но без совместимого ответа на них расширение может не завершить долгую операцию вкладки.

По умолчанию `audit submit` использует проект `https://chatgpt.com/g/g-p-6950376d4d8c8191a0fe600e98389912-electro2d/project`, backend `--browser-backend codex-chrome`, уже установленный `Codex Chrome Extension` в пользовательском Chrome, интервал обновления `60` секунд, общий предел ожидания отчёта `180` минут и предел ожидания входа в ChatGPT `10` минут. Это единственный штатный браузерный путь. Команда не копирует cookies, `localStorage`, пароли, файлы профиля Chrome или другие файлы сессии: C#-команда подключается к pipe установленного расширения, создаёт собственную управляемую вкладку и выполняет действия через протокол расширения. Собственная вкладка безусловно закрывается через `finalizeTabs` в `finally` после успеха, ошибки или отмены; пользовательские вкладки и чужие browser sessions не изменяются. Для воспроизводимости разрешены параметры `--codex-chrome-pipe <path>`, `--codex-session-id <id>` и `--codex-turn-id <id>`; пара identity-параметров передаётся только вместе.

Для локальной проверки команды разрешены параметры `--project-url <url>`, `--poll-seconds <seconds>`, `--timeout-minutes <minutes>`, `--login-timeout-minutes <minutes>`, `--codex-chrome-pipe <path>`, `--codex-session-id <id>` и `--codex-turn-id <id>`. Значение `--poll-seconds` меньше `60` допускается только вместе с `--allow-fast-poll`, чтобы тесты могли проверять цикл ожидания без изменения рабочего правила внешнего аудита. Диагностического режима сохранения созданной вкладки после завершения команды нет.

Команда `audit submit` не принимает параметр для каталога скриншотов, не создаёт PNG-скриншоты браузерного протокола и не принимает tool screenshots как доказательство состояния страницы; во внутренней реализации не должно оставаться `AuditSubmitCodexChromeScreenshotRecorder`, PNG capture helper или вызовов `CaptureAsync` в submit workflow. Визуальный снимок реального окна Chrome может быть только отдельной пользовательской диагностикой вне audit tooling и не заменяет сохранённый Markdown-отчёт. Машинная диагностика опирается на DOM-дамп, строгую проверку текущей итерации и structured diagnostics. Режим `--dump-dom-only` записывает root DOM, `frame-tree.json`, `target-info.json`, accessibility tree и доступные legacy target/frame diagnostics; он не проверяет и не выбирает Deep Research в composer. Порядок ранней записи DOM-диагностики проверяется поведенческим тестом через внутренний драйвер `IAuditSubmitDomDumpDriver`: тест вызывает тот же production-путь `DumpDomFromUrlAsync`, подменяет только браузерный клиент и проверяет созданные JSON-файлы на диске, а не строки исходного кода.

`--dump-dom-only` не является режимом сохранения verdict-а. Он не должен писать диагностический summary в `docs/verdicts/**/<task-id>-audit-rNN.md` или `docs/verdicts/**/<task-id>-audit-control-rNN.md`; такие пути зарезервированы только для сохранённых внешних отчётов, первая непустая строка которых является `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`. При попытке совместить `--dump-dom-only` с таким `--out` команда обязана отказать до подключения к Chrome. DOM-диагностика хранится в `--dom-dump-dir`; если нужен отдельный текстовый summary, его путь должен быть временным и не похожим на verdict report.

Встроенный операторский слой доказательств не входит в основной ZIP и не меняет его `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или набор записей после выполнения `audit package verify` и `audit package message`. Он создаётся в сопровождающем ZIP `<task-id>-audit-rNN.operator-workflow.zip` после финальной записи и внутренней проверки основного ZIP. Сопровождающий ZIP содержит `OPERATOR-WORKFLOW.md`, `payload/metadata.json`, `payload/sha256.txt`, `payload/AUDIT-MANIFEST.sha256`, `payload/SHA256SUMS.sha256`, `payload/archive-entries.sha256`, `payload/archive-entries.txt`, а также каталоги `evidence/<task-id>-<iteration>/checks/audit-package-verify/` и `evidence/<task-id>-<iteration>/checks/audit-package-message/` с файлами `command.txt`, `stdout.txt`, `stderr.txt`, `exit-code.txt`, `duration-ms.txt`, `metadata.json`, `cwd.txt`, `env.json` и `timeout-seconds.txt`. В `metadata.json` для этих двух каталогов `executionMode` должен быть `subprocess`, ожидаемый и фактический коды завершения должны быть `0`, длительность должна быть больше нуля, а `timeoutSeconds` должен совпадать с `timeout-seconds.txt` и фактическим лимитом времени операторской команды: `600` секунд. Этот лимит покрывает большие audit ZIP, где `audit package verify` восстанавливает тысячи generated API packet files.

`command.txt` для проверки восстановления обязан показывать переносимую команду с `--zip <relative-zip-path>`, `--baseline <sha>` и плейсхолдером `--repo <clean-repo-path>`, а не абсолютный локальный путь временной рабочей копии. Фактическое выполнение происходит только через отдельный процесс `dotnet run --project eng/Electron2D.Build --no-build --no-restore -- audit package verify --zip <path> --baseline <sha> --repo <clean-repo-path>`; прямой вызов внутреннего метода проверки не считается допустимым evidence. Аналогично `audit package message evidence` создаётся только через отдельный процесс `dotnet run --project eng/Electron2D.Build --no-build --no-restore -- audit package message --zip <path>`, а не через внутренний метод подготовки текста. Это `baseline availability evidence`, то есть проверяемое доказательство того, что пакет был проверен на отдельной чистой копии исходной ревизии. Флаги `--no-build` и `--no-restore` обязательны для этих operator workflow subprocesses: основной запуск `audit package` уже работает внутри собранного build-tool, поэтому sidecar-доказательства не должны заново восстанавливать зависимости или пересобирать проект только ради чтения готового ZIP. Если архив или изменённые документы утверждают, что исходная ревизия доступна только локально, непубликована или требует ручной догадки, отсутствие вложенного baseline-снимка всё ещё не является отдельным blocker-ом, но успешное `audit package verify evidence` в сопровождающем ZIP становится обязательным доказательством восстановления.

## Окончания строк

Аудиторский архив опирается на общую репозиторную политику из `docs/repository-policy/line-endings.md`: `.gitattributes` задаёт `* text=auto eol=lf`, обычные текстовые файлы должны храниться и восстанавливаться с LF, а CRLF разрешён только для явных исключений `.bat`, `.cmd` и временно для исторических verdict-документов в `docs/verdicts/`, которые текущая задача не изменяет. Настоящие двоичные форматы описываются по расширениям как `binary`; XML-карты `.tmx` и `.tsx` в reference-game данных считаются текстом и нормализуются в LF.

Одноразовая нормализация tracked текстовых файлов допускается только как замена CRLF или одиночного CR на LF без смыслового изменения содержимого. Проверка:

Для современных страниц ChatGPT действует то же уточнение про выбранный frame/target: кнопка экспорта выбирается внутри уже найденной отчётной поверхности, а пункт `Экспортировать в Markdown` после клика может быть найден в основном DOM страницы, если меню отрисовано вне внутреннего frame. Это не допускает координатный клик и не допускает выбор другой report-card.

```bash
dotnet run --project eng/Electron2D.Build -- verify line-endings
```

читает `git ls-files --eol` и отклоняет неожиданные `w/crlf` у tracked текстовых файлов. Перед повторным внешним аудитом ручная команда `git ls-files --eol | rg "w/crlf"` должна показывать только описанные исключения или не показывать строк вообще.

## Конфигурация

Источник истины для входных данных - JSON-файл. Путь по умолчанию для рабочих сценариев:

```text
.temp/audit/<task-id>/audit-package.input.json
```

YAML считается допустимым будущим форматом того же контракта, но текущая реализация читает JSON. Конфигурация включается в архив как `metadata/audit-package.input.json`; исходный путь из `.temp` в архив не попадает.

Минимальные поля:

| Поле | Назначение |
| --- | --- |
| `taskId` | Идентификатор задачи. Должен совпадать с `--task` и всеми сгенерированными артефактами. |
| `iteration` | Итерация аудита в форме `rNN`. Должна совпадать с `--iteration`. |
| `baseline` | Исходная ревизия Git. Должна совпадать с `--baseline`. |
| `branch` | Имя рабочей ветки или логическое имя изменения, которое попадёт в запрос аудитору. |
| `domain` | Домен изменения, например `release-management`. |
| `scopeTaskIds` | Машиночитаемая область задач, которые принимает текущий пакет. Если список пуст, область равна одиночному `taskId`; если список содержит несколько значений, пакет объявляет явно комбинированную область (`combined scope`) и должен включать основной `taskId`. |
| `scopeSummary` | Краткое пояснение области аудита. Поле предназначено для объяснения, почему несколько задач входят в один пакет, и не заменяет `repoFileAllowlist`, `previousVerdictChain` или `blockerClosureList`. |
| `repoFileGlobs` | Шаблоны файлов репозитория, которые восстанавливаются через patch. |
| `repoFileAllowlist` | Явный список файлов репозитория, которые разрешено включить даже без совпадения с шаблоном. Список может включать task document `.taskboard/tasks/<task-id>.e2task` или `data/dev-diary/...`, если они принадлежат задаче и нужны аудитору. |
| `archiveOnlyEvidenceGlobs` | Шаблоны файлов, которые попадают только в `evidence/` архива и не входят в patch. |
| `preflightChecks` | Машиночитаемый список уже выполненных проверок. Каждая запись объявляет имя, переносимую команду и evidence-globs с готовыми файлами вывода; `audit package` только импортирует эти файлы и не запускает команду повторно. |
| `checks` | Машиночитаемый список дешёвых package-local проверок, которые команда запускает до упаковки. Каждая запись обязана иметь непустой `rationale`, объясняющий, почему эта проверка должна быть доказательством внутри package-route, а не preflight. Тестовый раннер здесь запрещён; тяжёлые и средние тесты должны быть в `preflightChecks`. |

| `maxFileSize` | Максимальный размер одного входного файла в байтах. |
| `outputDirectory` | Ожидаемый каталог результата. Должен совпадать с `--out`, если указан. |

Дополнительные поля:

| Поле | Назначение |
| --- | --- |
| `previousVerdictChain` | Список предыдущих verdict-документов или пустой список. Каждый путь должен указывать на Markdown-файл внутри `docs/verdicts/`, иначе конфигурация считается ошибочной. ## Содержимое архива

Корневые файлы архива:

- `AUDIT-MANIFEST.md` - инвентарь, а не повествовательный отчёт.
- `<task-id>.patch` - patch, созданный штатным Git через `git diff --binary`.
- `repo-file-hashes.json` - ожидаемые SHA-256 файлов репозитория после применения patch.
- `metadata/repo-file-snapshots.json` - машинный индекс полных снимков файлов, доступных аудитору для чтения.
- `SHA256SUMS.txt` - контрольные суммы всех файлов архива, кроме самого `SHA256SUMS.txt`, в формате, совместимом с `sha256sum -c`.
- `AUDIT-REQUEST.md` - статический запрос аудитору, дословно скопированный из отслеживаемого Git-файла `docs/release-management/AUDIT-REQUEST.md`.
- `metadata/audit-package.input.json` - нормализованная копия конфигурации.
- `repo-after/` - полные итоговые снимки repo-owned файлов объявленной области задачи, которые существуют после изменения.
- `repo-before/` - полные снимки repo-owned файлов объявленной области задачи из исходной ревизии, если файл существовал в `baseline`.
- `evidence/` - только доказательства настроенных проверок и импортированные файлы доказательств; операторские доказательства `audit package verify` и `audit package message` хранятся в сопровождающем ZIP, чтобы не менять основной архив после проверки.

`repo-after/` является основной поверхностью чтения для внешнего content review: аудитору не нужно восстанавливать дерево репозитория, чтобы прочитать полный итоговый код, тесты и документацию выбранной области. Patch остаётся навигацией по изменённым местам и восстановительной моделью, но не заменяет полные итоговые снимки файлов. `repo-before/` нужен для проверки удалений, переименований и регрессий относительно исходной ревизии; он не является полным baseline-снимком репозитория. Снимки создаются только для repo-owned файлов объявленной области задачи: `repoFileGlobs`, `repoFileAllowlist`, `previousVerdictChain` и связанных записей `repo-file-hashes.json`. Dirty, untracked или out-of-scope файлы рабочей копии не должны попадать в `repo-after/`, `repo-before/` или snapshot index.

`metadata/repo-file-snapshots.json` содержит по одной записи на каждый repo-owned файл declared scope, то есть объявленной области задачи. Текущая schema version задаётся структурой `taskId`, `iteration`, `baseline` и массивом `files`. Каждая запись указывает `path`, `status`, `afterSnapshot`, `beforeSnapshot`, `afterSha256`, `beforeSha256`, `contentKind` и `fullContentIncluded`. Для текстовых файлов `contentKind` равен `text`, для двоичных файлов - `binary`. `afterSnapshot` или `beforeSnapshot` с непустым значением означает `exists=true` и `included=true` для соответствующей стороны; отсутствие поля означает `exists=false` для `added`/`deleted` стороны либо отсутствие baseline-содержимого. Переименования в текущей версии представлены как связанная пара `deleted` старого пути и `added` нового пути из declared scope; отдельное поле `oldPath` пока не используется и не должно ослаблять требование иметь оба snapshot-а. `fullContentIncluded` должен быть `true` для каждого файла, который входит в `repo-file-hashes.json` или список удалённых файлов. `status` должен соответствовать фактической паре сторон: `added` имеет только `after`, `deleted` имеет только `before`, `modified` имеет обе стороны с разными SHA-256, а `unchanged` имеет обе стороны с одинаковыми SHA-256. Если файл declared scope нельзя безопасно вложить полностью из-за реального секрета, конфиденциального содержимого, размера или неподдержанного двоичного содержания, текущий контракт T-0237 завершает `audit package` или `audit package verify` отказом до отправки архива. Если в будущем появится controlled omission для слишком большого, сгенерированного или чувствительного baseline-файла, индекс обязан явно указать `fullContentIncluded: false`, `omissionReason`, SHA-256, размер и допустимый способ аудита; T-0237 таких исключений не разрешает.

Матрица снимков:

| Status | `repo-before/` | `repo-after/` |
| --- | --- | --- |
| `added` | отсутствует, потому что файла нет в `baseline` | обязателен |
| `modified` | обязателен | обязателен |
| `unchanged` | обязателен, если файл выбран в scope и существует в `baseline` | обязателен |
| `deleted` | обязателен | отсутствует |
| `renamed` | представляется текущей моделью как пара `deleted` old path и `added` new path, если оба пути выбраны; будущий `oldPath` в индексе не должен ослаблять обязательность обоих снимков | обязателен для new path |
| `binary` | включается как полный byte snapshot, если проходит защитные правила и лимит размера | включается как полный byte snapshot, если проходит защитные правила и лимит размера |

Snapshot content проходит те же защитные правила, что patch и evidence: машинные абсолютные пути, реальные секреты, приватные ключи, токены и пароли запрещены в любом текстовом `repo-before/` или `repo-after/` файле. Реальный secret/sensitive content в `repo-after/` для changed text file всегда запрещает сборку package; его нельзя скрывать omission-записью, потому что это итоговое содержимое проверяемой задачи. Для `repo-before/` нет общего послабления на старые строки: допускается только точечная совместимость с уже известной старой формулировкой `docs/release-management/audit-package.md`, где замещающие значения были закрыты обратной кавычкой внутри документационного примера. Любой другой `repo-before/` snapshot с замещающим значением и произвольным suffix остаётся blocker-ом. Большой файл определяется тем же `maxFileSize` из конфигурации audit package; значение по умолчанию равно `1_048_576` байтам. Двоичный файл определяется байтовой проверкой содержимого, а сгенерированный файл пока не получает отдельного omission режима: если он входит в declared scope, он должен безопасно войти как полный snapshot или package завершится ошибкой. Текущая реализация не создаёт redacted или partial snapshot: если полный снимок нельзя безопасно вложить, package завершается ошибкой до создания финального ZIP. Это fail-closed правило важнее полноты audit ZIP, потому что неполный или отредактированный snapshot нельзя выдавать за полноценный content review. Если будущий контракт разрешит `fullContentIncluded: false`, `AUDIT-REQUEST.md` должен требовать считать omitted implementation/test/documentation snapshot evidence gap и blocker-ом, когда файл критичен для проверки задачи и нет другой полной поверхности чтения.

Тестовые файлы, которые проверяют сам scanner секретов, могут создавать синтетические строки вида ключ + разделитель + значение только во время выполнения теста. В исходнике такие fixture-значения должны храниться разрезанными фрагментами, чтобы полный текстовый snapshot выбранного тестового файла не содержал готового secret-like assignment. Это не исключение из secret scan: если в тестовом source-файле лежит непрерывная строка, похожая на реальный токен, пароль, приватный ключ или другое секретное присваивание, `audit package` обязан остановиться до создания ZIP. Оператор C# lambda `=>` не является разделителем присваивания секрета и не должен блокировать упаковку только из-за имени параметра вроде `token`.

Архив не содержит и по текущему контракту не обязан содержать Git bundle, архив tar или иной вложенный набор файлов всего исходного дерева. Восстановление проверяется через patch, `repo-file-hashes.json`, `AUDIT-MANIFEST.md`, `metadata/repo-file-snapshots.json`, отдельные `repo-before/` снимки файлов declared scope и отдельную чистую копию исходной ревизии, переданную в `audit package verify --repo <clean-repo-path>`.

Сопровождающий ZIP `<task-id>-audit-rNN.operator-workflow.zip` не является частью основного архива и не входит в его `SHA256SUMS.txt`. Он остаётся локальным доказательством предварительной проверки и не прикрепляется командой `audit submit` к внешнему ChatGPT-запросу. Его назначение - доказать локальной предварительной проверке, что уже финальный основной ZIP был проверен и что текст внешнего сообщения был получен штатной командой. Файл `payload/metadata.json` в сопровождающем ZIP содержит относительный путь основного ZIP, SHA-256 основного ZIP, SHA-256 `AUDIT-MANIFEST.md`, SHA-256 `SHA256SUMS.txt`, SHA-256 списка записей архива и сам список записей. Если после создания операторских доказательств основной ZIP будет переписан, эти значения перестанут совпадать с фактическим файлом.

`AUDIT-MANIFEST.md` обязан содержать только проверяемый инвентарь:

- `taskId`, `iteration`, `baseline`, `branch`, `domain`, `scopeTaskIds` и непустой `scopeSummary`, если он задан;
- исходный путь статического запроса аудитору: `docs/release-management/AUDIT-REQUEST.md`;
- `diff name-status`, то есть список изменённых файлов и типов изменения;
- список всех путей архива;
- список repo-owned файлов из `repo-file-hashes.json`, включая существующие предыдущие verdict-документы из `previousVerdictChain`;
- ссылки на `metadata/repo-file-snapshots.json`, `repo-after/` и `repo-before/`, чтобы внешний аудитор видел, где читать полные файлы;
- объявленную область задач, цепочку предыдущих verdict-ов и список закрытых blocker-ов;
- матрицу закрытия прошлых blocker-ов для существующих `NEEDS_FIXES` отчётов: путь отчёта, blocker id, имя configured check и closure-строку, которая его закрывает;
- модель восстановления;
- список проверок и ссылки на соответствующие файлы `evidence/`, включая найденные TRX-файлы с результатами `dotnet test`;
- политику запрещённых путей и секретов.

Patch строится с `--find-renames`, поэтому чистые переносы файлов могут быть представлены строками `rename from` и `rename to` без полного повторения содержимого. Это не ослабляет восстановление: `repo-file-hashes.json` всё равно фиксирует итоговые SHA-256, а `audit package verify` применяет patch к чистой копии baseline и сравнивает полный набор восстановленных файлов.

## Пути и ZIP

До любого изменения Git `audit package verify` физически канонизирует каждый компонент путей `--repo`, основного ZIP и соседнего operator-workflow ZIP с разрешением symbolic link и Windows junction. Неразрешимый путь отклоняется fail-closed; физическое нахождение любого ZIP внутри clean repository даёт `E2D-BUILD-AUDIT-REPO-NOT-ISOLATED` даже тогда, когда переданный архивный путь лексически находится снаружи.

Все пути внутри архива используют UTF-8 POSIX-форму `dir/file.ext`. Backslash, управляющие символы, абсолютные пути, `..`, пустые сегменты и дубли путей запрещены. Манифест использует те же пути, что и архив.

ZIP создаётся в стабильном порядке:

- записи отсортированы по ordinal-сравнению путей;
- у всех записей фиксированное время `2000-01-01T00:00:00Z`;
- внешние атрибуты ZIP-записей, зависящие от операционной системы, должны быть нулевыми;
- ZIP-запись с не-ASCII путём должна иметь `UTF-8 flag`, то есть признак кодировки UTF-8 в центральном каталоге ZIP; текущая реализация пишет имена через UTF-8 и `audit package verify` проверяет этот признак минимум для таких путей;
- текстовые файлы генерируются в стабильном порядке;
- пути машины, абсолютные временные каталоги и текущее время не записываются, кроме явно разрешённых относительных метаданных проверки; Windows-пути с диском по шаблонам `[A-Z]:\...` и `[A-Z]:/...` запрещены в любом текстовом файле архива.
- `audit package verify` отклоняет архив с каталогами-записями, несортированными записями, нестабильной временной меткой ZIP, ненулевыми внешними атрибутами ZIP-записей или отсутствующим UTF-8 flag у не-ASCII пути.

Сопровождающий ZIP операторских доказательств тоже записывается с отсортированными путями, фиксированными временными метками и нулевыми внешними атрибутами ZIP-записей. Его байты не обязаны быть стабильными между разными запусками, потому что `stdout.txt`, `stderr.txt`, `duration-ms.txt` и `metadata.json` отражают реальные отдельные процессы.

## Patch и восстановление

Patch создаётся через Git, без ручной правки заголовков hunk-ов. Для выбранных файлов используется временный индекс Git:

1. В индекс читается дерево `baseline`.
2. В индекс добавляется текущее состояние выбранных файлов репозитория.
3. Команда строит `git diff --cached --binary <baseline>`.

Такой подход включает новые файлы, изменения, удаления и бинарные файлы, которые поддерживает `git diff --binary`. Если выбранный бинарный файл нельзя восстановить через эту модель, команда должна завершиться ошибкой до создания финального архива.

Для текстовых файлов команда добавляет во временный индекс Git байты с LF-окончаниями строк. Это защищает аудиторский patch от массовой перезаписи существующих файлов, когда локальная рабочая копия на Windows хранит те же строки с CRLF-окончаниями. `repo-file-hashes.json` считает SHA-256 по тем же восстановимым байтам, поэтому проверка архива сравнивает состояние, которое реально создаёт patch, а не машинный формат строк в текущем каталоге.

Файлы из `repoFileAllowlist`, которые игнорируются правилами Git, всё равно считаются заявленным входом задачи. Команда обязана принудительно добавить их во временный индекс Git, включить в patch и доказать восстановление через `audit package verify`; правила игнорирования не должны молча исключать такой файл из архива.

Файлы из `previousVerdictChain` считаются частью цепочки внешнего аудита. Когда такой файл есть в рабочей копии, он восстанавливается как обычный файл репозитория даже без совпадения с `repoFileGlobs` и без отдельного `repoFileAllowlist`. Это делает цепочку verdict-ов проверяемой в восстановленной модели файлов репозитория (`restored repo-owned model`): ссылка в `AUDIT-REQUEST.md` совпадает с patch, `repo-file-hashes.json` и инвентарём `AUDIT-MANIFEST.md`. Отсутствующий файл не создаётся автоматически и остаётся только исторической ссылкой, чтобы старые внешние ссылки не блокировали упаковку.

Содержимое существующих verdict-файлов из `previousVerdictChain` сохраняется дословно как историческое внешнее свидетельство внутри восстановленной модели файлов репозитория. Текущий контракт не объявляет `previousVerdictChain` независимым нотариальным доказательством происхождения прошлых внешних отчётов и не требует предыдущие аудиторские ZIP-архивы или отдельные исторические контрольные суммы (`no previous audit packages or detached historical checksums required`), если такой слой явно не введён отдельным контрактом пакета. Поэтому проверка локальных машинных путей не применяет запрет к diff-блокам этих файлов внутри patch: старый ответ аудитора может цитировать синтетический или чужой локальный путь, который не относится к текущей упаковке. То же ограниченное исключение для локальных путей действует при финальной проверке его снимков `repo-after/` или `repo-before/` в ZIP. Проверка конкретных значений, похожих на секреты, остаётся включённой для самих verdict-файлов, их ZIP-снимков и patch: если прошлый отчёт содержит не замещённый токен, пароль или приватный ключ, пакет должен завершиться отказом, а не передавать это значение аудитору. Значения-заглушки в угловых скобках, например `<redacted>`, `<token>`, `<value>` или `<non-placeholder>`, считаются замещёнными только когда они стоят на месте всего значения; текст после такой заглушки не считается безопасным продолжением и остаётся блокером. Если сохранённый внешний отчёт уже содержит нейтральную reviewer phrase, синтетический reviewer example со словами `concrete-secret`/`concrete-value`, точную reviewer-фразу с secret-like marker-ом и описательным значением или старую reviewer-фразу, где Markdown code span закрывается после `pass` и дальше идёт пояснение аудитора, такой фрагмент может быть разрешён только как точное значение из allowlist-а и только в контексте файла из `previousVerdictChain`, его `repo-before`/`repo-after` снимка или его patch-блока. Самостоятельное secret-like присваивание со значением `pass`, включая строку с password-like ключом и значением `pass`, не считается безопасной historical reviewer phrase и должно блокироваться. Произвольный suffix после разрешённой фразы или значения остаётся блокером. Обычные файлы репозитория, файлы текущей задачи, архивные доказательства, вывод проверок, TRX-файлы, `AUDIT-MANIFEST.md`, `AUDIT-REQUEST.md`, `repo-file-hashes.json` и `metadata/audit-package.input.json` также по-прежнему блокируют пакет при новых локальных путях машины или конкретных секретоподобных значениях.

По умолчанию patch остаётся текстовым: чистые изменения обычных текстовых файлов пишутся как обычные diff-блоки. Если обычный diff-блок вне `previousVerdictChain` раскрывает значение, похожее на секрет, только через удалённую или контекстную строку исходной ревизии (`baseline`), прежняя бинарная защита patch уже не считается достаточной сама по себе, потому что текущий пакет также обязан включать полный `repo-before/` снимок изменённого файла. Если этот `repo-before/` снимок содержит приватный ключ, токен, пароль или другое конкретное секретоподобное значение, команда завершает упаковку ошибкой до создания финального ZIP. Двоичный diff (`GIT binary patch`) может применяться только как дополнительная защита patch для безопасного baseline-содержимого; он не разрешает передавать небезопасный `repo-before/` снимок. Остальные чистые текстовые изменения остаются обычными diff-блоками. Если секретоподобное значение есть в текущем файле задачи, упаковка также завершается ошибкой до построения patch.

Patch проверяется сразу после создания и до сборки ZIP, а затем повторно при `audit package verify`. Для проверки локальных машинных путей команда временно исключает только diff-блоки существующих исторических verdict-файлов из `previousVerdictChain`; остальные блоки patch сканируются полностью. Для проверки секретоподобных значений patch сканируется целиком, включая diff-блоки previous verdict-файлов. Это включает добавленные строки, контекстные строки и удалённые строки из `baseline`. Если после точечного двоичного восстановления для проблемных файлов обычный patch всё ещё содержит запрещённый текст, пакет завершается ошибкой вместо передачи такого текста аудитору.

`repo-file-hashes.json` описывает ожидаемое восстановленное состояние:

- `taskId`, `iteration`, `baseline`;
- список существующих repo-owned файлов и их SHA-256;
- список удалённых repo-owned файлов.

Проверка восстановления выполняет:

1. C#-проверку `SHA256SUMS.txt`.
2. Дополнительный запуск `sha256sum -c`, если утилита доступна.
3. `git apply --check`.
4. `git apply`.
5. Сравнение точного набора изменённых, удалённых и новых путей файлов репозитория после применения patch с `repo-file-hashes.json`. Новые Git-игнорируемые файлы из `repoFileAllowlist` тоже входят в это сравнение.
6. Сравнение восстановленных SHA-256 файлов с `repo-file-hashes.json`.
7. Сравнение корневого `AUDIT-REQUEST.md` из архива с восстановленным отслеживаемым Git-файлом `docs/release-management/AUDIT-REQUEST.md`.
8. Проверку, что новые текстовые файлы не являются однострочными заглушками, если они не указаны в `oneLineStubAllowlist`.

## Evidence

Каждая заранее выполненная проверка из `preflightChecks` импортируется в отдельный каталог:

```text
evidence/<task-id>-<iteration>/preflight/<check-name>/
```

Каждая запись `preflightChecks[]` обязана иметь непустые `name`, `command` и `evidenceGlobs`. Шаблоны evidence должны указывать на существующие файлы, обычно под `.temp/audit-evidence/`, и проходить те же проверки путей, локальных машинных путей, секретов и запрещённых шаблонов, что обычные archive-only evidence. Если шаблон не нашёл ни одного файла, `audit package` завершается `E2D-BUILD-AUDIT-CONFIG-INVALID`, потому внешний аудитор не получит доказательство уже выполненного preflight. Команда из `preflightChecks[].command` записывается в `command.txt` рядом с импортированными файлами, но не исполняется. Если импортируемый корневой `metadata.json` preflight-проверки содержит поле `command` или должен содержать его как машинное описание команды, `audit package` нормализует это поле к тому же тексту, который записан в соседний `command.txt`. Корневой `metadata.json` парсится как JSON object, и repo-root в его string values нормализуется структурно через JSON tree, а не через raw-text guessing. Новые preflight evidence не должны сохранять lossy-строки вроде `dotnetrun`, `dotnetbuild`, `dotnettest` или `gitdiff`; для уже сохранённых исторических audit ZIP это правило не требует переписывания. Новые импортируемые текстовые preflight artifacts, включая `output.txt` и `result.txt`, дополнительно нормализуют только целый локальный путь к корню репозитория в placeholder `<repo>` и сохраняют обычный текст без посимвольных вставок.

Raw preflight text делится на две поверхности. Compact JSON snippets, которые целиком парсятся как JSON object/array, имеют доказанную левую token-boundary и не имеют external same-line tail, защищаются как структурные участки: команда обходит decoded JSON string values, нормализует decoded exact repo-root и safe child path values в `<repo>`, включая literal slash, escaped solidus `\/` и unicode escaped solidus `\u002F`, а затем serializes snippet обратно как compact JSON. Левая граница snippet-а доказывается так же fail-closed, как non-JSON token boundary: snippet в начале строки/текста или после whitespace/opening delimiter-а допустим только если перед ним нет path separator-а внутри текущего path-token-а; snippet, встроенный в `/opt/e2d{"repositoryRoot":"repo"}`, не нормализуется структурно. Parseable, но незащищённый JSON-looking snippet не считается границей raw path-token-а: raw replacement по участкам слева и справа проверяет start/end boundary на координатах исходного полного текста, поэтому `repo{"ok":true}`, `repo[0]`, `{"ok":true}repo`, `{"ok":true}repo/logs.txt` и traversal через `repo/dir{"ok":true}/../../...` не могут быть частично превращены в `<repo>` и остаются под fail-closed validation. Любое decoded/raw/path-like значение после JSON decode или raw normalization проходит через единый machine-path classifier: safe repo-root/child value заменяется на `<repo>`, а repo-root внутри prose string content, traversal value, нерепозиторный machine path или Windows drive absolute path, включая decoded `G\u003A/...` / `G\u003A\u005C...`, останавливает пакет с `E2D-BUILD-AUDIT-ABSOLUTE-PATH`. Raw substring replacement не применяется внутри защищённого JSON snippet-а, поэтому single quote/backtick, comma/semicolon и JSON-looking punctuation внутри JSON string content не могут стать доказанными token boundaries. JSON-looking prefix с external same-line tail, например `{\"repositoryRoot\":\"repo\"},\"backup\":0`, также не нормализуется частично и остаётся под финальной проверкой локальных путей.

Non-JSON raw text нормализуется только как whole path-token. Exact repo-root допускается в конце строки/текста или как quoted/backtick token с matching opening delimiter-ом и без non-whitespace tail; structural comma/semicolon/key-looking tails не считаются доказательством. Child path допускается только когда сразу после repo-root идёт платформенный separator (`/` или `\` на Windows, только `/` на POSIX), а последующий path-token не содержит `..`-сегментов до следующего separator-а или конца строки. Пробелы, запятые, точки с запятой, скобки и другие допустимые символы имени файла не завершают traversal scan внутри child path. Совпадение root использует семантику платформы: case-insensitive на Windows/macOS и ordinal на case-sensitive платформах, а start-boundary перед root должен быть началом текста, whitespace или opening delimiter, который сам не встроен в предшествующий path segment; path separator перед whitespace/opening delimiter/candidate на той же строке делает replacement недопустимым даже при whitespace внутри POSIX filename segment-а. Filename-valid punctuation или opening delimiter внутри более длинного path-token-а не считается безопасной левой границей.

Ambiguous tails вроде `repo\",backup/...`, `repo\";backup/...`, `repo\",]backup/...`, `repo\",\"backup/...`, `repo\";}backup/...`, `repo\",\"backup\":/logs.txt`, `repo\",\"backup\":\"/logs.txt`, `repo\",\"backup\":[/logs.txt`, `repo\",\"backup\":\"value\"/logs.txt`, `repo\",\"backup\":abc,def/logs.txt`, `repo\";\"backup\":abc;def/logs.txt`, `repo\",\"backup\":\"value\",tail/logs.txt`, `repo\",\"backup\":0`, `repo\",\"backup\":\"value\"`, `[\"repo\",\"backup\":0]`, `[\"repo\",\"backup\":\"value\"]`, `{\"repositoryRoot\":\"repo\"},\"backup\":0`, `{\"repositoryRoot\":\"repo\"},\"backup\":\"value\"`, `[{\"repositoryRoot\":\"repo\"}],\"backup\":0`, `{\"message\":\" 'repo','backup':0\"}` или `{\"message\":\" \`repo\`,\`backup\`:0\"}` fail closed и остаются под absolute-path validation. Sibling path с общим prefix-ом, например `repo backup/...`, `repo)backup/...`, `repo/../repo backup/...`, `repo/dir with space/../../repo backup/...`, quoted `repo/..`, `prefix <repo>/...`, `prefix\t<repo>/...`, `prefix:<repo>/...`, `prefix=<repo>/...`, `prefix(<repo>/...`, `prefix[<repo>/...`, `prefix\"<repo>/...`, `prefix:\"<repo>/...`, `prefix=(\"<repo>/...`, `prefix dir:\"<repo>/...`, `prefix dir=(\"<repo>/...`, `/opt/e2d{\"repositoryRoot\":\"repo\"}`, `/opt/e2d{\"repositoryRoot\":\"repo-with-escaped-solidus\"}`, `repo{\"ok\":true}`, `repo[0]`, `{\"ok\":true}repo`, `{\"ok\":true}repo/logs.txt`, `repo/dir{\"ok\":true}/../../repo backup/...`, `repo\\backup/...` на POSIX, `repo\")backup/...` на POSIX, `repo\",backup/...` на POSIX, `repo\";backup/...` на POSIX или case-variant root на case-sensitive платформе, не заменяется и остаётся под существующим absolute-path blocker-ом. Финальная проверка локальных путей ищет literal paths, reversible JSON path encodings минимум `\/` и `\u002F`, а также decoded JSON string literals через тот же machine-path classifier, поэтому reverse-encoded machine-local path или Windows drive path с escaped colon не может пройти только потому, что raw substring не совпал. Windows drive detector допускает `D:\u0022` и `D:\u0027` только как isolated punctuation-like values: после marker-а `u0022`/`u0027` до whitespace/quote/backtick terminator-а не должно быть `/`, `\`, букв, цифр или `_`. Если marker продолжается filename segment-ом через `.`, `-`, `)` или другую пунктуацию, а затем тем же token-ом идёт path separator, это реальный drive-root path и пакет останавливается с `E2D-BUILD-AUDIT-ABSOLUTE-PATH`. Если входной или нормализованный preflight evidence уже содержит pattern вида `<repo>/n<repo>/a<repo>/m`, `audit package` останавливается с `E2D-BUILD-AUDIT-PREFLIGHT-SANITIZER`, потому такой артефакт нечитаем и должен быть пересоздан до внешней отправки. Нерепозиторные машинные пути, секреты и приватные значения остаются блокерами.

Каждая дешёвая проверка из `checks` создаёт отдельный каталог:

```text
evidence/<task-id>-<iteration>/checks/<check-name>/
```

Минимальные файлы проверки:

- `command.txt` - исполняемый файл и аргументы без shell-строки.
- `cwd.txt` - относительный рабочий каталог.
- `env.json` - только переменные из `envAllowlist`.
- `timeout-seconds.txt` - тайм-аут проверки.
- `exit-code.txt` - ожидаемый и фактический код завершения.
- `duration-ms.txt` - фактическая длительность запуска в миллисекундах, нормализованная как число больше нуля с инвариантным десятичным разделителем.
- `stdout.txt` и `stderr.txt` - полный стандартный вывод и поток ошибок.
- `metadata.json` - машиночитаемая сводка, включая команду, тайм-аут, ожидаемый и фактический код завершения, ту же длительность, SHA-256 файлов вывода и TRX-файлов, если они были найдены.

`metadata.json` должен совпадать с соседними сырыми файлами: `expectedExitCode` и `actualExitCode` берутся из `exit-code.txt`, `timeoutSeconds` - из `timeout-seconds.txt`, а `durationMs` - из `duration-ms.txt`. Эти поля не должны зависеть от порядка аргументов конструктора или от человекочитаемого текста.

Перед записью `stdout.txt` и `stderr.txt` команда заменяет локальный корень репозитория на `<repo-root>`. Это сохраняет полный диагностический вывод проверок, но не раскрывает путь рабочей машины и не требует от оператора вручную подавлять вывод команд.

Перед запуском configured checks команда печатает структурированную JSON-диагностику с планом: число проверок, имя каждой проверки, переносимая команда, рабочий каталог, тайм-аут, ожидаемый код завершения и `rationale`. После каждой проверки команда печатает отдельную строку результата: имя проверки, переносимую команду, фактический код завершения, ожидаемый код, длительность, путь `stdout.txt` и путь `stderr.txt` внутри будущего evidence-каталога. Эти строки не заменяют сырые доказательства в ZIP; они нужны, чтобы оператор видел стоимость, состав и причину тяжёлого этапа до и во время упаковки, а не только после завершения архива.

Для проверок `dotnet test` или `dotnet run -- ... test` конфигурация должна использовать `preflightChecks[]`, а не `checks[]`. `checks[]` для `audit package` не должен превращать упаковку в повторный прогон среднего или тяжёлого уровня: любые команды тестового раннера внутри `checks[]` запрещены и завершают упаковку диагностикой `E2D-BUILD-AUDIT-CONFIG-INVALID`. Проверка нормализует имя исполняемого файла, поэтому `dotnet`, `dotnet.exe`, абсолютный путь к `dotnet` и repo-relative shim считаются одним запрещённым классом. Shell-оболочки в `checks[]` запрещены целиком независимо от аргументов: `cmd`, `cmd.exe`, `powershell`, `powershell.exe`, `pwsh`, `sh` и `bash` не должны использоваться для package-local checks, потому через split-аргументы можно скрыть повторный тестовый прогон от статической проверки. Если нужна shell-команда, широкий прогон или локальная уверенность, они относятся к preflight до упаковки, а их stdout/stderr/TRX/summary импортируются через `preflightChecks[].evidenceGlobs`.

TRX-файлы проверяются как XML-доказательства. Сканирование секретов должно читать содержимое результатов тестов: текст стандартного вывода, ошибок, сообщений, трассировок и обычных атрибутов. Служебные отображаемые имена тестов, где тестовый фреймворк может показать параметры проверок вроде ключа `token` со значением `...`, не должны блокировать включение штатного TRX. Если настоящее секретное присваивание или блок приватного ключа попадает в текст результата теста, TRX всё равно отклоняется. Перед записью TRX в архив команда нормализует служебные XML-атрибуты с путями сборки, например `codeBase` и `storage`: путь к локальному корню репозитория заменяется на `<repo-root>`, чтобы архив не раскрывал путь машины. На macOS та же нормализация учитывает системные варианты одного временного пути через `/var` и `/private/var`, потому что test runner может записать один вариант, а проверка репозитория работать с другим.

TRX-доказательства получают стабильные имена внутри архива: `trx/test-result-001.trx`, `trx/test-result-002.trx` и далее по отсортированному исходному списку. Исходные имена файлов, которые тестовый раннер может составлять из имени пользователя, имени машины и времени запуска, не записываются в пути архива, `metadata.json`, `AUDIT-MANIFEST.md`, `SHA256SUMS.txt`, `stdout.txt` или `stderr.txt`. Если стандартный вывод или поток ошибок содержит исходный путь TRX, команда заменяет его на стабильный путь архива.

При записи TRX XML команда дополнительно нормализует машинные атрибуты раннера: `TestRun/@name` становится `<test-run>`, а атрибуты `runUser`, `computerName` и `Deployment/@runDeploymentRoot` удаляются из архивной копии TRX. Поэтому ZIP не содержит ни настоящие локальные значения, ни сами проблемные XML-атрибуты, которые внешняя проверка трактует как утечку машинного контекста. Текст результатов тестов не редактируется перед проверкой секретов: настоящее секретное значение в `StdOut`, `StdErr`, сообщении или трассировке по-прежнему блокирует пакет.

Для самого T-0228 внешний пакет должен содержать отдельные checks для:

- минимального запуска `audit package` на тестовом наборе файлов;
- отдельного `audit package verify` на чистой копии baseline.

Эти записи `checks` должны создавать самостоятельные каталоги доказательств в `AUDIT-MANIFEST.md`, а не только строки в общей команде или внешней заметке.

Файлы из `archiveOnlyEvidenceGlobs` копируются под `evidence/<task-id>-<iteration>/archive-only/` и покрываются контрольными суммами. Они не входят в patch и не считаются repo-owned файлами.

Выбор archive-only evidence не должен начинаться с полного рекурсивного обхода корня репозитория. Команда выводит минимальные корни поиска из статической части каждого шаблона `archiveOnlyEvidenceGlobs`: для `.temp/audit-evidence/**` она перечисляет только `.temp/audit-evidence`, при пустом списке сразу возвращает пустой набор, а к полному обходу корня переходит только для шаблонов без более узкого статического префикса, например `*.txt`. Это защищает упаковку от старых `.temp/audit-clean/**`, локальных clean checkout-ов и других больших временных деревьев, которые не относятся к доказательствам текущего пакета.

Для временных доказательств проверки разрешён специальный путь `.temp/audit-evidence/...`. Это единственное исключение из общего запрета `.temp` и применяется только к файлам, выбранным через `archiveOnlyEvidenceGlobs`, `preflightChecks[].evidenceGlobs`, и к TRX-файлам, выбранным через `checks[].trxGlobs`: такие файлы попадают в раздел `evidence/`, не восстанавливаются в рабочую копию и не создают дополнительных каталогов в корне репозитория. При записи archive-only evidence в ZIP префикс `.temp/` удаляется, поэтому архивный путь выглядит как `evidence/<task-id>-<iteration>/archive-only/audit-evidence/...` и сам ZIP не содержит сегмент `.temp`; при записи `preflightChecks[].evidenceGlobs` команда кладёт файлы в `evidence/<task-id>-<iteration>/preflight/<check-name>/`. Для `repoFileGlobs`, `repoFileAllowlist`, `checks.cwd` и любых других входных путей конфигурации `.temp` остаётся запрещённым путём.

После выполнения проверок инструмент фиксирует SHA-256 выбранных файлов репозитория и доказательств. Перед записью ZIP тот же набор пересчитывается снова. Если файлы изменились между проверками и упаковкой, команда завершается ошибкой.

После финальной записи основного ZIP команда больше не изменяет `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или набор его записей. Операторские доказательства создаются только после этого и попадают в сопровождающий ZIP. Поэтому `audit package verify evidence` и `audit package message evidence` относятся к неизменяемому основному ZIP, а не к промежуточной версии архива.

## Защитные правила

Команда отклоняет архив или входные файлы, если путь или содержимое попадает под встроенный запрет:

- `.git`;
- `bin`;
- `obj`;
- `.temp`, `tmp` и временные каталоги, кроме `.temp/audit-evidence/...` только для `archiveOnlyEvidenceGlobs`, `preflightChecks[].evidenceGlobs` и `checks[].trxGlobs`;
- вложенные `.zip`, `.tar`, `.tar.gz`, `.7z`;
- `.env` и производные имена;
- опасные имена пути с `secret`, `token` или `private-key`;
- блоки приватных ключей;
- присваивания или JSON/YAML-подобные значения для `secret`, `token`, `password`, `api_key` и `api-key`.

Документальные упоминания `secretScanPolicy`, `secret`, `token`, `password` или `api_key` без секретного значения не считаются утечкой. Синтетические замещающие примеры, где ключ `token` получает значение `...`, а ключ `password` получает значение `<redacted>`, тоже допустимы, когда они описывают защитное правило. Такой marker должен занимать всё значение: если после замещающего значения продолжается обычный текст, сканирование не считает это безопасной заглушкой. Любое конкретное непустое значение после `secret`, `token`, `password`, `api_key` или `api-key` всё ещё считается потенциальным секретом.

`forbiddenPatterns` расширяет этот список для конкретной задачи.

Сканирование содержимого выполняется и при сборке, и при `audit package verify`. Текстовые файлы архива не должны содержать путь к корню локального репозитория, системному временному каталогу или Windows-путь с диском вида `[A-Z]:\...` и `[A-Z]:/...`. JSON-escaped кавычки после drive-like пунктуации, например `[A-Z]:\u0022` или `[A-Z]:\u0027`, не считаются путём только как isolated values без filename segment continuation и без последующего slash/backslash до terminator-а; реальные drive-root paths с slash/backslash после двоеточия остаются запрещены, включая JSON-строки с удвоенным backslash. Для patch это правило применяется к оставшимся текстовым diff-блокам целиком, включая удалённые строки из `baseline`; исключаются только исторические diff-блоки verdict-файлов из `previousVerdictChain`.

Для TRX-доказательств это правило применяется после безопасной нормализации служебных XML-атрибутов тестового раннера, стабильного переименования TRX-файлов и замены исходных TRX-путей в выводе проверок. Секретные значения в тексте результатов тестов остаются запрещёнными; исключение касается только машинно созданных отображаемых имён тестов и технических путей сборки, которые не являются содержимым проверяемого вывода.

`audit package verify` также завершается ошибкой, если сгенерированные имена путей архива, имя patch, корень `evidence/<task-id>-<iteration>/`, структурные поля `metadata/audit-package.input.json` или блок метаданных `AUDIT-MANIFEST.md` не совпадают с текущими `taskId` и `iteration`. Проверка не сканирует весь Markdown/JSON как свободный текст: связанные задачи вроде `T-0232` допускаются в `scopeTaskIds`, а исторические ссылки на задачи вроде `T-0207` допускаются в `previousVerdictChain` и `blockerClosureList`, потому что эти поля описывают область текущего пакета, цепочку внешних verdict-ов и закрытые blocker-ы, а не имя текущего архива. Корневой `AUDIT-REQUEST.md` больше не содержит сгенерированный заголовок текущей задачи: его соответствие проверяется сравнением с восстановленным отслеживаемым источником.

## Запрос аудитору

`docs/release-management/AUDIT-REQUEST.md` является отслеживаемым исходным контрактом: это постоянный текст запроса внешнему аудитору, который хранится в репозитории, проходит проверку как обычная документация и копируется в корень ZIP как `AUDIT-REQUEST.md` без подстановок, генерации блоков или замены заполнителей. Этот файл также фиксирует контракт проверки области пакета и предыдущих внешних verdict-ов: аудитор обязан читать `metadata.scopeTaskIds`, `metadata.scopeSummary`, `metadata.previousVerdictChain`, `metadata.blockerClosureList`, предыдущие verdict-файлы, дословное сохранение исторических verdict-ов и закрытие каждого предыдущего blocker-а.

Статический запрос обязан требовать проверку содержимого изменения, тестов, документации, секретов и области задачи. Внешний аудитор не проверяет упаковочный слой: целостность ZIP, контрольные суммы, сопровождающий ZIP, операторские доказательства, локальную команду `audit package verify` и применимость patch. Эти проверки остаются локальным предварительным шагом команды репозитория до отправки.

Приёмка verdict-а использует безопасное операторское правило из раздела ниже. Статический запрос внешнему аудитору требует один финальный отчёт и запрещает промежуточные сообщения с `VERDICT:`, но не содержит DOM-селекторы или инструкции по чтению интерфейса.

Динамические сведения о конкретной задаче остаются в проверяемых машинных файлах:

- `AUDIT-MANIFEST.md` - инвентарь архива, исходная ревизия, задача, итерация, список файлов и доказательств;
- `metadata/audit-package.input.json` - нормализованная входная конфигурация;
- `repo-file-hashes.json` - модель восстановления файлов репозитория.

Перед созданием ZIP команда обязана прочитать `docs/release-management/AUDIT-REQUEST.md` из рабочей копии и отклонить пакет, если файл отсутствует, пустой, выглядит как однострочная заглушка, содержит конфиденциальное значение, машинный абсолютный путь или не содержит обязательные маркеры ответа:

- `VERDICT: ACCEPT`;
- `VERDICT: NEEDS_FIXES`;
- `TASK_ASSESSMENT`;
- `BLOCKERS`;
- `EVIDENCE_REVIEW`;
- `RISKS_AND_NOTES`;
- `CLOSURE_DECISION`;
- `metadata.scopeTaskIds`;
- `metadata.scopeSummary`;
- `combined scope`;
- `metadata.previousVerdictChain`;
- `metadata.blockerClosureList`;
- `previous verdict files`;
- `verbatim preservation`;
- `previous blockers closure`;
- `metadata/repo-file-snapshots.json`;
- `repo-after/`;
- `repo-before/`;
- `implementation content review`;
- `test coverage review`;
- `documentation review`;
- `task compliance review`;
- `secret scanning`;
- `scope scanning`;
- `evidence gap`;
- `patch-only inspection`;
- `single final report`;
- `no intermediate VERDICT`;
- `full current-scope engineering review`;
- `FOLLOW_UP_FINDING`;
- `OUT_OF_SCOPE_NOTE`;
- `ACCEPTED_RISK`;
- `INFO_NOTE`;
- `global safety blocker`;
- `architecture coherence`.

`AUDIT-MANIFEST.md` обязан явно указать исходный путь этого статического файла: `docs/release-management/AUDIT-REQUEST.md`. Корневой `AUDIT-REQUEST.md` в ZIP сохраняет текст отслеживаемого источника в той же LF-нормализованной форме, что `repo-after/` снимки и patch. При `audit package verify` после применения patch команда сравнивает `AUDIT-REQUEST.md` из архива с восстановленным `docs/release-management/AUDIT-REQUEST.md`; любое расхождение означает, что архив не соответствует отслеживаемому контракту и должен быть отклонён. Та же проверка сверяет `metadata/repo-file-snapshots.json` с `repo-file-hashes.json`: каждый существующий итоговый файл должен иметь `repo-after/` снимок с тем же SHA-256, каждый удалённый или изменённый файл, существовавший в `baseline`, должен иметь `repo-before/` снимок с SHA-256 исходной ревизии, а отсутствующий или повреждённый снимок блокирует отправку архива.

## Отправка внешнему аудитору

Внешний аудит разрешён только после успешного `audit package verify` на отдельной чистой копии исходной ревизии. Агент всегда запускает одну штатную команду `audit submit`; C#-инструмент по сохранённым verdict-ам и глобальному rNN сам выбирает новый primary, reuse существующего обсуждения или новый clean control. Он открывает одну управляемую вкладку, прикрепляет проверенный основной ZIP, при новом чате вставляет полный request, а при reuse оставляет composer без повторного prompt-а, затем проверяет payload, отправляет ordinary ChatGPT request и ждёт новый ответ текущей отправки. Финальный Markdown получается одним copy-click: системный clipboard принимается только после смены native sequence number, а source-scoped captured `writeText()`, `write(...)` или `copy` event payload служит вторым доказательством. Browser `readText()`, sentinel и произвольный старый clipboard text не используются. Режим `--download-report-only` не отправляет новый запрос. В `finally` инструмент закрывает только созданную им вкладку и не изменяет пользовательские вкладки.

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Если текст запроса уже подготовлен отдельным файлом, его можно передать явно:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --message <message-path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Если отчёт уже готов в существующем ChatGPT discussion URL, допускается диагностический режим только для скачивания результата без повторной отправки архива:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --download-report-only --project-url <chat-url> --out docs/verdicts/<domain>/<task-id>-audit[-control]-rNN.md --browser-backend codex-chrome
```

Для повторной исправительной или контрольной итерации используется та же штатная команда выше. Сохранённую ссылку и тип route выбирает инструмент, а не агент.

Если сохранённое обсуждение задачи нельзя продолжать через поддержанный submit-путь, используйте явный новый чат проекта. Это не меняет ZIP и не разрешает ручную отправку:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --new-conversation --zip <path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

После `ACCEPT` следующая команда автоматически требует clean-control ZIP и control-имя отчёта следующей глобальной итерации.

Короткий ручной запрос (`prompt`, то есть текст, написанный оператором вместо проверенного текста из архива), отправка без прикреплённого основного аудиторского ZIP и ручная отправка вместо `audit submit` запрещены. Нельзя заменять вывод `audit package message --zip <path>` фразами вроде `Приложен verified audit package`, пересказом задачи или служебным обрамлением `<task-id> audit rNN`: аудитор должен получить один проверенный основной ZIP и дословный текст запроса из архива без первого Markdown H1.

## Извлечение verdict-а - жёсткое правило

Это операторское правило для чтения результата внешнего аудита из текущего audit chat. Оно защищает приёмку от ошибки, когда `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES` найден в пользовательском запросе, примере формата, истории, боковой панели, обычном превью карточки или частично загруженном тексте страницы.

Финальный verdict = первая непустая строка единственного полного текста отчёта, полученного штатным `audit submit`. Для новой отправки это новый ordinary assistant-ответ текущей отправки. В режиме `--download-report-only` это последний assistant-ответ существующего обсуждения, если controlled ordinary copy action возвращает строгий verdict-отчёт, либо Markdown-файл старой отчётной карточки через legacy recovery. Всё остальное не verdict.

Чтобы старые отчётные карточки той же задачи не могли быть сохранены как новая итерация или поверх предыдущего verdict-а, внешний отчёт должен в `TASK_ASSESSMENT` явно называть проверенные `metadata.taskId` и `metadata.iteration`, а `audit submit` принимает только строгий repo-relative путь `--out` вида `docs/verdicts/<domain>/<task-id>-audit-rNN.md` для primary или `docs/verdicts/<domain>/<task-id>-audit-control-rNN.md` для control. Строгий путь требует точные сегменты `docs/verdicts`, точное расширение `.md`, доменный сегмент, отсутствие произвольного префикса, абсолютного пути, пустого сегмента, `.`, `..` и вариаций регистра фиксированных сегментов или расширения. Обычный submit сверяет этот путь с текущей ZIP identity и автоматически выведенным route, а `--download-report-only` выводит primary/control и текущую identity из filename и затем дополнительно сравнивает её с явными ссылками на evidence/ZIP в скачанном Markdown.

Порядок чтения:

1. Источник verdict-а новой отправки - только новый ordinary ответ ассистента текущей отправки после увеличения счётчика сообщений. Источник verdict-а в `--download-report-only` - только controlled ordinary copy action последнего assistant-ответа существующего обсуждения или Markdown-файл старой отчётной карточки, скачанный legacy recovery path.
2. Запрещено определять verdict по `main.textContent`, всему `article`, `conversation-turn`, sidebar/history, user/tool messages, обычному превью карточки, counters, `verdictCount`, regex over full page или `matches.at(-1)`.
3. Сначала убедитесь, что ChatGPT не показывает видимую активную кнопку остановки ответа текущей отправки; если она видна, генерация ещё активна, даже когда в истории уже есть старый завершённый отчёт. После завершения генерации новой отправки читайте только последний новый ordinary ответ ассистента. Legacy frame/target export разрешён только read-only recovery старого отчёта; при неоднозначности нескольких карточек или export actions источник не выбирается произвольно.
4. Первая непустая строка сохранённого текста должна быть ровно `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`.
5. Текст должен содержать обязательные секции `TASK_ASSESSMENT:`, `BLOCKERS:`, `EVIDENCE_REVIEW:`, `RISKS_AND_NOTES:` и `CLOSURE_DECISION:` в таком порядке.
6. Первый единственный текст отчёта, который прошёл строгую проверку и получен при неактивной генерации из разрешённого источника текущего режима, является готовым отчётом; команда не выполняет второй export только для подтверждения того же текста.
7. Если первая непустая строка не verdict, найдено несколько отчётных карточек или несколько candidate report texts, текст начинается с `Вы сказали`, извлечение включает пользовательский запрос, историю, боковую панель, обычное превью карточки или сообщения инструментов, либо была сжатая передача контекста (`context compaction`) или прерывание (`interruption`), остановитесь: не закрывайте задачу и перечитайте источник заново.
8. `ACCEPT` валиден только если первая строка полного текста отчётной карточки ровно `VERDICT: ACCEPT`, отчёт содержит обязательные секции в нужном порядке, секция `TASK_ASSESSMENT:` явно называет текущие `metadata.taskId` и `metadata.iteration`, а секция `BLOCKERS` не содержит `B1`..`Bn`. Текст `CLOSURE_DECISION` не должен становиться дополнительным словарным барьером после `VERDICT: ACCEPT`: он сохраняется как человеческое объяснение решения, а не как отдельный машинный verdict. Любой `VERDICT: ACCEPT` из prompt-инструкций, примеров формата, предыдущих сообщений, истории или превью карточки не имеет силы.

Если операторский инструмент не может получить валидный отчёт из разрешённого источника текущего режима, задача остаётся открытой до повторного чтения источника. Нельзя заменять это правило счётчиком совпадений, предварительным просмотром, regex over full page или любым агрегированным текстом страницы.

## Cleanup

Audit ZIP, сопровождающий ZIP операторских доказательств, временные каталоги и сырые доказательства не коммитятся. После `VERDICT: ACCEPT` рабочие архивы штатно удаляются вручную или будущей командой:

```bash
dotnet run --project eng/Electron2D.Build -- audit cleanup --task <task-id>
```

`audit package` не создаёт финальный ZIP после принятия задачи; новый архив нужен только для следующей итерации внешнего аудита.

Правило остановки, заморозка области, запрет менять принятую область после primary `ACCEPT`, сводки классов отказа и автоматическая защита от расползания области остаются отдельными правилами процесса и должны усиливаться отдельными задачами после final `ACCEPT` T-0238.

Дополнительно работает closure coverage для actionable `OUT_OF_SCOPE_NOTE` и `INFO_NOTE`: explicit marker `Actionable: true`, `Actionable: yes` или `[actionable]` переводит такую запись из `RISKS_AND_NOTES` в обязательный follow-up finding, который должен иметь closure note. `audit submit --download-report-only` также поддерживает текущую структуру ChatGPT: для обычного ответа он пробует controlled copy action последнего assistant-сообщения существующего обсуждения, видимый Deep Research iframe без готового отчёта и собственной export surface не блокирует fallback к готовой карточке выше по обсуждению, а page-level export может прокрутить единственную report-card кнопку экспорта в видимую область и нажать её через DOM без координатного клика. Legacy read-only recovery покрыт регрессиями, которые исполняют production export expressions и downloader-контракты. `AuditSubmitMarkdownMenuSelectorAcceptsPlainChatGptMenuRow` проверяет Markdown-export пункт старой отчётной карточки. `AuditSubmitReadyTargetSelectionWaitsForNewestNonBaselineTarget` и `AuditSubmitReportCandidateFlowBlocksPageFallbackWhileNewestTargetGenerates` проверяют выбор готовой legacy target surface без создания нового исследования. Итоговый Markdown всегда проходит строгий валидатор текущей task/iteration.

`AuditSubmitPromptSubmissionUsesOrdinaryChatByDefault` проверяет единственный контракт новой отправки: основной ZIP прикрепляется до вставки текста, текст и ZIP-плашка проверяются до чтения счётчика сообщений, после чего выполняется ordinary send. `AuditSubmitPromptSubmissionSkipsPromptFillForEmptyReuseMessage` фиксирует другой порядок для reuse: сначала очищается любой восстановленный черновик, затем прикрепляется основной ZIP, повторный prompt не вставляется, а финальный guard требует пустое поле. `AuditSubmitAlwaysUsesOrdinaryChatAndFinalizesOwnedTabsAcrossResults` исполняет production owned-tab orchestration и проверяет отсутствие Deep Research-ветки, а также ровно одну финализацию после успешного отчёта, ошибки отчёта и исключения callback сразу после создания вкладки. `AuditSubmitOrdinaryCopyResetsPreloadCaptureBeforeClick` исполняет тот же типизированный production copy orchestration для native clipboard, captured payload, отсутствующей кнопки, неготового capture и ошибки после клика; все post-click ветки содержат ровно один click. Статические assertions в этом тесте остаются только запретами browser clipboard read/permission и не используются как доказательство порядка действий. `AuditSubmitDeepResearchOptionIsAbsentBeforeBrowserLaunch` проверяет, что старой опции нет в parser allowlist. `AuditSubmitPromptPayloadReadyRequiresPromptTextAndAuditZipChip` исполняет производственное DOM-выражение `PromptPayloadReadyExpression` на контролируемой модели поля отправки и проверяет положительный случай, ZIP-плашку с вложенным элементом имени файла, текущую компактную плашку `имя ZIP + ZIP-архив`, пустое поле, отсутствующую ZIP-плашку, обычную метку рядом с полем с именем ZIP без признаков прикреплённого файла и ZIP-плашку из истории сообщений. Controlled result использует native clipboard sequence change либо source-scoped captured Markdown; `navigator.clipboard.readText()` отсутствует. Собственная вкладка закрывается подтверждённым `finalizeTabs` в `finally` при любом исходе, включая callback failure сразу после `createTab`; ошибка закрытия не выдаётся за успешное завершение. `--download-report-only` остаётся read-only recovery, `--deep-research` отклоняется, а `ACCEPT` проходит строгий identity/headings/blockers validator.

Перед отправкой штатный ordinary path дополнительно перечитывает состояние поля отправки и до `30` секунд ждёт, пока одновременно будут видны тело сообщения и плашка основного audit ZIP. Если тело сообщения или видимая ZIP-плашка не готовы, команда останавливается с `E2D-BUILD-AUDIT-SUBMIT-PAYLOAD-MISSING` и пишет последнюю структурную причину отказа. Эта проверка выполняется до чтения счётчика сообщений и до `ClickSendAsync`.

Прикрепление ZIP не открывает меню вложений. В новом primary/control чате перед side effect инструмент допускает только два состояния текущего composer: пустое поле либо точный уже восстановленный request-текст; в reuse-чате он сначала очищает любой текст и допускает только пустое поле. Если точный ожидаемый ZIP уже прикреплён к допустимому состоянию composer, готовый payload переиспользуется без второй установки. Для нового прикрепления инструмент сначала выбирает точный `#upload-files`, а при его отсутствии - единственный подходящий file input только внутри формы текущего composer. Произвольный глобальный `input[type=file]` не является fallback. Выбранный DOM-элемент получает уникальный marker и сохраняется как реальная ссылка в page-global registry по случайному token. Инструмент преобразует этот элемент в `backendNodeId` и ровно один раз вызывает `DOM.setFileInputFiles` с `allowTransientRecovery: false`, поэтому потеря ответа CDP не может автоматически повторить установку того же ZIP. Post-set guard получает тот же объект из registry, проверяет подключённость, точное число файлов и ожидаемое имя и удаляет marker без ручной генерации `input` или `change`. До заполнения request-текста инструмент ждёт появления единственной ZIP-плашки у допустимого состояния composer; компактная метка с точным именем принимается только вместе с тем же единственным именем в `input.files` текущей формы. Если input не найден, post-set guard не подтверждён или точная ZIP-плашка не появилась, отправка завершается с `E2D-BUILD-AUDIT-SUBMIT-ATTACHMENT-MISSING` до prompt/Send. Финальный payload guard непосредственно перед Send требует точный request-текст и ZIP для нового чата либо пустой composer и ZIP для reuse.

Обычный submit никогда не обращается к Deep Research targets. При необходимости прочитать старую отчётную карточку оператор запускает отдельный `--download-report-only` для конкретного conversation URL; этот read-only путь выбирает legacy target, экспортирует Markdown и отсекает stale task/iteration тем же строгим валидатором.

После `T-0235` `--download-report-only` принимает только конкретный ChatGPT conversation URL с сегментом `/c/<conversation-id>` и строгий repo-relative output path: primary `docs/verdicts/<domain>/<task-id>-audit-rNN.md` либо control `docs/verdicts/<domain>/<task-id>-audit-control-rNN.md`. Тип recovery выводится из filename без отдельного route-флага. Project/root/foreign URL, output path с произвольным префиксом перед `docs/verdicts/`, output без доменного сегмента, path traversal через `..`, неверный регистр сегментов `docs/verdicts` или расширение не ровно `.md` отклоняются до подключения к браузеру. Обычный submit после подтверждённой отправки ZIP ждёт перехода текущей вкладки на concrete conversation URL и записывает primary sidecars `.temp/audit/<task-id>/conversation-url-rNN.txt` / `conversation-url.txt` или control sidecars `control-conversation-url-rNN.txt` / `control-conversation-url.txt` только с таким URL, чтобы recovery не получал общий project URL; если concrete URL не появился или sidecar path нельзя вывести из штатного имени ZIP, команда завершается диагностикой `E2D-BUILD-AUDIT-SUBMIT-CONVERSATION-URL-MISSING`. Обычный submit сверяет `--out` с текущими `taskId`/`iteration` из ZIP и route, выведенным state machine до подключения к браузеру. После `NEEDS_FIXES` state machine выбирает primary sidecar либо control sidecar, после `ACCEPT` начинает новый clean-control чат проекта; `--new-conversation` остаётся только аварийным primary reset. Read-only извлечение Markdown сначала проверяет последний assistant-ответ через controlled ordinary copy action; legacy frame/target recovery используется только для старых отчётных карточек. Ошибка `E2D-BUILD-AUDIT-SUBMIT-REPORT-INVALID` дополняется конкретной причиной: неоднозначные кандидаты, неподходящий source, prompt echo/template, неверная первая строка, отсутствующий heading, отсутствие текущих `metadata.taskId`/`metadata.iteration` в `TASK_ASSESSMENT:` или numbered blocker в `ACCEPT`.

CDP reattach после recoverable detach требует, чтобы хотя бы одна короткая domain probe (`Runtime.enable`, `DOM.enable` или `Page.enable`) подтвердила рабочее подключение; если все probes всё ещё возвращают detach, reattach продолжается и не выдаётся за успех. После reattach исходная read-only CDP-команда повторяется. Исчерпание локального burst-recovery во время ordinary generation polling переводится в следующий poll той же вкладки и ограничивается общим timeout всей команды; для остальных операций сохраняется fail-closed диагностика `E2D-BUILD-AUDIT-SUBMIT-CODEX-CHROME-PROTOCOL`. DOM-клики отправки, copy action, прикрепления файла или выбора меню после recoverable detach не повторяются.

Ручная сборка ZIP для внешнего аудита запрещена. Внешний аудит допускается только после успешной пары команд `audit package` и `audit package verify` на отдельной чистой копии исходной ревизии; штатная отправка и извлечение отчёта выполняются через `audit submit`, который использует проверенный основной ZIP и текст из `audit package message --zip <path>`, но не прикрепляет сопровождающий ZIP операторских доказательств. Новая отправка всегда использует обычный запрос в ChatGPT; `--deep-research` отсутствует в CLI и отклоняется как неизвестная опция.

# Детерминированный `audit package`

Обновлено: 2026-07-03.

Этот файл является доменным документом для внутренней команды сборки аудиторского архива. Документ фиксирует ожидаемый контракт, текущую модель реализации, ограничения и проверки в одном месте.

## Ведение документа

- Перед изменением команды `audit package` сначала обновите этот документ.
- Затем добавьте точечные тесты в `RepositoryBuildToolTests`, реализуйте изменение в `eng/Electron2D.Build` и снова синхронизируйте этот документ, если фактическое поведение отличается от контракта.
- Доказательства аудита не собираются вручную: архив создаёт только C#-команда репозитория, а внешний аудит допускается только после проверки восстановления.

## Назначение

`audit package` заменяет ручную сборку ZIP-архива для внешнего аудита. Команда создаёт воспроизводимый основной архив из явной конфигурации, запускает заданные проверки, сохраняет сырые доказательства (`evidence`, то есть команды, вывод и коды завершения), строит patch штатным `git diff --find-renames --binary` через временный индекс Git и проверяет, что чистая копия репозитория восстанавливается из исходной ревизии (`baseline`) до ожидаемого состояния. Текущий контракт не вкладывает в основной ZIP снимок исходной ревизии: чистая копия исходной ревизии является обязательным внешним входом проверки восстановления (`external baseline REQUIRED input`), а отсутствие такого вложенного набора файлов не является требованием контракта (`no baseline payload requirement`). Чтобы отсутствие вложенного снимка не превращалось в неподтверждённое допущение, команда после финальной записи основного ZIP создаёт сопровождающий ZIP `<task-id>-audit-rNN.operator-workflow.zip` с `operator workflow evidence` - переносимыми сырыми доказательствами запуска `audit package verify` и `audit package message` для неизменяемого основного ZIP.

Инструмент является внутренней автоматизацией репозитория. Он не является публичной командой `e2d`, не входит в пакет среды выполнения и не должен публиковаться как часть релиза.

## Границы правил и запросов

`AGENTS.md` задаёт правила работы агента внутри репозитория: безопасность, ведение `TASKS.md`, дневник, проверки, сборку аудиторского архива и обязательный путь внешней приёмки. Он может ссылаться на этот доменный документ, но не должен описывать ручные клики, чтение страницы ChatGPT, работу с cookies или альтернативный браузерный способ вместо `audit submit`.

`docs/release-management/AUDIT-REQUEST.md` является только текстом для внешнего аудитора. Он описывает, какие материалы из одного приложенного audit ZIP нужно проверить и какой финальный отчёт вернуть. Этот файл не описывает Chrome, вкладки, cookies, локальную проверку ZIP, контрольные суммы, ручное скачивание отчёта или операторский порядок отправки.

`.codex/prompts/goal-prompt.md` является коротким active goal prompt-ом оркестратора и должен оставаться не длиннее 3500 символов. Он задаёт цель запуска одной компактной формулировкой и ссылается на workflow-модель вместо копирования её шагов.

`.codex/prompts/goal-task-workflow.md` является локальной workflow-моделью goal-задачи. Он подробно описывает порядок работы: предварительное чтение контекста, работу исполнителя, проверки, `audit package`, `audit package verify`, `audit submit`, обработку verdict-а, повторный цикл исправлений, принятие и commit без push. Все детали браузера, режима `Глубокое исследование`, прикрепления ZIP, обновления страницы, экспорта Markdown и закрытия собственной вкладки принадлежат только C#-команде `audit submit` и этому документу.

Любая задача, которую агент доводит до принятия после введения этого правила, остаётся открытой до внешнего отчёта `VERDICT: ACCEPT`, полученного штатной командой `audit submit` и сохранённого в `docs/verdicts/<domain>/<task-id>-audit-rNN.md`. Prompt-ы и агентские правила не должны принимать `VERDICT: ACCEPT` из текста запроса, примера формата, истории чата, обычного предпросмотра или ручного чтения страницы.

## Экономика локального цикла

Внешний аудит является приёмочным барьером, а не инструментом отладки локальной автоматизации. Он должен отвечать на вопрос «можно ли принять проверяемый пакет?», а не искать следующий дефект в выборе режима `Глубокое исследование`, прикреплении ZIP, восстановлении архива, выгрузке Markdown или распознавании старого отчёта. До создания нового пакета командой `audit package` должны быть закрыты локальные причины отказа, которые можно проверить без внешнего исследования.

Для локальной работы используются три уровня проверок:

- `Fast` - быстрый уровень после мелкой правки. Цель - проверить новый регрессионный тест и ближайший контракт за 1-2 минуты. Этот уровень не должен запускать `audit package`, `audit package verify`, восстановление чистой копии, браузерную отправку, загрузку Markdown-отчёта или широкие фильтры интеграционных тестов без прямой необходимости.
- `Medium` - средний уровень перед созданием пакета. Цель - доказать затронутый внутренний механизм через поведенческий тест, релевантный точечный набор тестов, документационные проверки, `verify audit-followups` при изменении отчётов или заметок о закрытии, проверку лицензий при изменении исходных файлов и `git diff --check`. Ориентир - до 10 минут; если средний уровень регулярно выходит за этот бюджет, это отдельный долг процесса.
- `Heavy` - тяжёлый приёмочный барьер перед реальной отправкой: представительский набор проверок, который реально создаёт audit ZIP, запускает `audit package verify` на отдельной чистой копии, проверяет `audit package message` и сохраняет обязательные `operator workflow evidence`, то есть доказательства запуска операторских команд. Этот уровень запускается только после зелёного `Medium` и не используется после каждой мелкой правки. Полный исчерпывающий набор всех `AuditTier=Heavy` сценариев остаётся отдельным срезом `audit-exhaustive` для изменений самой упаковки, восстановления, ZIP-структуры, сканирования секретов или перед ручным глубоким разбором качества аудиторского инструмента; обычный локальный и CI-коридор не должен платить за этот набор после каждой правки.

Штатная команда быстрого уровня для аудиторских контрактов - `dotnet run --project eng/Electron2D.Build -- verify audit-contracts`. Она читает `docs/release-management/AUDIT-REQUEST.md`, этот документ, `.codex/prompts/goal-prompt.md`, `.codex/prompts/goal-task-workflow.md`, `docs/repository/agent-workflow.md`, `AGENTS.md` и in-process проверяет обязательные маркеры, классификацию отчётов и контракт разбора `RISKS_AND_NOTES`. Команда не создаёт audit ZIP, не распаковывает ZIP, не создаёт чистую копию репозитория, не вызывает `audit package verify`, не запускает браузерную автоматизацию и не вызывает дочерние `dotnet run`. Ожидаемый объём - 20-40 лёгких проверок, ожидаемое внутреннее время выполнения - до 5 секунд, допустимый верхний предел для обычного операторского запуска через `dotnet run` после сборки - 30 секунд. Превышение этого бюджета без явного объяснения означает регрессию быстрого уровня.

Итоговая диагностика `verify audit-contracts` должна содержать человекочитаемую сводку: `AuditTier=Fast`, число проверок, сколько прошло, сколько упало, сколько пропущено, длительность и явное подтверждение, что тяжёлый путь не запускался. Эта сводка нужна для записи в дневник и `TASKS.md` в числовом виде, а не общей фразой "точечные проверки прошли".

Интеграционные тесты аудиторской автоматизации в `RepositoryBuildToolTests` должны иметь ровно одну метку `[Trait("AuditTier", "...")]` со значением `Fast`, `Medium` или `Heavy`. `Fast` разрешён только для проверок исходного текста, документов и внутренних контрактов без дочерних процессов; защитный тест `AuditWorkflowFastAuditTierDoesNotUseHeavyHelpers` запрещает в таких тестах вспомогательные методы, которые запускают `dotnet`, Node, `audit package`, `audit package verify`, чистую копию репозитория или браузерную автоматизацию. `Medium` разрешает точечные интеграционные проверки и контролируемые дочерние процессы, но защитный тест `AuditWorkflowMediumAuditTierDoesNotUseHeavyPackagingHelpers` запрещает helper-ы упаковки, восстановления чистой копии, `audit package verify` и `audit package message`. Для части тяжёлых тестов используется дополнительная метка `[Trait("AuditCadence", "Acceptance")]`: она означает не новый уровень качества, а частоту запуска, то есть короткий представительский набор, который должен проходить перед внешней упаковкой и в CI. Штатные срезы `dotnet run --project eng/Electron2D.Build -- test --integration-slice audit-medium` и `dotnet run --project eng/Electron2D.Build -- test --integration-slice audit-heavy` выбирают соответственно `AuditTier=Medium` и `AuditTier=Heavy` вместе с `AuditCadence=Acceptance`; срез `audit-exhaustive` выбирает полный `AuditTier=Heavy` без ограничения по частоте запуска. Совместимый срез `audit-package` остаётся объединением `AuditTier=Medium|AuditTier=Heavy`, но не используется как основной локальный коридор. При запуске `test --no-build` команда передаёт в дочерний `dotnet test` переменную `ELECTRON2D_BUILD_TOOL_NO_BUILD=1`, чтобы helper-ы внутри integration tests запускали `eng/Electron2D.Build` через `dotnet run --no-build` и не пересобирали build tool десятки раз в широком срезе. Каждый аудиторский срез `test --integration-slice audit-*` печатает `E2D-BUILD-TEST-SLICE-SUMMARY` с именем команды, именем среза, `AuditTier`, числом дочерних процессов, распознанным числом тестов, длительностью, тайм-аутом и причиной сбоя или `failure=none`. Широкие `FullyQualifiedName~AuditPackage`/`AuditSubmit`/`AuditWorkflow` не являются tier-контрактом. Срез `repository-tooling` исключает все три audit-уровня, чтобы не смешивать обычные проверки build-tool с аудиторским циклом.

`audit package` после создания operator workflow sidecar обязан печатать `E2D-BUILD-AUDIT-OPERATOR-WORKFLOW-SUMMARY`: какие sidecar-команды реально запускались, их коды выхода, длительность каждой команды, общий `totalDurationMs`, тайм-аут и путь сопровождающего ZIP. Эта диагностика не заменяет evidence-файлы внутри sidecar, а даёт оператору числовую стоимость оставшейся тяжёлой части прямо в выводе команды.

Перед `Heavy` у оператора должна быть краткая матрица доказательств (`proof matrix`) - проверяемая таблица или список вида «замечание или локальный отказ -> изменённый рабочий путь кода или стабильный внутренний драйвер/тестовая обвязка -> имя теста -> команда или фильтр -> ожидаемые доказательства». Если заявленное закрытие blocker-а или локального отказа не связано с поведенческим тестом либо стабильным внутренним контрактом, пакет не считается готовым к созданию.

Для исправительной итерации с непустым `metadata.previousVerdictChain` список `metadata.blockerClosureList` обязателен. Каждая запись закрытия должна называть хотя бы один фактически настроенный `checks[].name` из текущего `audit package` config, который даёт локальное доказательство закрытия. Для существующих прошлых `VERDICT: NEEDS_FIXES` отчётов из `previousVerdictChain` список также должен покрывать каждый найденный blocker: запись закрытия должна назвать путь прошлого отчёта, идентификатор blocker-а вида `B1`, `B2` и имя configured check текущего пакета. Если существующий прошлый отчёт `VERDICT: NEEDS_FIXES` не содержит в секции `BLOCKERS` распознаваемые `B*` blocker ids, упаковка должна завершиться диагностикой `E2D-BUILD-AUDIT-CONFIG-INVALID`: такой отчёт нельзя считать отсутствием прошлых blocker-ов, потому агент не сможет доказуемо закрыть их пачкой. Это не валидация Markdown-отчёта внешнего аудитора и не требование писать отчёт машинным форматом: проверяется только операторская конфигурация пакета до отправки. Если прошлый blocker закрыт только свободной фразой без связи с configured check, без пути отчёта или без идентификатора blocker-а, `audit package` должен завершиться диагностикой `E2D-BUILD-AUDIT-CONFIG-INVALID` до запуска внешнего аудита.

Обычный исправительный цикл внешнего аудита имеет бюджет внимания: первая отправка без прошлых отчётов, вторая отправка с одним прошлым verdict-отчётом и третья отправка с двумя прошлыми verdict-отчётами. Если для новой упаковки требуется больше двух записей в `metadata.previousVerdictChain`, задача не обрывается, но `audit package` переводит цикл в стабилизационный режим: среди configured checks должен быть отдельный check с именем `audit-loop-stabilization`. Этот check должен быть локальной пакетной проверкой классов причин, которые уже проявились во внешних отчётах, например driver/harness-тестом для browser DOM path, clean-control context, payload attachment guard, report extraction или restore/package model. Без такого check-а новая упаковка завершается `E2D-BUILD-AUDIT-CONFIG-INVALID`. Смысл правила - не запретить работу после `r03`, а не позволить продолжать `r04`, `r05` и далее как слепую серию внешних отладочных запусков.

Доказательство по тексту исходника (`source-level proof`), то есть тест, который читает C#-файл как текст и ищет строку или порядок строк, не доказывает поведение `audit submit`, DOM-выражений, выбора `Глубокое исследование`, выгрузки Markdown, ожидания отчёта, восстановления архива или сохранения verdict-а. Такой тест допустим только для статического документационного маркера или явного запрета текста, но не для закрытия поведенческого blocker-а.

## Правило остановки (`stop-loss`) для повторяющихся отказов

Класс локального отказа (`failure class`) - это одинаковая причина на одном участке цикла: одинаковый диагностический код, например `E2D-BUILD-AUDIT-SUBMIT-DEEP-RESEARCH-MISSING`, `E2D-BUILD-AUDIT-SUBMIT-REPORT-EXPORT-MISSING`, `E2D-BUILD-AUDIT-SUBMIT-REPORT-STALE`, `E2D-BUILD-AUDIT-RESTORE-MISMATCH`, либо одинаковый этап и смысловая причина, если диагностический код ещё недостаточно точен. Неудачная попытка, которая не создала сохранённый verdict-отчёт, не является полезной внешней итерацией.

Если один класс локального отказа повторился два раза в одной задаче или в одном цикле аудита (`audit-loop`), запрещено собирать или отправлять следующую `rNN` «на удачу»: нужно не увеличивать `rNN`, а остановить цикл упаковки и отправки. Сначала нужно зафиксировать класс отказа в `TASKS.md` и дневнике, добавить локальный регрессионный тест через внутренний драйвер или тестовую обвязку для рабочего пути кода либо стабильного внутреннего контракта, пройти зелёный `Fast`, затем зелёный `Medium`. Только после этого разрешён новый `Heavy`.

Если локальный драйвер или тестовая обвязка для повторяющегося класса отказа невозможны, оператор не должен продолжать увеличивать `rNN`. Нужно явно записать blocker процесса: какая часть не моделируется локально, какие данные нужны, и почему следующий внешний запуск не будет слепым повтором той же попытки.

## Заморозка области цикла аудита

После первого `audit package` или `audit submit` для задачи область пакета заморожена. Нельзя добавлять новые активные задачи, менять несвязанные prompt-ы, соседние документы, дорожную карту, инструменты сборки или критерии приёмки, если это не закрывает сохранённый blocker текущей итерации или явно не включено в `metadata.scopeTaskIds`, `metadata.scopeSummary`, manifest и матрицу доказательств как объединённая область (`combined scope`).

Новые идеи, улучшения процесса и соседние исправления после старта цикла аудита записываются как отдельная задача, запись дневника или `accepted-risk`, но не попадают в текущий пакет без явной объединённой области. Если объединённая область всё же нужна, она должна быть заявлена до упаковки и стать предметом текущей внешней проверки.

## После primary `VERDICT: ACCEPT`

После сохранённого primary `VERDICT: ACCEPT` принятую область нельзя менять до control audit. Запрещены изменения кода, тестов, доменных документов, доказательств, критериев и конфигурации пакета, которые меняют уже принятую область. Разрешены только сохранение отчёта, локальное состояние отправки, сборка отдельно проверенного чистого control ZIP без прошлого контекста, отправка control audit в новый чат проекта и учётное закрытие, которое не меняет принятую область.

Перед любым изменением после primary `ACCEPT` оператор сначала классифицирует `RISKS_AND_NOTES`. Если замечание можно закрыть через запись в `TASKS.md`, `duplicate`, `accepted-risk`, `tracked-existing`, `tracked-new`, `not-actionable` или другое учётное решение без изменения принятой области, код и документы не меняются. Если замечание действительно требует изменения кода, тестов, документов, доказательств или критериев принятой области, право на control audit сбрасывается: нужен новый пакет и новый primary audit.

## Контракт CLI

Быстрая проверка аудиторских контрактов:

```bash
dotnet run --project eng/Electron2D.Build -- verify audit-contracts
```

Сборка архива:

```bash
dotnet run --project eng/Electron2D.Build -- audit package --task <task-id> --iteration rNN --baseline <sha> --config <path> --out .temp/audit
```

Проверка уже собранного архива на чистой копии:

```bash
dotnet run --project eng/Electron2D.Build -- audit package verify --zip <path> --baseline <sha> --repo <clean-repo-path>
```

Подготовка текста сообщения для внешнего аудита после успешной проверки:

```bash
dotnet run --project eng/Electron2D.Build -- audit package message --zip <path>
```

Отправка проверенного архива во внешний аудит через Electron2D audit-проект ChatGPT:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --out <report-path> --browser-backend codex-chrome
```

Повторная отправка новой итерации той же задачи в уже созданное обсуждение ChatGPT:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --reuse-conversation --zip <path> --out <report-path> --browser-backend codex-chrome
```

Повторная отправка новой итерации в новый чат проекта, если сохранённое обсуждение задачи больше нельзя продолжать штатно:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --new-conversation --zip <path> --out <report-path> --browser-backend codex-chrome
```

Опциональная отправка через режим `Глубокое исследование`, если обычный ChatGPT-запрос не даёт пригодного аудиторского ответа и причина зафиксирована в задаче/дневнике:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --deep-research --zip <path> --out <report-path> --browser-backend codex-chrome
```

Контрольный независимый аудит после первого `VERDICT: ACCEPT`, без истории исправительных итераций:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --control-audit --zip <path> --out <report-path> --browser-backend codex-chrome
```

Проверка закрытия actionable findings из сохранённых отчётов внешнего аудита:

```bash
dotnet run --project eng/Electron2D.Build -- verify audit-followups
```

Контрольный аудит после primary `VERDICT: ACCEPT` отправляет отдельно проверенный чистый контрольный ZIP для той же задачи, той же rNN и той же принятой области. Отправка control audit всегда начинается с URL проекта ChatGPT и создаёт новый чат; запрещено использовать `--reuse-conversation`, существующий conversation URL или primary-обсуждение. Такой ZIP собирается только штатной командой `audit package`, затем проходит `audit package verify`, но его входная конфигурация должна иметь пустые `metadata.previousVerdictChain` и `metadata.blockerClosureList`, а модель файлов репозитория не должна включать сохранённые Markdown verdict-отчёты под `docs/verdicts/`. Локальный индекс документации `data/documentation/**` не должен индексировать `docs/verdicts/**`; активные контекстные артефакты контрольного ZIP, включая `AUDIT-MANIFEST.md`, `AUDIT-REQUEST.md`, `repo-after/data/documentation/**`, `evidence/**`, `metadata/**`, `repo-file-hashes.json`, `SHA256SUMS.txt` и добавленные строки patch для контекстных файлов, не должны содержать конкретные Markdown-пути сохранённых verdict-отчётов под `docs/verdicts/`, включая нестандартные имена файлов. Mutable process-history files, включая `TASKS.md` и `data/dev-diary/**`, не входят в clean-control ZIP как repo-owned content: их история содержит предыдущие verdict paths, operator notes и summaries, поэтому контрольный пакет должен переносить принятую область через metadata, domain docs, code/tests и generated evidence без этих ledger-файлов. Обычные repo-owned source/test files под `repo-after/**` могут содержать синтетические fixture paths для проверки самого audit tooling, если эти пути не попали в manifest, request, generated docs, evidence, metadata или repo-file model как контекст старых внешних отчётов. Исторические снимки `repo-before/**`, а также удалённые и контекстные строки patch являются доказательством исходного состояния diff-а, поэтому обычные ссылки из старого содержимого там не считаются текущим контекстом; при этом прямые entries `repo-before/docs/verdicts/**`, metadata-ссылки на `docs/verdicts/**`, snapshot index для сохранённых verdict-отчётов и добавленные строки patch для context-bearing paths остаются запрещены. Чистый контрольный ZIP может отличаться от primary ZIP только удалением контекста прошлых внешних отчётов; код, документы, тесты, доказательства и критерии принятой области менять нельзя. Если между primary `ACCEPT` и control audit изменились код, документы, тесты, доказательства или критерии, нужен новый package и новый primary audit. Между primary `ACCEPT` и control audit разрешены только сохранение verdict report, локальное состояние отправки и сборка чистого контрольного ZIP без прошлых отчётов. Если control audit возвращает `VERDICT: NEEDS_FIXES`, задача остаётся открытой; следующий package должен включить control verdict в `metadata.previousVerdictChain`, закрыть его blocker-ы через `metadata.blockerClosureList` и снова пройти обычный primary loop с сохранённой ссылкой задачи.

`audit package` создаёт файл `<task-id>-audit-rNN.zip` и сопровождающий файл `<task-id>-audit-rNN.operator-workflow.zip` в каталоге `--out`. Если любой из этих файлов уже существует, команда завершается ошибкой. Перезапись разрешена только через явный флаг `--force`; в таком случае диагностическое сообщение фиксирует, что перезапись была разрешена.

Если `--config` не передан, команда читает конфигурацию из `.temp/audit/<task-id>/audit-package.input.json`. Явный `--config` остаётся рекомендуемой формой для внешнего аудита, потому что путь входного файла виден в команде воспроизведения.

`audit package verify` распаковывает основной архив во временный каталог, проверяет контрольные суммы, структуру архива, применимость patch, восстановленные SHA-256 файлов репозитория, `metadata/repo-file-snapshots.json` и защитные правила. Та же команда автоматически требует соседний сопровождающий ZIP `<main-name>.operator-workflow.zip`: если файл отсутствует, не проходит детерминированные правила ZIP, не покрыт `SHA256SUMS.txt`, содержит неполный набор `payload/` или операторских доказательств, ссылается на другой SHA-256 основного ZIP, хранит неверные хэши `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или списка записей основного архива, либо содержит несогласованные метаданные команд, проверка завершается отказом. Команда изменяет переданную чистую копию, поэтому `--repo` должен указывать на отдельную рабочую копию, предназначенную только для проверки восстановления. В этой рабочей копии команда сама выставляет локальные настройки Git `core.autocrlf=false` и `core.eol=lf`, проверяет доступность исходной ревизии `--baseline`, возвращает дерево к этой ревизии, удаляет только отслеживаемые Git файлы (`tracked files`) из списка `git ls-files -z`, снова материализует их через `git reset --hard <baseline>`, удаляет игнорируемые результаты сборки и другие игнорируемые рабочие файлы, а затем проверяет чистоту дерева. Такой порядок принудительно заменяет ранее созданные CRLF-байты на LF даже тогда, когда Git считал рабочее дерево чистым. После успешного применения patch команда повторно очищает Git-игнорируемые результаты сборки, которые не перечислены в `repo-file-hashes.json`, и только затем сравнивает восстановленный набор файлов с моделью архива; намеренно восстановленные Git-игнорируемые файлы из `repoFileAllowlist` сохраняются. Snapshot index должен совпадать с declared scope из `repo-file-hashes.json` и списком удалённых файлов: файл вне этой области, orphan snapshot под `repo-after/` или `repo-before/`, отсутствующий обязательный snapshot, path traversal, duplicate entry или SHA-256 mismatch отклоняют пакет. Для каждой записи `afterSnapshot` обязан быть точным путём `repo-after/<path>`, `beforeSnapshot` - точным путём `repo-before/<path>`, а `status` вычисляется из наличия и SHA-256 `before`/`after`, а не принимается на доверии из JSON. Если после подготовки дерево всё ещё содержит изменения, диагностика `E2D-BUILD-AUDIT-REPO-DIRTY` обязана включать фактический вывод `git status --porcelain` без абсолютных путей машины. Команда не создаёт исходную ревизию из содержимого ZIP и не требует, чтобы ZIP содержал Git bundle, архив tar или другой снимок исходного дерева Git.

`audit package message` читает уже собранный ZIP, проверяет корневой `AUDIT-REQUEST.md`, `metadata/audit-package.input.json`, идентификатор задачи, номер итерации и имя файла `<task-id>-audit-rNN.zip`. При успехе команда пишет в `stdout` только готовый текст для вставки в ChatGPT: содержимое `AUDIT-REQUEST.md` из архива без первого Markdown-заголовка первого уровня. Успешный вывод не содержит JSON-диагностик, сгенерированного обрамления задачи и итерации, строк вроде `<task-id> audit rNN`, `Приложен verified audit package` или `Ниже дословный текст`. При ошибке команда завершает работу с ненулевым кодом и пишет только структурированную JSON-диагностику: отсутствующий ZIP, отсутствующий `metadata/audit-package.input.json`, отсутствующий корневой `AUDIT-REQUEST.md`, недействительные обязательные маркеры, недействительные `taskId` или `iteration`, а также несоответствие имени ZIP метаданным считаются блокирующими ошибками.

`audit submit` является операторской C#-автоматизацией поверх проверенного audit package. Команда обязана проверить существование основного ZIP, получить текст сообщения из `--message <path>` или через тот же контракт, что и `audit package message`, открыть одну свежую вкладку Electron2D audit-проекта ChatGPT, дождаться доступности поля сообщения без предварительного прокручивания страницы проекта, прикрепить ровно один основной ZIP-файл, вставить полный текст сообщения из корневого `AUDIT-REQUEST.md`, проверить видимость тела сообщения и ZIP-плашки и только затем отправить запрос. Базовый режим `audit submit` использует обычный запрос в ChatGPT: он не вставляет текстовый маркер `@Глубокое исследование`, не выбирает режим `Глубокое исследование` и не ждёт карточку Deep Research report.

После отправки обычного запроса команда запоминает число сообщений до отправки, ждёт новое пользовательское сообщение и новый ответ ассистента текущей отправки, затем работает только с последним видимым `data-message-author-role="assistant"` после этого увеличения счётчика. Поиск verdict-а по всей странице, истории чата, пользовательскому запросу, боковой панели, превью карточек или старым ответам запрещён. Обычный ответ сохраняется как Markdown через штатную кнопку копирования ответа: команда ещё до загрузки страницы ставит pre-load захват `navigator.clipboard.writeText()`, `navigator.clipboard.write(...)` и `event.clipboardData.setData('text/plain', ...)` внутри браузерного события копирования, затем находит кнопку `button[data-testid="copy-turn-action-button"]` с подписью `Копировать` / `Copy` именно под новым assistant-сообщением, записывает уникальный sentinel в системный буфер обмена, best-effort выдаёт странице права Chrome на чтение/запись clipboard, ставит дополнительный поздний захват аргумента штатного `navigator.clipboard.writeText()`, `text/plain` содержимого штатного `navigator.clipboard.write(...)` и Markdown payload из browser `copy` event той же кнопки, очищает оба состояния захвата перед кликом и нажимает кнопку через управляемый Chrome. Если координатный клик не приводит к новому Markdown в системном буфере, команда вызывает DOM `click()` у той же найденной copy-кнопки текущего assistant-сообщения перед финальным чтением captured payload; это остаётся штатным copy action ChatGPT, а не собственным восстановлением Markdown из DOM ответа. Основной источник обычного Markdown-отчёта после copy action - текст, который сама штатная copy-кнопка передала в `writeText()`, `write(...)` или `event.clipboardData.setData('text/plain', ...)` и который был захвачен pre-load или поздним hook-ом.

После copy action команда читает реальный системный буфер обмена и отвергает неизменившийся sentinel как stale-текст. Если системный буфер не дал новый Markdown, команда доверяет `navigator.clipboard.readText()` только при успешно установленном sentinel; если sentinel не был установлен, browser `readText()` не является доказательством текущей copy action и принимается только captured Markdown, который сама copy-кнопка передала в `writeText()`, `write(...)`, `event.clipboardData.setData('text/plain', ...)` или другой проверяемый `copy` event payload. Fallback на выделенный текст допускается только если anchor/focus selection находятся внутри текущего assistant-сообщения, для которого нажималась copy-кнопка. Временный active input/textarea payload допускается только как непустой selected range и только если это не active element до клика; полный `active.value`, полный `active.textContent` и глобальное выделение вне текущего assistant-сообщения не являются источником verdict-а. Собственный DOM-to-Markdown renderer при этом не используется. Базовый ordinary path не восстанавливает Markdown из `innerText`, `textContent` или собственного DOM-to-Markdown renderer-а. Разовый таймаут протокола Chrome во время copy/clipboard-чтения считается временным пропуском polling-цикла; если один и тот же отказ повторяется устойчиво не меньше 30 секунд, команда падает с локальной диагностикой `E2D-BUILD-AUDIT-SUBMIT-ORDINARY-COPY-UNAVAILABLE` и не записывает частичный отчёт. Отсутствие нового assistant-сообщения текущей отправки или недостаточный счётчик сообщений считается обычным ожиданием ответа и не запускает таймер отсутствующей copy-кнопки. После завершения генерации и появления нового ответа устойчивое отсутствие текущей copy-кнопки тоже считается отказом ordinary copy action и завершается той же bounded диагностикой, а не общим таймаутом отправки. Текст из copy action проходит тот же строгий валидатор: первая непустая строка должна быть ровно `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`, должны присутствовать обязательные секции `TASK_ASSESSMENT:`, `BLOCKERS:`, `EVIDENCE_REVIEW:`, `RISKS_AND_NOTES:` и `CLOSURE_DECISION:`, а секция `TASK_ASSESSMENT:` должна явно назвать текущие `metadata.taskId` и `metadata.iteration`. Для `VERDICT: ACCEPT` дополнительно требуется только отсутствие маркеров `B1`, `B2` и далее в секции `BLOCKERS:`; текст `CLOSURE_DECISION:` сохраняется как человеческое объяснение и не является отдельным словарным барьером. Если copy-кнопка недоступна, Markdown copy action нельзя прочитать или стабильный скопированный Markdown не соответствует этому контракту, команда завершает запуск с диагностикой и не записывает частичный отчёт как готовый.

Режим `--deep-research` сохраняет прежний механизм как явный резервный путь. В этом режиме команда перед вставкой текста открывает меню кнопкой `+` слева от поля, выбирает реальный пункт `Глубокое исследование` по видимой подписи или стабильным connector-атрибутам `connector_openai_deep_research`, после вставки текста повторно проверяет выбранный connector отдельной плашкой и только затем отправляет запрос. Далее `--deep-research` ждёт готовую карточку отчёта `Углубленный исследовательский отчет` / `Deep research report` и извлекает отчёт через DOM-клик по Markdown-export действию: `Экспортировать в Markdown`, `Скачать как Markdown`, `Export as Markdown`, `Download as Markdown` или равнозначному пункту. Если таких report/export candidates найдено больше одного, команда не выбирает первый элемент, а завершает текущую попытку как неоднозначную. `--download-report-only` остаётся диагностическим режимом для уже готового Deep Research отчёта по конкретному ChatGPT conversation URL и не отправляет ZIP повторно. За пределами controlled ordinary copy action команда не использует произвольный буфер обмена как источник verdict-а и не запускает PowerShell для чтения или записи результата: `--deep-research` и `--download-report-only` берут Markdown только из export/download path. При успехе команда сохраняет отчёт в `--out` и пишет тот же отчёт в `stdout`, затем закрывает только вкладку, созданную текущим запуском. При ошибке команда закрывает только вкладку, созданную текущим запуском, и возвращает структурированную JSON-диагностику. Команда не закрывает и не изменяет уже открытые пользовательские вкладки, даже если они открыты на том же ChatGPT-проекте: закрытие чужих или оставшихся от прежней сессии браузерного протокола (`browser session`) вкладок не является безопасно детерминированным действием через этот прямой путь расширения и поэтому считается невозможным в контракте инструмента.

Перед нажатием отправки команда проверяет текущее содержимое поля отправки. В течение короткого окна ожидания полный текст сообщения должен оставаться в поле отправки, а рядом с полем должна быть видна ровно одна плашка имени основного audit ZIP. Такая плашка должна иметь признаки прикреплённого файла в поле отправки, например `attachment`/`file`/`upload`-атрибуты, русские признаки прикрепления файла или кнопку удаления внутри компактной плашки; обычная текстовая метка рядом с полем не считается прикреплённым ZIP. В режиме `--deep-research` выбранная плашка `Глубокое исследование` дополнительно проверяется отдельным правилом после вставки текста. Если текст или ZIP-плашка потеряны перед отправкой, команда останавливается с `E2D-BUILD-AUDIT-SUBMIT-PAYLOAD-MISSING`, добавляет в диагностику последнюю причину отказа и не отправляет сомнительный запрос во внешний аудит.

Уточнение для выбранного Deep Research frame или target: кнопка `Экспорт` нажимается только в выбранной отчётной поверхности, но пункт Markdown после этого можно искать и в основном DOM страницы, потому ChatGPT может отрисовать меню поверх всего документа. Такой fallback не разрешает выбирать другую кнопку экспорта, другую карточку или старый отчёт; он относится только к пункту Markdown-экспорта после уже выбранной кнопки отчёта.

При выборе режима `Глубокое исследование` команда сначала проверяет, не открыт ли уже нужный пункт меню. Если пункт уже виден, она кликает по нему и не нажимает `+` повторно в этом же цикле. Если меню кнопки `+` уже открыто, но пункт ещё не найден, команда ждёт следующую проверку и не нажимает `+` повторно: на медленном интерфейсе второй клик закрывает меню и может бесконечно срывать выбор режима. Если видимых строк ещё нет, но кнопка `+` уже сообщает `aria-expanded="true"`, команда также ждёт следующую проверку, не называя это состояние открытым меню. Это защищает медленную отрисовку меню: автоматизация не должна закрывать уже открывающийся список вторым кликом по `+`. Реальная строка меню ChatGPT может быть не только `button` или `role="menuitem"`, но и обычным элементом `.__menu-item` либо элементом с парой `[data-fill][tabindex]`; эти варианты разрешены для клика по пункту `Глубокое исследование` и по пункту Markdown-export. Та же разметка запрещена для проверки выбранной плашки: `DeepResearchSelectedExpression` обязан отвергать `.__menu-item`, `[data-fill][tabindex]` и их потомков, чтобы строка меню не считалась уже выбранным режимом.

После baseline старых Deep Research targets режим `audit submit --deep-research` может выбрать последний готовый target, если ChatGPT показывает несколько готовых non-baseline targets в одном reused-обсуждении. Но этот готовый target должен быть последним среди non-baseline Deep Research targets: если после него есть более новый target, который ещё не готов, команда ждёт дальше и не экспортирует старый отчёт. Это ожидание блокирует page-level export fallback: пока более новый target генерируется, команда не должна кликать старую готовую карточку выше в истории. Это не ослабляет проверку результата: скачанный Markdown всё равно проходит строгую валидацию `metadata.taskId`, `metadata.iteration`, обязательных секций и признаков текущего отчёта. Если выбранный target отдаёт старый отчёт, команда должна завершиться как stale/invalid report, а не сохранять его в `--out`.

`verify audit-followups` проверяет, что сохранённые отчёты внешнего аудита не оставили actionable `FOLLOW_UP_FINDING` без tracked closure note. Команда читает `TASKS.md` и все Markdown-отчёты под `docs/verdicts/**/*.md`; в Git-репозитории она объединяет отслеживаемые пути с фактическими файлами на диске, чтобы только что сохранённые primary/control reports проверялись ещё до commit. Saved audit report считается входом проверки, когда первая непустая строка файла строго равна `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`; остальные Markdown-файлы под `docs/verdicts/` игнорируются как невалидные для closure-verifier входы. Для каждого такого отчёта команда извлекает structured entries из секции `RISKS_AND_NOTES`: `FOLLOW_UP_FINDING F1` является actionable по умолчанию; `OUT_OF_SCOPE_NOTE N1` и `INFO_NOTE I1` не требуют closure без явной пометки `Actionable: true`, `Actionable: yes` или `[actionable]`; текст вида `B1` внутри `RISKS_AND_NOTES` не превращается в blocker сам по себе. Уникальный ключ finding-а - пара `(saved report path, finding id)`, поэтому `FOLLOW_UP_FINDING F1` из primary report и `FOLLOW_UP_FINDING F1` из control report закрываются отдельными notes.

Closure note хранится в tracked `TASKS.md` и должен быть пригоден для машинной проверки. Рекомендуемый формат:

```markdown
- audit-followup-closure:
  - source: docs/verdicts/<domain>/<task-id>-audit-rNN.md
  - id: FOLLOW_UP_FINDING F1
  - state: tracked-new
  - target: <task-id>
  - rationale: кратко, почему выбран именно этот способ закрытия.
```

Обязательные поля closure note: `source`, `id`, `state`, `target`, `rationale`. Поле `state` принимает только `tracked-existing`, `tracked-new`, `accepted-risk`, `duplicate`, `not-actionable` или `promoted-blocker`. Для `accepted-risk` тот же note обязан содержать поля risk-register модели, совместимой с T-0105: `affected area`, `impact`, `likelihood`, `mitigation`, `owner/next decision point`, `decision state`. Если accepted risk не содержит rationale, next decision point или decision state, команда возвращает ошибку. Команда пишет только JSON-диагностики `BuildDiagnostic`; незакрытый finding, неверный closure state или неполный accepted risk завершают проверку ненулевым кодом.

Список `checks` в `audit-package.input.json` и notes задачи должен отражать фактически выполненные команды и их состояние. Нельзя записывать в список `checks` внешнюю упаковку, отправку или проверочный шаг, который не запускался в текущей итерации; локальный `verify audit-followups` указывается только после реального запуска.

Если после отправки ChatGPT показывает кнопку остановки ответа, команда считает генерацию активной: она продолжает минутный цикл, не создаёт PNG-скриншоты и всё равно обновляет страницу раз в минуту до готового отчёта. Пока генерация активна, команда не принимает отчёт как готовый даже тогда, когда текущий streaming-текст уже начинается с `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`, либо в истории уже видна старая завершённая отчётная карточка. Когда активной генерации нет, команда сначала прокручивает обсуждение вниз, затем пробует нажать уже открытый пункт `Экспортировать в Markdown`; если пункта нет, она открывает кнопку `aria-label="Экспорт"` и нажимает пункт `Экспортировать в Markdown`. После клика команда ждёт новый `.md` в управляемой папке загрузки или в реальной пользовательской папке загрузок Windows. Если генерация стала активной до или после чтения, скачанный текст отбрасывается. Если ChatGPT во время ожидания показывает временный диалог ограничения частоты запросов `Слишком много запросов` / `Too many requests`, команда нажимает видимую кнопку подтверждения `Понятно` / `OK` и продолжает тот же минутный цикл; обработка временного диалога ограничения частоты запросов не считается отчётом и не требует ручного шага. После каждого минутного интервала команда обновляет страницу, ждёт возвращения сообщений обсуждения и только затем продолжает извлечение отчёта. Если расширение Chrome во время длинного ожидания сообщает recoverable CDP detachment, клиент выполняет до пяти повторных `attach` с best-effort `detach` перед каждой попыткой и ждёт `Page.enable`, `Runtime.enable`, `DOM.enable` до 45 секунд на попытку.

Если `Codex Chrome Extension` временно теряет подключение к странице во время чтения состояния страницы или обновления страницы, backend `codex-chrome` выполняет ограниченное повторное подключение к собственной вкладке: сначала освобождает отладочное подключение этой вкладки через `detach`, затем снова выполняет `attach` и включает нужные CDP-домены. После успешного восстановления команда продолжает тот же цикл ожидания. Действия, которые могут изменить состояние ChatGPT, например выбор пункта меню, прикрепление файла, вставка текста и отправка, автоматически не повторяются после такого сбоя, чтобы не создать двойное действие.

После успешного нажатия отправки ZIP команда обязана записать локальную подсказку восстановления с текущим URL созданного audit-чата. Подсказка хранится как временный sidecar `conversation-url-<iteration>.txt` под `.temp/audit/<task-id>/`, а также как стабильный `conversation-url.txt` для следующей исправительной итерации той же задачи; эти файлы не входят в audit package. Conversation URL является только локальным состоянием отправки: он не включается в `SHA256SUMS.txt`, `AUDIT-MANIFEST.md`, `repo-file-hashes.json`, `metadata/repo-file-snapshots.json`, evidence, saved verdict files или другие repo-owned документы. В `TASKS.md` можно фиксировать место хранения локального состояния, например `.temp/audit/<task-id>/conversation-url.txt`, но не конкретный ChatGPT URL. Если browser/CDP-связь оборвалась после отправки, оператор должен иметь возможность запустить `audit submit --download-report-only --project-url <saved-url> ...` без ручного поиска вкладки. Режим `--download-report-only` принимает только конкретный ChatGPT conversation URL с сегментом `/c/<conversation-id>`; общий URL проекта, корень ChatGPT, пустой conversation id или ссылка без `/c/` отклоняются до подключения к браузеру.

`--reuse-conversation` означает, что текущая отправка продолжает обсуждение задачи. Если `--project-url` передан явно, он обязан быть конкретным ChatGPT conversation URL с `/c/<conversation-id>`. Если `--project-url` не передан, команда читает `.temp/audit/<task-id>/conversation-url.txt`, где `<task-id>` выводится из имени ZIP `<task-id>-audit-rNN.zip`; отсутствие файла или общий URL проекта блокируют запуск до подключения к браузеру. После saved primary `VERDICT: NEEDS_FIXES` или saved control `VERDICT: NEEDS_FIXES` следующая primary-итерация r02+ не имеет права уходить в project root: `audit submit` отклоняет запуск без `--reuse-conversation` или явно переданного concrete conversation URL. После отправки команда снова пишет итерационный `conversation-url-rNN.txt` и обновляет стабильный `conversation-url.txt`.

`--new-conversation` является явным аварийным путём для primary-итерации, когда сохранённое обсуждение задачи больше нельзя продолжить штатной отправкой, например из-за устойчивого отказа именно этого чата до отправки или получения отчёта. Команда открывает корневой URL проекта из собственной конфигурации и создаёт новый audit chat тем же штатным способом: прикрепляет ровно один проверенный ZIP, вставляет проверенный текст сообщения и сохраняет новый `conversation-url-rNN.txt` плюс стабильный `conversation-url.txt`. Режим `Глубокое исследование` выбирается только если вместе с этим явно передан `--deep-research`; базовый `--new-conversation` остаётся обычным запросом в ChatGPT. Флаг несовместим с `--reuse-conversation`, `--control-audit`, `--download-report-only` и любым явно переданным `--project-url`, потому он всегда означает новый чат проекта, а не переход в уже существующее обсуждение или чужой адрес. Использовать его можно только после локальной фиксации причины в `TASKS.md` и дневнике; предыдущие сохранённые verdict reports всё равно должны быть включены в `metadata.previousVerdictChain` самого ZIP, если это исправительная primary-итерация.

Выбор `Глубокое исследование` должен быть устойчив к медленной отрисовке меню. В цикле выбора команда сначала проверяет, не открыт ли уже пункт `Глубокое исследование` после предыдущего нажатия `+`; если пункт уже виден, команда кликает его и не нажимает `+` повторно, чтобы случайно не закрыть открывшееся меню. Если кнопка `+` уже сообщает `aria-expanded="true"`, но видимые строки меню ещё не отрисовались рядом с ней, команда ждёт следующую проверку и тоже не нажимает `+` повторно; это считается состоянием ожидания отрисовки, а не доказательством открытого меню. Только если пункта нет, меню не открыто и кнопка `+` не находится в состоянии раскрытия, команда нажимает `+` и ждёт появления списка. Реальный интерфейс ChatGPT может отрисовывать строки меню как обычные `div` без `role="menuitem"` и без connector-атрибутов; поэтому кликабельными считаются не только `button`, `role="menuitem"`, `role="option"` и `role="button"`, но и видимые строки `.__menu-item` или элементы с парой `data-fill` + `tabindex`, если их текст начинается с `Глубокое исследование` / `Deep Research` и они находятся рядом с кнопкой `+`.

`--control-audit` означает независимую проверку после первого `ACCEPT`. Команда обязана начинать из корневого URL проекта и отклоняет конкретный conversation URL, чтобы контрольный аудит не получил прошлое обсуждение исправлений. До подключения к браузеру команда проверяет, что для той же задачи и того же `rNN` уже сохранён primary verdict report, первая непустая строка которого ровно `VERDICT: ACCEPT`; отсутствие такого отчёта или primary `VERDICT: NEEDS_FIXES` блокируют control audit. Затем команда проверяет сам ZIP: `metadata.previousVerdictChain` и `metadata.blockerClosureList` должны быть массивами длиной ровно `0`, прямые entries `repo-after/docs/verdicts/**` и `repo-before/docs/verdicts/**` запрещены, `TASKS.md` и `data/dev-diary/**` запрещены как mutable process-history files, а все строки `repo-file-hashes.json` включая `deletedRepoFiles` и все строки `metadata/repo-file-snapshots.json` не должны содержать сохранённые verdict-отчёты `docs/verdicts/`. После структурной проверки команда сканирует контекстные текстовые артефакты ZIP и отклоняет конкретные Markdown-пути под `docs/verdicts/` в `AUDIT-MANIFEST.md`, `AUDIT-REQUEST.md`, `repo-after/data/documentation/**`, `evidence/**`, `metadata/**`, `repo-file-hashes.json`, `SHA256SUMS.txt` или добавленных строках patch для context-bearing repo paths. Обычные source/test snapshots под `repo-after/**`, исторические `repo-before/**` snapshots и удалённые или контекстные строки patch не считаются текущим verdict context. Нарушение этого правила отклоняется до подключения к браузеру диагностикой `E2D-BUILD-AUDIT-SUBMIT-CONTROL-CONTEXT`, потому контрольный аудитор должен получить нулевой контекст прошлых внешних отчётов и самостоятельно искать проблемы в текущей области. URL контрольного обсуждения сохраняется отдельно как `control-conversation-url-<iteration>.txt` и `control-conversation-url.txt`, чтобы не перезаписать основную ссылку задачи.

Все режимы `audit submit`, которые записывают verdict Markdown, принимают только строгий repo-relative путь `--out`: для primary это `docs/verdicts/<domain>/<task-id>-audit-rNN.md`, для control audit это `docs/verdicts/<domain>/<task-id>-audit-control-rNN.md`. Путь не должен иметь произвольный префикс перед `docs/verdicts/`, пропускать доменный сегмент, быть абсолютным или содержать `.`, `..` либо пустые сегменты. Для обычного submit путь дополнительно обязан совпадать с текущими `taskId`/`iteration` из ZIP и режимом primary/control; для `--download-report-only`, который не принимает `--zip`, ожидаемая task/iteration identity выводится из этого строгого пути. Если скачанный Markdown явно ссылается на evidence или ZIP другой итерации той же задачи, команда отклоняет его диагностикой `E2D-BUILD-AUDIT-SUBMIT-REPORT-STALE` и не записывает файл как готовый verdict.

Проверка выбранного Deep Research connector не должна зависеть от того, что выбранный элемент является потомком `#prompt-textarea`. Если ChatGPT отрисовывает выбранный инструмент как соседний pill-элемент рядом с полем ввода, команда обязана распознать его по `data-id`, `data-system-hint-type` или `data-keyword` для `connector_openai_deep_research` в зоне composer-а и не считать успешный выбор отказом.

DOM-активация кнопки экспорта и Markdown-пункта должна выполнять не только `element.click()`, но и обычную последовательность событий `pointerdown`/`mousedown`/`pointerup`/`mouseup` на найденном элементе. Это сохраняет запрет на координатный клик по отчётной карточке, но покрывает UI, где download action привязан к pointer/mouse event handlers, а не к одному synthetic `click()`. Упоминания `button.click()` в export-контракте ниже являются сокращением для этой полной DOM-активации найденного элемента.

Read-only восстановление отчёта через `--download-report-only` открывает уже существующее обсуждение, поэтому оно не должно считать все уже существующие Deep Research targets устаревшим baseline-ом. Если в таком обсуждении осталось несколько готовых Deep Research targets от предыдущих отчётных карточек, read-only путь выбирает последний готовый target в порядке `Target.getTargets`. Обычный submit после отправки ZIP сохраняет прежнее защитное поведение: targets, снятые до отправки текущего запроса, игнорируются, а несколько новых неигнорируемых ready targets не выбираются произвольно.

Проверка выбранного `Глубокое исследование` в composer должна отличать настоящую выбранную плашку от пункта меню и старого сообщения в истории. Стабильные connector-атрибуты `connector_openai_deep_research` засчитываются как выбранное состояние, когда они находятся на видимой компактной плашке рядом с полем ввода или внутри него; старый маркер `data-inline-selection-pill` остаётся допустимым, но не является обязательным для новых вариантов интерфейса. Обычный текст `@Глубокое исследование`, скрытые элементы, `button`, `role="button"`, `role="menuitem"`, элементы внутри `role="menu"`, `role="listbox"` или popover-контейнера не считаются выбранной плашкой, даже если внутри них есть connector metadata. Широкий контейнер страницы, карточка старого сообщения или ancestor-элемент с вложенной старой Deep Research-плашкой тоже не считаются выбранным режимом: вложенный connector допустим только для компактной плашки, а не для общего контейнера страницы или истории. Readiness gate для готовой отчётной поверхности и click selector используют один и тот же набор export/download подписей: `Экспорт`, `Export`, `Скачать` и `Download` с download icon или явным `aria-haspopup="menu"`, причём readiness допускает offscreen button, который click selector может прокрутить.

Извлечение готового отчёта привязано к текущей вкладке. Команда сначала проверяет видимый Deep Research `iframe` на открытой странице, сопоставляет его с единственным frame id из `Page.getFrameTree` этой же вкладки и пытается экспортировать Markdown в этом контексте. Глобальный список `Target.getTargets` используется только как резервный путь, когда текущая страница не дала единственный готовый frame; targets, существовавшие до отправки или не относящиеся к текущему чату, не должны становиться первичным источником отчёта. Если после baseline одновременно готовы несколько фоновых Deep Research targets, команда не выбирает один из них и продолжает через current-page DOM fallback либо следующий цикл ожидания. Если выбран frame или target, кнопка `Экспорт` нажимается в выбранном контексте отчёта. Пункт меню `Экспортировать в Markdown` после этого можно искать и в выбранном контексте, и в основном DOM страницы: текущий интерфейс ChatGPT может открыть меню поверх всей страницы, даже когда сама кнопка экспорта находится внутри Deep Research frame или target. Такой fallback допустим только для пункта Markdown после выбранной кнопки экспорта и не разрешает повторно выбирать чужую отчётную карточку или нажимать другую кнопку экспорта на странице.

Восстановление CDP-связи (`Chrome DevTools Protocol`, то есть протокола управления вкладкой Chrome) вокруг Deep Research target/frame применяется только к безопасным операциям чтения и подготовки контекста: attach target, enable-команды, `Page.getFrameTree`, создание isolated world и evaluate без побочного эффекта. DOM-клики отправки, выбора экспорта и выбора пункта Markdown считаются изменяющими операциями и не повторяются как обычные read-retry, чтобы не отправить запрос или экспорт дважды. Перед polling/export команда заново убеждается, что нужный tab или target подключён, а recoverable failures вроде `Debugger is not attached`, закрытого context или временного detached target приводят к ограниченному повторному attach/enable вместо немедленного отказа.

Строгий валидатор текста отчёта не смягчается, но его диагностика должна объяснять причину `E2D-BUILD-AUDIT-SUBMIT-REPORT-INVALID`. Сообщение обязано различать минимум такие случаи: отчёт ещё генерируется или кандидатов не один; candidate пришёл не из открытой отчётной карточки и не из нового ответа ассистента текущей отправки; Markdown начинается с echo пользовательского prompt-а или похож на prompt-шаблон; первая непустая строка не `VERDICT: ACCEPT` и не `VERDICT: NEEDS_FIXES`; отсутствует или нарушен порядок обязательных heading-ов; `TASK_ASSESSMENT:` не называет текущие `metadata.taskId` и `metadata.iteration`; `VERDICT: ACCEPT` содержит numbered blockers `B1`..`Bn`; submit запущен с `--out`, который не является строгим repo-relative verdict-путём или не совпадает с текущей ZIP identity/mode. Такая диагностика не считается verdict-ом и не разрешает закрытие задачи без полного сохранённого отчёта.

Pipe-клиент `codex-chrome` отвечает строкой `pong` на входящие служебные JSON-RPC запросы `ping` от `Codex Chrome Extension` во время ожидания своего ответа. Такие запросы не являются действиями браузера, но без совместимого ответа на них расширение может не завершить долгую операцию вкладки.

По умолчанию `audit submit` использует проект `https://chatgpt.com/g/g-p-6950376d4d8c8191a0fe600e98389912-electro2d/project`, backend `--browser-backend codex-chrome`, уже установленный `Codex Chrome Extension` в пользовательском Chrome, интервал обновления `60` секунд, общий предел ожидания отчёта `180` минут и предел ожидания входа в ChatGPT `10` минут. Это единственный штатный браузерный путь. Команда не копирует cookies, `localStorage`, пароли, файлы профиля Chrome или другие файлы сессии: C#-команда подключается к pipe установленного расширения `Codex Chrome Extension`, выбирает backend с типом `extension`, создаёт собственную управляемую вкладку в текущей пользовательской Chrome-сессии, выполняет действия через протокол расширения и закрывает только эту вкладку после результата или ошибки. Она не использует `getUserTabs`, `claimUserTab`, `Page.close` для вкладок пользователя и не завершает чужие сессии браузерного протокола (`browser session`) через `finalizeTabs`. Для воспроизводимости разрешены параметры `--codex-chrome-pipe <path>`, `--codex-session-id <id>` и `--codex-turn-id <id>`; пара `--codex-session-id` и `--codex-turn-id` должна передаваться только вместе. Если пара не передана и в окружении нет одновременно `CODEX_SESSION_ID` и `CODEX_TURN_ID`, команда создаёт независимую сессию браузерного протокола (`browser session`) сама и не использует `CODEX_THREAD_ID` как суррогат такой сессии. Если pipe не передан, команда обнаруживает доступный `codex-browser-use` pipe сама и выбирает только backend `extension`. Отдельная команда проверки входа, другой браузерный backend, закрытие уже открытых пользовательских вкладок и обход через скопированные файлы браузерной сессии не входят в контракт.

Для локальной проверки команды разрешены параметры `--project-url <url>`, `--poll-seconds <seconds>`, `--timeout-minutes <minutes>`, `--login-timeout-minutes <minutes>`, `--codex-chrome-pipe <path>`, `--codex-session-id <id>`, `--codex-turn-id <id>` и `--keep-tab-open-on-error`. Значение `--poll-seconds` меньше `60` допускается только вместе с `--allow-fast-poll`, чтобы тесты могли проверять цикл ожидания без изменения рабочего правила внешнего аудита. Флаг `--keep-tab-open-on-error` предназначен только для диагностики реального сайта: при ошибке он оставляет открытой собственную вкладку текущего запуска, чтобы оператор видел фактическое состояние Chrome; при успешном получении отчёта вкладка всё равно закрывается.

Команда `audit submit` не принимает параметр для каталога скриншотов, не создаёт PNG-скриншоты браузерного протокола и не принимает tool screenshots как доказательство состояния страницы. Визуальный снимок реального окна Chrome может быть только отдельной пользовательской диагностикой вне audit tooling и не заменяет сохранённый Markdown-отчёт. Машинная диагностика submit-пути опирается на DOM-дамп, строгий Markdown-export, проверку текущей итерации и structured diagnostics. Режим `--dump-dom-only` обязан сначала записать root DOM, `frame-tree.json`, `target-info.json`, `deep-research-selected-result.json` и `deep-research-selected-diagnostics.json`; отсутствие или неготовность Deep Research frame не должно блокировать базовый dump, потому именно этот режим нужен для диагностики ситуации, когда плашка `Глубокое исследование` не была получена. Порядок ранней записи DOM-диагностики проверяется поведенческим тестом через внутренний драйвер `IAuditSubmitDomDumpDriver`: тест вызывает тот же production-путь `DumpDomFromUrlAsync`, подменяет только браузерный клиент и проверяет созданные JSON-файлы на диске, а не строки исходного кода.

`--dump-dom-only` не является режимом сохранения verdict-а. Он не должен писать диагностический summary в `docs/verdicts/**/<task-id>-audit-rNN.md` или `docs/verdicts/**/<task-id>-audit-control-rNN.md`; такие пути зарезервированы только для сохранённых внешних отчётов, первая непустая строка которых является `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`. При попытке совместить `--dump-dom-only` с таким `--out` команда обязана отказать до подключения к Chrome. DOM-диагностика хранится в `--dom-dump-dir`; если нужен отдельный текстовый summary, его путь должен быть временным и не похожим на verdict report.

Встроенный операторский слой доказательств не входит в основной ZIP и не меняет его `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или набор записей после выполнения `audit package verify` и `audit package message`. Он создаётся в сопровождающем ZIP `<task-id>-audit-rNN.operator-workflow.zip` после финальной записи и внутренней проверки основного ZIP. Сопровождающий ZIP содержит `OPERATOR-WORKFLOW.md`, `payload/metadata.json`, `payload/sha256.txt`, `payload/AUDIT-MANIFEST.sha256`, `payload/SHA256SUMS.sha256`, `payload/archive-entries.sha256`, `payload/archive-entries.txt`, а также каталоги `evidence/<task-id>-<iteration>/checks/audit-package-verify/` и `evidence/<task-id>-<iteration>/checks/audit-package-message/` с файлами `command.txt`, `stdout.txt`, `stderr.txt`, `exit-code.txt`, `duration-ms.txt`, `metadata.json`, `cwd.txt`, `env.json` и `timeout-seconds.txt`. В `metadata.json` для этих двух каталогов `executionMode` должен быть `subprocess`, ожидаемый и фактический коды завершения должны быть `0`, длительность должна быть больше нуля, а `timeoutSeconds` должен совпадать с `timeout-seconds.txt` и фактическим лимитом времени операторской команды: `180` секунд.

`command.txt` для проверки восстановления обязан показывать переносимую команду с `--zip <relative-zip-path>`, `--baseline <sha>` и плейсхолдером `--repo <clean-repo-path>`, а не абсолютный локальный путь временной рабочей копии. Фактическое выполнение происходит только через отдельный процесс `dotnet run --project eng/Electron2D.Build --no-build --no-restore -- audit package verify --zip <path> --baseline <sha> --repo <clean-repo-path>`; прямой вызов внутреннего метода проверки не считается допустимым evidence. Аналогично `audit package message evidence` создаётся только через отдельный процесс `dotnet run --project eng/Electron2D.Build --no-build --no-restore -- audit package message --zip <path>`, а не через внутренний метод подготовки текста. Это `baseline availability evidence`, то есть проверяемое доказательство того, что пакет был проверен на отдельной чистой копии исходной ревизии. Флаги `--no-build` и `--no-restore` обязательны для этих operator workflow subprocesses: основной запуск `audit package` уже работает внутри собранного build-tool, поэтому sidecar-доказательства не должны заново восстанавливать зависимости или пересобирать проект только ради чтения готового ZIP. Если архив или изменённые документы утверждают, что исходная ревизия доступна только локально, непубликована или требует ручной догадки, отсутствие вложенного baseline-снимка всё ещё не является отдельным blocker-ом, но успешное `audit package verify evidence` в сопровождающем ZIP становится обязательным доказательством восстановления.

## Окончания строк

Аудиторский архив опирается на общую репозиторную политику из `docs/repository-policy/line-endings.md`: `.gitattributes` задаёт `* text=auto eol=lf`, обычные текстовые файлы должны храниться и восстанавливаться с LF, а CRLF разрешён только для явных исключений `.bat`, `.cmd` и временно для исторических verdict-документов в `docs/verdicts/`, которые текущая задача не изменяет. Настоящие двоичные форматы описываются по расширениям как `binary`; XML-карты `.tmx` и `.tsx` в reference-game данных считаются текстом и нормализуются в LF.

Одноразовая нормализация tracked текстовых файлов допускается только как замена CRLF или одиночного CR на LF без смыслового изменения содержимого. Проверка:

Для современных страниц ChatGPT действует то же уточнение про выбранный frame/target: кнопка экспорта выбирается внутри уже найденной отчётной поверхности, а пункт `Экспортировать в Markdown` после клика может быть найден в основном DOM страницы, если меню отрисовано вне внутреннего frame. Это не допускает координатный клик и не допускает выбор другой report-card.

```bash
dotnet run --project eng/Electron2D.Build -- verify line-endings
```

читает `git ls-files --eol` и отклоняет неожиданные `w/crlf` у tracked текстовых файлов. Перед повторным внешним аудитом ручная команда `git ls-files --eol | rg "w/crlf"` должна показывать только описанные исключения или не показывать строк вообще.

## Конфигурация

Источник истины для входных данных - JSON-файл. Путь по умолчанию для рабочих сценариев:

```text
.temp/audit/<task-id>/audit-package.input.json
```

YAML считается допустимым будущим форматом того же контракта, но текущая реализация читает JSON. Конфигурация включается в архив как `metadata/audit-package.input.json`; исходный путь из `.temp` в архив не попадает.

Минимальные поля:

| Поле | Назначение |
| --- | --- |
| `taskId` | Идентификатор задачи. Должен совпадать с `--task` и всеми сгенерированными артефактами. |
| `iteration` | Итерация аудита в форме `rNN`. Должна совпадать с `--iteration`. |
| `baseline` | Исходная ревизия Git. Должна совпадать с `--baseline`. |
| `branch` | Имя рабочей ветки или логическое имя изменения, которое попадёт в запрос аудитору. |
| `domain` | Домен изменения, например `release-management`. |
| `scopeTaskIds` | Машиночитаемая область задач, которые принимает текущий пакет. Если список пуст, область равна одиночному `taskId`; если список содержит несколько значений, пакет объявляет явно комбинированную область (`combined scope`) и должен включать основной `taskId`. |
| `scopeSummary` | Краткое пояснение области аудита. Поле предназначено для объяснения, почему несколько задач входят в один пакет, и не заменяет `repoFileAllowlist`, `previousVerdictChain` или `blockerClosureList`. |
| `repoFileGlobs` | Шаблоны файлов репозитория, которые восстанавливаются через patch. |
| `repoFileAllowlist` | Явный список файлов репозитория, которые разрешено включить даже без совпадения с шаблоном. Список может включать Git-игнорируемые рабочие файлы вроде `TASKS.md` или `data/dev-diary/...`, если они принадлежат задаче и нужны аудитору. |
| `archiveOnlyEvidenceGlobs` | Шаблоны файлов, которые попадают только в `evidence/` архива и не входят в patch. |
| `checks` | Машиночитаемый список проверок, которые команда запускает до упаковки. |
| `forbiddenPatterns` | Дополнительные запрещённые шаблоны путей сверх встроенной защиты. |
| `secretScanPolicy` | Политика поиска секретов. Поддерживаемое значение текущей реализации: `basic`. |
| `maxFileSize` | Максимальный размер одного входного файла в байтах. |
| `outputDirectory` | Ожидаемый каталог результата. Должен совпадать с `--out`, если указан. |

Дополнительные поля:

| Поле | Назначение |
| --- | --- |
| `previousVerdictChain` | Список предыдущих verdict-документов или пустой список. Каждый путь должен указывать на Markdown-файл внутри `docs/verdicts/`, иначе конфигурация считается ошибочной. Если указанный файл существует в рабочей копии репозитория, команда автоматически включает его в модель восстановления файлов репозитория: patch, `repo-file-hashes.json` и инвентарь `AUDIT-MANIFEST.md`. Если файл отсутствует, путь остаётся только исторической ссылкой в запросе аудитору. |
| `blockerClosureList` | Список закрытых blocker-ов, который попадёт в запрос аудитору. Для существующих прошлых `NEEDS_FIXES` отчётов каждая найденная запись `B1`, `B2` и далее из секции `BLOCKERS` должна иметь closure-строку с путём отчёта, этим blocker id и именем configured check текущего пакета. |

## Содержимое архива

Корневые файлы архива:

- `AUDIT-MANIFEST.md` - инвентарь, а не повествовательный отчёт.
- `<task-id>.patch` - patch, созданный штатным Git через `git diff --binary`.
- `repo-file-hashes.json` - ожидаемые SHA-256 файлов репозитория после применения patch.
- `metadata/repo-file-snapshots.json` - машинный индекс полных снимков файлов, доступных аудитору для чтения.
- `SHA256SUMS.txt` - контрольные суммы всех файлов архива, кроме самого `SHA256SUMS.txt`, в формате, совместимом с `sha256sum -c`.
- `AUDIT-REQUEST.md` - статический запрос аудитору, дословно скопированный из отслеживаемого Git-файла `docs/release-management/AUDIT-REQUEST.md`.
- `metadata/audit-package.input.json` - нормализованная копия конфигурации.
- `repo-after/` - полные итоговые снимки repo-owned файлов объявленной области задачи, которые существуют после изменения.
- `repo-before/` - полные снимки repo-owned файлов объявленной области задачи из исходной ревизии, если файл существовал в `baseline`.
- `evidence/` - только доказательства настроенных проверок и импортированные файлы доказательств; операторские доказательства `audit package verify` и `audit package message` хранятся в сопровождающем ZIP, чтобы не менять основной архив после проверки.

`repo-after/` является основной поверхностью чтения для внешнего content review: аудитору не нужно восстанавливать дерево репозитория, чтобы прочитать полный итоговый код, тесты и документацию выбранной области. Patch остаётся навигацией по изменённым местам и восстановительной моделью, но не заменяет полные итоговые снимки файлов. `repo-before/` нужен для проверки удалений, переименований и регрессий относительно исходной ревизии; он не является полным baseline-снимком репозитория. Снимки создаются только для repo-owned файлов объявленной области задачи: `repoFileGlobs`, `repoFileAllowlist`, `previousVerdictChain` и связанных записей `repo-file-hashes.json`. Dirty, untracked или out-of-scope файлы рабочей копии не должны попадать в `repo-after/`, `repo-before/` или snapshot index.

`metadata/repo-file-snapshots.json` содержит по одной записи на каждый repo-owned файл declared scope, то есть объявленной области задачи. Текущая schema version задаётся структурой `taskId`, `iteration`, `baseline` и массивом `files`. Каждая запись указывает `path`, `status`, `afterSnapshot`, `beforeSnapshot`, `afterSha256`, `beforeSha256`, `contentKind` и `fullContentIncluded`. Для текстовых файлов `contentKind` равен `text`, для двоичных файлов - `binary`. `afterSnapshot` или `beforeSnapshot` с непустым значением означает `exists=true` и `included=true` для соответствующей стороны; отсутствие поля означает `exists=false` для `added`/`deleted` стороны либо отсутствие baseline-содержимого. Переименования в текущей версии представлены как связанная пара `deleted` старого пути и `added` нового пути из declared scope; отдельное поле `oldPath` пока не используется и не должно ослаблять требование иметь оба snapshot-а. `fullContentIncluded` должен быть `true` для каждого файла, который входит в `repo-file-hashes.json` или список удалённых файлов. `status` должен соответствовать фактической паре сторон: `added` имеет только `after`, `deleted` имеет только `before`, `modified` имеет обе стороны с разными SHA-256, а `unchanged` имеет обе стороны с одинаковыми SHA-256. Если файл declared scope нельзя безопасно вложить полностью из-за реального секрета, конфиденциального содержимого, размера или неподдержанного двоичного содержания, текущий контракт T-0237 завершает `audit package` или `audit package verify` отказом до отправки архива. Если в будущем появится controlled omission для слишком большого, сгенерированного или чувствительного baseline-файла, индекс обязан явно указать `fullContentIncluded: false`, `omissionReason`, SHA-256, размер и допустимый способ аудита; T-0237 таких исключений не разрешает.

Матрица снимков:

| Status | `repo-before/` | `repo-after/` |
| --- | --- | --- |
| `added` | отсутствует, потому что файла нет в `baseline` | обязателен |
| `modified` | обязателен | обязателен |
| `unchanged` | обязателен, если файл выбран в scope и существует в `baseline` | обязателен |
| `deleted` | обязателен | отсутствует |
| `renamed` | представляется текущей моделью как пара `deleted` old path и `added` new path, если оба пути выбраны; будущий `oldPath` в индексе не должен ослаблять обязательность обоих снимков | обязателен для new path |
| `binary` | включается как полный byte snapshot, если проходит защитные правила и лимит размера | включается как полный byte snapshot, если проходит защитные правила и лимит размера |

Snapshot content проходит те же защитные правила, что patch и evidence: машинные абсолютные пути, реальные секреты, приватные ключи, токены и пароли запрещены в любом текстовом `repo-before/` или `repo-after/` файле. Реальный secret/sensitive content в `repo-after/` для changed text file всегда запрещает сборку package; его нельзя скрывать omission-записью, потому что это итоговое содержимое проверяемой задачи. Для `repo-before/` нет общего послабления на старые строки: допускается только точечная совместимость с уже известной старой формулировкой `docs/release-management/audit-package.md`, где замещающие значения были закрыты обратной кавычкой внутри документационного примера. Любой другой `repo-before/` snapshot с замещающим значением и произвольным suffix остаётся blocker-ом. Большой файл определяется тем же `maxFileSize` из конфигурации audit package; значение по умолчанию равно `1_048_576` байтам. Двоичный файл определяется байтовой проверкой содержимого, а сгенерированный файл пока не получает отдельного omission режима: если он входит в declared scope, он должен безопасно войти как полный snapshot или package завершится ошибкой. Текущая реализация не создаёт redacted или partial snapshot: если полный снимок нельзя безопасно вложить, package завершается ошибкой до создания финального ZIP. Это fail-closed правило важнее полноты audit ZIP, потому что неполный или отредактированный snapshot нельзя выдавать за полноценный content review. Если будущий контракт разрешит `fullContentIncluded: false`, `AUDIT-REQUEST.md` должен требовать считать omitted implementation/test/documentation snapshot evidence gap и blocker-ом, когда файл критичен для проверки задачи и нет другой полной поверхности чтения.

Архив не содержит и по текущему контракту не обязан содержать Git bundle, архив tar или иной вложенный набор файлов всего исходного дерева. Восстановление проверяется через patch, `repo-file-hashes.json`, `AUDIT-MANIFEST.md`, `metadata/repo-file-snapshots.json`, отдельные `repo-before/` снимки файлов declared scope и отдельную чистую копию исходной ревизии, переданную в `audit package verify --repo <clean-repo-path>`.

Сопровождающий ZIP `<task-id>-audit-rNN.operator-workflow.zip` не является частью основного архива и не входит в его `SHA256SUMS.txt`. Он остаётся локальным доказательством предварительной проверки и не прикрепляется командой `audit submit` к внешнему ChatGPT-запросу. Его назначение - доказать локальной предварительной проверке, что уже финальный основной ZIP был проверен и что текст внешнего сообщения был получен штатной командой. Файл `payload/metadata.json` в сопровождающем ZIP содержит относительный путь основного ZIP, SHA-256 основного ZIP, SHA-256 `AUDIT-MANIFEST.md`, SHA-256 `SHA256SUMS.txt`, SHA-256 списка записей архива и сам список записей. Если после создания операторских доказательств основной ZIP будет переписан, эти значения перестанут совпадать с фактическим файлом.

`AUDIT-MANIFEST.md` обязан содержать только проверяемый инвентарь:

- `taskId`, `iteration`, `baseline`, `branch`, `domain`, `scopeTaskIds` и непустой `scopeSummary`, если он задан;
- исходный путь статического запроса аудитору: `docs/release-management/AUDIT-REQUEST.md`;
- `diff name-status`, то есть список изменённых файлов и типов изменения;
- список всех путей архива;
- список repo-owned файлов из `repo-file-hashes.json`, включая существующие предыдущие verdict-документы из `previousVerdictChain`;
- ссылки на `metadata/repo-file-snapshots.json`, `repo-after/` и `repo-before/`, чтобы внешний аудитор видел, где читать полные файлы;
- объявленную область задач, цепочку предыдущих verdict-ов и список закрытых blocker-ов;
- матрицу закрытия прошлых blocker-ов для существующих `NEEDS_FIXES` отчётов: путь отчёта, blocker id, имя configured check и closure-строку, которая его закрывает;
- модель восстановления;
- список проверок и ссылки на соответствующие файлы `evidence/`, включая найденные TRX-файлы с результатами `dotnet test`;
- политику запрещённых путей и секретов.

Patch строится с `--find-renames`, поэтому чистые переносы файлов могут быть представлены строками `rename from` и `rename to` без полного повторения содержимого. Это не ослабляет восстановление: `repo-file-hashes.json` всё равно фиксирует итоговые SHA-256, а `audit package verify` применяет patch к чистой копии baseline и сравнивает полный набор восстановленных файлов.

## Пути и ZIP

Все пути внутри архива используют UTF-8 POSIX-форму `dir/file.ext`. Backslash, управляющие символы, абсолютные пути, `..`, пустые сегменты и дубли путей запрещены. Манифест использует те же пути, что и архив.

ZIP создаётся в стабильном порядке:

- записи отсортированы по ordinal-сравнению путей;
- у всех записей фиксированное время `2000-01-01T00:00:00Z`;
- внешние атрибуты ZIP-записей, зависящие от операционной системы, должны быть нулевыми;
- ZIP-запись с не-ASCII путём должна иметь `UTF-8 flag`, то есть признак кодировки UTF-8 в центральном каталоге ZIP; текущая реализация пишет имена через UTF-8 и `audit package verify` проверяет этот признак минимум для таких путей;
- текстовые файлы генерируются в стабильном порядке;
- пути машины, абсолютные временные каталоги и текущее время не записываются, кроме явно разрешённых относительных метаданных проверки; Windows-пути с диском по шаблонам `[A-Z]:\...` и `[A-Z]:/...` запрещены в любом текстовом файле архива.
- `audit package verify` отклоняет архив с каталогами-записями, несортированными записями, нестабильной временной меткой ZIP, ненулевыми внешними атрибутами ZIP-записей или отсутствующим UTF-8 flag у не-ASCII пути.

Сопровождающий ZIP операторских доказательств тоже записывается с отсортированными путями, фиксированными временными метками и нулевыми внешними атрибутами ZIP-записей. Его байты не обязаны быть стабильными между разными запусками, потому что `stdout.txt`, `stderr.txt`, `duration-ms.txt` и `metadata.json` отражают реальные отдельные процессы.

## Patch и восстановление

Patch создаётся через Git, без ручной правки заголовков hunk-ов. Для выбранных файлов используется временный индекс Git:

1. В индекс читается дерево `baseline`.
2. В индекс добавляется текущее состояние выбранных файлов репозитория.
3. Команда строит `git diff --cached --binary <baseline>`.

Такой подход включает новые файлы, изменения, удаления и бинарные файлы, которые поддерживает `git diff --binary`. Если выбранный бинарный файл нельзя восстановить через эту модель, команда должна завершиться ошибкой до создания финального архива.

Для текстовых файлов команда добавляет во временный индекс Git байты с LF-окончаниями строк. Это защищает аудиторский patch от массовой перезаписи существующих файлов, когда локальная рабочая копия на Windows хранит те же строки с CRLF-окончаниями. `repo-file-hashes.json` считает SHA-256 по тем же восстановимым байтам, поэтому проверка архива сравнивает состояние, которое реально создаёт patch, а не машинный формат строк в текущем каталоге.

Файлы из `repoFileAllowlist`, которые игнорируются правилами Git, всё равно считаются заявленным входом задачи. Команда обязана принудительно добавить их во временный индекс Git, включить в patch и доказать восстановление через `audit package verify`; правила игнорирования не должны молча исключать такой файл из архива.

Файлы из `previousVerdictChain` считаются частью цепочки внешнего аудита. Когда такой файл есть в рабочей копии, он восстанавливается как обычный файл репозитория даже без совпадения с `repoFileGlobs` и без отдельного `repoFileAllowlist`. Это делает цепочку verdict-ов проверяемой в восстановленной модели файлов репозитория (`restored repo-owned model`): ссылка в `AUDIT-REQUEST.md` совпадает с patch, `repo-file-hashes.json` и инвентарём `AUDIT-MANIFEST.md`. Отсутствующий файл не создаётся автоматически и остаётся только исторической ссылкой, чтобы старые внешние ссылки не блокировали упаковку.

Содержимое существующих verdict-файлов из `previousVerdictChain` сохраняется дословно как историческое внешнее свидетельство внутри восстановленной модели файлов репозитория. Текущий контракт не объявляет `previousVerdictChain` независимым нотариальным доказательством происхождения прошлых внешних отчётов и не требует предыдущие аудиторские ZIP-архивы или отдельные исторические контрольные суммы (`no previous audit packages or detached historical checksums required`), если такой слой явно не введён отдельным контрактом пакета. Поэтому проверка локальных машинных путей не применяет запрет к diff-блокам этих файлов внутри patch: старый ответ аудитора может цитировать синтетический или чужой локальный путь, который не относится к текущей упаковке. То же ограниченное исключение для локальных путей действует при финальной проверке его снимков `repo-after/` или `repo-before/` в ZIP. Проверка конкретных значений, похожих на секреты, остаётся включённой для самих verdict-файлов, их ZIP-снимков и patch: если прошлый отчёт содержит не замещённый токен, пароль или приватный ключ, пакет должен завершиться отказом, а не передавать это значение аудитору. Значения-заглушки в угловых скобках, например `<redacted>`, `<token>`, `<value>` или `<non-placeholder>`, считаются замещёнными только когда они стоят на месте всего значения; текст после такой заглушки не считается безопасным продолжением и остаётся блокером. Если сохранённый внешний отчёт уже содержит нейтральную reviewer phrase или синтетический reviewer example со словами `concrete-secret`/`concrete-value`, он может быть разрешён только как точное значение из allowlist-а и только в контексте файла из `previousVerdictChain`, его `repo-before/`/`repo-after/` снимка или его patch-блока. Произвольный suffix после такой фразы остаётся блокером. Обычные файлы репозитория, файлы текущей задачи, архивные доказательства, вывод проверок, TRX-файлы, `AUDIT-MANIFEST.md`, `AUDIT-REQUEST.md`, `repo-file-hashes.json` и `metadata/audit-package.input.json` также по-прежнему блокируют пакет при новых локальных путях машины или конкретных секретоподобных значениях.

По умолчанию patch остаётся текстовым: чистые изменения обычных текстовых файлов пишутся как обычные diff-блоки. Если обычный diff-блок вне `previousVerdictChain` раскрывает значение, похожее на секрет, только через удалённую или контекстную строку исходной ревизии (`baseline`), прежняя бинарная защита patch уже не считается достаточной сама по себе, потому что текущий пакет также обязан включать полный `repo-before/` снимок изменённого файла. Если этот `repo-before/` снимок содержит приватный ключ, токен, пароль или другое конкретное секретоподобное значение, команда завершает упаковку ошибкой до создания финального ZIP. Двоичный diff (`GIT binary patch`) может применяться только как дополнительная защита patch для безопасного baseline-содержимого; он не разрешает передавать небезопасный `repo-before/` снимок. Остальные чистые текстовые изменения остаются обычными diff-блоками. Если секретоподобное значение есть в текущем файле задачи, упаковка также завершается ошибкой до построения patch.

Patch проверяется сразу после создания и до сборки ZIP, а затем повторно при `audit package verify`. Для проверки локальных машинных путей команда временно исключает только diff-блоки существующих исторических verdict-файлов из `previousVerdictChain`; остальные блоки patch сканируются полностью. Для проверки секретоподобных значений patch сканируется целиком, включая diff-блоки previous verdict-файлов. Это включает добавленные строки, контекстные строки и удалённые строки из `baseline`. Если после точечного двоичного восстановления для проблемных файлов обычный patch всё ещё содержит запрещённый текст, пакет завершается ошибкой вместо передачи такого текста аудитору.

`repo-file-hashes.json` описывает ожидаемое восстановленное состояние:

- `taskId`, `iteration`, `baseline`;
- список существующих repo-owned файлов и их SHA-256;
- список удалённых repo-owned файлов.

Проверка восстановления выполняет:

1. C#-проверку `SHA256SUMS.txt`.
2. Дополнительный запуск `sha256sum -c`, если утилита доступна.
3. `git apply --check`.
4. `git apply`.
5. Сравнение точного набора изменённых, удалённых и новых путей файлов репозитория после применения patch с `repo-file-hashes.json`. Новые Git-игнорируемые файлы из `repoFileAllowlist` тоже входят в это сравнение.
6. Сравнение восстановленных SHA-256 файлов с `repo-file-hashes.json`.
7. Сравнение корневого `AUDIT-REQUEST.md` из архива с восстановленным отслеживаемым Git-файлом `docs/release-management/AUDIT-REQUEST.md`.
8. Проверку, что новые текстовые файлы не являются однострочными заглушками, если они не указаны в `oneLineStubAllowlist`.

## Evidence

Каждая проверка из `checks` создаёт отдельный каталог:

```text
evidence/<task-id>-<iteration>/checks/<check-name>/
```

Минимальные файлы проверки:

- `command.txt` - исполняемый файл и аргументы без shell-строки.
- `cwd.txt` - относительный рабочий каталог.
- `env.json` - только переменные из `envAllowlist`.
- `timeout-seconds.txt` - тайм-аут проверки.
- `exit-code.txt` - ожидаемый и фактический код завершения.
- `duration-ms.txt` - фактическая длительность запуска в миллисекундах, нормализованная как число больше нуля с инвариантным десятичным разделителем.
- `stdout.txt` и `stderr.txt` - полный стандартный вывод и поток ошибок.
- `metadata.json` - машиночитаемая сводка, включая команду, тайм-аут, ожидаемый и фактический код завершения, ту же длительность, SHA-256 файлов вывода и TRX-файлов, если они были найдены.

`metadata.json` должен совпадать с соседними сырыми файлами: `expectedExitCode` и `actualExitCode` берутся из `exit-code.txt`, `timeoutSeconds` - из `timeout-seconds.txt`, а `durationMs` - из `duration-ms.txt`. Эти поля не должны зависеть от порядка аргументов конструктора или от человекочитаемого текста.

Перед записью `stdout.txt` и `stderr.txt` команда заменяет локальный корень репозитория на `<repo-root>`. Это сохраняет полный диагностический вывод проверок, но не раскрывает путь рабочей машины и не требует от оператора вручную подавлять вывод команд.

Перед запуском configured checks команда печатает структурированную JSON-диагностику с планом: число проверок, имя каждой проверки, переносимая команда, рабочий каталог, тайм-аут и ожидаемый код завершения. После каждой проверки команда печатает отдельную строку результата: имя проверки, фактический код завершения, ожидаемый код, длительность, путь `stdout.txt` и путь `stderr.txt` внутри будущего evidence-каталога. Эти строки не заменяют сырые доказательства в ZIP; они нужны, чтобы оператор видел стоимость и состав тяжёлого этапа до и во время упаковки, а не только после завершения архива.

Для проверок `dotnet test` конфигурация обязана задавать TRX-вывод через штатный logger и включать соответствующий `trxGlobs`, например `TestResults/**/*.trx`. Рекурсивный сегмент `**/` совпадает и с прямыми дочерними файлами, и с файлами во вложенных каталогах, поэтому `TestResults/**/*.trx` включает как `TestResults/result.trx`, так и `TestResults/nested/result.trx`. Вывод команд нельзя подавлять через shell-редиректы вроде `*> $null`, потому что внешний аудитор должен видеть `stdout.txt`, `stderr.txt`, `exit-code.txt`, `timeout-seconds.txt`, `metadata.json` и найденные TRX-файлы без локальных догадок.

`checks[]` для `audit package` не должен превращать упаковку в повторный широкий прогон среднего уровня. Для `dotnet test --filter` запрещены широкие фильтры по семействам аудиторских тестов вроде `FullyQualifiedName~AuditSubmit`, `FullyQualifiedName~AuditPackage`, `FullyQualifiedName~AuditPackageMessage`, `FullyQualifiedName~AuditRequest`, `FullyQualifiedName~AuditMessage` и `FullyQualifiedName~AuditWorkflow`, а также такие же короткие варианты с `RepositoryBuildToolTests.`. Проверка-доказательство пакета должна использовать точные имена тестов, уже введённую `Trait`-метку уровня вроде `AuditTier=Fast`, либо отдельную задаче-специфичную метку. Если нужен широкий прогон для локальной уверенности, он относится к `Medium` до упаковки, но не к evidence checks внутри ZIP.

TRX-файлы проверяются как XML-доказательства. Сканирование секретов должно читать содержимое результатов тестов: текст стандартного вывода, ошибок, сообщений, трассировок и обычных атрибутов. Служебные отображаемые имена тестов, где тестовый фреймворк может показать параметры проверок вроде ключа `token` со значением `...`, не должны блокировать включение штатного TRX. Если настоящее секретное присваивание или блок приватного ключа попадает в текст результата теста, TRX всё равно отклоняется. Перед записью TRX в архив команда нормализует служебные XML-атрибуты с путями сборки, например `codeBase` и `storage`: путь к локальному корню репозитория заменяется на `<repo-root>`, чтобы архив не раскрывал путь машины. На macOS та же нормализация учитывает системные варианты одного временного пути через `/var` и `/private/var`, потому что test runner может записать один вариант, а проверка репозитория работать с другим.

TRX-доказательства получают стабильные имена внутри архива: `trx/test-result-001.trx`, `trx/test-result-002.trx` и далее по отсортированному исходному списку. Исходные имена файлов, которые тестовый раннер может составлять из имени пользователя, имени машины и времени запуска, не записываются в пути архива, `metadata.json`, `AUDIT-MANIFEST.md`, `SHA256SUMS.txt`, `stdout.txt` или `stderr.txt`. Если стандартный вывод или поток ошибок содержит исходный путь TRX, команда заменяет его на стабильный путь архива.

При записи TRX XML команда дополнительно нормализует машинные атрибуты раннера: `TestRun/@name` становится `<test-run>`, а атрибуты `runUser`, `computerName` и `Deployment/@runDeploymentRoot` удаляются из архивной копии TRX. Поэтому ZIP не содержит ни настоящие локальные значения, ни сами проблемные XML-атрибуты, которые внешняя проверка трактует как утечку машинного контекста. Текст результатов тестов не редактируется перед проверкой секретов: настоящее секретное значение в `StdOut`, `StdErr`, сообщении или трассировке по-прежнему блокирует пакет.

Для самого T-0228 внешний пакет должен содержать отдельные checks для:

- минимального запуска `audit package` на тестовом наборе файлов;
- отдельного `audit package verify` на чистой копии baseline.

Эти записи `checks` должны создавать самостоятельные каталоги доказательств в `AUDIT-MANIFEST.md`, а не только строки в общей команде или внешней заметке.

Файлы из `archiveOnlyEvidenceGlobs` копируются под `evidence/<task-id>-<iteration>/archive-only/` и покрываются контрольными суммами. Они не входят в patch и не считаются repo-owned файлами.

Выбор archive-only evidence не должен начинаться с полного рекурсивного обхода корня репозитория. Команда выводит минимальные корни поиска из статической части каждого шаблона `archiveOnlyEvidenceGlobs`: для `.temp/audit-evidence/**` она перечисляет только `.temp/audit-evidence`, при пустом списке сразу возвращает пустой набор, а к полному обходу корня переходит только для шаблонов без более узкого статического префикса, например `*.txt`. Это защищает упаковку от старых `.temp/audit-clean/**`, локальных clean checkout-ов и других больших временных деревьев, которые не относятся к доказательствам текущего пакета.

Для временных доказательств проверки разрешён специальный путь `.temp/audit-evidence/...`. Это единственное исключение из общего запрета `.temp` и применяется только к файлам, выбранным через `archiveOnlyEvidenceGlobs`, и к TRX-файлам, выбранным через `checks[].trxGlobs`: такие файлы попадают в раздел `evidence/`, не восстанавливаются в рабочую копию и не создают дополнительных каталогов в корне репозитория. При записи archive-only evidence в ZIP префикс `.temp/` удаляется, поэтому архивный путь выглядит как `evidence/<task-id>-<iteration>/archive-only/audit-evidence/...` и сам ZIP не содержит сегмент `.temp`. Для `repoFileGlobs`, `repoFileAllowlist`, `checks.cwd` и любых других входных путей конфигурации `.temp` остаётся запрещённым путём.

После выполнения проверок инструмент фиксирует SHA-256 выбранных файлов репозитория и доказательств. Перед записью ZIP тот же набор пересчитывается снова. Если файлы изменились между проверками и упаковкой, команда завершается ошибкой.

После финальной записи основного ZIP команда больше не изменяет `AUDIT-MANIFEST.md`, `SHA256SUMS.txt` или набор его записей. Операторские доказательства создаются только после этого и попадают в сопровождающий ZIP. Поэтому `audit package verify evidence` и `audit package message evidence` относятся к неизменяемому основному ZIP, а не к промежуточной версии архива.

## Защитные правила

Команда отклоняет архив или входные файлы, если путь или содержимое попадает под встроенный запрет:

- `.git`;
- `bin`;
- `obj`;
- `.temp`, `tmp` и временные каталоги, кроме `.temp/audit-evidence/...` только для `archiveOnlyEvidenceGlobs` и `checks[].trxGlobs`;
- вложенные `.zip`, `.tar`, `.tar.gz`, `.7z`;
- `.env` и производные имена;
- опасные имена пути с `secret`, `token` или `private-key`;
- блоки приватных ключей;
- присваивания или JSON/YAML-подобные значения для `secret`, `token`, `password`, `api_key` и `api-key`.

Документальные упоминания `secretScanPolicy`, `secret`, `token`, `password` или `api_key` без секретного значения не считаются утечкой. Синтетические замещающие примеры, где ключ `token` получает значение `...`, а ключ `password` получает значение `<redacted>`, тоже допустимы, когда они описывают защитное правило. Такой marker должен занимать всё значение: если после замещающего значения продолжается обычный текст, сканирование не считает это безопасной заглушкой. Любое конкретное непустое значение после `secret`, `token`, `password`, `api_key` или `api-key` всё ещё считается потенциальным секретом.

`forbiddenPatterns` расширяет этот список для конкретной задачи.

Сканирование содержимого выполняется и при сборке, и при `audit package verify`. Текстовые файлы архива не должны содержать путь к корню локального репозитория, системному временному каталогу или Windows-путь с диском вида `[A-Z]:\...` и `[A-Z]:/...`. Для patch это правило применяется к оставшимся текстовым diff-блокам целиком, включая удалённые строки из `baseline`; исключаются только исторические diff-блоки verdict-файлов из `previousVerdictChain`.

Для TRX-доказательств это правило применяется после безопасной нормализации служебных XML-атрибутов тестового раннера, стабильного переименования TRX-файлов и замены исходных TRX-путей в выводе проверок. Секретные значения в тексте результатов тестов остаются запрещёнными; исключение касается только машинно созданных отображаемых имён тестов и технических путей сборки, которые не являются содержимым проверяемого вывода.

`audit package verify` также завершается ошибкой, если сгенерированные имена путей архива, имя patch, корень `evidence/<task-id>-<iteration>/`, структурные поля `metadata/audit-package.input.json` или блок метаданных `AUDIT-MANIFEST.md` не совпадают с текущими `taskId` и `iteration`. Проверка не сканирует весь Markdown/JSON как свободный текст: связанные задачи вроде `T-0232` допускаются в `scopeTaskIds`, а исторические ссылки на задачи вроде `T-0207` допускаются в `previousVerdictChain` и `blockerClosureList`, потому что эти поля описывают область текущего пакета, цепочку внешних verdict-ов и закрытые blocker-ы, а не имя текущего архива. Корневой `AUDIT-REQUEST.md` больше не содержит сгенерированный заголовок текущей задачи: его соответствие проверяется сравнением с восстановленным отслеживаемым источником.

## Запрос аудитору

`docs/release-management/AUDIT-REQUEST.md` является отслеживаемым исходным контрактом: это постоянный текст запроса внешнему аудитору, который хранится в репозитории, проходит проверку как обычная документация и копируется в корень ZIP как `AUDIT-REQUEST.md` без подстановок, генерации блоков или замены заполнителей. Этот файл также фиксирует контракт проверки области пакета и предыдущих внешних verdict-ов: аудитор обязан читать `metadata.scopeTaskIds`, `metadata.scopeSummary`, `metadata.previousVerdictChain`, `metadata.blockerClosureList`, предыдущие verdict-файлы, дословное сохранение исторических verdict-ов и закрытие каждого предыдущего blocker-а.

Статический запрос обязан требовать проверку содержимого изменения, тестов, документации, секретов и области задачи. Внешний аудитор не проверяет упаковочный слой: целостность ZIP, контрольные суммы, сопровождающий ZIP, операторские доказательства, локальную команду `audit package verify` и применимость patch. Эти проверки остаются локальным предварительным шагом команды репозитория до отправки.

Приёмка verdict-а использует безопасное операторское правило из раздела ниже. Статический запрос внешнему аудитору требует один финальный отчёт и запрещает промежуточные сообщения с `VERDICT:`, но не содержит DOM-селекторы или инструкции по чтению интерфейса.

Динамические сведения о конкретной задаче остаются в проверяемых машинных файлах:

- `AUDIT-MANIFEST.md` - инвентарь архива, исходная ревизия, задача, итерация, список файлов и доказательств;
- `metadata/audit-package.input.json` - нормализованная входная конфигурация;
- `repo-file-hashes.json` - модель восстановления файлов репозитория.

Перед созданием ZIP команда обязана прочитать `docs/release-management/AUDIT-REQUEST.md` из рабочей копии и отклонить пакет, если файл отсутствует, пустой, выглядит как однострочная заглушка, содержит конфиденциальное значение, машинный абсолютный путь или не содержит обязательные маркеры ответа:

- `VERDICT: ACCEPT`;
- `VERDICT: NEEDS_FIXES`;
- `TASK_ASSESSMENT`;
- `BLOCKERS`;
- `EVIDENCE_REVIEW`;
- `RISKS_AND_NOTES`;
- `CLOSURE_DECISION`;
- `metadata.scopeTaskIds`;
- `metadata.scopeSummary`;
- `combined scope`;
- `metadata.previousVerdictChain`;
- `metadata.blockerClosureList`;
- `previous verdict files`;
- `verbatim preservation`;
- `previous blockers closure`;
- `metadata/repo-file-snapshots.json`;
- `repo-after/`;
- `repo-before/`;
- `implementation content review`;
- `test coverage review`;
- `documentation review`;
- `task compliance review`;
- `secret scanning`;
- `scope scanning`;
- `evidence gap`;
- `patch-only inspection`;
- `single final report`;
- `no intermediate VERDICT`;
- `full current-scope engineering review`;
- `FOLLOW_UP_FINDING`;
- `OUT_OF_SCOPE_NOTE`;
- `ACCEPTED_RISK`;
- `INFO_NOTE`;
- `global safety blocker`;
- `architecture coherence`.

`AUDIT-MANIFEST.md` обязан явно указать исходный путь этого статического файла: `docs/release-management/AUDIT-REQUEST.md`. Корневой `AUDIT-REQUEST.md` в ZIP сохраняет текст отслеживаемого источника в той же LF-нормализованной форме, что `repo-after/` снимки и patch. При `audit package verify` после применения patch команда сравнивает `AUDIT-REQUEST.md` из архива с восстановленным `docs/release-management/AUDIT-REQUEST.md`; любое расхождение означает, что архив не соответствует отслеживаемому контракту и должен быть отклонён. Та же проверка сверяет `metadata/repo-file-snapshots.json` с `repo-file-hashes.json`: каждый существующий итоговый файл должен иметь `repo-after/` снимок с тем же SHA-256, каждый удалённый или изменённый файл, существовавший в `baseline`, должен иметь `repo-before/` снимок с SHA-256 исходной ревизии, а отсутствующий или повреждённый снимок блокирует отправку архива.

## Отправка внешнему аудитору

Внешний аудит разрешён только после успешного `audit package verify` на отдельной чистой копии исходной ревизии. Штатная отправка выполняется командой `audit submit`: она открывает одну свежую вкладку проекта в пользовательском Chrome через установленный `Codex Chrome Extension`, ждёт доступности поля сообщения без предварительного прокручивания страницы проекта, прикрепляет проверенный основной аудиторский ZIP и не прикрепляет сопровождающий ZIP `<task-id>-audit-rNN.operator-workflow.zip`, вставляет полный вывод команды `audit package message --zip <path>` или текст из `--message <path>`, проверяет текст и ZIP-плашку, запоминает текущее число сообщений, отправляет сообщение, ждёт нового ответа ассистента текущей отправки и сохраняет результат после строгой проверки. Если для повторной итерации той же задачи передан конкретный ChatGPT conversation URL с `/c/<conversation-id>`, команда открывает именно это обсуждение и добавляет новый ZIP в него; общий URL проекта для reuse-режима отклоняется до подключения к браузеру. Первая итерация и независимый контрольный аудит после первого `ACCEPT` используют корневой URL проекта, а control-режим дополнительно отклоняет ZIP с прошлой verdict-цепочкой, closure-списком или сохранёнными verdict-отчётами, чтобы начать обсуждение без контекста исправительных итераций. В ordinary-режиме финальный текст результата команда получает только через штатную copy-кнопку нового ответа ассистента текущей отправки и controlled clipboard proof: system clipboard принимается только после успешного sentinel, browser `readText()` не может вернуть sentinel как candidate, captured `writeText()`, `write(...)` или `copy` event value используется как доказанный результат той же copy-кнопки; произвольный старый clipboard text не является источником verdict-а. В режиме `--deep-research` команда до вставки текста выбирает реальный пункт `Глубокое исследование`, проверяет connector `connector_openai_deep_research` после вставки текста, ждёт готовую отчётную карточку и получает финальный текст как Markdown-файл, скачанный пунктом `Экспортировать в Markdown`. Режим `--download-report-only` остаётся диагностическим режимом для уже готового Deep Research отчёта и делает одну детерминированную попытку выгрузки Markdown. Команда не копирует cookies, `localStorage` или файлы сессии Chrome, не использует произвольный буфер обмена как источник verdict-а, не закрывает и не изменяет уже открытые пользовательские вкладки и закрывает только вкладку, созданную текущим запуском.

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Если текст запроса уже подготовлен отдельным файлом, его можно передать явно:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --zip <path> --message <message-path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Если отчёт уже готов в существующем ChatGPT discussion URL, допускается диагностический режим только для скачивания результата без повторной отправки архива:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --download-report-only --project-url <chat-url> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Для повторной исправительной итерации той же задачи используйте сохранённую ссылку задачи:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --reuse-conversation --zip <path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Если сохранённое обсуждение задачи нельзя продолжать через поддержанный submit-путь, используйте явный новый чат проекта. Это не меняет ZIP и не разрешает ручную отправку:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --new-conversation --zip <path> --out docs/verdicts/<domain>/<task-id>-audit-rNN.md --browser-backend codex-chrome
```

Для контрольного аудита после первого `ACCEPT` используйте отдельный запуск без истории:

```bash
dotnet run --project eng/Electron2D.Build -- audit submit --control-audit --zip <path> --out docs/verdicts/<domain>/<task-id>-audit-control-rNN.md --browser-backend codex-chrome
```

Короткий ручной запрос (`prompt`, то есть текст, написанный оператором вместо проверенного текста из архива), отправка без прикреплённого основного аудиторского ZIP и ручная отправка вместо `audit submit` запрещены. Нельзя заменять вывод `audit package message --zip <path>` фразами вроде `Приложен verified audit package`, пересказом задачи или служебным обрамлением `<task-id> audit rNN`: аудитор должен получить один проверенный основной ZIP и дословный текст запроса из архива без первого Markdown H1.

## Извлечение verdict-а - жёсткое правило

Это операторское правило для чтения результата внешнего аудита из текущего audit chat. Оно защищает приёмку от ошибки, когда `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES` найден в пользовательском запросе, примере формата, истории, боковой панели, обычном превью карточки или частично загруженном тексте страницы.

Финальный verdict = первая непустая строка единственного полного текста отчёта, полученного штатным `audit submit`. В базовом режиме это новый ответ ассистента текущей отправки. В режиме `--deep-research` и `--download-report-only` это Markdown-файл, который ChatGPT скачивает из открытой отчётной карточки `Углубленный исследовательский отчет` / `Deep research report` через пункт меню `Экспортировать в Markdown` или равнозначное Markdown-download действие этой отчётной карточки. Всё остальное не verdict.

Чтобы старые отчётные карточки той же задачи не могли быть сохранены как новая итерация или поверх предыдущего verdict-а, внешний отчёт должен в `TASK_ASSESSMENT` явно называть проверенные `metadata.taskId` и `metadata.iteration`, а `audit submit` принимает только строгий repo-relative путь `--out` вида `docs/verdicts/<domain>/<task-id>-audit-rNN.md` для primary или `docs/verdicts/<domain>/<task-id>-audit-control-rNN.md` для control. Строгий путь требует точные сегменты `docs/verdicts`, точное расширение `.md`, доменный сегмент, отсутствие произвольного префикса, абсолютного пути, пустого сегмента, `.`, `..` и вариаций регистра фиксированных сегментов или расширения. Обычный submit сверяет этот путь с текущей ZIP identity и режимом, а `--download-report-only` принимает только primary filename, выводит из него текущую identity и затем дополнительно сравнивает её с явными ссылками на evidence/ZIP в скачанном Markdown.

Порядок чтения:

1. Источник verdict-а в базовом режиме - только новый ответ ассистента текущей отправки после увеличения счётчика сообщений. Источник verdict-а в `--deep-research` и `--download-report-only` - только Markdown-файл, скачанный пунктом `Экспортировать в Markdown` или равнозначным Markdown-download действием, для единственной отчётной карточки `Углубленный исследовательский отчет` / `Deep research report` в текущем audit chat.
2. Запрещено определять verdict по `main.textContent`, всему `article`, `conversation-turn`, sidebar/history, user/tool messages, обычному превью карточки, counters, `verdictCount`, regex over full page или `matches.at(-1)`.
3. Сначала убедитесь, что ChatGPT не показывает видимую активную кнопку остановки ответа текущей отправки; если она видна, генерация ещё активна, даже когда в истории уже есть старый завершённый отчёт. В базовом режиме после завершения генерации читайте только последний новый ответ ассистента текущей отправки. В режиме `--deep-research` для встроенного фрейма используйте CDP-контекст этого фрейма и ищите реальную кнопку экспорта внутри него, потому что основной DOM страницы не видит внутренние кнопки cross-origin фрейма. Root-wrapper фрейм и ровно один внутренний готовый фрейм считаются одной отчётной поверхностью; несколько внутренних готовых фреймов или несколько page-level Deep Research frame ids считаются неоднозначностью. Если фрейм или target выбран, кнопка экспорта нажимается только в выбранном контексте; основной DOM страницы не используется до этого клика. После успешного клика выбранной кнопки `Экспорт` Markdown-пункт можно искать как в выбранном контексте, так и в основном DOM страницы, потому меню может быть page-level overlay. Если фрейм не найден, используйте основной DOM страницы. Не вычисляйте координату правого верхнего угла карточки и не кликайте по неизвестной точке внутри фрейма. Если найдено несколько готовых фреймов, несколько отчётных карточек, несколько export-кнопок или несколько пунктов Markdown, не выбирайте первый кандидат: это неоднозначность источника.
4. Первая непустая строка сохранённого текста должна быть ровно `VERDICT: ACCEPT` или `VERDICT: NEEDS_FIXES`.
5. Текст должен содержать обязательные секции `TASK_ASSESSMENT:`, `BLOCKERS:`, `EVIDENCE_REVIEW:`, `RISKS_AND_NOTES:` и `CLOSURE_DECISION:` в таком порядке.
6. Первый единственный текст отчёта, который прошёл строгую проверку и получен при неактивной генерации из разрешённого источника текущего режима, является готовым отчётом; команда не выполняет второй export только для подтверждения того же текста.
7. Если первая непустая строка не verdict, найдено несколько отчётных карточек или несколько candidate report texts, текст начинается с `Вы сказали`, извлечение включает пользовательский запрос, историю, боковую панель, обычное превью карточки или сообщения инструментов, либо была сжатая передача контекста (`context compaction`) или прерывание (`interruption`), остановитесь: не закрывайте задачу и перечитайте источник заново.
8. `ACCEPT` валиден только если первая строка полного текста отчётной карточки ровно `VERDICT: ACCEPT`, отчёт содержит обязательные секции в нужном порядке, секция `TASK_ASSESSMENT:` явно называет текущие `metadata.taskId` и `metadata.iteration`, а секция `BLOCKERS` не содержит `B1`..`Bn`. Текст `CLOSURE_DECISION` не должен становиться дополнительным словарным барьером после `VERDICT: ACCEPT`: он сохраняется как человеческое объяснение решения, а не как отдельный машинный verdict. Любой `VERDICT: ACCEPT` из prompt-инструкций, примеров формата, предыдущих сообщений, истории или превью карточки не имеет силы.

Если операторский инструмент не может получить валидный отчёт из разрешённого источника текущего режима, задача остаётся открытой до повторного чтения источника. Нельзя заменять это правило счётчиком совпадений, предварительным просмотром, regex over full page или любым агрегированным текстом страницы.

## Cleanup

Audit ZIP, сопровождающий ZIP операторских доказательств, временные каталоги и сырые доказательства не коммитятся. После `VERDICT: ACCEPT` рабочие архивы штатно удаляются вручную или будущей командой:

`audit package` не создаёт финальный ZIP после принятия задачи; новый архив нужен только для следующей итерации внешнего аудита.

Source-aware значит, что global selection принимается только внутри текущего assistant-сообщения, а active input/textarea принимается только как выбранный временный payload после клика, без полного `active.value` / `active.textContent`. Direct `tab_clipboard_read_text` / `tab_clipboard_write_text` через текущий pipe `Codex Chrome Extension` не используется, потому установленная extension-поверхность отвечает, что Chrome не поддерживает эти команды. `--deep-research` остаётся явным резервным режимом: команда выбирает настоящий режим `Глубокое исследование` кнопкой `+` до вставки текста, повторно проверяет выбранный режим перед отправкой и извлекает verdict только из полного Markdown-файла отчётной карточки без поиска по всей странице. `ACCEPT` принимается при первой строке `VERDICT: ACCEPT`, наличии обязательных секций, явном указании текущих `metadata.taskId` и `metadata.iteration` в `TASK_ASSESSMENT:` и отсутствии `B1`..`Bn` в `BLOCKERS:`. `CLOSURE_DECISION:` не требует специальной фразы после `VERDICT: ACCEPT`. Все режимы, которые записывают verdict Markdown, принимают только строгий repo-relative `--out` в `docs/verdicts/<domain>/` с точными сегментами `docs/verdicts` и расширением `.md`; обычный submit сверяет filename с текущей ZIP identity и primary/control mode, а `--download-report-only` принимает только primary filename и выводит из него текущую identity даже без ZIP. `--control-audit` до подключения к браузеру проверяет пустые metadata-массивы, отсутствие прямых `repo-after/docs/verdicts/**` / `repo-before/docs/verdicts/**` entries, отсутствие `docs/verdicts/**` в `repo-file-hashes.json` и `metadata/repo-file-snapshots.json`, запрет `TASKS.md` и `data/dev-diary/**`, а также контекстные текстовые артефакты ZIP (`AUDIT-MANIFEST.md`, `AUDIT-REQUEST.md`, `repo-after/data/documentation/**`, `evidence/**`, `metadata/**`, `repo-file-hashes.json`, `SHA256SUMS.txt`, добавленные строки patch для context-bearing repo paths) на конкретные Markdown-пути сохранённых verdict-отчётов. Source/test snapshots под `repo-after/**` могут содержать синтетические fixture paths, исторические `repo-before/**` снимки и удалённые строки patch не сканируются как текущий контекст, но прямые `repo-before/docs/verdicts/**` entries, metadata-ссылки и patch additions для context-bearing paths остаются запрещены. Закрытие чужих или оставшихся от старой сессии браузерного протокола (`browser session`) вкладок больше не считается частью реализации: этот путь признан невозможным для детерминированного C#-инструмента через прямой протокол расширения, поэтому команда не использует его и не обещает очистку уже открытых вкладок пользователя.

Перед отправкой штатный путь дополнительно перечитывает состояние поля отправки и до `30` секунд ждёт, пока одновременно будут видны тело сообщения и плашка основного audit ZIP. В режиме `--deep-research` выбранная плашка `Глубокое исследование` проверяется отдельным правилом после вставки текста. Если тело сообщения или видимая ZIP-плашка не готовы, команда останавливается с `E2D-BUILD-AUDIT-SUBMIT-PAYLOAD-MISSING` и пишет последнюю структурную причину отказа. Эта проверка выполняется до чтения счётчика сообщений и до `ClickSendAsync`.

Обычный submit больше не использует `--download-report-only` как единственный способ восстановления, когда в reused-обсуждении накопилось несколько готовых Deep Research reports. После baseline старых targets он может взять последний готовый target текущей страницы только тогда, когда за ним нет более нового неготового target. Если новый target текущей отправки ещё генерируется, команда ждёт его и блокирует page-level export fallback, а не скачивает старый ready report. Скачанный Markdown затем отсекается тем же валидатором, который проверяет текущие `taskId` и `iteration`. Это нужно, чтобы downloader не зависал на одном и том же готовом rNN-отчёте и не превращал `rNN` во внутренний счётчик отладки.

После `T-0235` `--download-report-only` принимает только конкретный ChatGPT conversation URL с сегментом `/c/<conversation-id>` и строгий repo-relative primary output path `docs/verdicts/<domain>/<task-id>-audit-rNN.md`; project/root/foreign URL, control filename, output path с произвольным префиксом перед `docs/verdicts/`, output без доменного сегмента, path traversal через `..`, неверный регистр сегментов `docs/verdicts` или расширение не ровно `.md` отклоняются до подключения к браузеру. Обычный submit после подтверждённой отправки ZIP ждёт перехода текущей вкладки на concrete conversation URL и записывает `.temp/audit/<task-id>/conversation-url-rNN.txt` и `.temp/audit/<task-id>/conversation-url.txt` только с таким URL, чтобы recovery через `--download-report-only` и следующая исправительная итерация через `--reuse-conversation` не получали общий project URL; если concrete URL не появился или sidecar path нельзя вывести из штатного имени ZIP, команда завершается диагностикой `E2D-BUILD-AUDIT-SUBMIT-CONVERSATION-URL-MISSING` вместо тихого продолжения без recovery-файла. Обычный submit, `--new-conversation`, `--deep-research` и `--control-audit` дополнительно сверяют `--out` с текущими `taskId`/`iteration` из ZIP и выбранным primary/control mode до подключения к браузеру. `--reuse-conversation` открывает сохранённый conversation URL задачи или явный `/c/<conversation-id>` URL, `--new-conversation` явно открывает корневой URL проекта из собственной конфигурации для нового primary-чата после локально зафиксированного лимита старого обсуждения и отклоняет любой явно переданный `--project-url`, а `--control-audit` отклоняет conversation URL и сохраняет контрольную ссылку отдельно как `control-conversation-url-rNN.txt` и `control-conversation-url.txt`. Извлечение Markdown сначала использует видимый Deep Research frame текущей вкладки через `Page.getFrameTree`; глобальный `Target.getTargets` остаётся резервным путём, если текущая страница не дала единственный usable frame context, а видимый iframe без CDP-контекста не считается выбранной surface и не блокирует дальнейшие fallback-ветки. Target-level recovery теперь повторяет только безопасные attach/read операции, а DOM-клики отправки и экспорта вызываются с `allowTransientRecovery: false`; read-only проверка видимого rate-limit dialog тоже допускает восстановление CDP-связи до отдельного клика подтверждения. Ошибка `E2D-BUILD-AUDIT-SUBMIT-REPORT-INVALID` дополняется конкретной причиной: неоднозначные кандидаты, неподходящий source, prompt echo/template, неверная первая строка, отсутствующий heading, отсутствие текущих `metadata.taskId`/`metadata.iteration` в `TASK_ASSESSMENT:` или numbered blocker в `ACCEPT`.

Ручная сборка ZIP для внешнего аудита запрещена. Внешний аудит допускается только после успешной пары команд `audit package` и `audit package verify` на отдельной чистой копии исходной ревизии; штатная отправка и извлечение отчёта выполняются через `audit submit`, который использует проверенный основной ZIP и текст из `audit package message --zip <path>`, но не прикрепляет сопровождающий ZIP операторских доказательств. Базовый путь использует обычный запрос в ChatGPT; режим `Глубокое исследование` включается только явным `--deep-research`.

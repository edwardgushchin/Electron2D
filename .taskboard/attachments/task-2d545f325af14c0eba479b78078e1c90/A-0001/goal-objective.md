## Блокеры

1. **Задача всё ещё может принять сама себя.**

   Агент может записать:

   * `status: Done`;
   * `acceptanceState: Accepted`;
   * произвольный `acceptedBy`;
   * activity с `actorKind: Human`.

   Это будет валидно. `actorKind: Human` — обычная редактируемая строка, а не подтверждённое полномочие.

   Для вашего primary/control audit особенно важно: `AcceptanceResult` разрешён только для `Human`, поэтому доверенный AI-аудитор не может быть корректно представлен, но обычный агент может притвориться человеком.

   Нужен transition API с реальными ролями `Worker`, `Auditor`, `Owner`, проверкой identity и запретом self-acceptance.

2. **Командная модель обходится через `Process`.**

   Валидна команда:

   ```json
   {
     "kind": "Process",
     "executable": "bash",
     "arguments": ["-lc", "rm -rf ..."],
     "requiredAccess": "ReadOnly",
     "confirmation": "PolicyDecides"
   }
   ```

   То есть `LegacyShell: ForbiddenUntilReviewed` обходится через `bash`, PowerShell или `cmd`. Кроме того, команда сама заявляет, какие права ей нужны.

   `requiredAccess` нужно считать только запросом, а реальные права должен независимо определять sandbox/executor. Shell-интерпретаторы требуют отдельной политики. Права должны быть массивом capabilities — сейчас нельзя выразить одновременно `WorkspaceWrite + Network`.

3. **Lifecycle всё ещё пропускает невозможные состояния.**

   Схема принимает:

   * `Done` с `Failed`/`Open` criterion;
   * `Passed` без evidence;
   * `Done` с активным blocker;
   * `Done` с `submittedAt: null`;
   * активные blockers в `Ready`, `Review`, `Done`;
   * `Active` blocker с заполненными `resolvedAt/resolvedBy`;
   * `Resolved` blocker без данных разрешения.

   Также `AcceptanceResult.decision: Rejected` не имеет соответствующего `acceptanceState`.

4. **Semantic validator пока только упомянут в тексте.**

   `task-board-v3.allOf[0]` содержит только `description` и ничего не проверяет. По-прежнему валидны дубли ID, циклы групп, одинаковые ranks, потерянные placements и несуществующие ссылки.

   Если semantic validator уже существует отдельно — его нужно считать обязательной версионированной частью формата и покрыть negative contract-тестами. По двум приложенным схемам его контракта нет.

5. **Пути вложений улучшены, но ещё небезопасны.**

   Валиден путь:

   ```text
   .taskboard/attachments/task-other/different/..
   ```

   Сегменты пути не обязаны совпадать с текущими `taskUid` и `attachmentId`. Также разрешены `.`, `..`, управляющие символы, Windows-имена вроде `CON`; не закрыты symlink/reparse-point escape.

   Нужны semantic equality, безопасный filename pattern, `lstat/realpath`, проверка regular file, фактических SHA-256/размера и квоты доски.

## Конкретные локальные дефекты

* `idPolicy.prefix` разрешает начало с `-`, `_`, `.`, но `displayId` требует букву или цифру. Например `prefix: "-"` создаст невалидный `-0001`.
* `workingDirectory` не принимает `"."`, поэтому нельзя запустить команду из корня проекта.
* `platforms: ["Any", "Windows"]` валидно, хотя противоречиво.
* `typedLink` типизирован только названием: `kind: Uri` всё ещё принимает произвольный `value`.
* Поле `assignee` полностью исчезло. Если его не заменяет отдельный lease/ownership-документ, это потеря функциональности.
* `externalAudit` остался обычной строкой и не может формально описать primary/control audit, независимость аудитора и цепочку verdict.
* `migration.reportSha256` не содержит ссылки на сам отчёт; `sourceDigests` допускает неоднозначные пути.
* Ajv strict ругается на `contains` без локального `"type": "array"`.
* Проверку `date`, `date-time` и `uri` нужно явно включать как format assertion.

## Итог

[task-board-v3.schema.json](sandbox:/workspace/scratch/e9197046486f/upload/task-board-v3.schema.json) и [task-file-v3.schema.json](sandbox:/workspace/scratch/e9197046486f/upload/task-file-v3.schema.json) уже можно считать хорошим **shape-слоем**. Но полный v3-контракт должен включать ещё:

1. `TaskBoardSemanticValidatorV3`;
2. `TaskTransitionValidatorV3`;
3. доверенную identity/role-модель;
4. независимую execution/sandbox policy;
5. негативные тесты на все перечисленные контрпримеры.

Без этих компонентов автоматический `ACCEPT` и выполнение команд по schema-valid задаче остаются небезопасными.

Задачи в этом Agent-native taskboard должны хранить весь контекст задачи. В карточке задачи можно переписываться с агентом для уточнения или вопросов по задаче, прикладывать там же в чате файлы, и все это должно быть доступно любому агенту как контекст из карточки задачи.

Тогда требование принципиальное: **задача — это не просто карточка с activity, а полноценная контекстная капсула**. В текущей v3 это ещё не реализовано.

Главное уточнение к моему прошлому review: историю нельзя удалять или заменять summary. Её можно физически вынести из основного JSON ради производительности, но она должна оставаться частью task aggregate и быть доступной без потерь.

## Как должна выглядеть модель

Нужно разделить:

* `conversation` — переписка пользователя и агентов;
* `activity` — системные события: статусы, тесты, блокеры, аудит;
* `attachments` — все приложенные файлы;
* `contextSnapshot` — необязательная производная сводка для быстрого входа агента.

Пример сообщения:

```json
{
  "messageId": "msg-000042",
  "sequence": 42,
  "author": {
    "actorId": "agent-codex-17",
    "actorKind": "Agent",
    "role": "Worker"
  },
  "createdAt": "2026-07-13T10:30:00Z",
  "replyToMessageId": "msg-000041",
  "content": [
    {
      "kind": "Markdown",
      "markdown": "Нужно уточнить поведение при отмене."
    },
    {
      "kind": "Attachment",
      "attachmentId": "att-log"
    }
  ],
  "agentRunId": "run-123"
}
```

## Чего не хватает в v3

1. `Comment` в `activity` — это не полноценный чат:

   * нет порядка `sequence`;
   * нет ответа на сообщение;
   * нет нескольких content blocks;
   * файл не привязан к конкретному сообщению;
   * нет связи с agent run;
   * не определена неизменяемость истории.

2. Вложения хранят только файл и метаданные. Для агента нужна доступность содержимого:

   * исходный файл;
   * SHA-256;
   * извлечённый текст для PDF/DOCX;
   * OCR для изображений;
   * информация о версии extractor;
   * возможность открыть оригинал.

3. `activity.maxItems: 10000` создаёт момент, когда переписку больше нельзя продолжать. Для чата такой предел недопустим без lossless-сегментации.

4. `AgentSummary` не должен заменять сообщения. Summary может ускорять загрузку, но обязан содержать:

   * покрытый диапазон сообщений;
   * revision контекста;
   * digest источников;
   * автора/модель;
   * возможность перейти к оригиналам.

## Правильная сборка контекста агента

Любой агент, открывающий задачу, должен получать:

1. описание и execution contract;
2. acceptance criteria;
3. зависимости и блокеры;
4. актуальное состояние;
5. всю переписку либо её lossless-доступную историю;
6. последние сообщения непосредственно;
7. все файлы и их извлечённое содержимое;
8. системную activity;
9. предыдущие решения, тесты и аудиты.

При очень большой истории не нужно пытаться засунуть всё в один prompt. Агент получает summary + последние сообщения, но имеет инструмент для чтения любого старого сообщения или вложения. **Полный контекст должен быть сохранён и доступен, а не обязательно одновременно помещён в окно модели.**

## Физическое хранение

Лучше считать задачей агрегат:

```text
task-uid/
├── task.json
├── conversation.jsonl
├── activity.jsonl
├── attachments/
└── derived/
    ├── extracted-text/
    └── previews/
```

Это логически одна задача, даже если физически не один файл. Так сообщения можно дописывать атомарно, не переписывая огромный `.e2task` при каждом ответе.

Если принципиально нужен один JSON, `conversation.messages[]` можно встроить, но будут расти merge-конфликты, стоимость сериализации и риск повреждения всей истории.

## Ещё два обязательных правила

* Сообщения и activity должны быть append-only. Редактирование — новым событием `MessageCorrected`, а не тихой перезаписью истории.
* Запуск агента должен фиксировать `taskRevision`, `lastMessageSequence` и `contextDigest`. Если во время работы пришло новое уточнение, агент должен получить обновление или выполнить rebase контекста.

Итого: в v3 нужно добавить отдельную модель `conversation`, связь сообщений с attachments и нормативный `AgentContextBuilderV3`. Без этого заявленное вами ключевое свойство Agent-native taskboard — передача полного контекста любому следующему агенту — схемами пока не обеспечено.

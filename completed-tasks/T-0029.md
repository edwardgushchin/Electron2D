# T-0029 [x] P0: Реализовать текстовый backend через SDL_ttf: `Font`, glyph layout, fallback и cache

## Итог

Реализован text backend baseline для `0.1.0 Preview`: публичный measurement API для целевого `Font`, `VerticalAlignment`, `Control`, `Label`, internal glyph layout, fallback font resolution, layout cache и internal SDL_ttf boundary через SDL3-CS.

`CanvasItem.DrawString()` теперь создаёт text command с `TextLayout`: размер, basic RTL direction, alignment offset, glyph positions, выбранные fallback fonts и destination rect. Реальный raster/GPU draw call остаётся будущей renderer-задачей, но command stream уже содержит данные для backend.

## Проверенное поведение

- `Font.GetStringSize()`, `GetHeight()`, `GetAscent()`, `GetDescent()` и `HasChar()` работают через целевого public API.
- `Control` наследуется от `CanvasItem` и поддерживает `Position`, `Size`, `font`/`font_size` theme overrides.
- `Label` наследуется от `Control`, рисует одну строку через `DrawString()` и поддерживает `Text`, `HorizontalAlignment`, `VerticalAlignment`, `Uppercase`.
- Unicode surrogate pairs считаются одним codepoint через `System.Text.Rune`.
- Basic RTL для Hebrew/Arabic ranges фиксируется в layout direction и visual glyph order.
- Если primary font не содержит glyph, layout выбирает fallback font.
- Повторный layout с тем же ключом переиспользует internal cache.
- `Compatibility` и `Standard` backend profiles объявляют поддержку `RenderingFeature.Text`.
- SDL_ttf API спрятан за internal `ISdlTtfApi`/`SdlTtfApi`; SDL handles не попадают в public API.

## Заметки агента

T-0029 не добавляет public `FontFile`, `ResourceLoader`, import pipeline, multiline wrapping, complex bidi shaping, clipping или real text raster/GPU draw call. Эти части должны идти отдельными целевыми задачами. `HorizontalAlignment.Fill` в текущем baseline ведёт себя как left alignment и задокументирован как ограничение.
